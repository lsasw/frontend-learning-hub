分布式锁核心原理 —— 为什么需要它？

从单机锁到分布式锁

在单机 Java 应用中，我们使用 synchronized 或 ReentrantLock 来保护共享资源。但当应用部署到多台服务器时，JVM 级别的锁就失效了 —— 每个 JVM 都有自己的锁对象，无法跨进程互斥。

单机锁 vs 分布式锁 架构对比

分布式锁必须满足的四个条件

🔒

互斥性

任意时刻只有一个客户端持有锁

💀

防死锁

即使持有锁的客户端崩溃，锁也能被自动释放

🛡️

解铃还须系铃人

只有加锁的客户端才能解锁，不可被他人误释放

⚡

高可用

锁服务本身不能是单点故障

⚠️ 致命误区

误以为 SETNX + EXPIRE 就足够安全

❌ 错误做法：先 SETNX 再 EXPIRE，两步非原子，中间崩溃→死锁。

✅ 正确做法：使用 SET key value NX EX seconds 一条原子命令。

Redisson 分布式锁实现机制（重点深度剖析）

Redisson 是什么？

Redisson 是 Redis 官方推荐的 Java 客户端，它不仅封装了 Redis 的基本操作，更在分布式锁领域提供了生产级的实现。底层依赖 Redis 的 EVAL 命令执行 Lua 脚本，保证操作的原子性。

ℹ️
核心优势：所有锁操作都通过 Lua 脚本在 Redis 服务端原子执行，避免了"先查后改"的竞态窗口。Redisson 的分布式锁接口完全兼容 JUC 的 Lock 接口，学习成本极低。
2.1

加锁流程与 Lua 脚本

加锁 Lua 脚本精讲

Redisson 的加锁本质是执行一段 Lua 脚本，原子地完成"检查锁是否存在 → 不存在则加锁 / 存在且是自己则重入"的判断。

LUA
-- KEYS[1] = 锁的 key（如 "myLock"）
-- ARGV[1] = 锁自动释放时间（默认 30 秒）
-- ARGV[2] = 客户端唯一标识（UUID:threadId）

-- 判断锁是否存在
if (redis.call('exists', KEYS[1]) == 0) then
    -- 锁不存在 → 直接加锁
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;  -- 返回 nil 表示加锁成功
end;

-- 锁存在 → 判断是否是自己加的
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    -- 是自己的锁 → 可重入，计数 +1
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;  -- 返回 nil 表示重入成功
end;

-- 别人的锁 → 返回剩余 TTL（毫秒）
return redis.call('pttl', KEYS[1]);
💡
设计精妙之处：Redisson 使用 Redis Hash 结构存储锁信息，而非简单的 String。Hash 的 field 是 UUID:threadId，value 是重入次数。这天然支持了可重入锁！后文会详细展开。

2.2

看门狗（Watchdog）—— 锁续期机制

为什么需要看门狗？

假设你设置了 30 秒的锁过期时间，但业务逻辑跑了 35 秒。如果没有看门狗，锁在第 30 秒自动过期，另一个线程拿到锁，两个线程同时操作共享数据——灾难。

看门狗续期原理图

看门狗核心参数与源码逻辑

参数默认值说明
lockWatchdogTimeout30000 ms锁默认过期时间（看门狗守护上限）
续期间隔lockWatchdogTimeout / 3 = 10s每 10 秒检查并续期一次
触发条件—未指定 leaseTime 时自动启用；指定则禁用看门狗
JAVA
// 方式1：不指定超时 → 看门狗自动续期（推荐）
RLock lock = redisson.getLock("order:123");
lock.lock();  // 看门狗启动，每10秒续期到30秒

// 方式2：指定 leaseTime → 看门狗不启用
lock.lock(10, TimeUnit.SECONDS);  // 10秒后自动释放，不续期

// 看门狗定时任务的核心逻辑（简化版）
private void scheduleExpirationRenewal(long threadId) {
    Timeout task = timer.newTimeout(timeout -> {
        // 执行续期 Lua 脚本：如果锁仍被当前线程持有 → PEXPIRE 30s
        if (renewLock(threadId)) {
            scheduleExpirationRenewal(threadId); // 递归调度下一次续期
        }
    }, internalLockLeaseTime / 3, TimeUnit.MILLISECONDS);
}
⚠️
特别注意：如果指定了 leaseTime，Redisson 认为你清楚业务执行时间，不会启动看门狗。业务超时后锁直接释放，可能导致并发安全问题。
2.3

锁释放流程

释放锁 Lua 脚本

释放时也需要原子操作：验证是当前线程的锁 → 减少重入计数 → 计数归零则删除 key → 向等待线程发布解锁消息。

LUA
-- KEYS[1] = 锁 key
-- KEYS[2] = Redisson 的解锁频道（用于通知等待线程）
-- ARGV[1] = 解锁消息（0 表示解锁）
-- ARGV[2] = 锁超时时间
-- ARGV[3] = 客户端标识 UUID:threadId

if (redis.call('hexists', KEYS[1], ARGV[3]) == 0) then
    return nil;  -- 锁不属于我，不能释放
end;

local counter = redis.call('hincrby', KEYS[1], ARGV[3], -1);

if (counter > 0) then
    -- 还有重入计数，只减计数不删 key
    redis.call('pexpire', KEYS[1], ARGV[2]);
    return 0;
else
    -- 重入计数归零 → 删除 key → 通知等待者
    redis.call('del', KEYS[1]);
    redis.call('publish', KEYS[2], ARGV[1]);
    return 1;
end;

2.4

可重入锁（ReentrantLock）支持

Hash 结构实现可重入

Redisson 利用 Redis Hash 的 field-value 对来追踪重入次数。这是它区别于简单 SETNX 方案的核心设计。

Redis Hash 存储模型

可重入锁使用示例

JAVA
public void processOrder(String orderId) {
    RLock lock = redisson.getLock("order:" + orderId);
    try {
        lock.lock();  // 第1次加锁，Hash field 计数 = 1
        updateInventory(orderId);  // 内部也会加同一把锁
    } finally {
        lock.unlock();  // 计数 -1 → 0 → 删除 key
    }
}

private void updateInventory(String orderId) {
    RLock lock = redisson.getLock("order:" + orderId);
    try {
        lock.lock();  // 第2次加锁，Hash field 计数 = 2（重入成功！）
        // 扣减库存...
    } finally {
        lock.unlock();  // 计数 -1 → 1
    }
}
💡
与 JUC ReentrantLock 完全一致的使用体验！同一个线程可以多次获取同一把锁，释放时也需要相应次数。这个设计让从单机迁移到分布式几乎不需要改变代码思维。
2.5

公平锁（FairLock）

公平锁 vs 非公平锁

⚡

非公平锁（默认）

所有等待者同时竞争，随机获胜。吞吐量更高，但可能造成"饥饿"。

⚖️

公平锁

按照请求顺序排队，先到先得。公平但吞吐量略低（维护队列开销）。

公平锁实现原理

Redisson 公平锁基于 Redis 的 List + 优先级队列 实现排队机制：

公平锁排队机制

每个等待客户端在 Redis List 中排队，当前锁释放后，队头（最早请求的客户端）获得锁。同时支持 超时放弃——超过等待时间的客户端自动退出队列。

JAVA
RLock fairLock = redisson.getFairLock("fairLock");
fairLock.lock();  // 公平排队获取

// 带超时等待
boolean acquired = fairLock.tryLock(3, 10, TimeUnit.SECONDS);
// 最多等 3 秒，锁最多持有 10 秒
⚠️
公平锁注意：公平锁需要额外的 Redis 命令维护队列，性能约为非公平锁的 70%-80%。高并发场景下慎重选择。
2.6

红锁（RedLock）—— Redis 集群下的安全锁

为什么需要 RedLock？

在 Redis 主从 + 哨兵架构下，如果主节点宕机时锁数据尚未同步到从节点，新选出的主节点可能让另一个客户端获取同一把锁。RedLock 通过在 N 个独立 Redis 节点上都成功加锁来解决此问题。

RedLock 加锁流程（N=5，多数=3）
JAVA
// 配置 3 个独立 Redis 节点
Config config1 = new Config();
config1.useSingleServer().setAddress("redis://192.168.1.1:6379");
Config config2 = new Config();
config2.useSingleServer().setAddress("redis://192.168.1.2:6379");
Config config3 = new Config();
config3.useSingleServer().setAddress("redis://192.168.1.3:6379");

RedissonRedLock redLock = redisson.getRedLock(
    redisson1.getLock("myLock"),
    redisson2.getLock("myLock"),
    redisson3.getLock("myLock")
);

try {
    if (redLock.tryLock(500, 30000, TimeUnit.MILLISECONDS)) {
        // 执行业务...
    }
} finally {
    redLock.unlock();  // 依次向所有节点发送解锁命令
}
🚨 重要争议

RedLock 并非银弹

❌ Redis 作者 antirez 提出的 RedLock 算法被分布式系统专家 Martin Kleppmann 公开质疑：时钟跳跃、GC 停顿都可能导致锁安全被破坏。

✅ 如果你的业务能容忍极低概率的锁冲突（如缓存更新），RedLock 可用。涉及资金等强一致性场景，建议用 ZooKeeper。

三种分布式锁方案全景对比

🔴

方案一：Redis + Redisson（基于缓存）

利用 Redis 的单线程 + Lua 原子脚本实现分布式锁。Redisson 封装了看门狗、可重入、公平锁等高级特性。

Redis 分布式锁架构
✅ 高性能（微秒级）
✅ 自动续期
✅ 可重入
✅ 公平锁
❌ 主从切换丢锁风险
❌ 强一致性场景不适用
✅ 适用场景：高并发扣库存、接口防重、缓存击穿保护、短时互斥操作
🦓

方案二：ZooKeeper + Curator（基于 CP 协调服务）

利用 ZK 的临时顺序节点 + Watcher 机制实现。节点删除即释放锁，天然防死锁。Curator 是 Netflix 开源的高层封装。

ZooKeeper 分布式锁原理
JAVA
// Curator 分布式锁（推荐）
InterProcessMutex lock = new InterProcessMutex(
    curatorClient, "/locks/myLock"
);

if (lock.acquire(10, TimeUnit.SECONDS)) {
    try {
        // 执行业务...
    } finally {
        lock.release();
    }
}
✅ 强一致性（CP模型）
✅ 自动释放（Session断开）
✅ 天然公平排队
❌ 性能较低（毫秒级）
❌ 运维成本高
⚠️ 注意：ZK 锁的性能瓶颈在写操作（创建/删除节点），不适合超高并发。Session 超时期间可能短暂无锁。
🗄️

方案三：MySQL（基于数据库行锁）

利用 MySQL 的唯一索引 + 行级锁或 SELECT ... FOR UPDATE 实现。最简单但性能最差。

SQL / JAVA
-- 方式 A：唯一索引 + INSERT（推荐）
CREATE TABLE distributed_lock (
    lock_key VARCHAR(128) PRIMARY KEY,
    holder    VARCHAR(128),
    expire_at DATETIME,
    INDEX idx_expire (expire_at)
);

-- 加锁：插入成功 = 获取锁
INSERT INTO distributed_lock (lock_key, holder, expire_at)
VALUES ('order:123', 'server-1', NOW() + INTERVAL 30 SECOND);

-- 解锁：删除记录
DELETE FROM distributed_lock
WHERE lock_key = 'order:123' AND holder = 'server-1';
✅ 零额外依赖
✅ 天然持久化
❌ 性能最差
❌ 无自动续期
❌ 手动清理过期锁
❌ 单点瓶颈
⚠️ 注意：MySQL 锁只适合低并发 + 已有数据库的场景。高并发下连接池耗尽风险极高。

常见踩坑与最佳实践

💣 踩坑 #1

指定 leaseTime 后忘记看门狗不会启动

❌ lock.lock(10, TimeUnit.SECONDS); 业务跑了 15 秒，锁在第 10 秒自动释放，另一个线程拿到锁

✅ 不指定超时让看门狗自动管理，或用 tryLock(waitTime, leaseTime, unit) 并确保 leaseTime 足够大

💣 踩坑 #2

unlock() 未放在 finally 块

❌ 业务抛异常后锁不释放，其他线程永久阻塞

✅ lock.lock(); try { ... } finally { lock.unlock(); } 铁律

💣 踩坑 #3

锁粒度太大导致性能瓶颈

❌ 对整个商品列表加一把锁 "product:all" —— 所有商品操作串行化

✅ 按资源 ID 细粒度加锁 "product:" + productId

💣 踩坑 #4

Redis 主从切换导致锁丢失

❌ 主节点宕机，锁数据未同步到从节点，新主节点允许另一客户端加锁

✅ ① 接受极小概率冲突（大部分场景可行）② 用 RedLock ③ 切到 ZooKeeper

💣 踩坑 #5

锁重入与释放次数不匹配

❌ lock() 了 3 次但只 unlock() 了 2 次，锁永远不会释放

✅ 使用 lock.lock(); try { ... } finally { lock.unlock(); } 配对，或使用 lock.forceUnlock() 强制清理

最佳实践检查清单

#实践理由
1默认不指定 leaseTime让看门狗自动续期，避免锁提前释放
2lock + try/finally 铁三角确保异常时锁也能释放
3控制锁粒度按资源 ID 加锁，避免全局锁
4设置合理的等待超时tryLock(waitTime, ...) 避免无限阻塞
5避免在锁内执行 RPC远程调用不可控，锁持有时间不可预测
6监控锁竞争情况使用 Redis MONITOR 或 Redisson 内置统计
7资金场景用 ZK强一致性保证，不允许锁冲突
