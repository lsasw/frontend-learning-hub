ThreadLocal 脏数据 · 线程池场景复现

📖 背景介绍

核心问题 线程池复用线程时，如果前一个任务在 ThreadLocal 中设置了值但未清理，下一个被分配到同一线程的任务会意外读到上一个任务残留的数据，导致业务逻辑错误或安全漏洞。

典型场景 Web 应用中用 ThreadLocal 存储当前登录用户信息（如 UserContext），请求结束时忘记 remove()。线程池回收线程后，下一个请求可能获取到别人的用户信息。

⚠️
Tomcat 默认线程池 200 线程；一次未清理的 ThreadLocal，同一线程上的所有后续请求都能读到脏数据，直到该线程被销毁或 ThreadLocal 值被覆盖。

🔬 原理说明

核心数据结构：每个 Thread 内部维护一个 ThreadLocalMap，Key 是 ThreadLocal 对象的弱引用，Value 是存储的值。线程池复用线程 = 复用同一个 Thread → 复用同一个 ThreadLocalMap → 前一个任务未 remove() 的 Entry 依然存在。

💻 关键代码

JAVA
// ❌ 错误写法——未清理ThreadLocal
public class UserContext {
    private static final ThreadLocal<String> CURRENT_USER = new ThreadLocal<>();
    public static void setUser(String user) { CURRENT_USER.set(user); }
    public static String getUser() { return CURRENT_USER.get(); }
    // 缺少 remove() 方法！
}

// ✅ 正确写法——finally 中清理
public void handleRequest(String user) {
    try {
        UserContext.setUser(user);
        // 业务处理...
        String current = UserContext.getUser();
    } finally {
        UserContext.CURRENT_USER.remove();  // ⬅ 关键！
    }
}
✅
最佳实践：ThreadLocal 的使用必须成对——set() 和 remove() 绑定在同一个 try-finally 中。框架层面推荐用拦截器/过滤器统一清理。

HashMap 1.7 vs 1.8 扩容过程对比

📖 背景介绍

JDK 1.7 致命缺陷 HashMap 扩容使用"头插法"转移链表元素。在多线程并发 put 触发扩容时，头插法会导致链表反转 + 成环，后续 get() 操作进入死循环，CPU 飙升至 100%。

JDK 1.8 改进 扩容改用"尾插法"——维护头尾两个指针按原顺序迁移链表，从根本上消除成环。同时引入红黑树优化哈希冲突严重时的查询性能（链表 ≥ 8 且数组 ≥ 64 时树化）。

🔥
血泪教训：这是 Java 并发编程最经典的坑之一，淘宝/美团等早期线上事故都与此有关。1.8 尾插法只是避免了死循环，并不保证线程安全——并发场景仍应用 ConcurrentHashMap。

💻 关键代码对比

JDK 1.7 transfer()
void transfer(Entry[] newTable) {
  for (Entry e : table) {
    while (e != null) {
      Entry next = e.next;
      int i = indexFor(e.hash, newTable.length);
      // ⬇ 头插法：新节点插在头部
      e.next = newTable[i];
      newTable[i] = e;
      e = next;
    } // 链表反转！A→B→C 变 C→B→A
  }
}
JDK 1.8 resize()
// 尾插法：维护 loHead/loTail
Node loHead=null, loTail=null;
Node hiHead=null, hiTail=null;
Node next;
do {
  next = e.next;
  if ((e.hash & oldCap) == 0) {
    // 低位链表（原位置）
    if (loTail==null) loHead=e;
    else loTail.next=e;
    loTail = e;  // ⬅ 尾部追加
  } else {
    // 高位链表（原位置+oldCap）
    if (hiTail==null) hiHead=e;
    else hiTail.next=e;
    hiTail = e;  // ⬅ 尾部追加
  }
} while ((e = next) != null);
// 保持原顺序 A→B→C ✓

synchronized 锁升级流程 · 实验演示

📖 背景介绍

为什么需要锁升级？ JDK 1.6 之前，synchronized 直接使用操作系统的 mutex lock（重量级锁），涉及用户态↔内核态切换，开销巨大。但实际上大部分锁竞争并不激烈——大多数情况下只有一个线程访问。

锁升级策略 JVM 根据竞争激烈程度动态升级锁：从开销最小的偏向锁开始，逐步升级到重量级锁。注意：HotSpot 默认不降级（只在 GC 的 STW 期间可能批量撤销偏向锁），一旦升级到重量级锁就一直是重量级锁。

💡
核心思想："用空间换时间"——大多数情况锁根本没人抢，没有必须一开始就用操作系统的重量级锁。锁升级 = 根据竞争情况动态选择最轻量的实现。

💻 关键验证代码

JAVA
// 验证偏向锁延迟：JVM 启动后 4 秒才启用偏向锁（BiasedLockingStartupDelay）
// JVM 参数: -XX:BiasedLockingStartupDelay=0 立即启用偏向锁
public class LockUpgradeDemo {
    static final Object lock = new Object();

    public static void main(String[] args) throws Exception {
        // 打印锁对象的 Mark Word（需要 JOL 依赖）
        System.out.println(ClassLayout.parseInstance(lock).toPrintable());
        // 输出: ... 00000001 (unlocked, non-biasable 或 biasable)

        synchronized (lock) {
            System.out.println(ClassLayout.parseInstance(lock).toPrintable());
            // 输出: ... 00000101 (biased, ThreadID=xxx)  ← 偏向锁
        }

        // 启动第二个线程竞争同一把锁
        new Thread(() -> {
            synchronized (lock) {
                System.out.println(ClassLayout.parseInstance(lock).toPrintable());
                // 输出: ... 01101000 (thin/lw lock)  ← 升级为轻量级锁
            }
        }).start();
    }
}

锁状态Mark Word 标志位存储内容触发条件

JVM OOM 三种场景 · 制造与分析

📖 背景介绍

OOM（OutOfMemoryError）是 Java 开发中最常见的生产事故之一。根据发生区域不同，OOM 分为多种类型，每种类型的成因、排查方式和解决方案都不一样。我们聚焦最核心的三种场景。

💥
堆 OOM
java.lang.OutOfMemoryError: Java heap space
对象分配过多，GC 无法回收足够空间
🧠
元空间 OOM
java.lang.OutOfMemoryError: Metaspace
动态加载类过多，元空间不足
⚡
直接内存 OOM
java.lang.OutOfMemoryError: Direct buffer memory
NIO ByteBuffer 分配超出直接内存上限

🎮 三大 OOM 场景可视化

② 元空间 OOM
③ 直接内存 OOM

堆 OOM：对象分配过多

堆内存使用: 850MB / 1024MB83%

触发方式 无限创建对象并持有引用，使 GC 无法回收。最常见于：大批量数据一次性加载到内存、缓存未设上限、死循环中创建对象。

JAVA
// -Xmx512m -Xms512m
public class HeapOOM {
    public static void main(String[] args) {
        List<byte[]> list = new ArrayList<>();
        while (true) {
            list.add(new byte[1024 * 1024]); // 每次分配 1MB，永不释放
        }
        // 输出: java.lang.OutOfMemoryError: Java heap space
    }
}
🔧
排查方法：jmap -dump:live,format=b,file=heap.hprof <pid> 导出堆转储 → MAT/JProfiler 分析支配树找到占用最大的对象 → 追踪到创建该对象的代码。

元空间 OOM：动态类加载过多

元空间使用: 52MB / 256MB20%

触发方式 CGLIB/Javassist 动态生成类、Groovy 脚本引擎反复编译、大量 JSP → Servlet 编译，每个类占用元空间且不会被卸载（除非 ClassLoader 被 GC）。

JAVA
// -XX:MaxMetaspaceSize=64m -XX:MetaspaceSize=16m
public class MetaspaceOOM {
    public static void main(String[] args) {
        while (true) {
            Enhancer enhancer = new Enhancer();
            enhancer.setSuperclass(OOMObject.class);
            enhancer.setUseCache(false);
            enhancer.setCallback((MethodInterceptor)(obj,method,args1,proxy)->proxy.invokeSuper(obj,args1));
            enhancer.create(); // 每次生成一个新类
        }
        // 输出: java.lang.OutOfMemoryError: Metaspace
    }
}
🔧
排查方法：jstat -gc <pid> 1000 观察 MU/MC 列的增长趋势 → -XX:+TraceClassLoading 追踪类加载 → 检查是否有循环创建动态代理 / CGLib 增强。

直接内存 OOM：NIO ByteBuffer 超限

直接内存使用: 64MB / 256MB25%

独特之处 直接内存不在 JVM 堆内，不受 -Xmx 限制也不受 GC 管理。由 -XX:MaxDirectMemorySize 控制（默认 = -Xmx）。只有 ByteBuffer 对象被 GC 时，其引用的直接内存才会被 Cleaner 回收。

JAVA
// -XX:MaxDirectMemorySize=128m
public class DirectMemoryOOM {
    public static void main(String[] args) {
        List<ByteBuffer> list = new ArrayList<>();
        while (true) {
            list.add(ByteBuffer.allocateDirect(1024 * 1024)); // 每次 1MB
        }
        // 输出: java.lang.OutOfMemoryError: Direct buffer memory
    }
}
🔧
排查方法：直接内存 OOM 不会在 Heap Dump 中体现（堆可能很小但 OOM）。用 pmap -x <pid> 或 NMT（-XX:NativeMemoryTracking=summary）查看进程总内存。注意 Netty 的 PooledByteBufAllocator 也使用直接内存。
