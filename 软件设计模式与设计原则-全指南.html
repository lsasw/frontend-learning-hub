⭐ SOLID 原则

SOLID 是面向对象设计的五大基本原则，由 Robert C. Martin（Uncle Bob）提出，是构建可维护面向对象系统的基石。

单一职责原则 (SRP)

S — Single Responsibility

一个类应该只有一个引起它变化的原因。换言之，一个类只做一件事。

定义：就一个类而言，应该仅有一个引起它变化的原因。如果一个类承担的职责过多，就等于把这些职责耦合在一起，一个职责的变化可能会削弱或抑制这个类完成其他职责的能力。

适用场景：
  • 类中方法过多，且各方法操作的数据不同
  • 类承担了多个不相关的职责（如数据持久化 + 业务逻辑 + 日志记录）
  • 修改一个功能时经常影响其他不相关功能
❌ 违反 SRPTypeScript
// 一个类承担了用户管理、验证、邮件通知三种职责
class UserService {
  register(user: User): void {
    // 职责1：验证
    if (!user.email || !user.password) {
      throw new Error("邮箱和密码不能为空");
    }
    if (user.password.length < 8) {
      throw new Error("密码至少8位");
    }
    // 职责2：存储
    db.save("users", user);
    // 职责3：发邮件
    smtp.send(user.email, "欢迎注册");
  }
}
✅ 遵循 SRPTypeScript
// 拆分为三个各司其职的类
class UserValidator {
  validate(user: User): void {
    if (!user.email || !user.password)
      throw new Error("邮箱和密码不能为空");
    if (user.password.length < 8)
      throw new Error("密码至少8位");
  }
}

class UserRepository {
  save(user: User): void {
    db.save("users", user);
  }
}

class EmailService {
  sendWelcome(email: string): void {
    smtp.send(email, "欢迎注册");
  }
}

class UserService {
  constructor(
    private validator: UserValidator,
    private repo: UserRepository,
    private email: EmailService
  ) {}
  register(user: User): void {
    this.validator.validate(user);  // 委托验证
    this.repo.save(user);           // 委托存储
    this.email.sendWelcome(user.email); // 委托通知
  }
}

开闭原则 (OCP)

O — Open-Closed

软件实体应该对扩展开放，对修改关闭。通过抽象和多态实现新增功能时不修改已有代码。

定义：软件实体（类、模块、函数等）应该可以扩展，但是不可修改。即在不修改原有代码的前提下，通过增加新代码来实现新功能。
适用场景：
  • 需要频繁添加新类型或新策略时
  • 不同分支处理逻辑相似但类型不同
  • 希望发布后无需重新编译已有代码就能扩展功能
❌ 违反 OCP：每加一种形状就要改 AreaCalculatorTypeScript
class AreaCalculator {
  calculate(shape: any): number {
    if (shape.type === "circle") {
      return Math.PI * shape.radius ** 2;
    } else if (shape.type === "rectangle") {
      return shape.width * shape.height;
    }
    // 新增三角形？必须修改这个方法 → 违反 OCP
    throw new Error("不支持的形状");
  }
}
✅ 遵循 OCP：通过抽象 + 多态扩展TypeScript
// 定义抽象接口
interface Shape {
  area(): number;
}

// 具体实现：各自计算面积
class Circle implements Shape {
  constructor(private radius: number) {}
  area(): number {
    return Math.PI * this.radius ** 2;
  }
}

class Rectangle implements Shape {
  constructor(private w: number, private h: number) {}
  area(): number {
    return this.w * this.h;
  }
}

// 新增三角形 —— 无需修改任何已有代码
class Triangle implements Shape {
  constructor(private base: number, private height: number) {}
  area(): number {
    return 0.5 * this.base * this.height;
  }
}

// 计算器只依赖抽象，对扩展开放
class AreaCalculator {
  calculate(shape: Shape): number {
    return shape.area();
  }
}

里氏替换原则 (LSP)

L — Liskov Substitution

所有引用父类的地方必须能透明地使用子类对象，且程序行为不会发生变化。

定义：子类型必须能够替换掉它们的基类型，而不影响程序的正确性。核心是子类不能违背父类的行为契约（前置条件不能更强，后置条件不能更弱）。
适用场景：
  • 设计继承体系时验证子类是否合理
  • 子类重写父类方法时检查行为一致性
  • 判断是否应该用继承还是组合
❌ 违反 LSP：正方形继承长方形，但 setWidth 破坏了长方形契约TypeScript
class Rectangle {
  protected width: number = 0;
  protected height: number = 0;
  setWidth(w: number): void { this.width = w; }
  setHeight(h: number): void { this.height = h; }
  area(): number { return this.width * this.height; }
}

class Square extends Rectangle {
  setWidth(w: number): void { this.width = w; this.height = w; }
  setHeight(h: number): void { this.width = h; this.height = h; }
}

// 调用方期望 Rectangle 的行为
function resize(r: Rectangle): void {
  r.setWidth(5);
  r.setHeight(10);
  console.log(r.area()); // Rectangle→50, Square→100 行为不一致！
}
✅ 遵循 LSP：用共同基类代替继承TypeScript
// 提取共同抽象，不再强行继承
abstract class Shape {
  abstract area(): number;
}

class Rectangle extends Shape {
  constructor(private w: number, private h: number) { super(); }
  setWidth(w: number): void { this.w = w; }
  setHeight(h: number): void { this.h = h; }
  area(): number { return this.w * this.h; }
}

class Square extends Shape {
  constructor(private side: number) { super(); }
  setSide(s: number): void { this.side = s; }
  area(): number { return this.side ** 2; }
}

// 现在子类替换父类不会产生行为不一致
function printArea(s: Shape): void {
  console.log(s.area()); // 无论传入什么形状都正确
}

接口隔离原则 (ISP)

I — Interface Segregation

客户端不应该被迫依赖它不使用的方法。将臃肿的接口拆分为更小、更具体的接口。

定义：客户端不应该依赖它不需要的接口。一个类对另一个类的依赖应该建立在最小的接口上。接口过大会导致实现类被迫实现不需要的方法。
适用场景：
  • 接口包含过多方法，部分实现类只需要其中几个
  • 实现类被迫提供空方法或抛异常来满足接口
  • 不同客户端只使用接口的不同子集
❌ 违反 ISP：胖接口强迫打印机实现不需要的方法TypeScript
// 臃肿接口
interface Machine {
  print(doc: string): void;
  scan(doc: string): void;
  fax(doc: string): void;
}

// 简单打印机被迫实现 scan 和 fax
class SimplePrinter implements Machine {
  print(doc: string): void { console.log("打印:", doc); }
  scan(doc: string): void { throw new Error("不支持扫描"); }
  fax(doc: string): void { throw new Error("不支持传真"); }
}
✅ 遵循 ISP：拆分为细粒度接口TypeScript
// 按职责拆分
interface Printer { print(doc: string): void; }
interface Scanner { scan(doc: string): void; }
interface FaxMachine { fax(doc: string): void; }

// 简单打印机只实现需要的接口
class SimplePrinter implements Printer {
  print(doc: string): void { console.log("打印:", doc); }
}

// 多功能设备实现多个接口
class MultiFunctionDevice implements Printer, Scanner, FaxMachine {
  print(doc: string): void { console.log("打印:", doc); }
  scan(doc: string): void { console.log("扫描:", doc); }
  fax(doc: string): void { console.log("传真:", doc); }
}

依赖倒转原则 (DIP)

D — Dependency Inversion

高层模块不应依赖低层模块，二者都应依赖抽象。抽象不应依赖细节，细节应依赖抽象。

定义：① 高层模块不应该依赖低层模块，两者都应该依赖其抽象。② 抽象不应该依赖细节，细节应该依赖抽象。核心是面向接口编程，而非面向实现编程。
适用场景：
  • 高层业务逻辑直接依赖低层数据访问或外部服务
  • 需要切换底层实现（如更换数据库、消息队列）
  • 单元测试时需要 Mock 依赖项
❌ 违反 DIP：高层直接依赖低层具体实现TypeScript
// 低层模块
class MySQLDatabase {
  save(data: string): void { console.log("MySQL保存:", data); }
}

// 高层模块直接依赖 MySQLDatabase（具体实现）
class UserService {
  private db = new MySQLDatabase(); // 硬编码依赖
  saveUser(name: string): void {
    this.db.save(name);
  }
}
// 换成 PostgreSQL？必须修改 UserService → 耦合
✅ 遵循 DIP：通过抽象解耦TypeScript
// 抽象接口
interface IDatabase {
  save(data: string): void;
}

// 低层实现依赖抽象
class MySQLDatabase implements IDatabase {
  save(data: string): void { console.log("MySQL保存:", data); }
}
class PostgreSQLDatabase implements IDatabase {
  save(data: string): void { console.log("PostgreSQL保存:", data); }
}

// 高层也依赖抽象 —— 通过构造注入
class UserService {
  constructor(private db: IDatabase) {} // 依赖抽象
  saveUser(name: string): void {
    this.db.save(name);
  }
}

// 使用时注入具体实现
const svc = new UserService(new MySQLDatabase());
const svc2 = new UserService(new PostgreSQLDatabase());
// 切换数据库无需修改 UserService

DRY 原则 (Don't Repeat Yourself)

DRY

系统中的每一项知识都必须有唯一、明确、权威的表示。消除重复，提高可维护性。

定义：每一个知识点在系统中都应该有唯一的、权威的表示。重复的代码意味着修改时需要改多处，遗漏一处就会产生 Bug。DRY 不仅适用于代码，也适用于数据库 Schema、测试逻辑、构建脚本等。
适用场景：
  • 多处出现几乎相同的代码块
  • 相同的业务规则在不同地方分别实现
  • 常量、配置值散落在代码各处
❌ 违反 DRY：重复的校验逻辑TypeScript
function createUser(name: string, email: string) {
  if (!name || name.length < 2) throw new Error("姓名无效");
  if (!email || !email.includes("@")) throw new Error("邮箱无效");
  // ...创建用户
}
function updateUser(name: string, email: string) {
  if (!name || name.length < 2) throw new Error("姓名无效"); // 重复！
  if (!email || !email.includes("@")) throw new Error("邮箱无效"); // 重复！
  // ...更新用户
}
✅ 遵循 DRY：提取公共校验函数TypeScript
// 唯一的校验逻辑来源
function validateUser(name: string, email: string): void {
  if (!name || name.length < 2) throw new Error("姓名无效");
  if (!email || !email.includes("@")) throw new Error("邮箱无效");
}

function createUser(name: string, email: string) {
  validateUser(name, email);
  // ...创建用户
}
function updateUser(name: string, email: string) {
  validateUser(name, email); // 复用
  // ...更新用户
}

KISS 原则 (Keep It Simple, Stupid)

KISS

保持简单。用最简单的方式解决问题，避免不必要的复杂性。

定义：大多数系统在保持简单而不是变得复杂时运作得更好。简单性应该是设计的一个关键目标，不必要的复杂性应该被避免。不要为了展示技术而过度设计。
适用场景：
  • 过度使用设计模式导致代码难以理解
  • 简单问题用了复杂的框架或架构
  • 引入不必要的抽象层、配置或间接引用
❌ 违反 KISS：简单需求过度设计TypeScript
// 只需要从配置读一个值，却搞了完整的状态机
abstract class ConfigState {
  abstract getValue(key: string): string;
}
class LoadingState extends ConfigState {
  getValue(key: string): string { throw new Error("加载中"); }
}
class ReadyState extends ConfigState {
  constructor(private data: Record) { super(); }
  getValue(key: string): string { return this.data[key]; }
}
class ConfigManager {
  private state: ConfigState = new LoadingState();
  load(data: Record) { this.state = new ReadyState(data); }
  get(key: string): string { return this.state.getValue(key); }
}
✅ 遵循 KISS：直接解决TypeScript
// 读取配置就是读一个对象
const config: Record = JSON.parse(
  fs.readFileSync("config.json", "utf-8")
);
const value = config["api_key"]; // 完事

YAGNI 原则 (You Aren't Gonna Need It)

YAGNI

不要实现当前不需要的功能。避免为想象中的未来需求编写代码。

定义：你不会需要它。永远不要因为"将来可能需要"就去实现一个功能。只在真正需要的时候才去写，这样可以减少代码量、降低维护成本、避免死代码堆积。
适用场景：
  • 为"将来可能扩展"预留接口或抽象层
  • 添加当前需求不需要的配置项或参数
  • 提前实现"通用框架"而当前只有一个具体场景
❌ 违反 YAGNI：提前实现用不到的通用性TypeScript
// 当前只需要支持 HTTP，却为 FTP、SMTP 预留了抽象
interface ProtocolAdapter {
  connect(): void;
  send(data: string): void;
  disconnect(): void;
}
class HttpAdapter implements ProtocolAdapter { /*...*/ }
class FtpAdapter implements ProtocolAdapter { /*...*/ } // 没人用
class SmtpAdapter implements ProtocolAdapter { /*...*/ } // 没人用
class ProtocolFactory {
  create(type: string): ProtocolAdapter { /*...*/ }
}
// 但实际上项目中只用到了 HttpAdapter
✅ 遵循 YAGNI：只做当前需要的TypeScript
// 当前只需要 HTTP，就直接用
class HttpClient {
  async get(url: string): Promise {
    return fetch(url);
  }
  async post(url: string, body: unknown): Promise {
    return fetch(url, { method: "POST", body: JSON.stringify(body) });
  }
}
// 将来真的需要 FTP 时再提取抽象也不迟

迪米特法则 (Law of Demeter)

LoD — 最少知识原则

一个对象应该对其他对象保持最少的了解。只与"直接朋友"通信。

定义：一个对象应该对其他对象有最少的了解。具体规则：一个方法只应该调用以下对象的方法——①自身 ②方法参数 ③自身创建的对象 ④自身的直接成员。不应该调用"朋友的朋友"的方法。
适用场景：
  • 出现链式调用 a.getB().getC().doSomething() 的深层访问
  • 类需要了解其他类内部结构才能工作
  • 模块间耦合度过高，修改一个类影响多个不相关模块
❌ 违反 LoD：通过链式调用深入访问陌生人TypeScript
class Customer {
  constructor(private wallet: Wallet) {}
  getWallet(): Wallet { return this.wallet; }
}
class Wallet {
  balance: number = 100;
  deduct(amount: number): void { this.balance -= amount; }
}

// 收银员直接操作顾客钱包 —— 知道太多了
class Cashier {
  checkout(customer: Customer, amount: number): void {
    const wallet = customer.getWallet(); // 访问了 Customer 的内部
    wallet.deduct(amount);               // 又访问了 Wallet 的方法
    // Cashier 同时依赖 Customer 和 Wallet 两个类
  }
}
✅ 遵循 LoD：只跟直接朋友说话TypeScript
class Wallet {
  private balance: number = 100;
  deduct(amount: number): boolean {
    if (this.balance >= amount) {
      this.balance -= amount;
      return true;
    }
    return false;
  }
}
class Customer {
  constructor(private wallet: Wallet) {}
  // Customer 封装了支付逻辑，不暴露 Wallet
  pay(amount: number): boolean {
    return this.wallet.deduct(amount);
  }
}

// 收银员只与 Customer 交互（直接朋友）
class Cashier {
  checkout(customer: Customer, amount: number): boolean {
    return customer.pay(amount); // 不需要知道 Wallet 的存在
  }
}

🏗️ 设计模式总览

设计模式是前人总结的、在特定场景下解决软件设计问题的经验方案。GoF（Gang of Four）归纳了 23 种经典设计模式，分为创建型、结构型、行为型三大类。

分类关注点包含模式
创建型对象的创建机制单例、工厂方法、抽象工厂、建造者、原型
结构型类与对象的组合适配器、桥接、组合、装饰器、外观、享元、代理
行为型对象间的通信与职责分配责任链、命令、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者

🟢 创建型模式

创建型模式关注对象的创建过程，将对象的创建与使用分离，使系统更灵活。

单例模式 (Singleton)

创建型
定义：确保一个类只有一个实例，并提供一个全局访问点。
适用场景：
  • 全局配置管理
  • 数据库连接池
  • 日志记录器
  • 缓存服务
单例模式实现TypeScript
class Singleton {
  private static instance: Singleton | null = null;

  // 私有构造函数，防止外部 new
  private constructor(
    private config: Record = {}
  ) {}

  // 全局访问点
  static getInstance(): Singleton {
    if (!Singleton.instance) {
      Singleton.instance = new Singleton({ env: "production" });
    }
    return Singleton.instance;
  }

  get(key: string): T {
    return this.config[key] as T;
  }
}

// 使用
const s1 = Singleton.getInstance();
const s2 = Singleton.getInstance();
console.log(s1 === s2); // true —— 同一个实例
⚠️ 注意单例模式可能引入全局状态，增加耦合度，在并发环境下需注意线程安全。现代实践中可用依赖注入替代。

工厂方法模式 (Factory Method)

创建型
定义：定义一个创建对象的接口，但由子类决定实例化哪个类。工厂方法使一个类的实例化延迟到其子类。
适用场景：
  • 创建对象时不想指定具体类
  • 需要根据条件创建不同子类实例
  • 希望将对象创建逻辑集中管理
工厂方法模式实现TypeScript
// 产品接口
interface Logger {
  log(message: string): void;
}

// 具体产品
class ConsoleLogger implements Logger {
  log(message: string): void { console.log("[Console]", message); }
}
class FileLogger implements Logger {
  log(message: string): void { console.log("[File]", message); }
}

// 工厂接口
interface LoggerFactory {
  createLogger(): Logger;
}

// 具体工厂：每个子类决定创建哪种产品
class ConsoleLoggerFactory implements LoggerFactory {
  createLogger(): Logger { return new ConsoleLogger(); }
}
class FileLoggerFactory implements LoggerFactory {
  createLogger(): Logger { return new FileLogger(); }
}

// 使用
const factory: LoggerFactory = new FileLoggerFactory();

建造者模式 (Builder)

创建型
定义：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。
适用场景：
  • 对象有大量可选参数，构造函数参数爆炸
  • 需要分步骤创建复杂对象
  • 同一构建过程需要产生不同表示
建造者模式实现TypeScript
class HttpRequest {
  constructor(
    public method: string,
    public url: string,
    public headers: Record,
    public body: string | null,
    public timeout: number
  ) {}
}

// 建造者：流式 API 分步构建
class HttpRequestBuilder {
  private method = "GET";
  private url = "";
  private headers: Record = {};
  private body: string | null = null;
  private timeout = 30000;

  setMethod(method: string): this {
    this.method = method; return this;
  }
  setUrl(url: string): this {
    this.url = url; return this;
  }
  addHeader(key: string, value: string): this {
    this.headers[key] = value; return this;
  }
  setBody(body: string): this {
    this.body = body; return this;
  }
  setTimeout(ms: number): this {
    this.timeout = ms; return this;
  }
  build(): HttpRequest {
    return new HttpRequest(
      this.method, this.url, this.headers, this.body, this.timeout
    );
  }
}

// 使用：清晰、可读、可选参数灵活
const req = new HttpRequestBuilder()
  .setMethod("POST")
  .setUrl("https://api.example.com/users")
  .addHeader("Content-Type", "application/json")
  .addHeader("Authorization", "Bearer token123")
  .setBody(JSON.stringify({ name: "张三" }))
  .setTimeout(5000)
  .build();

创建型
定义：用原型实例指定创建对象的种类，并通过拷贝这些原型来创建新对象。
适用场景：
  • 创建新对象成本较大（如需要数据库查询或网络请求）
  • 需要保持对象结构一致但值不同
  • 避免重复初始化复杂对象
原型模式实现TypeScript
// 原型接口
interface Cloneable {

}

🟡 结构型模式

结构型模式关注如何将类或对象组合成更大的结构，以实现新功能或更好的灵活性。

适配器模式 (Adapter)

结构型
定义：将一个类的接口转换成客户希望的另一个接口。使得原本由于接口不兼容而不能一起工作的类可以一起工作。
适用场景：
  • 需要使用已有的类，但其接口与需求不匹配
  • 对接第三方库，接口不兼容
  • 统一多个不同接口的调用方式
适配器模式实现TypeScript
// 目标接口：系统期望的接口
interface PaymentGateway {
  pay(amount: number, currency: string): boolean;
}

// 已有类：第三方支付库，接口不同
class StripeSDK {
  charge(usdCents: number): { success: boolean } {
    console.log(`Stripe扣款 ${usdCents} 美分`);
    return { success: true };
  }
}

// 适配器：把 Stripe 接口转换成系统期望的接口
class StripeAdapter implements PaymentGateway {
  constructor(private stripe: StripeSDK) {}
  pay(amount: number, currency: string): boolean {
    // 转换：金额转美分 + 只支持美元
    const cents = Math.round(amount * 100);
    const result = this.stripe.charge(cents);
    return result.success;
  }
}

// 使用：系统只跟 PaymentGateway 打交道
function checkout(gateway: PaymentGateway, amount: number) {
  gateway.pay(amount, "USD");
}

const adapter = new StripeAdapter(new StripeSDK());
checkout(adapter, 99.99); // Stripe扣款 9999 美分

结构型
定义：将抽象部分与它的实现部分分离，使它们都可以独立地变化。
适用场景：
  • 需要在两个独立维度上扩展（如形状×颜色、消息×发送渠道）
  • 避免类爆炸（继承组合过多）
  • 运行时切换实现
桥接模式实现TypeScript

组合模式 (Composite)

结构型
定义：将对象组合成树形结构以表示"部分-整体"的层次结构。使得客户端可以一致地处理单个对象和组合对象。
适用场景：
  • 文件系统（文件/文件夹）
  • UI 组件树（容器/叶子组件）
  • 组织架构管理
组合模式实现TypeScript
// 统一接口
interface FileSystemNode {
  getName(): string;
  getSize(): number;
  print(indent?: string): void;
}

// 叶子节点：文件
class File implements FileSystemNode {
  constructor(private name: string, private size: number) {}
  getName(): string { return this.name; }
  getSize(): number { return this.size; }
  print(indent: string = ""): void {
    console.log(`${indent}📄 ${this.name} (${this.size}KB)`);
  }
}

// 组合节点：文件夹
class Folder implements FileSystemNode {
  private children: FileSystemNode[] = [];
  constructor(private name: string) {}
  getName(): string { return this.name; }
  getSize(): number {
    return this.children.reduce((sum, c) => sum + c.getSize(), 0);
  }
  add(node: FileSystemNode): this {
    this.children.push(node); return this;
  }
  print(indent: string = ""): void {
    console.log(`${indent}📁 ${this.name}/ (${this.getSize()}KB)`);
    this.children.forEach(c => c.print(indent + "  "));
  }
}

// 使用：客户端一致地处理文件和文件夹
const root = new Folder("project")
  .add(new File("index.html", 5))
  .add(new File("style.css", 3))
  .add(new Folder("src")
    .add(new File("app.ts", 12))
    .add(new File("utils.ts", 4))
  );
root.print();
// 📁 project/ (24KB)
//   📄 index.html (5KB)
//   📄 style.css (3KB)
//   📁 src/ (16KB)
//     📄 app.ts (12KB)
//     📄 utils.ts (4KB)

定义：动态地给一个对象添加额外的职责。比生成子类（继承）更加灵活。

享元模式 (Flyweight)

结构型
定义：运用共享技术有效地支持大量细粒度的对象。将对象内部状态（可共享）与外部状态（不可共享）分离。
适用场景：
  • 大量相似对象，内存开销大
  • 对象大部分状态可外部化
  • 文本编辑器字符渲染、游戏中的粒子/树木
享元模式实现TypeScript
// 享元对象：共享的内在状态（树的类型）
class TreeType {
  constructor(
    public name: string,
    public color: string,
    public texture: string
  ) {}
  draw(x: number, y: number): void {
    console.log(`${this.color}${this.name} 绘制于 (${x}, ${y})`);
  }
}

// 享元工厂：缓存并复用
class TreeFactory {
  private static pool = new Map();
  static get(name: string, color: string, texture: string): TreeType {
    const key = `${name}_${color}_${texture}`;
    if (!this.pool.has(key)) {
      this.pool.set(key, new TreeType(name, color, texture));
      console.log(`创建新树种: ${name}`);
    }
    return this.pool.get(key)!;
  }
}

// 上下文：外部状态（位置）+ 享元引用
class Tree {
  constructor(private x: number, private y: number, private type: TreeType) {}
  plant(): void { this.type.draw(this.x, this.y); }
}

// 使用：1000 棵树只有 3 个 TreeType 对象
const trees: Tree[] = [];
trees.push(new Tree(1, 1, TreeFactory.get("橡树", "🟢", "粗糙")));
trees.push(new Tree(2, 5, TreeFactory.get("橡树", "🟢", "粗糙"))); // 复用
trees.push(new Tree(3, 3, TreeFactory.get("松树", "🌲", "鳞片")));
trees.forEach(t => t.plant());

代理模式 (Proxy)

结构型
定义：为其他对象提供一种代理以控制对这个对象的访问。代理对象在客户端和目标对象之间起到中介作用。
适用场景：
  • 延迟加载（虚拟代理）
  • 权限控制（保护代理）
  • 缓存请求结果（智能代理）
  • 远程服务调用（远程代理）
代理模式实现：缓存代理TypeScript
// 主题接口
interface ImageService {
  loadImage(url: string): Promise;
}

// 真实主题：耗时的图片加载
class RealImageService implements ImageService {
  async loadImage(url: string): Promise {
    console.log(`⏳ 从网络下载: ${url}`);
    await new Promise(r => setTimeout(r, 1000)); // 模拟网络延迟
    return `图片数据(${url})`;
  }
}

// 代理：添加缓存
class ImageProxy implements ImageService {
  private cache = new Map();
  constructor(private real: RealImageService) {}

  async loadImage(url: string): Promise {
    if (this.cache.has(url)) {
      console.log(`✅ 缓存命中: ${url}`);
      return this.cache.get(url)!;
    }
    const data = await this.real.loadImage(url);
    this.cache.set(url, data);
    return data;
  }
}

// 使用
const proxy = new ImageProxy(new RealImageService());
await proxy.loadImage("a.png"); // ⏳ 从网络下载: a.png
await proxy.loadImage("a.png"); // ✅ 缓存命中: a.png

🟣 行为型模式

行为型模式关注对象之间的通信和职责分配，描述对象之间如何协作完成任务。

责任链模式 (Chain of Responsibility)

行为型
定义：将请求的发送者和接收者解耦，使多个对象都有机会处理请求。将这些对象连成一条链，沿链传递请求，直到有一个对象处理为止。
适用场景：
  • 多层审批流程
  • 事件冒泡处理
  • HTTP 中间件管道
  • 日志级别过滤
责任链模式实现TypeScript
abstract class Handler {
  private next: Handler | null = null;
  setNext(h: Handler): Handler { this.next = h; return h; }
  handle(request: string): string | null {
    if (this.next) return this.next.handle(request);
    return null;
  }
}

// 具体处理者
class AuthHandler extends Handler {
  handle(request: string): string | null {
    if (!request.includes("token"))
      return "❌ 认证失败：缺少 token";
    return super.handle(request);
  }
}
class RateLimitHandler extends Handler {
  handle(request: string): string | null {
    if (request.includes("spam"))
      return "❌ 限流：请求过于频繁";
    return super.handle(request);
  }
}
class BizHandler extends Handler {
  handle(request: string): string | null {
    return "✅ 业务处理完成";
  }
}

// 构建责任链
const chain = new AuthHandler();
chain.setNext(new RateLimitHandler()).setNext(new BizHandler());

console.log(chain.handle("GET /api token"));      // ✅ 业务处理完成
console.log(chain.handle("GET /api spam token")); // ❌ 限流
console.log(chain.handle("GET /api"));            // ❌ 认证失败

命令模式 (Command)

行为型
定义：将一个请求封装为一个对象，从而允许用不同的请求对客户进行参数化、排队请求或记录日志，以及支持撤销操作。
适用场景：
  • 需要撤销/重做功能
  • 任务队列、宏命令
  • GUI 操作（按钮点击 → 命令）
  • 事务管理
命令模式实现：支持撤销的编辑器TypeScript
// 命令接口
interface Command {
  execute(): void;
  undo(): void;
}

// 接收者
class TextEditor {
  private text = "";
  getText(): string { return this.text; }
  insert(text: string): void { this.text += text; }
  delete(n: number): string {
    const removed = this.text.slice(-n);
    this.text = this.text.slice(0, -n);
    return removed;
  }
}

// 具体命令：插入文本
class InsertCommand implements Command {
  constructor(private editor: TextEditor, private text: string) {}
  execute(): void { this.editor.insert(this.text); }
  undo(): void { this.editor.delete(this.text.length); }
}

// 调用者：管理命令历史
class CommandManager {
  private history: Command[] = [];
  execute(cmd: Command): void {
    cmd.execute();
    this.history.push(cmd);
  }
  undo(): void {
    const cmd = this.history.pop();
    cmd?.undo();
  }
}

// 使用
const editor = new TextEditor();
const manager = new CommandManager();
manager.execute(new InsertCommand(editor, "Hello "));
manager.execute(new InsertCommand(editor, "World!"));
console.log(editor.getText()); // Hello World!
manager.undo();
console.log(editor.getText()); // Hello

迭代器模式 (Iterator)

行为型
定义：提供一种方法顺序访问一个聚合对象中的各个元素，而又不暴露该对象的内部表示。
适用场景：
  • 自定义集合需要遍历能力
  • 多种遍历方式（前序/中序/后序、深度/广度）
  • 统一不同数据结构的遍历接口
迭代器模式实现TypeScript
// 迭代器接口
interface Iterator {
  hasNext(): boolean;
  next(): T;
}

// 聚合接口
interface Aggregate {
  createIterator(): Iterator;
}

// 具体聚合：书架
class Bookshelf implements Aggregate {
  constructor(private books: string[]) {}
  createIterator(): Iterator {
    return new BookIterator(this.books);
  }
}

// 具体迭代器

中介者模式 (Mediator)

行为型
定义：用一个中介对象来封装一系列的对象交互。中介者使各对象不需要显式地相互引用，从而使其耦合松散，而且可以独立地改变它们之间的交互。
适用场景：
  • UI 组件间通信（表单联动）
  • 聊天室消息分发
  • 机场调度系统
  • 多个对象之间存在复杂的网状引用
中介者模式实现：聊天室TypeScript
// 中介者接口
interface ChatMediator {
  sendMessage(msg: string, user: User): void;
  register(user: User): void;
}

// 同事类
class User {
  constructor(public name: string, private mediator: ChatMediator) {}
  send(msg: string): void {
    console.log(`${this.name} 发送: ${msg}`);
    this.mediator.sendMessage(msg, this);
  }
  receive(msg: string, from: string): void {
    console.log(`${this.name} 收到来自 ${from} 的消息: ${msg}`);
  }
}

// 具体中介者
class ChatRoom implements ChatMediator {
  private users: User[] = [];
  register(user: User): void { this.users.push(user); }
  sendMessage(msg: string, sender: User): void {
    // 中介者负责分发，用户之间不直接引用
    this.users
      .filter(u => u !== sender)
      .forEach(u => u.receive(msg, sender.name));
  }
}

// 使用：用户之间通过聊天室通信
const room = new ChatRoom();
const alice = new User("Alice", room);
const bob = new User("Bob", room);
const charlie = new User("Charlie", room);
room.register(alice); room.register(bob); room.register(charlie);
alice.send("大家好！");
// Alice 发送: 大家好！
// Bob 收到来自 Alice 的消息: 大家好！
// Charlie 收到来自 Alice 的消息: 大家好！

备忘录模式 (Memento)

行为型
定义：在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便以后恢复。
适用场景：
  • 编辑器撤销功能
  • 游戏存档/读档
  • 事务回滚
  • 状态快照
备忘录模式实现：游戏存档TypeScript
// 备忘录：保存的状态快照
class GameMemento {
  constructor(
    public readonly level: number,
    public readonly score: number,
    public readonly hp: number
  ) {}
}

// 发起人：需要保存/恢复状态的对象
class GameCharacter {
  constructor(
    private level: number,
    private score: number,
    private hp: number
  ) {}
  // 保存状态
  save(): GameMemento {
    return new GameMemento(this.level, this.score, this.hp);
  }
  // 恢复状态
  restore(m: GameMemento): void {
    this.level = m.level;
    this.score = m.score;
    this.hp = m.hp;
  }
  play(): void { this.level++; this.score += 100; this.hp -= 10; }
  status(): string { return `Lv.${this.level} 分数:${this.score} HP:${this.hp}`; }
}

// 管理者：管理备忘录
class SaveManager {
  private saves: Map = new Map();
  save(name: string, m: GameMemento): void { this.saves.set(name, m); }
  load(name: string): GameMemento | undefined { return this.saves.get(name); }
}

// 使用
const player = new GameCharacter(1, 0, 100);
const saves = new SaveManager();
saves.save("关卡1前", player.save());
player.play();
console.log(player.status()); // Lv.2 分数:100 HP:90
player.restore(saves.load("关卡1前")!);
console.log(player.status()); // Lv.1 分数:0 HP:100

观察者模式 (Observer)

行为型
定义：定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并被自动更新。
适用场景：
  • 事件订阅/发布系统
  • 数据绑定（MVVM 框架）
  • 股价变动通知
  • 消息队列消费
观察者模式实现：股价变动通知TypeScript
// 观察者接口
interface Observer {
  update(stock: string, price: number): void;
}

// 主题接口
interface Subject {
  attach(o: Observer): void;
  detach(o: Observer): void;
  notify(): void;
}

// 具体主题：股票
class Stock implements Subject {
  private observers: Observer[] = [];
  private price: number;
  constructor(private symbol: string, initialPrice: number) {
    this.price = initialPrice;
  }
  attach(o: Observer): void { this.observers.push(o); }
  detach(o: Observer): void {
    this.observers = this.observers.filter(x => x !== o);
  }
  notify(): void {
    this.observers.forEach(o => o.update(this.symbol, this.price));
  }
  setPrice(newPrice: number): void {
    const old = this.price;
    this.price = newPrice;
    console.log(`📈 ${this.symbol}: ${old} → ${newPrice}`);
    this.notify();
  }
}

// 具体观察者
class PriceAlert implements Observer {
  constructor(private name: string) {}
  update(stock: string, price: number): void {
    console.log(`  [${this.name}] ${stock} 当前价格: ${price}`);
  }
}

// 使用
const apple = new Stock("AAPL", 180);
apple.attach(new PriceAlert("张三"));
apple.attach(new PriceAlert("李四"));
apple.setPrice(185);
// 📈 AAPL: 180 → 185
//   [张三] AAPL 当前价格: 185
//   [李四] AAPL 当前价格: 185

状态模式 (State)

行为型
定义：允许一个对象在其内部状态改变时改变它的行为。对象看起来似乎修改了它的类。
适用场景：
  • 对象行为随状态变化（订单状态机：待付款→已付款→已发货→已完成）
  • 大量条件分支判断状态
  • 自动售货机、红绿灯
状态模式实现：订单状态机TypeScript
interface OrderState {
  pay(ctx: OrderContext): void;
  ship(ctx: OrderContext): void;
  deliver(ctx: OrderContext): void;
  cancel(ctx: OrderContext): void;
  getName(): string;
}

// 上下文
class OrderContext {
  private state: OrderState;
  constructor() { this.state = new PendingState(); }
  setState(s: OrderState): void { this.state = s; }
  getState(): string { return this.state.getName(); }
  pay(): void { this.state.pay(this); }
  ship(): void { this.state.ship(this); }
  deliver(): void { this.state.deliver(this); }
  cancel(): void { this.state.cancel(this); }
}

// 具体状态
class PendingState implements OrderState {
  getName(): string { return "待付款"; }
  pay(ctx: OrderContext): void {
    console.log("✅ 支付成功");
    ctx.setState(new PaidState());
  }
  ship(): void { console.log("❌ 未付款，不能发货"); }
  deliver(): void { console.log("❌ 未付款，不能送达"); }
  cancel(): void { console.log("✅ 订单已取消"); ctx.setState(new CancelledState()); }
}
class PaidState implements OrderState {
  getName(): string { return "已付款"; }
  pay(): void { console.log("❌ 已付款，无需重复支付"); }
  ship(ctx: OrderContext): void {
    console.log("✅ 已发货"); ctx.setState(new ShippedState());
  }
  deliver(): void { console.log("❌ 尚未发货"); }
  cancel(): void { console.log("✅ 已退款并取消"); ctx.setState(new CancelledState()); }
}
class ShippedState implements OrderState {
  getName(): string { return "已发货"; }
  pay(): void { console.log("❌ 已付款"); }
  ship(): void { console.log("❌ 已发货"); }
  deliver(ctx: OrderContext): void {
    console.log("✅ 已送达"); ctx.setState(new DeliveredState());
  }
  cancel(): void { console.log("❌ 已发货，无法取消"); }
}
class DeliveredState implements OrderState {
  getName(): string { return "已完成"; }
  pay(): void { console.log("❌ 已完成"); }
  ship(): void { console.log("❌ 已完成"); }
  deliver(): void { console.log("❌ 已完成"); }
  cancel(): void { console.log("❌ 已完成，无法取消"); }
}
class CancelledState implements OrderState {
  getName(): string { return "已取消"; }
  pay(): void { console.log("❌ 订单已取消"); }
  ship(): void { console.log("❌ 订单已取消"); }
  deliver(): void { console.log("❌ 订单已取消"); }
  cancel(): void { console.log("❌ 已经是取消状态"); }
}

// 使用
const order = new OrderContext();
console.log(order.getState()); // 待付款
order.ship();    // ❌ 未付款，不能发货
order.pay();     // ✅ 支付成功
order.ship();    // ✅ 已发货
order.deliver(); // ✅ 已送达
console.log(order.getState()); // 已完成

策略模式 (Strategy)

行为型
定义：定义一系列算法，将每一个算法封装起来，并使它们可以相互替换。策略模式使得算法可以独立于使用它的客户端而变化。
适用场景：
  • 多种支付方式（微信/支付宝/银行卡）
  • 排序算法切换
  • 促销折扣策略
  • 消除大量 if-else 分支
策略模式实现：折扣策略TypeScript
// 策略接口
interface DiscountStrategy {
  calculate(price: number): number;
}

// 具体策略
class NoDiscount implements DiscountStrategy {
  calculate(price: number): number { return price; }
}
class PercentageDiscount implements DiscountStrategy {
  constructor(private rate: number) {} // rate: 0.8 = 八折
  calculate(price: number): number { return price * this.rate; }
}
class FullReductionDiscount implements DiscountStrategy {
  constructor(private threshold: number, private reduction: number) {}
  calculate(price: number): number {
    return price >= this.threshold ? price - this.reduction : price;
  }
}

// 上下文
class PriceCalculator {
  constructor(private strategy: DiscountStrategy) {}
  setStrategy(s: DiscountStrategy): void { this.strategy = s; }
  finalPrice(price: number): number {
    return this.strategy.calculate(price);
  }
}

// 使用
const calc = new PriceCalculator(new NoDiscount());
console.log(calc.finalPrice(100)); // 100

calc.setStrategy(new PercentageDiscount(0.8));
console.log(calc.finalPrice(100)); // 80

calc.setStrategy(new FullReductionDiscount(100, 20));
console.log(calc.finalPrice(100)); // 80 (满100减20)

模板方法模式 (Template Method)

行为型
定义：定义一个操作中算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。
适用场景：
  • 框架定义流程骨架，子类填充细节
  • 数据导出（CSV/JSON/Excel）流程统一
  • 构建查询（SQL/NoSQL）步骤一致
  • 生命周期钩子
模板方法模式实现：数据导出TypeScript
abstract class DataExporter {
  // 模板方法：定义算法骨架，final 防止子类覆盖
  export(data: Record[]): string {
    const header = this.buildHeader(data[0]);
    const body = this.buildBody(data);
    const footer = this.buildFooter();
    return header + body + footer;
  }
  // 抽象方法：子类必须实现
  protected abstract buildBody(data: Record[]): string;
  // 钩子方法：子类可选覆盖
  protected buildHeader(sample: Record): string {
    return Object.keys(sample).join(",") + "\n";
  }
  protected buildFooter(): string { return ""; }
}

// CSV 导出
class CsvExporter extends DataExporter {
  protected buildBody(data: Record[]): string {
    return data.map(row =>
      Object.values(row).join(",")
    ).join("\n");
  }
}
// JSON 导出
class JsonExporter extends DataExporter {
  protected buildHeader(): string { return "["; }
  protected buildBody(data: Record[]): string {
    return data.map(d => JSON.stringify(d)).join(",");
  }
  protected buildFooter(): string { return "]"; }
}

// 使用
const data = [{ name: "张三", age: 25 }, { name: "李四", age: 30 }];
console.log(new CsvExporter().export(data));
// name,age
// 张三,25
// 李四,30
console.log(new JsonExporter().export(data));
// [{"name":"张三","age":25},{"name":"李四","age":30}]

访问者模式 (Visitor)

行为型
定义：表示一个作用于某对象结构中的各元素的操作。它使你可以在不改变各元素的类的前提下定义作用于这些元素的新操作。
适用场景：
  • 编译器 AST 遍历（语法树节点 → 代码生成/类型检查/优化）
  • 对一组不同类型的对象执行同一类操作
  • 报表导出（同一组数据 → 不同格式）
访问者模式实现：文档元素导出TypeScript
// 元素接口
interface DocumentElement {
  accept(visitor: Visitor): void;
}

// 访问者接口
interface Visitor {
  visitParagraph(p: Paragraph): void;
  visitImage(img: ImageElement): void;
  visitTable(t: Table): void;
}

// 具体元素
class Paragraph implements DocumentElement {
  constructor(public text: string) {}
  accept(v: Visitor): void { v.visitParagraph(this); }
}
class ImageElement implements DocumentElement {
  constructor(public url: string) {}
  accept(v: Visitor): void { v.visitImage(this); }
}
class Table implements DocumentElement {
  constructor(public rows: string[][]) {}
  accept(v: Visitor): void { v.visitTable(this); }
}

// 具体访问者：HTML 导出
class HtmlExportVisitor implements Visitor {
  private html = "";
  getResult(): string { return this.html; }
  visitParagraph(p: Paragraph): void {
    this.html += `
${p.text}
\n`;
  }
  visitImage(img: ImageElement): void {
    this.html += `\n`;
  }
  visitTable(t: Table): void {
    const rows = t.rows.map(r =>
      `${r.map(c => `${c}`).join("")}`
    ).join("");
    this.html += `${rows}
\n`;
  }
}

// 具体访问者：统计字数
class WordCountVisitor implements Visitor {
  count = 0;
  visitParagraph(p: Paragraph): void { this.count += p.text.length; }
  visitImage(): void { /* 图片不计字数 */ }
  visitTable(t: Table): void {
    t.rows.forEach(r => r.forEach(c => this.count += c.length));
  }
}

// 使用：不修改元素类就能新增操作
const doc: DocumentElement[] = [
  new Paragraph("Hello World"),
  new ImageElement("logo.png"),
  new Table([["A", "B"], ["1", "2"]])
];
const htmlVisitor = new HtmlExportVisitor();
doc.forEach(el => el.accept(htmlVisitor));
console.log(htmlVisitor.getResult());

const countVisitor = new WordCountVisitor();
doc.forEach(el => el.accept(countVisitor));
console.log(`总字数: ${countVisitor.count}`); // 总字数: 13

📌 总结：原则与模式的关系

设计原则是"道"，设计模式是"术" 设计原则是指导思想，告诉你"应该怎样思考"；设计模式是具体方案，告诉你"遇到某类问题可以这样做"。模式是原则在特定场景下的落地实践。
设计原则相关设计模式
SRP 单一职责外观模式（简化接口）、命令模式（封装操作）
OCP 开闭原则策略模式、观察者模式、装饰器模式、工厂模式
LSP 里氏替换所有依赖继承的模式（模板方法、工厂方法）
ISP 接口隔离适配器模式、代理模式（按需暴露接口）
DIP 依赖倒转策略模式、工厂模式、观察者模式（均依赖抽象接口）
LoD 迪米特法则中介者模式、外观模式（减少直接耦合）
实践建议 ① 不要为了用模式而用模式——先理解问题，再选模式。② 模式不是银弹，过度使用反而增加复杂度（违背 KISS/YAGNI）。③ 优先用语言原生特性（如 JS 的 Proxy、Python 的装饰器语法）。④ 在重构中逐步引入模式，而非一开始就套用。
