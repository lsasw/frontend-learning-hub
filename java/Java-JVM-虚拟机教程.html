1JVM 整体架构

🏛️JVM 架构全景图

JVM 由三大子系统构成：类加载子系统（负责加载字节码）、运行时数据区（内存模型，所有数据存放处）、执行引擎（解释执行 + JIT 编译 + GC）。三者协作完成「加载 → 链接 → 初始化 → 执行 → 回收」的完整生命周期。

2类加载机制

🧬类生命周期与双亲委派模型

类从被加载到卸载，经历 7 个阶段。加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载。其中验证/准备/解析统称为「链接」阶段。

📥 加载
读 .class 入内存→ ✅ 验证
格式/语义校验→ 📋 准备
分配内存/零值→ 🔗 解析
符号→直接引用→ 🚀 初始化
执行 <clinit>→ ▶️ 使用→ 🗑️ 卸载

双亲委派模型（Parents Delegation Model）

类加载器收到加载请求后，不自己加载，而是逐级向上委托，直到 Bootstrap ClassLoader。父加载器找不到才由子加载器尝试。这保证了核心类库的唯一性和安全性。

🔬 类加载阶段详解 ▶
阶段做什么关键细节
1. 加载通过全限定名获取 .class 字节流 → 转为方法区数据结构 → 生成 Class 对象可来自 jar/网络/动态代理；数组类由 JVM 直接创建
2. 验证文件格式验证 / 元数据验证 / 字节码验证 / 符号引用验证-Xverify:none 可关闭（不推荐）
3. 准备为 static 变量分配内存并设零值（不执行赋值语句）static int a=123 → 此时 a=0；final static 编译期确定的值会直接赋初始值
4. 解析将常量池中符号引用替换为直接引用（内存地址）类/方法/字段的解析，可能触发其他类加载
5. 初始化执行类构造器 <clinit>()（static 块 + static 变量赋值）线程安全（JVM 加锁）· 父类优先 · 接口不要求父接口先初始化
Java 
// 类初始化触发时机
public class InitDemo {
    static int a = 1;
    static { System.out.println("clinit 执行"); }

    public static void main(String[] args) {
        // 以下操作均触发初始化：
        InitDemo obj = new InitDemo();  // new 实例
        Class.forName("InitDemo");       // 反射
        int x = InitDemo.a;                // 访问 static 变量
        // 以下不触发：
        Class<?> c = InitDemo.class;        // 只获取 Class 对象
        InitDemo[] arr = new InitDemo[10]; // 数组只创建数组类
    }
}

4垃圾回收算法

🧹四大基础 GC 算法 · Tab 切换对比

标记-清除 (Mark-Sweep)

流程：① 从 GC Roots 出发标记所有可达对象 ② 统一清除未标记的对象

最基础的 GC 算法，后续算法均基于其改进。

✅ 优点
实现简单 · 无需移动对象
❌ 缺点
内存碎片严重 · 标记和清除效率都不高
🎯 适用场景
老年代（CMS 收集器使用）
⚠️ 关键问题
碎片导致大对象分配失败 → 提前 Full GC

复制算法 (Copying)

流程：① 内存分为两块（From + To）② GC 时将存活对象复制到 To 区 ③ 回收 From 整块

新生代默认算法（Eden + S0 + S1），S0 和 S1 总有一块为空。

✅ 优点
无碎片 · 分配高效（指针碰撞） · 实现简单
❌ 缺点
内存利用率只有 50%（实际 90%） · 存活对象多时复制开销大
🎯 适用场景
新生代（对象朝生夕灭，存活率低）
🎯 核心参数
-XX:SurvivorRatio=8 → Eden:S0:S1=8:1:1

标记-整理 (Mark-Compact)

流程：① 从 GC Roots 标记存活对象 ② 将存活对象向一端移动 ③ 清理边界外的内存

解决标记-清除的碎片问题，但移动对象需要 STW（Stop The World）。

✅ 优点
无碎片 · 内存利用率高
❌ 缺点
STW 停顿长 · 移动对象 + 更新引用开销大
🎯 适用场景
老年代（Serial Old、Parallel Old 收集器）
🏷️ 变体
滑动整理、双指针整理

分代收集 (Generational Collection) —— 当前 JVM 默认策略

假设：绝大多数对象朝生夕灭 → 按对象年龄分代，不同代用不同算法。

代GC 类型算法特点
新生代Minor GC / Young GC复制算法频繁触发（几十秒一次）· 停顿短 · 回收大量对象
老年代Major GC / Full GC标记-清除 或 标记-整理触发少 · 停顿长 · 回收整个堆+方法区
new 对象→ Eden 分配→ Minor GC
存活→S0/S1→ 年龄+1
来回切换→ ≥15 或
S区放不下→ 晋升老年代
💡晋升规则：① 年龄计数器（Age）达到 MaxTenuringThreshold（默认 15）→ 晋升 · ② 动态年龄判断：同年龄对象大小超过 Survivor 区一半 → 大于等于该年龄的对象直接晋升 · ③ 大对象（如长数组）直接进入老年代（-XX:PretenureSizeThreshold）

5GC 收集器对比

🔄HotSpot 7 大经典 GC 收集器

收集器是 GC 算法的实现，不同收集器适合不同场景。下表按分代排列：

收集器位置算法线程STW适用场景
Serial新生代复制单线程全程 STW客户端 · 单核 · 小堆（< 100M）
ParNew新生代复制多线程全程 STW配合 CMS 使用 · 多核服务器
Parallel Scavenge新生代复制多线程全程 STW吞吐量优先 · 后台批处理
Serial Old老年代标记-整理单线程全程 STW客户端 · CMS 后备方案
Parallel Old老年代标记-整理多线程全程 STW配合 Parallel Scavenge
CMS老年代标记-清除多线程部分并发低延迟 · 互联网应用 (已弃用 JDK14+)
G1全堆标记-整理+复制多线程可控停顿大堆（4G+）· JDK9+ 默认 · 服务端
ZGC全堆染色指针+复制多线程< 1ms超大堆（TB级）· 极低延迟 · JDK15+

CMS 工作流程（老年代并发收集）

① 初始标记
STW · 标记 GC Roots→ ② 并发标记
并发 · 从 Roots 追踪→ ③ 重新标记
STW · 修正并发变化→ ④ 并发清除
并发 · 回收垃圾

CMS 追求最短 STW，但存在碎片问题 + 浮动垃圾（并发标记期间新产生的垃圾本次无法回收）。JDK14 已移除。

💡选型口诀：小堆串行 Serial · 吞吐优先 Parallel · 低延迟 CMS/G1 · 大堆超低延迟 ZGC。JDK9+ 默认 G1，JDK21+ 默认 ZGC（部分发行版）。
🔥 G1 (Garbage First) 收集器详解 · JDK9+ 默认 ▶

G1 抛弃了传统物理分代，改用 Region（1~32MB 的逻辑分区）将堆划分为多个大小相等的区域。每个 Region 可扮演 Eden、Survivor 或 Old 角色，动态切换。

核心机制：G1 追踪每个 Region 的垃圾堆积量，优先回收垃圾最多的 Region（Garbage First）。通过 Remembered Set 解决跨 Region 引用问题。用户可设置期望停顿时间 -XX:MaxGCPauseMillis=200，G1 自动调整。

bash 
# G1 常用参数
-XX:+UseG1GC                     # 启用 G1
-XX:MaxGCPauseMillis=200         # 期望最大停顿 200ms
-XX:G1HeapRegionSize=4M          # Region 大小（1~32M，2 的次方）
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用 45% 触发并发标记

6字节码执行流程

🔧从 .java 到机器码

📝 .java
源代码→ 🔨 javac
前端编译→ 📦 .class
字节码→ 📥 类加载器
加载验证→ ⚙️ 执行引擎→ 🖥️ 机器码
🔬 字节码示例：看看 javac 编译结果 ▶
Java 源码 
public class SimpleCalc {
    public int add(int a, int b) {
        return a + b;
    }
}
字节码 (javap -c SimpleCalc) 
public int add(int, int);
  Code:
     0: iload_1        // 从局部变量表位置 1 加载 int (a)
     1: iload_2        // 从局部变量表位置 2 加载 int (b)
     2: iadd           // 栈顶两 int 相加，结果压栈
     3: ireturn        // 返回栈顶 int

JVM 基于栈的指令集（与 x86 的寄存器架构不同），操作数在操作数栈上传递。每行字节码指令只占 1 字节（操作码）+ 若干操作数。

7JIT 编译原理

⚡Just-In-Time 编译：热点代码 → 机器码缓存

JVM 先解释执行，同时监控方法调用频率。当某个方法被调用超过阈值（默认 10,000 次），JIT 编译器将其编译为本地机器码存入 CodeCache，后续调用直接执行机器码，跳过解释环节。

🎯 分层编译（Tiered Compilation）
JDK8+ 默认模式，5 个编译层级：
L0: 解释执行
L1: C1 无 profiling
L2: C1 + 简单 profiling
L3: C1 + 完整 profiling
L4: C2 激进优化
🔧 C2 核心优化
· 方法内联（Inlining）
· 逃逸分析（Escape Analysis）
· 锁消除（Lock Elision）
· 标量替换（Scalar Replacement）
· 去虚拟化（Devirtualization）
📐 内联实战
小方法内联到大方法中，消除方法调用开销（栈帧创建、参数传递）。-XX:MaxInlineSize=35（默认 35 字节以内的方法直接内联）
🔄 逆优化
C2 的激进假设（如"该引用永远指向 A 类"）若被打破，则回退到解释执行重新收集 profiling → 再编译。这就是 JVM "热得更热"的原因。
bash 
# 查看 JIT 编译情况
java -XX:+PrintCompilation -XX:+UnlockDiagnosticVMOptions \
     -XX:+PrintInlining YourApp

# 输出示例：
  169   1    3  java.lang.String::hashCode (55 bytes)
  170   2    4  com.example.Task::process (132 bytes)
# 列1=编译ID  列2=编译层级(3=C1,4=C2)  列3=方法名
# 字节数=方法大小（后面的55/132）
🎯逃逸分析经典案例：对象只在方法内部使用（未逃逸），C2 可以消除堆分配，对象直接拆散为栈上变量（标量替换）。例如 new Point(x,y) 如果 Point 不逃逸 → 直接优化成两个局部变量，无需在堆上分配。

9命令行诊断工具

🛠️六大核心诊断命令

JDK 自带的 bin 目录下有丰富的命令行诊断工具，无需额外安装，生产环境排查首选。每个工具聚焦一个维度：进程、统计、内存、线程、配置、综合。

📋 jps — JVM 进程状态工具进程▶

功能最基础的工具，列出所有正在运行的 Java 进程及其主类名。通常作为其他工具的「第一步」——获取进程 PID。

bash 
# 列出本地所有 Java 进程
jps -l
# 输出：
12345 com.example.BootApplication    # -l: 显示完整主类名
12346 org.eclipse.jetty.start.Main
12389 sun.tools.jps.Jps

# 常用参数
jps -v        # 打印启动时的 JVM 参数
jps -m        # 打印传递给 main 方法的参数
jps -lvm      # 三合一：全类名 + 参数 + JVM 参数
📊 jstat — 统计监控工具监控▶

实时查看 JVM 的运行数据：类加载、GC 统计、编译统计。最常用于监控 GC 频率和耗时，判断是否需要调优。

bash 
# 格式：jstat -<option> <pid> <间隔ms> <次数>
# 监控 GC：每 1 秒输出一次，共 10 次
jstat -gc 12345 1000 10

# 输出列解读（关键列）：
# S0C/S1C: Survivor 0/1 容量    S0U/S1U: Survivor 使用量
# EC: Eden 容量    EU: Eden 使用量    OC: Old 容量    OU: Old 使用量
# MC: Metaspace 容量    MU: Metaspace 使用量
# YGC: Young GC 次数    YGCT: Young GC 耗时
# FGC: Full GC 次数     FGCT: Full GC 耗时     GCT: GC 总耗时

# 其他常用 option：
jstat -class 12345    # 类加载统计（装载/卸载数量+耗时）
jstat -compiler 12345 # JIT 编译统计（编译方法数+耗时）
jstat -gcutil 12345 1000  # GC 使用率百分比（更直观）
💡实战用法：发布新版本后跑 jstat -gcutil PID 1000 100 观察 100 秒。如果 FGC 在增长 → 可能内存泄漏；如果 EU/OU 持续接近 100% → 需要调大堆。
🗺️ jmap — 内存映像工具内存▶

查看堆内存中的对象统计信息，导出 heap dump 二进制快照文件，供 MAT 等工具离线分析。OOM 排查的核心入口。

bash 
# 查看堆中对象统计（按占用空间降序）
jmap -histo:live 12345 | head -20
# 输出： num  #instances  #bytes  class name
#        1:    500000   40000000  [C (char 数组，注意看大小)
#        2:    120000   28000000  java.lang.String
#        3:     80000   16000000  com.example.User

# 导出 heap dump（生产慎用，会触发 Full GC）
jmap -dump:live,format=b,file=heap.hprof 12345

# 查看堆内存配置
jmap -heap 12345
# 输出：各代大小、使用率、GC 算法配置

# JDK 9+ 替代方案（推荐）
jcmd 12345 GC.heap_dump heap.hprof
⚠️生产注意事项：jmap -dump:live 会先触发 Full GC（回收死对象），然后导出。大堆（>4G）可能卡顿数秒到数分钟。线上优先用 -XX:+HeapDumpOnOutOfMemoryError 自动 dump，或 JDK 9+ 的 jcmd GC.heap_dump。
🧵 jstack — 线程快照工具线程▶

打印 JVM 中所有线程的堆栈轨迹（thread dump），是排查死锁、线程饥饿、CPU 飙升问题的利器。

bash 
# 打印线程快照
jstack 12345 > thread_dump.txt

# 关键信息解读：
# "http-nio-8080-exec-1" #25 daemon prio=5  → 线程名+守护状态+优先级
#   java.lang.Thread.State: WAITING (parking) → 线程状态
#   at sun.misc.Unsafe.park(Native Method)
#   at java.util.concurrent.locks.LockSupport.park(LockSupport.java:175)

# 快速排查步骤：
# 1. top -Hp PID       → 找到 CPU 最高的线程 nid（十六进制）
# 2. jstack PID|grep nid → 定位到具体代码行
# 3. jstack -l PID      → -l 输出锁的额外信息
# 4. 看末尾 "Found 1 deadlock" → 死锁检测
💡死锁自动检测：jstack -l PID 的输出末尾会明确标注 Found one Java-level deadlock 并列出死锁涉及的线程和锁。连续执行 2-3 次 jstack 间隔 5 秒可以排除瞬时状态。
⚙️ jinfo — 配置信息工具配置▶

查看和动态修改 JVM 运行参数。排查「这个参数到底生效了没有」的神器。

bash 
# 查看所有 JVM 参数（含默认值）
jinfo -flags 12345
# -XX:MaxHeapSize=4294967296  -XX:+UseG1GC ...

# 查看单个参数
jinfo -flag MaxHeapSize 12345    # 输出：-XX:MaxHeapSize=4294967296
jinfo -flag UseG1GC 12345        # 输出：-XX:+UseG1GC（+ 表示开启）

# 动态修改参数（只能改 manageable 标志位）
jinfo -flag +PrintGC 12345       # 打开 GC 日志（动态生效！）
jinfo -flag -PrintGC 12345       # 关闭 GC 日志
🎯 jcmd — 综合诊断命令推荐▶

JDK 7+ 推出的全能工具，目标是「一个命令取代 jmap/jstack/jinfo 的大部分功能」。功能最全且对进程影响最小。

bash 
# 列出所有 Java 进程
jcmd -l

# 列出该进程支持的所有诊断命令
jcmd 12345 help

# 常用诊断命令
jcmd 12345 VM.version      # JVM 版本信息
jcmd 12345 VM.flags        # 所有 JVM 参数（同 jinfo -flags）
jcmd 12345 VM.uptime       # 运行时长
jcmd 12345 Thread.print    # 线程快照（同 jstack）
jcmd 12345 GC.class_histogram # 类统计（同 jmap -histo）
jcmd 12345 GC.heap_dump /tmp/heap.hprof  # 堆转储
jcmd 12345 VM.system_properties   # 系统属性
jcmd 12345 GC.run         # 触发 Full GC
jcmd 12345 VM.native_memory summary  # 堆外内存统计

10快照分析工具

🔬Heap Dump 分析：MAT / JProfiler / VisualVM

将 jmap 或 -XX:+HeapDumpOnOutOfMemoryError 生成的 .hprof 文件进行离线分析，定位内存泄漏、大对象、重复对象。

功能说明
Leak Suspects自动分析内存泄漏嫌疑点，按占用空间排序给出嫌疑报告，会指出哪个类/哪条引用链导致泄漏。
Histogram按类统计对象数量和占用空间，快速定位「哪个类实例最多/占用最大」。
Dominator Tree支配树分析：显示每个对象「支配」了多少内存（即它被 GC 后能释放多少空间），比 Histogram 更准确。
Path to GC Roots追踪对象到 GC Root 的引用路径，找出「为什么这个对象没有被回收」的根本原因。
OQL类 SQL 的对象查询语言，精确检索堆中满足条件的对象。如 SELECT * FROM com.example.User WHERE age > 30
bash 
# MAT 使用流程
# 1. 获取 heap dump
jmap -dump:live,format=b,file=heap.hprof 12345
# 2. 用 MAT 打开 heap.hprof
# 3. 选择 "Leak Suspects Report" 自动分析
# 4. 进入 Dominator Tree 找占用最大的对象
# 5. Path to GC Roots → 定位引用链 → 修复代码
# 提示：大堆(>4G)需调 MAT 启动参数 MemoryAnalyzer.ini 中 -Xmx
功能说明
Live Memory实时查看堆中所有对象及其引用关系，可按类/包过滤，支持录制对象分配。
CPU Profiler采样或插桩分析 CPU 热点方法，定位性能瓶颈。支持按线程/调用树展示。
Thread Profiler监控线程状态变化，检测死锁和线程阻塞，按时间线可视化展示。
DB/JDBC Profiler追踪数据库调用：SQL 语句、执行耗时、调用次数，定位慢 SQL。
Heap Walker堆快照离线分析，支持按最大对象/最大保留集合/类统计等多种视图。
功能说明
MonitorCPU、堆内存、Metaspace、线程数、已加载类数的实时曲线图。即开即用无需配置。
Threads线程列表 + 时间线视图，一键生成 Thread Dump，可视化检测死锁。
Sampler低开销 CPU 采样 + 内存采样，无需重启 JVM，适合生产环境临时排查。
ProfilerCPU/内存插桩 Profiler，数据更精确但开销较大，适合测试环境。
Plugins插件生态：GC Viewer（GC 日志可视化）、BTrace（动态字节码注入）等。
💡JDK 9+ 注意：VisualVM 从 JDK 9 起不再随 JDK 分发。需要从 visualvm.github.io 单独下载。替代方案：JDK 自带的 JMC（JDK Mission Control）或 IDEA Ultimate 内置的 Profiler。

🦾Arthas — 阿里开源在线诊断神器

无需重启、无需修改代码、无需预先埋点 —— 在线诊断 JVM 的瑞士军刀。在生产环境定位问题的最佳选择。

bash 
# 一键安装启动（下载 arthas-boot.jar 并 attach 到目标进程）
curl -O https://arthas.aliyun.com/arthas-boot.jar
java -jar arthas-boot.jar

# === 核心命令 ===

# 1. dashboard — 实时仪表盘（类似 jstat + top 的可视化版）
dashboard
# 显示：线程列表(按CPU排序) + 堆内存非堆内存 + GC + 运行时信息

# 2. thread — 线程诊断
thread                        # 显示所有线程（按 CPU 排序）
thread -n 3                   # 显示 CPU 最高的 3 个线程
thread -b                     # 检测死锁（自动找出死锁线程）
thread 42                    # 显示指定线程 ID 的堆栈

# 3. watch — 方法调用观测（在线"断点"）
watch com.example.UserService getUser '{params,returnObj}' -x 3
# 实时打印 getUser 方法的入参(params)和返回值(returnObj)，展开3层

# 4. trace — 方法调用链路追踪
trace com.example.UserService getUser
# 打印整个调用链及各环节耗时，快速定位慢方法

# 5. jad — 在线反编译
jad com.example.UserService
# 反编译运行中的类，验证线上代码是否是最新版本

# 6. stack — 查看方法调用路径
stack com.example.UserService getUser
# 当某个方法被调用时打印调用它的完整堆栈

# 7. vmoption — 查看/修改 JVM 参数
vmoption PrintGC true             # 动态打开 GC 日志

# 8. heapdump — 导出堆快照
heapdump /tmp/heap.hprof       # 同 jmap 但不会卡顿
dashboard
实时仪表盘→ thread -n 3
找 CPU 热门线程→ thread 42
查看堆栈→ trace / watch
定位慢方法→ jad 反编译
确认代码版本→ heapdump
内存分析

典型线上排查路径：dashboard 全局 → thread 找热点 → trace 定位 → watch 验证 → jad 确认

✅Arthas 三大优势：① 零侵入——不修改启动参数、不重启进程；② 交互式——类似调试器的 REPL 体验；③ 生产安全——单个命令执行完即退出，不留 Agent 常驻。Classloader 隔离保证不会污染应用的类加载。

🖥️JFR + JMC：JDK 内置性能分析套装

JDK Flight Recorder (JFR) 是 HotSpot 内置的低开销事件记录引擎，JMission Control (JMC) 是配套的可视化分析客户端。

特性说明
开销默认配置下 CPU 开销 < 1%，极致优化，生产环境可持续开启。
事件类型涵盖 GC、类加载、方法采样、线程、IO、网络、锁竞争等 150+ 种事件。
启动方式-XX:StartFlightRecording=filename=rec.jfr（JVM 参数）或 jcmd PID JFR.start（动态开启）
分析维度热点方法、GC 详细时间线、线程延迟、锁竞争、Socket IO、异常统计、内存分配速率。
JDK 版本JDK 8u262+ 内置；JDK 11+ 默认开启。JMC 需单独下载。
bash 
# 方式一：启动时开启（JVM 参数）
java -XX:StartFlightRecording:duration=60s,filename=recording.jfr -jar app.jar

# 方式二：运行时动态开启（jcmd）
jcmd 12345 JFR.start name=myrec duration=60s filename=/tmp/rec.jfr

# 方式三：持续录制（生产推荐）
jcmd 12345 JFR.start name=prod settings=profile maxsize=100m
# ... 问题复现后 dump 最近 1 小时的数据
jcmd 12345 JFR.dump name=prod filename=issue.jfr
jcmd 12345 JFR.stop name=prod

📊GC 日志分析：GCViewer & GCEasy

将 GC 日志可视化，直观判断 GC 频率、暂停时间分布、内存回收趋势。

GCViewer

  • 开源桌面工具，直接打开 GC 日志文件
  • 展示 GC 事件时间线、暂停耗时分布
  • 支持多文件对比（优化前后对比）
  • CSV/图表导出

GCEasy (在线)

  • gceasy.io 上传日志即可分析
  • 自动生成 GC 健康报告 + 调优建议
  • 可视化图表：暂停时间、回收量、内存趋势
  • 支持所有 GC 收集器的日志格式
bash 
# 开启 GC 日志（JDK 8）
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log

# 开启 GC 日志（JDK 9+）
-Xlog:gc*:file=gc.log:time,level,tags

# 然后用 GCViewer 或上传 gceasy.io 分析 gc.log
