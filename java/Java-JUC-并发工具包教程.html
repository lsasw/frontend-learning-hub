1JUC 核心组件总览

Java 并发工具包（java.util.concurrent，简称 JUC）由 Doug Lea 设计，是 Java 平台处理并发编程的核心武器库。它按职责划分为 5 个子包，覆盖了从底层锁到高层并发模式的完整栈。

2locks 锁体系

🔐

ReentrantLock — 可重入独占锁

java.util.concurrent.locks ▶

ReentrantLock java.util.concurrent.locks.ReentrantLock

可重入的独占锁，功能上类似 synchronized 但更灵活：支持公平/非公平模式、可中断获取 (lockInterruptibly)、可超时获取 (tryLock)、可绑定多个条件变量 (Condition)。底层基于 AQS 实现。

使用场景需要 tryLock 超时控制、需要公平排队、需要多个 Condition 等待队列的复杂同步
核心方法lock() / unlock() / tryLock(timeout) / lockInterruptibly() / newCondition()
⚠️ 注意必须在 finally 中 unlock，否则死锁！推荐始终用 try-finally 模式
Java 
// ReentrantLock 标准使用模式 —— 必须 try-finally
import java.util.concurrent.locks.ReentrantLock;

public class Account {
    private final ReentrantLock lock = new ReentrantLock(); // 默认非公平
    private double balance;

    // 转账：保证原子性
    public void transfer(Account target, double amount) {
        lock.lock(); // 加锁
        try {
            if (this.balance >= amount) {
                this.balance -= amount;
                target.deposit(amount);
            }
        } finally {
            lock.unlock(); // 确保释放！
        }
    }

    public void deposit(double amount) {
        lock.lock();
        try { balance += amount; }
        finally { lock.unlock(); }
    }
}

// tryLock 超时获取 —— 避免死锁等待
if (lock.tryLock(3, TimeUnit.SECONDS)) {
    try { /* 临界区 */ }
    finally { lock.unlock(); }
} else {
    // 3 秒内未获取到锁，走降级逻辑
}
⚠️ synchronized vs ReentrantLock：如果不需要 ReentrantLock 的高级特性（超时、中断、多 Condition），优先用 synchronized —— JDK 6+ 之后 synchronized 性能已大幅优化（偏向锁→轻量级锁→重量级锁升级），且不会忘记释放锁。ReentrantLock 适合需要精细控制的场景。
📖

ReadWriteLock — 读写分离锁

ReentrantReadWriteLock ▶

ReentrantReadWriteLock java.util.concurrent.locks.ReentrantReadWriteLock

维护一对锁：读锁（共享）和写锁（独占）。多个读线程可同时持有读锁，但写线程独占。适合读多写少场景，大幅提升并发读吞吐量。

使用场景缓存系统、配置中心、只读数据库副本 —— 读远多于写的场景
核心规则读-读不互斥 · 读-写互斥 · 写-写互斥 · 支持锁降级（写→读）但不支持锁升级
⚠️ 注意写锁饥饿问题（读多时写线程可能长时间拿不到锁）→ JDK 8 引入 StampedLock 解决
Java 
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class Cache<K, V> {
    private final Map<K, V> map = new HashMap<>();
    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();

    // 读操作：多线程并发读
    public V get(K key) {
        readLock.lock();
        try { return map.get(key); }
        finally { readLock.unlock(); }
    }

    // 写操作：独占写入
    public void put(K key, V value) {
        writeLock.lock();
        try { map.put(key, value); }
        finally { writeLock.unlock(); }
    }

    // 锁降级：写锁 → 读锁（安全的）
    public V computeIfAbsent(K key, Function<K, V> loader) {
        writeLock.lock();
        try {
            V val = map.get(key);
            if (val == null) {
                val = loader.apply(key);
                map.put(key, val);
            }
            // 锁降级：先拿读锁，再释放写锁
            readLock.lock();
            return val;
        } finally {
            writeLock.unlock(); // 此时仍持有读锁
            readLock.unlock();
        }
    }
}
🎟️

StampedLock — 乐观读锁（JDK 8+）

性能更高的读写锁 ▶

StampedLock java.util.concurrent.locks.StampedLock

JDK 8 引入，在 ReentrantReadWriteLock 基础上增加乐观读模式：读操作不加锁，先获取一个 stamp，读取后再 validate，如果期间没有写操作则直接使用，否则升级为悲观读锁。性能比 ReentrantReadWriteLock 高 10%~50%。

使用场景极高并发读 + 偶尔写的场景（如实时行情数据缓存）
⚠️ 注意不可重入！同一线程不能重复获取读锁。不支持 Condition。推荐用模板代码。
Java 
private final StampedLock sl = new StampedLock();

public double getValue() {
    long stamp = sl.tryOptimisticRead(); // 乐观读，不加锁
    double current = value;
    if (!sl.validate(stamp)) {     // 验证期间是否有写操作
        stamp = sl.readLock();       // 升级为悲观读锁
        try { current = value; }
        finally { sl.unlockRead(stamp); }
    }
    return current;
}

public void setValue(double newValue) {
    long stamp = sl.writeLock();
    try { value = newValue; }
    finally { sl.unlockWrite(stamp); }
}

3atomic 原子类

原子类基于 CAS（Compare-And-Swap）机制实现无锁线程安全，核心是 Unsafe.compareAndSwapXxx → JDK 9+ 使用 VarHandle。比加锁更轻量，适合简单操作的原子更新。

⚡

AtomicInteger / AtomicLong — 原子计数器

CAS 无锁原子操作 ▶

AtomicInteger java.util.concurrent.atomic.AtomicInteger

对 int 的原子操作，内部通过 CAS 实现自增、自减、比较设置等操作的无锁线程安全。替代 synchronized 或 volatile + synchronized 的计数器方案。

使用场景全局计数器、序列号生成、统计接口调用次数
核心方法get() / set() / incrementAndGet() / compareAndSet(expect, update) / getAndUpdate(fn)
Java 
import java.util.concurrent.atomic.AtomicInteger;

public class Counter {
    private final AtomicInteger count = new AtomicInteger(0);

    // 原子自增
    public void increment() {
        count.incrementAndGet(); // ++count
    }

    // CAS 自旋：自定义更新逻辑
    public void updateMax(int newValue) {
        int old;
        do {
            old = count.get();
            if (newValue <= old) return; // 不需要更新
        } while (!count.compareAndSet(old, newValue)); // CAS 自旋
    }

    // JDK 8+ 函数式更新
    public void doubleIt() {
        count.getAndUpdate(x -> x * 2);
    }
}
💡 CAS 原理：compareAndSet(expect, update) —— 如果当前值 == expect，则更新为 update 并返回 true，否则返回 false。底层是一条 CPU 原子指令（x86 的 CMPXCHG），无需加锁。
📊

LongAdder — 高并发计数器（JDK 8+）

比 AtomicLong 更快 ▶

LongAdder java.util.concurrent.atomic.LongAdder

在超高并发写场景下，AtomicLong 的 CAS 自旋会导致大量竞争失败重试。LongAdder 采用分段 CAS（cell 数组），各线程先写入不同 cell，最终 sum() 时累加，减少竞争。

使用场景高并发统计（QPS/TPS 计数器）、热数据计数、metrics 采集
⚠️ 注意sum() 不是精确的瞬时值（遍历 cell 期间可能有更新）→ 适合统计近似值，不适合做精确的序号生成
Java 
// 高并发 QPS 统计
private final LongAdder requestCount = new LongAdder();

public void onRequest() {
    requestCount.increment(); // 分段写入，几乎无竞争
}

// 每秒打印 QPS
public void reportQps() {
    long count = requestCount.sumThenReset(); // 累加并清零
    System.out.println("QPS: " + count);
}

/*
 * 性能对比（8 线程 1 亿次 increment）
 * AtomicLong: ~3000ms（大量 CAS 失败重试）
 * LongAdder:  ~200ms（分段减少竞争，快 15 倍）
 */
🔄

AtomicReference — 引用类型原子更新

无锁数据结构基石 ▶

AtomicReference <V> java.util.concurrent.atomic.AtomicReference

对对象引用的原子更新，常用于实现无锁数据结构（如链表头部插入）、CAS 状态切换（INIT → RUNNING → DONE）。

Java 
// 无锁状态机：任务状态 CAS 切换
enum TaskState { INIT, RUNNING, DONE, FAILED }

class Task {
    private final AtomicReference<TaskState> state =
        new AtomicReference<>(TaskState.INIT);

    public boolean start() {
        return state.compareAndSet(TaskState.INIT, TaskState.RUNNING);
    }

    public boolean complete() {
        return state.compareAndSet(TaskState.RUNNING, TaskState.DONE);
    }

    public boolean fail() {
        // 从 RUNNING 或 INIT 都可以转为 FAILED
        return state.updateAndGet(s ->
            s == TaskState.RUNNING || s == TaskState.INIT ? TaskState.FAILED : s
        ) == TaskState.FAILED;
    }
}

// 无锁栈（Treiber Stack）
class ConcurrentStack<T> {
    private final AtomicReference<Node<T>> head = new AtomicReference<>();

    public void push(T value) {
        Node<T> newHead = new Node<>(value);
        Node<T> oldHead;
        do {
            oldHead = head.get();
            newHead.next = oldHead;
        } while (!head.compareAndSet(oldHead, newHead));
    }
}

4collections 并发集合

📦

ConcurrentHashMap — 并发哈希表

JUC 集合之王 ▶

ConcurrentHashMap <K, V> java.util.concurrent.ConcurrentHashMap

线程安全的 HashMap，替代 Collections.synchronizedMap 和 Hashtable。JDK 7 用 Segment 分段锁，JDK 8+ 改用 Node 数组 + CAS + synchronized 锁单个桶 → 并发度大幅提升。

使用场景全局缓存、配置存储、计数统计、并发去重 —— 几乎所有需要并发 Map 的地方
核心特性读操作完全无锁 · 写操作只锁单个桶 · 弱一致性（迭代时能看到部分最新数据）· 不允许 null key/value
⚠️ 注意size() 是近似值 · 迭代不支持 remove（用 entrySet().removeIf() 替代）· 与 HashMap 不同，不允许 null
Java 
// 场景：线程安全的本地缓存
ConcurrentHashMap<String, Object> cache = new ConcurrentHashMap<>();

// 基本操作 —— 完全线程安全
cache.put("config", configObj);
Object val = cache.get("config");

// 原子操作：不存在才写入（避免 put-ifAbsent 竞态）
cache.putIfAbsent("init", new Object());

// computeIfAbsent：原子计算 + 写入
cache.computeIfAbsent("expensive", key -> {
    return computeExpensiveValue(key); // 只在缺失时执行
});

// merge：原子合并（计数器场景）
ConcurrentHashMap<String, LongAdder> wordCounts = new ConcurrentHashMap<>();
wordCounts.computeIfAbsent(word, k -> new LongAdder()).increment();

// 批量遍历（forEach 是线程安全的弱一致性遍历）
cache.forEach((key, value) -> {
    System.out.println(key + " = " + value);
});

// 并行聚合（JDK 8+ reduce）
long totalSize = cache.reduceValuesToInt(8, v -> 1, 0, Integer::sum);
✅ JDK 8 vs 7 关键差异：JDK 7 的 Segment（默认 16 段）意味着最大并发度 16；JDK 8 改为锁单个桶（Node），并发度 = 桶数，大幅提升。且 JDK 8 在桶为空时用 CAS 无锁写入，链表长度 ≥8 转红黑树（O(n) → O(log n)）。
📋

CopyOnWriteArrayList — 写时复制列表

读多写少场景 ▶

CopyOnWriteArrayList <E> java.util.concurrent.CopyOnWriteArrayList

写操作时复制整个数组（加 ReentrantLock），读操作完全无锁直接访问旧数组。适合读远多于写且列表较小的场景。

使用场景监听器列表、配置项列表、事件订阅者 —— 写极少读极多
⚠️ 注意每次写都复制整个数组 → 写性能 O(n) · 内存占用翻倍 · 弱一致性（迭代看到的是创建时的快照）
Java 
// 事件监听器管理 —— 典型读多写少场景
class EventBus {
    private final CopyOnWriteArrayList<EventListener> listeners =
        new CopyOnWriteArrayList<>();

    // 注册 / 注销（写操作少，复制开销可接受）
    public void register(EventListener l) { listeners.add(l); }
    public void unregister(EventListener l) { listeners.remove(l); }

    // 通知所有监听器（读操作多，完全无锁）
    public void fireEvent(Event e) {
        for (EventListener l : listeners) { // 迭代时拿到的是快照
            l.onEvent(e);
        }
    }
}
// 注意：遍历时不需要加锁，也不会 ConcurrentModificationException！
📨

ConcurrentLinkedQueue / BlockingQueue — 并发队列

生产者-消费者模型核心 ▶
队列类型特性适用场景
ConcurrentLinkedQueue无界非阻塞，CAS 无锁，基于 Michael & Scott 算法无界工作队列、CAS 并发任务分发
ArrayBlockingQueue有界数组 + ReentrantLock生产者-消费者、限流缓冲池
LinkedBlockingQueue可选有界链表 + 两把锁（读写分离）线程池任务队列（Executors 默认）
SynchronousQueue无容量，每个 put 必须等 take直接传递任务（newCachedThreadPool）
PriorityBlockingQueue无界优先级堆优先级任务调度
DelayQueue延时出队，元素到期才能取定时任务、缓存过期、延迟消息
Java 
// 生产者-消费者：ArrayBlockingQueue
BlockingQueue<Task> taskQueue = new ArrayBlockingQueue<>(100);

// 生产者
new Thread(() -> {
    while (true) {
        Task t = generateTask();
        taskQueue.put(t); // 队满则阻塞等待
    }
}).start();

// 消费者
new Thread(() -> {
    while (true) {
        Task t = taskQueue.take(); // 队空则阻塞等待
        processTask(t);
    }
}).start();

// DelayQueue：定时执行
DelayQueue<DelayedTask> delayQueue = new DelayQueue<>();
delayQueue.put(new DelayedTask("发报告", 5, TimeUnit.SECONDS));
// 5 秒后 take() 才能取到
DelayedTask task = delayQueue.take(); // 阻塞直到到期

5executor 线程池

🏗️

ThreadPoolExecutor — 核心线程池

最最重要的 JUC 类 ▶

ThreadPoolExecutor java.util.concurrent.ThreadPoolExecutor

JUC 线程池的核心实现。通过 7 个参数精细控制线程创建、队列排队、拒绝策略。是生产环境管理线程的标准方案，替代手动 new Thread()。

7 大参数corePoolSize · maximumPoolSize · keepAliveTime · unit · workQueue · threadFactory · handler
执行流程核心线程未满 → 创建核心线程 · 核心满 → 入队列 · 队列满 → 创建非核心线程到 maxPool · maxPool 满 → 触发拒绝策略
⚠️ 注意禁止用 Executors.newFixedThreadPool（无界队列 OOM 风险）和 newCachedThreadPool（无界线程 OOM 风险）→ 必须手动 new ThreadPoolExecutor
Java 
// 生产环境标准线程池创建 —— 禁止用 Executors！
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    8,                                      // corePoolSize: CPU 密集型 = N+1, IO 密集型 = 2N
    32,                                     // maximumPoolSize: 最大线程数
    60, TimeUnit.SECONDS,                    // keepAliveTime: 非核心线程空闲 60s 回收
    new ArrayBlockingQueue<Runnable>(200),  // 有界队列！防止 OOM
    new ThreadFactoryBuilder()                 // 自定义线程名（Guava 或手写）
        .setNameFormat("biz-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略：让提交者自己跑
);

// 提交任务
executor.submit(() -> doWork());
Future<String> future = executor.submit(() -> computeResult());

// 优雅关闭
executor.shutdown();      // 不接受新任务，等已有任务完成
executor.awaitTermination(30, TimeUnit.SECONDS);
if (!executor.isTerminated()) {
    executor.shutdownNow(); // 强制停
}
🚨 阿里规约：线程池不允许使用 Executors 去创建！newFixedThreadPool 和 newSingleThreadExecutor 用的是 LinkedBlockingQueue（无界，Integer.MAX_VALUE），堆积任务导致 OOM；newCachedThreadPool 的 maxPoolSize 是 Integer.MAX_VALUE，创建过多线程导致 OOM。必须手动 new ThreadPoolExecutor 并指定有界队列。
🔀

ForkJoinPool — 分治并行计算

work-stealing 算法 ▶

ForkJoinPool java.util.concurrent.ForkJoinPool

专为分治任务（Divide and Conquer）设计的线程池。每个线程有自己的工作队列（双端队列），线程空闲时从其他线程队列尾部窃取任务（work-stealing），减少竞争。是 parallelStream() 和 CompletableFuture 的底层线程池。

使用场景大数组/大集合并行处理、归并排序、大数据聚合、CPU 密集型分治计算
核心类RecursiveTask<V>（有返回值）· RecursiveAction（无返回值）
Java 
// 大数组求和 —— 分治 + work-stealing
class SumTask extends RecursiveTask<Long> {
    private final long[] arr;
    private final int start, end;
    private static final int THRESHOLD = 10_000; // 小于 1 万直接算

    SumTask(long[] arr, int start, int end) {
        this.arr = arr; this.start = start; this.end = end;
    }

    @Override
    protected Long compute() {
        int len = end - start;
        if (len <= THRESHOLD) {           // 够小，直接计算
            long sum = 0;
            for (int i = start; i < end; i++) sum += arr[i];
            return sum;
        }
        // 够大，分治
        int mid = start + (end - start) / 2;
        SumTask left = new SumTask(arr, start, mid);
        SumTask right = new SumTask(arr, mid, end);
        left.fork();              // 异步执行左半
        return right.compute() + left.join(); // 右半同步 + 等左半
    }
}

// 使用
long[] data = new long[100_000_000];
// ... 填充数据 ...
ForkJoinPool pool = new ForkJoinPool();
Long total = pool.invoke(new SumTask(data, 0, data.length));

// parallelStream 底层就是 ForkJoinPool.commonPool()
long sum = Arrays.stream(data).parallel().sum();

6tools 同步工具

🧮

CountDownLatch — 倒计时门栓

一次性等待 N 个线程完成 ▶

CountDownLatch java.util.concurrent.CountDownLatch

初始化一个计数 N，每个线程完成后 countDown() 使计数减 1，主线程 await() 阻塞直到计数归零。不可重置（一次性使用）。

使用场景并行初始化（等 N 个服务就绪再启动主流程）、批量任务等待汇总、多源数据加载完成后再渲染
⚠️ 注意计数到 0 后不可重用 · countDown 必须在 finally 中调用 · await 支持超时版本
Java 
// 场景：并行加载多个服务，全部就绪后启动主流程
public void startup() throws InterruptedException {
    String[] services = {"DB", "Redis", "MQ", "Config"};
    CountDownLatch ready = new CountDownLatch(services.length);

    for (String svc : services) {
        new Thread(() -> {
            try {
                initService(svc);        // 初始化服务
                System.out.println(svc + " ready");
            } finally {
                ready.countDown();        // 计数 -1（必须在 finally）
            }
        }).start();
    }

    // 等待所有服务就绪，最多 30 秒
    if (ready.await(30, TimeUnit.SECONDS)) {
        System.out.println("✅ 全部就绪，启动主流程");
    } else {
        System.out.println("⚠️ 超时，部分服务未就绪");
    }
}
🔄

CyclicBarrier — 循环屏障

可重置 · 分阶段同步 ▶

CyclicBarrier java.util.concurrent.CyclicBarrier

N 个线程到达屏障后阻塞，等全部到达后一起放行。可循环使用（放行后自动重置）。可在构造时传入一个 barrierAction，在所有线程到达后由最后一个到达的线程执行。

vs CountDownLatchCountDownLatch 是"一个等 N 个完成"，CyclicBarrier 是"N 个互等到齐再走" · CyclicBarrier 可重用 · CyclicBarrier 无异常时不会超时自动放行
使用场景多线程分阶段计算（如矩阵分块计算、分批数据校验后汇总）、并行测试同步
⚠️ 注意任一线程异常退出会导致 BrokenBarrierException · reset() 会打破当前 barrier
Java 
// 场景：多线程分块计算，每轮完成后汇总
int workers = 4;
CyclicBarrier barrier = new CyclicBarrier(workers, () -> {
    System.out.println("--- 所有分块完成，开始汇总 ---");
});

for (int i = 0; i < workers; i++) {
    final int workerId = i;
    new Thread(() -> {
        for (int round = 1; round <= 3; round++) {
            computeChunk(workerId, round);   // 计算自己的分块
            try {
                barrier.await();              // 等其他线程完成
            } catch (Exception e) { return; }
        }
    }).start();
}
// 输出：4 个 worker 各自算完 → 汇总 → 下一轮 → 重复 3 轮
🚦

Semaphore — 信号量 / 限流器

控制并发访问数 ▶

Semaphore java.util.concurrent.Semaphore

维护一组许可证（permits），acquire() 获取许可（无可用则阻塞），release() 释放许可。本质是一个计数器，用于控制同时访问某资源的线程数。

使用场景限流（如数据库连接池限流、API 并发数控制）、资源池（停车场模型）、限流网关
⚠️ 注意release 不要求由 acquire 的线程调用 → 可在任意线程释放 · 但要确保 acquire-release 配对 · 可用 tryAcquire 实现非阻塞限流
Java 
// 场景：数据库连接池限流（最多 10 个并发连接）
public class ConnectionPool {
    private final Semaphore permits = new Semaphore(10);
    private final DataSource dataSource;

    public Connection getConnection() throws InterruptedException {
        permits.acquire(); // 获取许可（满了就等）
        try {
            return dataSource.getConnection();
        } catch (SQLException e) {
            permits.release(); // 异常时释放
            throw new RuntimeException(e);
        }
    }

    public void returnConnection(Connection conn) {
        try { conn.close(); }
        finally { permits.release(); } // 归还许可
    }
}

// 非阻塞限流：tryAcquire
if (permits.tryAcquire()) {
    try { doWork(); }
    finally { permits.release(); }
} else {
    // 快速失败 / 降级
    throw new RateLimitExceededException("系统繁忙");
}

8AQS 核心原理

🧱

AbstractQueuedSynchronizer — JUC 的灵魂

理解 AQS = 理解 JUC 一半 ▶

AQS（AbstractQueuedSynchronizer）是 ReentrantLock、Semaphore、CountDownLatch、ReentrantReadWriteLock 等几乎所有同步工具的底层实现框架。核心思想：

核心思想模板方法模式 —— 子类只需实现 tryAcquire/tryRelease（独占）或 tryAcquireShared/tryReleaseShared（共享），AQS 负责队列管理和线程阻塞/唤醒
statevolatile int，不同的同步工具赋予不同含义：ReentrantLock 中=锁重入次数 · Semaphore 中=剩余许可数 · CountDownLatch 中=剩余计数
CLH 队列一个 FIFO 双向链表，存放等待获取锁的线程节点（Node），每个 Node 有一个 waitStatus（SIGNAL/CONDITION/CANCELLED）
独占 vs 共享独占模式：同一时刻只有一个线程能获取资源（ReentrantLock）· 共享模式：多个线程可同时获取（Semaphore、CountDownLatch、读锁）
Java 
// AQS 简化原理：ReentrantLock 的 tryAcquire 逻辑
protected final boolean tryAcquire(int acquires) {
    Thread current = Thread.currentThread();
    int c = getState();
    if (c == 0) {
        // state=0 表示无人持有，CAS 抢锁
        if (compareAndSetState(0, acquires)) {
            setExclusiveOwnerThread(current);
            return true;
        }
    }
    else if (current == getExclusiveOwnerThread()) {
        // 重入：当前线程已持有，state+1
        setState(c + acquires);
        return true;
    }
    return false; // 失败 → AQS 框架将线程加入 CLH 队列
}

/*
 * AQS 获取锁的完整流程：
 * 1. tryAcquire() 尝试 CAS 获取
 * 2. 失败 → addWaiter() 创建 Node 加入 CLH 队列尾部
 * 3. acquireQueued() 自旋检查前驱是否是 head
 * 4. 如果前驱是 head 且 tryAcquire 成功 → 出队
 * 5. 否则 LockSupport.park() 阻塞等待前驱唤醒
 *
 * 释放锁：
 * 1. tryRelease() CAS state-1
 * 2. 如果 state 归零 → unparkSuccessor() 唤醒后继节点
 */
✅ 一句话总结 AQS：一个 volatile state（表示同步状态）+ 一个 CLH FIFO 队列（管理等待线程）+ CAS（保证 state 修改原子性）+ park/unpark（线程阻塞唤醒）。所有同步工具只是定义了 state 的含义和 tryAcquire/tryRelease 的逻辑。

9最佳实践与避坑指南

原则做法反面案例 / 风险
禁用 Executors 手动 new ThreadPoolExecutor，指定有界队列 Executors.newFixedThreadPool 用无界队列 → 任务堆积 OOM
锁必须释放 ReentrantLock 的 unlock() 放在 finally 块 临界区抛异常 → lock 永不释放 → 所有等待线程死锁
锁顺序一致 多锁场景按固定顺序获取（如按 hash 值排序） 线程 A 先锁 1 再锁 2，线程 B 先锁 2 再锁 1 → 死锁
带超时获取锁 用 tryLock(timeout) 替代 lock() lock() 无限等待 → 死锁时永久卡死
并发集合优先 用 ConcurrentHashMap 替代 synchronizedMap Collections.synchronizedMap 全局锁 → 并发度极低
原子类替代锁 简单计数用 AtomicInteger/LongAdder synchronized + int 计数 → 性能差 10 倍+
线程命名 自定义 ThreadFactory 设置有意义的线程名 默认 pool-1-thread-1 → 排查问题时无法区分业务
优雅关闭 shutdown() + awaitTermination() + shutdownNow() 直接 kill → 任务丢失 · 不 shutdown → 线程泄漏
countDown 在 finally CountDownLatch 的 countDown 放 finally 异常导致 countDown 不执行 → await 永久阻塞
合理设置线程数 CPU 密集型 = N+1 · IO 密集型 = 2N（N=CPU 核数） 线程太少 → CPU 利用率低 · 太多 → 上下文切换开销大
