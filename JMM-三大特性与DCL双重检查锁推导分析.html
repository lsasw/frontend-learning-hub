1JMM 概述与三大特性

🧠什么是 Java Memory Model（JMM）

JMM（Java 内存模型）是 JSR-133 规范中定义的一组规则，它描述了多线程环境下变量的访问规则——即一个线程对共享变量的写入，何时对其他线程可见，以及编译器和处理器可以在什么范围内对指令进行重排序。

JMM 的核心目的是：在性能与正确性之间取得平衡。它既允许编译器和 CPU 进行优化（重排、缓存），又通过 synchronized、volatile、final 等关键字提供"安全保证"的开关。

JMM 抽象了两个内存区域：

  • 主内存（Main Memory）：所有共享变量存储的地方，所有线程共享。
  • 工作内存（Working Memory / Local Cache）：每个线程私有的寄存器/缓存副本，线程不能直接读写主内存。
💡
一句话理解 JMM

JMM 规定了线程如何与主内存交互：线程读 → 从主内存拷贝到工作内存 → 读工作内存副本；线程写 → 改工作内存 → 刷新回主内存。三大特性描述的就是这个过程中可能出现的三类"不一致"问题。

⚡三大特性一览

特性 核心含义 问题来源 保障手段
可见性 一个线程修改了共享变量，其他线程能立即看到 工作内存缓存导致"看不到最新值" volatile、synchronized、final
原子性 一个或多个操作不可分割，要么全做要么全不做 非原子操作中途被其他线程打断 synchronized、Lock、原子类
有序性 程序执行的顺序符合预期（防止指令重排破坏逻辑） 编译器/CPU 为了性能重排指令 volatile（禁止重排）、synchronized、happens-before
🔑
三大特性的关系

三者并非完全独立。在 DCL 场景中，synchronized 保证了同步块内的原子性和可见性，但同步块之外的第一次检查（null 判断）既没有原子性也没有可见性保障，而有序性（指令重排）的问题恰恰出在同步块内部的赋值操作上。这正是 volatile 的用武之地。

2可见性（Visibility）

👁️定义与 JMM 中的体现

可见性：当一个线程修改了共享变量的值，其他线程能够立即得知这个修改。

在 JMM 中，可见性问题的根源在于工作内存。每个线程有自己的 CPU 缓存 / 寄存器副本，线程 A 修改变量后，可能只更新了自己的工作内存，尚未刷新回主内存；线程 B 也可能仍在读取自己工作内存中的旧值——于是 B"看不到" A 的修改。

💡
volatile 如何保障可见性

volatile 变量的写操作会立即刷新到主内存，读操作会直接从主内存读取（绕过工作内存缓存）。这确保了所有线程看到的是同一份最新值。

🔗可见性在 DCL 中的体现

在 DCL 中，instance 字段的可见性涉及两个层面：

  • synchronized 块内：synchronized 会在锁释放时将工作内存刷新到主内存，在获取锁时让工作内存失效——所以同步块内的读写是可见的。
  • 同步块外的第一次检查：if (instance == null) 这行代码在 synchronized 块外面，没有锁的保护。如果 instance 不加 volatile，其他线程可能读到工作内存中的旧值（仍为 null），导致重复进入同步块。
Java 
// ❶ 同步块外的检查 —— 可见性没有保障（不加 volatile 时）
if (instance == null) {          // ← 线程B 可能读到旧值 null，即使线程A 已经赋值
    synchronized (Singleton.class) {
        // ❷ 同步块内的检查 —— synchronized 保障可见性
        if (instance == null) {      // ← 进入锁后重新读主内存，拿到最新值
            instance = new Singleton();  // ← 赋值
        }
    }
}
⚠️
可见性问题的影响

不加 volatile 时，第一次 null 检查可能读到缓存中的旧值，导致多个线程同时通过第一次检查、竞争锁。虽然第二次检查 + synchronized 能防止重复创建，但性能退化（每个线程都要争锁）。不过，可见性问题本身不会导致错误的对象——真正的致命问题是"有序性"（指令重排），下一节详述。

3原子性（Atomicity）

🔒定义与 JMM 中的体现

原子性：一个或多个操作在执行时不会被其他线程打断，要么全部执行完毕，要么完全不执行。不存在中间状态被其他线程观察到。

在 JMM 中：

  • 基本类型的简单读写（如 int、boolean 的赋值）在大多数平台上是原子的（64 位 long/double 在 32 位平台上可能不是，但 JMM 规定 volatile long/double 读写是原子的）。
  • 复合操作（如 i++ = 读-改-写三步）不是原子的。
  • synchronized 提供了互斥锁，保证同步块内的代码整体原子执行。
Java 
// 非原子操作示例
int count = 0;
count++;   // 实际是三步：① 读 count → ② count+1 → ③ 写回 count
            // 多线程下可能丢失更新

// synchronized 保证原子性
synchronized (lock) {
    count++;  // 在锁保护下，三步不可被打断
}

🔗原子性在 DCL 中的体现

DCL 中的原子性问题聚焦于 instance = new Singleton() 这一行。这个看似一行的赋值操作，在 JVM 层面实际分三步：

1
分配内存

JVM 在堆上为 Singleton 对象分配内存空间

memory = allocate()
2
初始化对象

调用构造方法，填充各字段默认值/初始值

ctorInstance(memory)
3
引用赋值

将 instance 引用指向刚分配的内存地址

instance = memory
💡
引用赋值本身是原子的，但三步整体不是

步骤③ instance = memory（引用赋值）在 32/64 位 JVM 上是原子操作——不会出现"半个引用"。但问题在于：这三步可能被重排序，导致步骤③先于步骤②执行。这就引出了下一个特性：有序性。

⚠️
synchronized 只保证原子性，不保证有序性

synchronized 让同步块内的代码互斥执行，但不会禁止编译器/CPU 对同步块内的指令进行重排序（只要不改变单线程语义）。所以即使有锁保护，instance = new Singleton() 的三步仍然可能被重排。

4有序性（Ordering）

🔀定义与 JMM 中的体现

有序性：程序执行的顺序与代码编写的顺序一致。在没有同步机制的情况下，编译器和 CPU 为了提升性能，可能会对指令进行重排序（Reordering）。

JMM 中的重排序分为三类：

类型执行者说明
编译器重排JIT 编译器JIT 在编译时为优化性能调整指令顺序
指令级重排CPU 流水线CPU 乱序执行（Out-of-Order Execution）
内存重排内存系统CPU 缓存导致内存操作对其他线程可见顺序不同
💡
as-if-serial 语义

重排序遵守 as-if-serial 语义：在单线程环境下，重排序不会改变程序的执行结果。但多线程环境下，重排序可能破坏线程间的协作逻辑——DCL 就是典型受害者。

✅
volatile 如何保障有序性

volatile 通过插入内存屏障（Memory Barrier）来禁止特定方向的重排序。具体来说：volatile 写操作之前的普通写不会被重排到 volatile 写之后；volatile 读操作之后的普通读不会被重排到 volatile 读之前。这构成了 happens-before 关系的核心。

🔀指令重排导致 DCL 失效的根源

回到 instance = new Singleton() 的三步操作。在 as-if-serial 语义下，步骤②和③的顺序对单线程没有影响——所以编译器/CPU 可以将它们重排为 ①→③→②：

✅ 正常顺序（期望）
① 分配内存空间
② 调用构造方法，初始化对象
③ instance → 指向内存地址
结果：其他线程看到的是完整对象 ✅
❌ 重排后顺序（危险）
① 分配内存空间
③ instance → 指向内存地址 ⚠️
② 调用构造方法，初始化对象
结果：其他线程可能拿到未初始化的对象 ❌
🚨
致命场景：半初始化对象

重排后，线程 A 执行到步骤③（已赋值，但构造方法尚未执行），此时 instance != null。线程 B 在同步块外的第一次检查 if (instance == null) 发现不为 null，直接返回这个半初始化的对象。调用者拿到的对象字段可能全是默认值（null / 0），导致空指针、逻辑错误等难以复现的 Bug。

5DCL 初衷与基本结构

STEP 1 · 起源

🏗️为什么需要双重检查锁？

DCL（Double-Checked Locking）是为了实现线程安全的单例模式，同时避免每次调用 getInstance() 都加锁带来的性能损耗。让我们从最简单的方案逐步演进：

Java 
// ❌ 方案一：直接加锁（性能差）
public class Singleton {
    private static Singleton instance;
    public synchronized static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
// 问题：每次调用都加锁，99.9% 的情况下锁是多余的

方案一的问题很明显：单例创建后，后续所有调用仍要争锁，严重拖慢性能。DCL 的思路是——先检查再锁，锁住后再检查一次：

Java 
// DCL 基本结构（不加 volatile —— 有隐患）
public class Singleton {
    private static Singleton instance;   // ← 缺少 volatile

    public static Singleton getInstance() {
        if (instance == null) {                // ❶ 第一次检查（无锁）
            synchronized (Singleton.class) {   // ❷ 加锁
                if (instance == null) {            // ❸ 第二次检查（持锁）
                    instance = new Singleton();     // ❹ 创建实例
                }
            }
        }
        return instance;
    }
}
STEP 1 · 结构分析

📋DCL 四步拆解

❶
第一次检查（无锁）

快速路径：如果 instance 已非 null，直接返回，不争锁

if (instance == null)
❷
加锁

只有 instance 为 null 时才进入同步块

synchronized (Singleton.class)
❸
第二次检查（持锁）

防止多个线程同时通过第一次检查后重复创建

if (instance == null) // again
❹
创建实例

真正实例化单例对象

instance = new Singleton()
💡
"双重检查"的含义

第一次检查（❶）是性能优化——避免不必要的锁竞争；第二次检查（❸）是正确性保障——防止多个线程依次拿到锁后重复创建对象。这就是 "Double-Checked" 的由来。

6不加 volatile 时的线程安全问题

STEP 2 · 深入问题

🚨逐步推导：指令重排如何导致半初始化对象

现在让我们模拟两个线程并发执行 DCL 的场景，逐步推导问题如何发生：

时间线推演：线程 A 与 线程 B 的竞态

T1
线程 A 执行第一次检查

instance == null → true，准备进入同步块

if (instance == null) // true
↓
T2
线程 A 获取锁

进入 synchronized 块，第二次检查 instance == null → true，开始创建对象

synchronized → if (null) → new Singleton()
↓ ⚠️ 此处发生指令重排
T3
线程 A：分配内存 + 直接赋值（跳过初始化）

JVM 执行了 ①分配内存 → ③instance = memory（引用已指向内存），但 ②构造方法 尚未执行。此时 instance ≠ null，但对象未初始化！

① allocate() → ③ instance = memory （② 未执行）
↓ 此时线程 B 介入
T4
线程 B 执行第一次检查（同步块外）

线程 B 读到 instance != null（因为 A 已经赋值了引用），直接返回！

if (instance == null) // false → return instance
↓
T5
线程 B 使用半初始化对象 💀

线程 A 的构造方法 ② 可能还没执行完。线程 B 拿到的对象字段全为默认值（null / 0），后续调用产生各种诡异 Bug。

instance.getField() → NullPointerException 💥
STEP 2 · 根因定位

🔍问题根因总结

为什么 synchronized 没能阻止这个问题？

关键点：线程 B 的读取操作在 synchronized 块外面。

  • 线程 A 在同步块内执行 instance = new Singleton()，受 synchronized 保护。
  • 但线程 B 的第一次检查 if (instance == null) 在同步块外，不持有锁。
  • 即使 A 释放锁时会把工作内存刷新到主内存（可见性），但 A 还没出同步块——A 仍在执行 ②构造方法，锁未释放。
  • 线程 B 在锁外读到了 A 写入的 instance 引用（因为引用赋值已经发生），但此时对象的构造尚未完成。

核心矛盾：synchronized 只保证同一时刻只有一个线程在同步块内，但无法阻止同步块外的线程读到同步块内尚未完成的写入结果。

🚨
三个特性各自的"锅"

有序性 instance = new Singleton() 的指令被重排（①③②），是问题的直接原因。
可见性 线程 B 在锁外读到 instance 的引用，是可见性没有正确同步的传导环节。
原子性 对象创建的三步操作不是原子的，是问题的前提条件——如果三步不可分割，就不会有中间态。

7volatile 解决方案

STEP 3 · 解决方案

✅正确的 DCL 实现

解决方法只需一个关键字：volatile。加上它后，instance = new Singleton() 的三步操作中，步骤②和③不会被重排——volatile 写操作之前的普通写不会被重排到 volatile 写之后。

Java 
// ✅ 正确的 DCL 单例（加 volatile）
public class Singleton {
    @volatile   // ← 这一行就是关键
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {                  // ❶ 第一次检查
            synchronized (Singleton.class) {   // ❷ 加锁
                if (instance == null) {              // ❸ 第二次检查
                    instance = new Singleton();       // ❹ 安全创建
                }
            }
        }
        return instance;
    }
}
STEP 3 · 原理解析

🛡️volatile 如何修复三大特性问题

① 有序性修复 —— 内存屏障禁止重排

volatile 写之前插入 StoreStore 屏障，确保 ②构造方法 的写入不会被重排到 ③instance 赋值之后：

JVM 内存屏障 
// volatile 写之前的屏障效果：
① allocate()          // 分配内存
② ctorInstance()      // 初始化对象 —— ← StoreStore 屏障确保此步先于③完成
   ─── StoreStore Barrier ───   // 禁止 ② 重排到 ③ 之后
③ instance = memory   // volatile 写 —— 此时对象已完全初始化
   ─── StoreLoad Barrier ───    // 禁止后续读重排到 ③ 之前
✅
有序性保障效果

有了内存屏障，instance = memory（步骤③）一定在 ctorInstance()（步骤②）之后执行。其他线程只要看到 instance != null，就一定能拿到完全初始化的对象。

② 可见性修复 —— 强制主内存读写

volatile 变量的写会立即刷新到主内存，读会直接从主内存获取。线程 A 执行 instance = new Singleton() 后，结果立刻对主内存可见；线程 B 在同步块外读取 instance 时，直接读主内存而非工作内存缓存：

线程A
写入 instance
volatile → 立即刷新主内存
→
主内存
instance = obj
最新值 ✅
←
线程B
读取 instance
volatile → 直接读主内存

③ 原子性补充 —— happens-before 语义

volatile 虽然不保证 instance = new Singleton() 整体原子，但通过 happens-before 规则建立了操作间的偏序关系：

volatile 的 happens-before 规则

// 对同一个 volatile 变量的写 happens-before 后续的读 Thread-A: instance = new Singleton(); // volatile 写 ─── happens-before ─── Thread-B: if (instance == null) { ... } // volatile 读 // 结论：A 的写（含构造方法初始化）对 B 的读完全可见

happens-before 是 JMM 定义的偏序关系，它保证：如果 A happens-before B，那么 A 的操作结果对 B 可见，且 A 的执行顺序先于 B。volatile 写 → volatile 读这条 happens-before 链，同时覆盖了可见性（结果可见）和有序性（顺序保证）。

9关键总结

📌 一图记住 DCL 的安全公式

1. 有序性是致命问题

instance = new Singleton() 不是一行代码，而是三步：分配内存 → 初始化 → 赋值引用。编译器可能重排为 分配→赋值→初始化，导致其他线程拿到"空壳对象"。

2. synchronized 管不住同步块外

第一次 null 检查在 synchronized 块外，不受锁保护。synchronized 只保证互斥和块内可见性，不禁止块内指令重排。

3. volatile 用内存屏障禁止重排

volatile 写前插入 StoreStore 屏障，确保初始化一定先于引用赋值。写后插入 StoreLoad 屏障，确保后续线程读到的是完整对象。

4. happens-before 是理论保证

volatile 写 happens-before 后续 volatile 读。这条规则同时保障了可见性（写结果对读可见）和有序性（写之前的操作不会被重排到写之后）。

5. 最终公式

synchronized（互斥+原子） + volatile（有序+可见） = 安全的 DCL 单例
