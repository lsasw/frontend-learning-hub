🌱 Spring AI 概述

Spring AI 是 Spring 官方推出的 AI 应用开发框架，目标是将 Spring 的设计理念（依赖注入、可移植抽象、约定优于配置）引入 AI 领域。它提供统一的 ChatModel、EmbeddingModel、VectorStore 等抽象接口，使开发者可以在 OpenAI、Azure OpenAI、Ollama、Anthropic 等不同模型提供商之间无缝切换，只需更改配置无需修改代码。

📌
核心定位 Spring 生态的 AI 集成层 —— 不是重新造轮子，而是用 Spring 的方式整合 AI 能力。类似于 Spring Data 统一了不同数据库的访问方式，Spring AI 统一了不同 AI 模型的调用方式。

            .user(userMessage)

    public BookInfo extractBookInfo(String text) {

            .call()
            .entity(BookInfo.class);
    }

    public record BookInfo(String title, String author, String isbn) {}

RAG 检索增强生成实现

Spring AI 通过 QuestionAnswerAdvisor 封装 RAG 流程，自动检索相关文档并注入到 Prompt 中：

@Service
public class RagService {

    private final ChatClient chatClient;
    private final VectorStore vectorStore;

    public RagService(ChatClient.Builder builder, VectorStore vectorStore) {
        this.vectorStore = vectorStore;
        this.chatClient = builder
            .defaultAdvisors(new QuestionAnswerAdvisor(vectorStore))
            .build();
    }

    // 文档入库
    public void ingestDocument(String content) {
        Document doc = new Document(content);
        vectorStore.add(List.of(doc));
    }

    // RAG 检索 + 生成
    public String askWithContext(String question) {
        return chatClient.prompt()
            .user(question)
            .call()
            .content();
        // Advisor 自动执行: 检索 → 拼接到 system prompt → 发送给 LLM
    }
}
✅
最佳实践 使用 ChatClient.Builder 在构造时配置 defaultAdvisors，将 RAG 逻辑与业务代码完全解耦。业务层只管调用 chatClient.prompt().user(question).call()，检索逻辑由 Advisor 链自动完成。

🟠 Spring AI Alibaba 概述

Spring AI Alibaba 是阿里巴巴基于 Spring AI 框架开发的扩展项目，深度整合了阿里云的 AI 能力（通义千问 DashScope、通义万相、百炼平台等）。它不仅提供了对阿里云 AI 模型的访问支持，还内置了一套完整的 Agent 框架（基于图编排的多步骤智能体执行引擎），这是它区别于原生 Spring AI 的最大亮点。

📌
与 Spring AI 的关系 Spring AI Alibaba 是 Spring AI 的超集 —— 它依赖 Spring AI 核心接口，在此基础上添加了：(1) 阿里云模型适配 (2) Graph Agent 编排引擎 (3) 开箱即用的应用模板 (4) 阿里云中间件集成。可以理解为 Spring AI 的"阿里云增强版"。

核心架构

Graph Agent 编排引擎

Spring AI Alibaba 内置的 Agent 框架基于有向图 (Directed Graph) 编排，将复杂的多步骤 AI 任务拆解为图中的节点和边。每个节点执行一个原子操作（如 LLM 调用、工具调用、条件判断），边定义执行流转（包括条件分支）。

概念说明类比
Node（节点）执行单元，封装一个具体操作类似函数 / Lambda
Edge（边）节点间流转关系类似流程图箭头
Conditional Edge根据状态动态选择下一节点类似 if/switch 分支
State（状态）在节点间传递的共享数据类似 Context Map
Graph（图）节点和边的完整编排拓扑类似 DAG 工作流

📄 定义 Graph 节点与编排
@Configuration
public class AgentGraphConfig {

    @Bean
    public StateGraph agentGraph(ChatClient chatClient) {
        
        // 定义共享状态
        OverAllState.Factory stateFactory = () -> {
            Map<String, Object> init = new HashMap<>();
            init.put("messages", new ArrayList<>());
            return new OverAllState(init);
        };

        return new StateGraph("agent", stateFactory)
            // 添加节点
            .addNode("understand", understandNode(chatClient))
            .addNode("tool_call", toolCallNode(chatClient))
            .addNode("generate", generateNode(chatClient))
            // 添加边
            .addEdge(START, "understand")
            .addConditionalEdges("understand", routeByIntent())
            .addEdge("tool_call", "generate")
            .addEdge("generate", END);
    }

    private GraphNode understandNode(ChatClient chatClient) {
        return state -> {
            String userInput = state.value("input").toString();
            String intent = chatClient.prompt()
                .user("分析用户意图，返回 'TOOL' 或 'RAG' 或 'DIRECT': " + userInput)
                .call().content();
            return state.update("intent", intent.trim());
        };
    }

    private ConditionalEdgeFunction routeByIntent() {
        return state -> {
            String intent = state.value("intent").toString();
            return switch (intent) {
                case "TOOL" -> "tool_call";
                case "RAG" -> "generate"; // 走 RAG Advisor
                default -> "generate";
            };
        };
    }
}
⚠️
注意事项 Graph 中不能出现孤立的节点（没有入边或出边），否则编译时会抛出 GraphStateException。条件边的返回值必须精确匹配已定义的节点名称。

🔗 LangChain4j 概述

LangChain4j 是 LangChain（Python 生态最流行的 LLM 框架）的 Java 移植版。它致力于在 Java 生态中提供与 LangChain 对等的能力，包括 LLM 调用、对话记忆、RAG、工具调用、AI 服务代理等。与 Spring AI 不同，LangChain4j 不依赖 Spring 生态，可以独立使用，但也提供了 Spring Boot starter。

📌
设计哲学差异 Spring AI 强调"Spring 风格的抽象与约定"，而 LangChain4j 更接近 Python LangChain 的"直接可用"风格——它用 AiServices 动态代理让接口定义即为 AI 调用，代码极其简洁。

组件职责关键特性
ChatLanguageModel底层 LLM 调用接口同步调用，返回完整响应
StreamingChatLanguageModel流式 LLM 调用Token 级别回调推送
AiServicesAI 服务动态代理工厂接口即声明，自动实现
ChatMemory对话记忆管理MessageWindow / TokenWindow

// 1. 定义 AI 服务接口 —— 无需手写实现类
public interface Assistant {

    String chat(String userMessage);

    // 结构化输出 —— 返回 Java 对象
    Sentiment analyzeSentiment(String text);

    // 带 System 消息定制
    @SystemMessage("你是一个翻译助手，将中文翻译为{{language}}")
    String translate(String text, @V("language") String language);

    enum Sentiment { POSITIVE, NEGATIVE, NEUTRAL }
}

// 2. 通过 AiServices 创建代理实例
ChatLanguageModel model = OpenAiChatModel.builder()
    .apiKey("sk-xxx")
    .modelName("gpt-4o")
    .build();

ChatMemory memory = MessageWindowChatMemory.withMaxMessages(20);

Assistant assistant = AiServices.builder(Assistant.class)
    .chatLanguageModel(model)
    .chatMemory(memory)
    .tools(new WeatherTools())  // 工具注册
    .build();

// 3. 像调用普通 Java 方法一样调用 AI
String answer = assistant.chat("你好，介绍一下你自己");
Sentiment sentiment = assistant.analyzeSentiment("这个产品太棒了！");
String translated = assistant.translate("你好世界", "English");

🔮 LangGraph4j 概述

LangGraph4j 是 LangGraph（LangChain 生态中的图编排引擎）的 Java 实现。它专注于有状态、多步骤、可循环的 AI 工作流编排，特别适合构建需要多轮推理、条件分支、人工介入、循环修正的复杂 Agent 系统。如果说 LangChain4j 解决的是"单次 LLM 调用的优雅封装"，那 LangGraph4j 解决的是"多步骤 AI 工作流的编排与状态管理"。

📌
与 LangChain4j 的关系 LangGraph4j 构建在 LangChain4j 之上，使用 LangChain4j 的 ChatLanguageModel、Tool 等组件作为图节点的执行单元。两者是互补关系：LangChain4j 提供 AI 原子能力，LangGraph4j 提供编排能力。

核心概念

概念说明核心类/接口
StateGraph有向图容器，定义节点和边StateGraph<State>
AgentState (State)工作流共享状态，Map 结构Map<String,Object> + Channels
Channel状态字段的更新策略（覆盖/追加）AppendableChannel / ReplaceChannel
Node图中的执行单元函数式接口 (state) -> state
Edge无条件流转addEdge(from, to)
Conditional Edge根据状态动态路由addConditionalEdges(from, func)
CompiledGraph编译后的可执行图graph.compile()

📄 ReAct Agent 图定义
import static org.bsc.langgraph4j.action.AsyncNodeAction.node_async;
import static org.bsc.langgraph4j.action.AsyncEdgeAction.edge_async;

public class ReActAgent {

    public StateGraph<AgentState> buildGraph(ChatLanguageModel model) {

        // 定义状态通道
        Channel messages = AppendableChannel.<AiMessage>of();

        return new StateGraph<>("react_agent", AgentState.class)
            .addChannel("messages", messages)
            
            // 节点1: Agent 推理（调用 LLM 决定下一步）
            .addNode("agent", node_async(state -> {
                List<AiMessage> msgs = state.value("messages");
                AiMessage response = model.generate(msgs);
                return Map.of("messages", response);
            }))
            
            // 节点2: 工具执行
            .addNode("tools", node_async(state -> {
                AiMessage lastMsg = state.value("messages").get(-1);
                // 执行工具调用
                ToolExecutionResult result = executeTools(lastMsg);
                return Map.of("messages", AiMessage.from(result.text()));
            }))
            
            // 边: START → agent
            .addEdge(START, "agent")
            
            // 条件边: agent → tools 或 END
            .addConditionalEdges("agent", edge_async(state -> {
                AiMessage lastMsg = state.value("messages").get(-1);
                if (lastMsg.hasToolExecutionRequests()) {
                    return "tools";  // 需要调用工具
                }
                return "END";     // 已完成，直接输出
            }))
            
            // 边: tools → agent (循环回推理节点)
            .addEdge("tools", "agent");
    }

    public void run(String userInput) throws Exception {
        StateGraph<AgentState> graph = buildGraph(model);
        
        // 编译图（可配置 Checkpointer 做持久化）
        CompiledGraph<AgentState> app = graph
            .compile();  // 生产环境加 .withCheckpointer(...)
        
        // 执行
        Map<String,Object> init = Map.of(
            "messages", UserMessage.from(userInput)
        );
        AgentState finalState = app.invoke(init).get();
        
        System.out.println(finalState.value("messages"));
    }
}
✅
ReAct 循环的核心 tools → agent 这条边形成了一个循环。Agent 每次推理后，如果发现需要调用工具，就跳转到 tools 节点执行工具，然后回到 agent 节点继续推理。这就是 ReAct (Reasoning + Acting) 模式的图编排实现。

Channel 状态更新策略

Channel 是 LangGraph4j 状态管理的核心概念，决定了节点返回的状态如何与当前状态合并：

📝 AppendableChannel

新值追加到已有列表中，适合消息历史、操作日志等累积型数据。

Channel<Message> ch = 
  AppendableChannel.of();
// node 返回 {messages: msg3}
// 状态: [msg1, msg2] 
// → [msg1, msg2, msg3]

🔄 ReplaceChannel

新值直接覆盖旧值，适合当前步骤、中间结果等最新值优先的数据。

Channel<String> ch = 
  ReplaceChannel.of();
// node 返回 {step: "generate"}
// 状态: {step: "understand"}
// → {step: "generate"}

Checkpointer 持久化与恢复

LangGraph4j 支持通过 Checkpointer 保存图执行的状态快照，实现断点恢复、人工介入（Human-in-the-loop）和历史回溯：

// 使用 MemorySaver（内存级，开发用）
CompiledGraph<AgentState> app = graph
    .compile()
    .withCheckpointer(new MemorySaver());

// 带线程 ID 调用 —— 同一线程的状态会持续累积
AgentState state = app.invoke(
    Map.of("messages", UserMessage.from("继续刚才的对话")),
    Map.of(RunnableConfig.THREAD_ID, "session-001")
).get();

// 获取状态历史快照
List<StateSnapshot> snapshots = app.getStateHistory(
    Map.of(RunnableConfig.THREAD_ID, "session-001")
);

// 回退到某个历史快照继续执行
app.fork(snapshot.config());
💡
Human-in-the-loop 实现 在条件边处暂停图执行，等待人工审批后继续。通过 Checkpointer 暂停 → 外部审批 → 恢复执行的流程，实现需要人工确认的 AI 工作流（如高风险操作前的审批）。
2 开发常见问题总结

针对每个框架在实际开发中高频遇到的问题、报错场景及解决方案

1. API Key 配置不生效 / NoSuchBeanException: ChatModel
高频 Spring AI

报错场景：启动时抛出 NoSuchBeanDefinitionException: No qualifying bean of type 'org.springframework.ai.chat.model.ChatModel'，或者配置了 API Key 但调用时报 401 未授权。

根因：(1) 未在 pom.xml 中引入对应模型的 starter 依赖；(2) application.yml 中 API Key 的层级路径写错；(3) 使用环境变量但未正确注入。

解决方案：
① 确认引入了 spring-ai-starter-model-openai（注意 artifactId 必须匹配使用的模型）；
② YAML 路径必须是 spring.ai.openai.api-key（不是 spring.ai.openai.api_key 或 openai.api-key）；
③ 推荐使用 ${OPENAI_API_KEY} 环境变量注入，避免硬编码。
2. 流式响应 (Flux) 在 Controller 中不触发 / 客户端收不到数据
高频 Spring AI

报错场景：Controller 返回 Flux<String> 但前端收不到流式数据，或者直接拿到一个完整的 JSON 数组而非逐条推送。

根因：(1) 未设置正确的 Content-Type；(2) 未使用 ServerSentEvent 包装；(3) Spring WebFlux 未正确配置。

@GetMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> stream(@RequestParam String msg) {
    return chatClient.prompt().user(msg).stream().content()
        .map(chunk -> ServerSentEvent.<String>builder(chunk).build());
}
关键点：
① 必须设置 produces = MediaType.TEXT_EVENT_STREAM_VALUE；
② 推荐用 ServerSentEvent 包装，兼容性更好；
③ 前端用 EventSource 或 fetch + ReadableStream 消费。
3. Advisor 执行顺序不符合预期
中频 Spring AI

报错场景：配置了多个 Advisor（如 RAG + 日志 + 安全检查），但执行顺序不对，导致 RAG 检索结果未注入到最终 Prompt 中。

根因：Spring AI 中 Advisor 的执行顺序由 Order 注解或 getOrder() 方法决定，数字越小优先级越高（先执行）。

解决方案：在自定义 Advisor 上使用 @Order 注解显式指定执行顺序。RAG Advisor 应在日志 Advisor 之前执行（Order 值更小），确保检索结果在日志记录前已注入。
4. VectorStore 连接超时 / 查询结果为空
中频 Spring AI

报错场景：使用 PgVector 或 Redis VectorStore 时，启动报连接超时，或查询相似度结果始终为空列表。

根因：(1) 向量维度不匹配（如 EmbeddingModel 输出 1536 维但 PgVector 表定义为 768 维）；(2) 未执行 schema 初始化；(3) 相似度阈值设置过高。

解决方案：
① 确认 spring.ai.vectorstore.pgvector.dimensions 与 EmbeddingModel 输出维度一致（OpenAI text-embedding-3-small = 1536，text-embedding-ada-002 = 1536）；
② 设置 spring.ai.vectorstore.pgvector.initialize-schema: true 自动建表；
③ 检查 SearchRequest.similarityThreshold，默认 0.0 可能过滤掉所有结果，建议设为 0.5~0.7。
5. Structured Output 转换失败 / JSON 解析异常
低频 Spring AI

报错场景：使用 entity(Class.class) 获取结构化输出时，抛出 JSON 反序列化异常。

根因：LLM 返回的文本中包含 Markdown 代码块标记（```json），导致 JSON 解析器无法直接解析。

解决方案：Spring AI 1.0.0+ 已内置处理 Markdown 包裹的 JSON。如果使用旧版本，在 System Prompt 中明确要求"返回纯 JSON，不要包含 Markdown 代码块标记"。或使用 BeanOutputConverter 手动处理。
6. Function Calling 工具未被 LLM 调用
中频 Spring AI

报错场景：定义了 @Tool 方法并注册到 ChatClient，但 LLM 回答"我无法执行此操作"而不是调用工具。

根因：(1) Tool 描述太模糊，LLM 无法判断何时调用；(2) 使用的模型不支持 Function Calling（如 GPT-3.5 部分版本）；(3) .tools() 未在 prompt 中绑定。

解决方案：
① @Tool 的 description 必须清晰描述功能和适用场景，如"获取指定城市当前天气，输入城市名称"；
② 确认模型支持 Function Calling（GPT-4o / GPT-4-turbo / Claude 3+ 均支持）；
③ 确认调用链：chatClient.prompt().user(msg).tools(myTools).call()。

高频 Alibaba

4. Graph 状态在节点间丢失 / State 为 null
中频 Alibaba

报错场景：在后续节点中通过 state.value("key") 获取值时返回 null，但前一个节点明明已写入。

根因：(1) 节点返回的 Map key 与读取的 key 不一致；(2) 状态未通过 state.update() 正确更新而是直接修改了局部变量；(3) 并行节点间的状态合并冲突。

解决方案：
① 节点必须返回 Map.of("key", value) 或调用 state.update("key", value)，不能直接修改 state 对象；
② 确认 key 名称完全一致（区分大小写）；
③ 并行执行节点不要写同一个 key，使用不同 key 或在汇聚节点合并。
5. Graph 循环导致无限执行 / 栈溢出
低频 Alibaba

报错场景：Agent 在工具调用和推理之间无限循环，最终超时或栈溢出。

根因：条件边的路由逻辑有缺陷，导致 LLM 持续认为需要调用工具而无法收敛到 END。

解决方案：
① 设置最大迭代次数：graph.compile().withMaxIterations(10)；
② 在条件边中增加收敛判断：如果消息历史超过 N 轮仍无最终答案，强制路由到 END；
③ 在 System Prompt 中明确指示 LLM"最多调用 3 次工具后给出最终回答"。
1. AiServices 代理创建失败 / NullPointerException
高频 LangChain4j

报错场景：调用 AiServices.builder(Interface.class).build() 后，调用方法时抛出 NPE 或 IllegalStateException: chatLanguageModel is not set。

根因：(1) 未调用 .chatLanguageModel(model) 设置模型；(2) 接口方法返回类型不被支持；(3) @SystemMessage 中的模板变量未通过 @V 提供。

解决方案：
① 确认 builder 链中设置了 .chatLanguageModel(model)；
② AiServices 支持的返回类型：String / int / double / boolean / enum / POJO（有默认构造器）/ List<String>；
③ @SystemMessage 中 {{varName}} 必须有对应的 @V("varName") 参数。
2. @Tool 方法参数注解 @P 缺失导致工具调用失败
高频 LangChain4j

报错场景：LLM 尝试调用工具但传参错误，或工具未被 LLM 识别为可用工具。

根因：LangChain4j 要求 @Tool 方法的每个参数必须标注 @P("描述")，否则 LLM 无法理解参数含义。缺少 @P 注解会导致工具描述不完整。

解决方案：每个参数必须加 @P 注解，描述要具体清晰： 
// ✅ 正确写法
@Tool("查询指定日期的汇率")
String getRate(@P("日期，格式 yyyy-MM-dd") String date,
           @P("货币对，如 USD/CNY") String pair) { ... }

// ❌ 错误写法 —— 缺少 @P
@Tool("查询汇率")
String getRate(String date, String pair) { ... }
3. ChatMemory 消息累积导致 Token 爆炸
中频 LangChain4j

报错场景：长时间对话后突然报 Maximum context length exceeded 或调用变慢且费用飙升。

根因：ChatMemory 管理不当，所有历史消息都被发送给 LLM，Token 数累积超过模型上下文窗口。

解决方案：
① 使用 MessageWindowChatMemory.withMaxMessages(20) 限制消息条数；
② 或使用 TokenWindowChatMemory.withMaxTokens(3000, tokenizer) 按 Token 限制（更精确）；
③ 对于多用户场景，每个用户使用独立的 memory：ChatMemoryProvider + memoryId。 
ChatMemoryProvider provider = memoryId -> 
    MessageWindowChatMemory.builder()
        .id(memoryId)
        .maxMessages(20)
        .build();

Assistant assistant = AiServices.builder(Assistant.class)
    .chatLanguageModel(model)
    .chatMemoryProvider(provider)
    .build();
4. RAG 检索结果不相关 / 检索为空
中频 LangChain4j

报错场景：配置了 EmbeddingStore + ContentRetriever，但 RAG 回答没有引用知识库内容，或检索结果完全不相关。

根因：(1) 文档切分粒度不合理（太大或太小）；(2) Embedding 模型与查询不匹配；(3) minScore 设置过高导致过滤掉所有结果；(4) 查询向量化与文档向量化使用了不同模型。

解决方案：
① 文档切分推荐 300~500 Token，overlap 50~100 Token：DocumentSplitters.recursive(500, 50)；
② minScore 建议设 0.6~0.75，不要设 0.9 以上；
③ 确保入库和查询使用同一个 EmbeddingModel 实例；
④ 考虑添加 Query Transformer（如同义词扩展、HyDE 等技术）提升检索质量。
5. StreamingChatLanguageModel 回调不触发
低频 LangChain4j

报错场景：使用 StreamingChatLanguageModel 但 onPartialResponse 回调从未触发。

根因：(1) 部分模型提供商不支持流式输出；(2) StreamingResponseHandler 未正确实现 onComplete 和 onError；(3) 网络代理缓冲了 SSE 流。

解决方案：确认模型支持流式（OpenAI gpt-4o 等支持），检查网络环境是否有代理缓冲 SSE。在 handler 中务必实现 onError 以捕获异常。
1. 图编译失败 / 环检测异常
高频 LangGraph4j

报错场景：调用 graph.compile() 时抛出 IllegalArgumentException: Cycle detected。

根因：图中存在无法终止的循环（如 agent → tools → agent，但没有条件边判断何时跳出循环）。LangGraph4j 默认不允许未受控的循环。

解决方案：
① 循环路径上必须有条件边来打破循环（如 agent 节点判断是否已完成，是则路由到 END）；
② 编译时指定最大迭代次数：graph.compile().withMaxIterations(15)；
③ 如果是有意为之的循环（ReAct 模式），确保条件边的逻辑在合理步数内能收敛到 END。
2. Channel 状态合并异常 / ConcurrentModificationException
高频 LangGraph4j

报错场景：并行执行多个节点时，状态合并抛出 ConcurrentModificationException 或状态数据丢失。

根因：多个并行节点同时写入同一个 Channel（特别是 AppendableChannel），导致并发修改异常或写入互相覆盖。

解决方案：
① 并行节点不要写同一个 Channel key，每个并行分支写独立 key，在汇聚节点合并；
② 如果必须写同一 key，使用线程安全的 Channel 实现；
③ 尽量避免在并行节点中修改共享状态，改为在后续串行节点中处理。
3. Checkpointer 恢复状态丢失 / 序列化失败
中频 LangGraph4j

报错场景：使用 Checkpointer 保存状态后恢复时，部分数据丢失或抛出序列化异常。

根因：(1) State 中存储了不可序列化的对象（如 InputStream、Connection）；(2) MemorySaver 仅存储在内存中，重启后丢失；(3) Thread ID 管理不当导致状态混淆。

解决方案：
① State 中只存储可序列化数据（String / Map / List / 基本类型），不要存储连接对象或流；
② 生产环境使用持久化 Checkpointer（如 JDBC Checkpointer），而非 MemorySaver；
③ 每个会话/用户使用唯一的 Thread ID，避免状态混淆。
4. 条件边路由错误 / 跳转到未定义节点
中频 LangGraph4j

报错场景：条件边返回的节点名称不在图中定义，抛出 IllegalArgumentException: Node 'xxx' not found。

根因：条件边的路由函数返回了未通过 addNode 注册的节点名，或字符串拼写不一致。

解决方案：
① 条件边路由函数的返回值必须是已注册的节点名或 END 常量；
② 使用枚举或常量类管理节点名，避免硬编码字符串拼写错误；
③ 在路由函数中添加 default 分支，确保兜底路由到 END 或安全节点。
5. 图执行超时 / 异步调用不返回
低频 LangGraph4j

报错场景：app.invoke(input).get() 长时间阻塞不返回，最终超时。

根因：(1) LLM 调用超时未设置；(2) 图中有无限循环；(3) 异步线程池耗尽。

解决方案：
① 设置 LLM 调用超时：OpenAiChatModel.builder().timeout(Duration.ofSeconds(30))；
② 设置图最大迭代次数防止无限循环；
③ 使用 app.invoke(input).get(60, TimeUnit.SECONDS) 设置整体超时。

Q1 Spring AI 的核心设计理念是什么？它解决了什么问题？ 基础 ▶
💡 参考答案

Spring AI 的核心设计理念是"可移植的抽象 + Spring 约定"。它解决了 AI 应用开发中的两个核心痛点：

  1. 模型锁定问题：传统开发中直接调用 OpenAI SDK，代码与特定模型深度耦合。Spring AI 提供 ChatModel 统一接口，切换模型只需改配置（类似 Spring Data 统一数据库访问）。
  2. AI 能力与 Spring 生态的整合：将 AI 能力以 Spring Boot Starter 形式提供，支持自动配置、依赖注入、Actuator 监控等 Spring 原生能力。

关键设计：ChatClient（高级 API）→ ChatModel（低层接口）→ Provider Adapter（模型适配层），三层分离，每层可独立替换。

Q2 Spring AI 中 Advisor 的作用是什么？与 Servlet Filter 有何异同？ 中级 ▶
💡 参考答案

Advisor 是 Spring AI 的请求/响应拦截器链，可以在 LLM 调用前修改 Prompt（如注入 RAG 检索结果），在调用后处理响应（如日志记录、敏感信息脱敏）。

与 Servlet Filter 的异同：

  • 相同点：都是链式执行、可排序（Order）、可拦截前后双向。
  • 不同点：Filter 拦截 HTTP 请求，Advisor 拦截 LLM 调用；Filter 操作的是 HttpServletRequest/Response，Advisor 操作的是 Prompt 和 ChatResponse；Advisor 可以直接修改发给 LLM 的 Prompt 内容。

典型应用：QuestionAnswerAdvisor（RAG）、SafeGuardAdvisor（内容安全）、自定义日志/审计 Advisor。

Q3 Spring AI 如何实现 RAG？整个流程是怎样的？ 中级 ▶
💡 参考答案

Spring AI 通过 QuestionAnswerAdvisor 封装 RAG 流程，整体分为 5 步：

  1. 文档加载：使用 DocumentReader 读取 PDF/Markdown/文本等
  2. 文档切分：通过 TokenTextSplitter 将长文档切分为语义片段
  3. 向量化入库：EmbeddingModel 将文本片段转为向量，存入 VectorStore
  4. 检索注入：用户提问时，Advisor 先将问题向量化，从 VectorStore 检索 Top-K 相关片段，拼接到 System Prompt 中
  5. LLM 生成：将增强后的 Prompt 发送给 LLM，生成基于上下文的回答

关键优势：RAG 逻辑完全封装在 Advisor 中，业务代码只需 chatClient.prompt().user(question).call()，实现关注点分离。

Q4 Spring AI 的 ChatClient 和 ChatModel 有什么区别？分别什么场景使用？ 中级 ▶
💡 参考答案

ChatModel 是底层模型调用抽象接口，直接操作消息列表，返回 ChatResponse，需要手动处理 Prompt 构造、消息历史、工具注册等。适合需要精细控制的场景。

ChatClient 是高级 API，提供流式 Builder 风格（prompt().user().call()），内置 Advisor 链、结构化输出、工具调用支持。适合大多数业务场景。

Q5 Spring AI 的 Function Calling 如何实现？底层原理是什么？ 高级 ▶
💡 参考答案

Spring AI 的 Function Calling 通过 @Tool 注解 + 自动注册机制实现：

  1. 注册阶段：Spring AI 扫描 @Tool 注解的方法，生成工具描述（名称、描述、参数 JSON Schema），注册到 ToolCallback 中
  2. 请求阶段：调用 .tools(myTools) 时，将工具描述附加到 LLM 请求的 tools 参数中
  3. LLM 决策：LLM 分析用户意图，决定是否需要调用工具，返回 tool_calls 字段（包含工具名和参数）
  4. 执行阶段：Spring AI 解析 tool_calls，通过反射调用对应的 Java 方法，获取返回值
  5. 融合阶段：将工具执行结果作为 ToolMessage 发回 LLM，LLM 基于结果生成最终回答

本质是两轮 LLM 调用：第一轮 LLM 决定调用什么工具 → 执行工具 → 第二轮 LLM 基于工具结果生成回答。

Q1 Spring AI Alibaba 与 Spring AI 是什么关系？为什么要用 Alibaba 版本？ 基础 ▶
💡 参考答案

Spring AI Alibaba 是 Spring AI 的超集扩展，关系类似于 Spring Cloud Alibaba 与 Spring Cloud：

  • 依赖关系：Alibaba 版本依赖 Spring AI 核心接口，在其上添加扩展
  • 模型适配：提供通义千问（DashScope）、通义万相等阿里云模型的原生适配
  • Graph Agent 引擎：内置基于图编排的多步骤 Agent 框架，Spring AI 原生没有
  • 应用模板：提供开箱即用的 JTem 模板，快速搭建 AI 应用
  • 云生态集成：与阿里云中间件（Nacos、Sentinel、OSS 等）深度整合

使用 Alibaba 版本的核心原因：(1) 需要使用通义千问等国产模型；(2) 需要 Graph Agent 编排能力；(3) 已有阿里云基础设施。

Q2 详细说明 Spring AI Alibaba Graph Agent 的编排原理 高级 ▶
💡 参考答案

Graph Agent 基于有向图（Directed Graph）编排，核心原理：

  1. 图定义：通过 StateGraph API 定义节点（Node）和边（Edge），构建有向图拓扑
  2. 状态传递：图执行时维护一个 OverAllState（Map 结构），每个节点读取输入状态、返回状态更新
  3. 条件路由：addConditionalEdges 接受一个路由函数，根据当前状态动态决定下一跳节点
  4. 循环支持：图允许有环（循环），通过条件边控制何时跳出循环（ReAct 模式的核心）
  5. 编译执行：graph.compile() 将图定义编译为可执行的 CompiledGraph，调用 invoke() 执行

与 Spring AI 原生 Advisor 链的区别：Advisor 是线性管道（A→B→C→LLM→C→B→A），Graph 是有向图（支持分支、并行、循环、汇聚），表达能力更强。

Q3 Spring AI Alibaba Graph 中的 State 管理机制是什么？如何避免状态冲突？ 中级 ▶
💡 参考答案

State 是 Graph 中节点间传递的共享数据，采用 Map 结构存储。每个节点通过返回 Map<String, Object> 来更新状态。

避免状态冲突的策略：

  • Key 命名规范：使用前缀区分不同节点的数据，如 "understand.intent"、"tool.result"
  • 不可变更新：节点不直接修改 State 对象，而是返回新的 Map，由框架合并
  • 并行节点隔离：并行执行的节点写不同的 key，汇聚节点再合并
  • 条件边读取：条件路由函数只读取 State，不写入，避免路由逻辑产生副作用

Q5 如何实现一个支持人工审批（Human-in-the-loop）的 Graph Agent？ 高级 ▶
💡 参考答案

核心思路是在条件边处暂停图执行，等待人工审批后恢复：

  1. 设计审批节点：在需要审批的操作前添加一个 "await_approval" 节点
  2. 状态暂停：该节点将状态设为 PENDING_APPROVAL，通过 Checkpointer 持久化当前状态
  3. 异步通知：通过消息队列或 Webhook 通知审批人
  4. 恢复执行：审批人通过 API 提交审批结果（approve/reject），使用 graph.fork(config) 从暂停点恢复执行
  5. 条件路由：恢复后根据审批结果路由到 "execute" 或 "cancel" 节点

关键依赖：Checkpointer（状态持久化）、Thread ID（会话关联）、外部触发机制（Webhook/MQ）。

Q1 LangChain4j 的 AiServices 动态代理原理是什么？ 中级 ▶
💡 参考答案

AiServices 使用 JDK 动态代理（Proxy.newProxyInstance）实现。核心流程：

  1. 接口扫描：AiServices.builder(Interface.class) 扫描接口的所有方法，解析注解（@SystemMessage、@UserMessage、@V、@MemoryId 等）
  2. 代理创建：调用 build() 时生成 InvocationHandler，每个方法调用被拦截
  3. 请求组装：拦截到方法调用时，根据注解信息组装 Prompt（System 消息 + User 消息 + Memory 历史 + RAG 上下文）
  4. 模型调用：将组装好的 Prompt 发送给 ChatLanguageModel
  5. 结果映射：根据方法返回类型，将 LLM 响应转换为对应 Java 类型（String / Enum / POJO / int 等）

本质是"接口即声明"模式 —— 你定义接口签名和注解，框架自动实现 LLM 调用逻辑，类似于 MyBatis 的 Mapper 代理。

Q4 LangChain4j 中 RAG 的完整流程是什么？如何提升检索质量？ 高级 ▶
💡 参考答案

完整流程：文档加载 → 文档切分 → 向量化 → 入库 → 查询向量化 → 相似度检索 → 上下文拼接 → LLM 生成

提升检索质量的策略：

  • 切分优化：使用 DocumentSplitters.recursive() 递归切分，保持语义完整（按段落 → 句子 → 词）
  • 查询改写：对用户查询进行扩展（同义词、HyDE 假设文档生成），提升召回率
  • Rerank 重排：先向量检索 Top-20，再用 Cross-Encoder 模型重排取 Top-5
  • 元数据过滤：在检索时按元数据（时间、类别）过滤，减少噪声
  • 混合检索：向量检索 + 关键词检索 (BM25)，取并集或加权融合
  • Embedding 微调：对特定领域数据微调 Embedding 模型
Q5 LangChain4j 如何处理 Tool Calling 的多轮交互？ 高级 ▶
💡 参考答案

LangChain4j 的 Tool Calling 多轮交互流程：

  1. 第一轮：用户消息 → LLM 返回 ToolExecutionRequest（工具名 + 参数）
  2. 工具执行：LangChain4j 通过反射调用对应 @Tool 方法，获取返回值
  3. 第二轮：将工具结果作为 ToolExecutionResultMessage 追加到消息历史，再次调用 LLM
  4. 循环判断：LLM 可能再次返回工具调用请求（多工具链式调用），LangChain4j 会循环执行直到 LLM 返回纯文本回答

关键机制：AiServices 内部自动处理多轮循环，开发者只需注册工具。但需注意 ChatMemory 会累积所有中间消息，需合理设置窗口大小。

Q1 LangGraph4j 解决了什么问题？为什么需要图编排而不直接用顺序代码？ 中级 ▶
💡 参考答案

LangGraph4j 解决的是复杂 AI 工作流的编排与状态管理问题。顺序代码的局限性：

  • 循环困难：ReAct Agent 需要在"推理→工具调用→推理"间循环，顺序代码需要手写 while 循环和状态管理
  • 条件分支：根据 LLM 输出动态决定下一步，顺序代码变成大量 if-else
  • 并行执行：多个工具同时调用，顺序代码需要手写线程池
  • 状态持久化：断点恢复、历史回溯，顺序代码没有内置支持
  • 人工介入：暂停等待审批后恢复，顺序代码难以实现

图编排将流程定义为可视化拓扑，节点是原子操作，边定义流转逻辑，状态由框架管理。代码更清晰、可维护、可观测。

Q2 LangGraph4j 的 Channel 机制是什么？为什么需要不同的 Channel 类型？ 中级 ▶
💡 参考答案

Channel 是 LangGraph4j 状态管理的核心抽象，定义了状态字段的更新策略：

  • AppendableChannel：新值追加到列表尾部。适用于消息历史、操作日志等需要完整记录的字段。节点返回 {messages: newMsg}，框架自动执行 list.add(newMsg)。
  • ReplaceChannel：新值直接覆盖旧值。适用于当前状态、中间结果等只需最新值的字段。节点返回 {step: "generate"}，框架执行 map.put("step", "generate")。

需要不同类型的原因：不同字段有不同的语义需求。消息历史如果用 ReplaceChannel，每轮对话只剩最后一条消息，丢失上下文；当前步骤如果用 AppendableChannel，会变成一个无意义的列表。

Q3 LangGraph4j 如何实现 ReAct Agent？与 LangChain4j 的 AiServices Agent 有何区别？ 高级 ▶
💡 参考答案

LangGraph4j 实现 ReAct Agent：

  1. 定义 "agent" 节点（LLM 推理）和 "tools" 节点（工具执行）
  2. 边：START → agent，agent → (条件) → tools 或 END，tools → agent
  3. 条件边检查 LLM 响应是否包含 ToolExecutionRequest：是则路由到 tools，否则到 END
  4. tools → agent 形成循环，实现 Reasoning-Acting 交替

Q4 LangGraph4j 的 Checkpointer 有什么作用？如何实现 Human-in-the-loop？ 高级 ▶
💡 参考答案

Checkpointer 作用：

  • 状态持久化：在图执行过程中定期保存状态快照
  • 断点恢复：应用崩溃后从最近快照恢复执行
  • 历史回溯：查看任意时刻的状态快照，支持回退到历史节点重新执行
  • 会话隔离：通过 Thread ID 区分不同用户/会话的状态

Human-in-the-loop 实现：

  1. 在需要人工介入的节点前设置暂停点
  2. 图执行到暂停点时，通过 Checkpointer 保存当前状态，返回暂停信号给调用方
  3. 调用方通知人工审批（Webhook/MQ/邮件）
  4. 人工通过 API 提交决策后，调用 graph.fork(config) 从暂停点恢复执行
  5. 条件边根据人工决策路由到不同分支

核心 API：MemorySaver（开发环境）、JdbcCheckpointer（生产环境持久化到数据库）。
