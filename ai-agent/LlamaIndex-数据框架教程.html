LlamaIndex 是什么 —— 与 LangChain 的定位差异

一句话定义

LlamaIndex（前身为 GPT Index）是一个 以数据为中心 的 LLM 应用框架。它的核心使命是：让你的数据可以被 LLM 理解和使用——把你散落在各处的文档、数据库、API 变成 LLM 可查询的知识。

🦙 LlamaIndex 的设计哲学

  • Data-first：数据接入 → 索引 → 查询，三步搞定
  • 索引为核心：提供 10+ 种索引类型应对不同场景
  • 开箱即用：5 行代码搭建一个 RAG 问答系统
  • 擅长：文档问答、知识库、结构化+非结构化混合查询

⛓️ 与 LangChain 对比

维度LlamaIndexLangChain
核心关注数据索引与检索组件编排与 Agent
上手门槛较低（API 设计直观）中等（概念较多）
RAG 能力⭐⭐⭐⭐⭐ 王者级⭐⭐⭐ 需较多自定义
Agent 能力⭐⭐⭐ 够用⭐⭐⭐⭐⭐ 王者级
最佳拍档两者可以一起用！LlamaIndex 做 RAG + LangChain 做 orchestration

数据接入 —— 从 160+ 种数据源加载

SimpleDirectoryReader（最常用）

PYTHON
from llama_index.core import SimpleDirectoryReader

# 一行加载整个目录（支持 PDF, Markdown, TXT, CSV, 代码文件...）
documents = SimpleDirectoryReader(
    input_dir="./data",
    recursive=True,               # 递归子目录
    required_exts=[".pdf", ".md", ".txt"],
    filename_as_id=True             # 文件名作为 doc ID（便于追溯）
).load_data()

print(f"加载了 {len(documents)} 个文档")

坑 #1：PDF 加载后全是乱码或空文本

LlamaIndex 默认用 PyPDF2 解析 PDF，但 PyPDF2 对中文 PDF、扫描件、复杂排版的解析效果很差，经常返回空字符串或乱码。

安装 llama-index-readers-file 并使用 PDFReader：from llama_index.readers.file import PDFReader。对于扫描件 PDF，建议先用 OCR（如 pytesseract）预处理。中文 PDF 优先用 pymupdf（fitz）。

Node Parser —— 分块策略

PYTHON
from llama_index.core.node_parser import (
    SentenceSplitter,
    SemanticSplitterNodeParser,
    MarkdownNodeParser,
    CodeSplitter,
)

# 方式 1：按句子分块（最通用）
parser = SentenceSplitter(
    chunk_size=512,
    chunk_overlap=64,
    paragraph_separator="\n\n"
)

# 方式 2：语义分块（推荐！）
# 根据 embedding 相似度自动切分，比固定大小效果好很多
from llama_index.embeddings.openai import OpenAIEmbedding
semantic_parser = SemanticSplitterNodeParser(
    embed_model=OpenAIEmbedding(),
    breakpoint_percentile_threshold=95,  # 相似度断点阈值
    buffer_size=1,
)

# 方式 3：Markdown 结构分块（按标题层级）
md_parser = MarkdownNodeParser()

# 使用
nodes = parser.get_nodes_from_documents(documents)

坑 #2：SemanticSplitter 调用太频繁导致 embedding 费用爆炸

SemanticSplitterNodeParser 会对每个句子做 embedding 来计算语义相似度。如果你的文档有 10 万个句子，这就是 10 万次 API 调用。

对超大文档，先用 SentenceSplitter 做一个粗分块（chunk_size=2048），再对每个粗分块用 SemanticSplitter 细切。或者直接用简单的 SentenceSplitter + 合理的 overlap，大部分场景已经够用。

坑 #3：代码文件直接用普通分块 → 函数被拦腰截断

Python 文件用 SentenceSplitter 分块，可能把一半函数体分到 chunk A、另一半分到 chunk B。检索时要么找不到，要么找到的是不完整的代码片段。

代码文件使用 CodeSplitter，它会按函数/类边界分块，保持代码完整性。

VectorStoreIndex —— 90% 场景的首选

PYTHON
from llama_index.core import VectorStoreIndex, Settings
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.llms.openai import OpenAI
from llama_index.vector_stores.chroma import ChromaVectorStore
import chromadb

# 全局配置
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.llm = OpenAI(model="gpt-4o-mini", temperature=0.1)

# 方式 1：默认内存索引（原型阶段）
index = VectorStoreIndex.from_documents(documents)

# 方式 2：持久化到 ChromaDB（生产环境）
chroma_client = chromadb.PersistentClient(path="./chroma_db")
chroma_collection = chroma_client.get_or_create_collection("my_knowledge")
vector_store = ChromaVectorStore(chroma_collection=chroma_collection)
index = VectorStoreIndex.from_vector_store(vector_store)

# 保存 & 加载（方式 1 的持久化）
index.storage_context.persist(persist_dir="./index_store")
# 下次加载：
from llama_index.core import StorageContext, load_index_from_storage
storage_context = StorageContext.from_defaults(persist_dir="./index_store")
index = load_index_from_storage(storage_context)

坑 #4：Settings 配置时机错误 → 索引重建才发现用错了 embedding 模型

Settings.embed_model 必须在 VectorStoreIndex.from_documents() 之前 设置。如果在创建索引之后才改 Settings，已创建的索引不会自动更新。等查询时发现检索质量差，只能重建整个索引。

在脚本/应用的最开头（import 之后、建索引之前）就配置好 Settings。建索引后如果要切换模型，必须重建索引。

坑 #5：向量存储和索引分开保存 → 加载后只恢复了一半

LlamaIndex 的持久化分两部分：向量存储（vector embeddings）和 索引元数据（docstore、index_store）。如果用 ChromaDB 存向量但忘记 persist index_store，下次加载时只有向量没有元数据映射。

使用外部向量库（ChromaDB/Pinecone）时：向量由外部库管理，但 storage_context.persist() 仍需调用（保存 docstore 映射）。或者全部存到外部库，不用 llama_index 的持久化。

混合索引 —— 多索引联合检索

PYTHON
# 同时建立 VectorStoreIndex + KeywordTableIndex
from llama_index.core import VectorStoreIndex, KeywordTableIndex
from llama_index.core.query_engine import RouterQueryEngine
from llama_index.core.selectors import LLMSingleSelector
from llama_index.core.tools import QueryEngineTool

vector_index = VectorStoreIndex.from_documents(docs)
keyword_index = KeywordTableIndex.from_documents(docs)

# 包装成工具
vector_tool = QueryEngineTool.from_defaults(
    query_engine=vector_index.as_query_engine(),
    description="适合语义模糊、概念性问题"
)
keyword_tool = QueryEngineTool.from_defaults(
    query_engine=keyword_index.as_query_engine(),
    description="适合精确关键词匹配"
)

# Router 自动选择最合适的索引
router = RouterQueryEngine(
    selector=LLMSingleSelector.from_defaults(),
    query_engine_tools=[vector_tool, keyword_tool],
)
result = router.query("数据库连接池的配置参数有哪些？")
print(result)

Query Engine —— 从检索到回答

四种查询引擎对比

引擎检索策略适用场景
as_query_engine()检索 top-k 文档，LLM 合成答案通用问答（默认）
as_chat_engine()带对话记忆的检索多轮对话场景
RetrieverQueryEngine自定义检索器 + 后处理需要精细控制检索流程
SubQuestionQueryEngine拆分子问题 → 逐个检索 → 合成复杂/对比类问题
PYTHON
# 1. 基础查询
query_engine = index.as_query_engine(
    similarity_top_k=5,            # 检索 top-5
    response_mode="compact",          # compact / tree_summarize / refine
    streaming=True,
)
response = query_engine.query("LlamaIndex 支持哪些索引类型？")
for node in response.source_nodes:
    print(f"来源: {node.metadata.get('file_name')}, 相关度: {node.score:.3f}")

# 2. SubQuestion —— 拆解复杂问题
from llama_index.core.query_engine import SubQuestionQueryEngine
from llama_index.core.tools import QueryEngineTool

sub_engine = SubQuestionQueryEngine.from_defaults(
    query_engine_tools=[
        QueryEngineTool.from_defaults(query_engine, description="技术文档"),
    ],
    llm=OpenAI(model="gpt-4o")  # 子问题拆分需要强模型
)
# 自动拆分："对比 LlamaIndex v0.9 和 v0.10 的变化"
# → ["LlamaIndex v0.9 特点", "LlamaIndex v0.10 新功能", "总结差异"]
response = sub_engine.query("对比 LlamaIndex v0.9 和 v0.10 的核心变化")

坑 #6：response_mode 选错 → 回答不完整或幻觉严重

LlamaIndex 有 6 种 response_mode，默认 compact 适合大部分场景。但 tree_summarize 适合"总结全文"类问题，refine 适合"逐段精炼"。用错了可能导致只回答了第一个 chunk 的内容。

compact（默认，通用）→ tree_summarize（总结性、全文覆盖）→ refine（逐步精炼、长文档细节）→ no_text（只返回检索结果不合成）。根据问题类型选择。

坑 #7：SubQuestionQueryEngine 拆分不准 → 回答文不对题

SubQuestionQueryEngine 依赖 LLM 把复杂问题拆成子问题。如果原问题表述模糊或专有名词多，拆分结果可能偏差很大，导致检索的内容和用户想问的完全不相关。

在问题中提供足够的上下文和专有名词解释。对于已知复杂的问题类型，建议人工预设子问题模板（用 QueryTransform），而不是完全依赖 LLM 自动拆分。

Chat Engine —— 多轮对话 RAG

PYTHON
from llama_index.core.memory import ChatMemoryBuffer

# 带记忆的多轮对话
chat_engine = index.as_chat_engine(
    chat_mode="condense_plus_context",  # 推荐模式
    memory=ChatMemoryBuffer.from_defaults(token_limit=3000),
    verbose=True,
)

# 多轮对话
response1 = chat_engine.chat("LlamaIndex 支持哪些索引类型？")
print(response1)

response2 = chat_engine.chat("刚才说的第一种有什么特点？")  # 自动关联上轮
print(response2)

# 重置对话
chat_engine.reset()

坑 #8：chat_mode 选 "simple" 导致多轮关联断裂

chat_mode="simple" 不会用对话历史来改写查询。用户第二轮说"那它的缺点是什么？"，引擎不知道"它"指什么，检索结果全错。

始终使用 "condense_plus_context"（先用对话历史改写问题再检索）或 "context"（直接检索 + 传入对话历史）。生产环境推荐前者。

高级模式 —— 生产级优化

5.1 Ingestion Pipeline —— 数据摄取管线

PYTHON
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.extractors import (
    TitleExtractor,
    QuestionsAnsweredExtractor,
    SummaryExtractor,
)

pipeline = IngestionPipeline(
    transformations=[
        SentenceSplitter(chunk_size=512, chunk_overlap=64),
        TitleExtractor(),                    # 自动提取标题
        QuestionsAnsweredExtractor(),        # 为每个 chunk 生成"可能被问的问题"
        SummaryExtractor(),                  # 为每个 chunk 生成摘要
        OpenAIEmbedding(),
    ]
)

# 运行管线
nodes = pipeline.run(documents=documents)
print(f"生成了 {len(nodes)} 个节点")
💡
QuestionsAnsweredExtractor 是 RAG 质量的秘密武器
它用 LLM 为每个 chunk 生成几个"可能被问到的问题"，并在索引中同时索引这些问题。当用户提问时，问题比原文更容易匹配到相关 chunk，显著提升检索召回率。

坑 #9：IngestionPipeline 中 QuestionsAnsweredExtractor 开销巨大

每个 chunk 都要调一次 LLM 生成问题，1000 个 chunk 就是 1000 次调用。如果不加缓存，每次重建索引都是一笔不小的 API 费用。

对已有的索引做增量更新而非全量重建。使用 IngestionCache 缓存已处理的节点。对于不需要高精度的场景，跳过 QuestionExtractor，直接用合理的 chunk 大小 + overlap。

LlamaIndex vs LangChain：什么时候用谁

✅ 优先 LlamaIndex

  • 构建文档问答 / 知识库系统
  • 需要多种数据源接入（PDF / DB / API / Notion...）
  • 需要高级索引策略（摘要索引、知识图谱索引）
  • 对检索质量有高要求（reranker、混合检索）
  • 希望快速出原型：5 行代码跑通 RAG

✅ 优先 LangChain

  • 复杂的 Agent 工作流（条件分支、循环）
  • 需要 Human-in-the-loop
  • 多模型编排、链式调用
  • 需要灵活的 Prompt 管理
  • 已有 LangChain 技术栈的团队
🚀
最佳组合：LlamaIndex + LangChain 协同工作 实际生产中，很多团队选择"双剑合璧"：
1. 用 LlamaIndex 做数据接入、索引构建、高质量 RAG 检索
2. 用 LangChain / LangGraph 做 Agent 编排、工作流控制、工具调用
3. 用 LangSmith 做全链路 Tracing 和监控

三者不是竞争关系，是互补关系。选择对场景最合适的工具，而不是绑定单一框架。
