全景概览

什么是 AI 智能体（Agent）？

AI 智能体是一种能够自主感知环境、规划任务、调用工具并执行动作的软件系统。与传统聊天机器人不同，智能体具备「手」和「脚」——它能操作文件系统、执行命令、浏览网页、编写代码，并在多轮交互中持续优化行为。

2025-2026 年，AI 智能体从概念走向工程落地，形成了四条截然不同的技术路线。本教程将深入剖析其中最具代表性的四款产品。

OpenClaw

规范可控的编程专家

以 Plan Mode 为核心 UX，先出完整计划经用户批准后才动手，强调用户控制与代码安全。

四条技术路线的核心分歧

四大智能体的设计原点差异：覆盖广度 vs 自我进化 vs 用户控制 vs 工程效率

OpenClaw
OpenClaw

定位：多平台消息路由器 + 工具执行引擎

OpenClaw 由 Peter Steinberger 发起，是社区驱动、生态最成熟的智能体框架。它的北极星指标是「覆盖多少平台、能调用多少工具」，目前支持 50+ 通讯平台原生接入，插件数量最多。

核心架构：Gateway 统一调度 + 多 Runtime 并行

OpenClaw 把 4 个 Runtime 当作一等公民，由 Gateway 进程统一接收所有渠道消息并路由到对应 Session：

  • PI (Process Intelligence) —— 本地进程级工具执行
  • Codex CLI —— OpenAI 代码代理委托
  • Claude CLI —— Anthropic 代码代理委托
  • ACP (Agent Control Protocol) —— 标准化代理通信协议

关键设计特点

  • 每个 Session 串行执行：防止竞争条件，确保消息处理有序
  • Multi-agent Routing：不同 Agent 可处理不同渠道，实现负载均衡
  • LLM 热切换：切换底层模型不影响其他组件运行
  • Spawn Subagent：隔离会话，零上下文成本创建子代理

Skill 系统：简单但够用

OpenClaw 的 Skill 采用多级目录结构（workspace / project / personal / bundled），通过 SKILL.md 自描述，支持自动发现和自动注入。它的优势在于热加载——修改 Skill 文件立即生效，无需重启服务。

目录结构
skills/
├── workspace/     # 工作区级
├── project/       # 项目级
├── personal/      # 个人级
└── bundled/       # 内置级
    └── git-ops/
        └── SKILL.md   # 自描述文件，自动发现注入
Hermes Agent
Hermes

定位：会自我进化的数字员工

Hermes 由 Nous Research 实验室开发，2026 年 2 月上线即获得 66K+ GitHub Stars。它的北极星指标是「用户使用 3 个月后，Agent 能省多少重复指令」——强调自我进化与技能沉淀。

核心架构：单循环 + 自我改进闭环

Hermes 的核心是一个原生 Agent Loop + 内置学习闭环，形成「越用越聪明」的正反馈：

ReAct 循环与自省机制

Hermes 的核心是 AIAgent 类（run_agent.py），实现 ReAct 循环（推理 → 行动 → 观察），并具备独特的自省机制——每步评估工具输出是否合理，不合理则自动重试或调整策略。

Skill 系统：活的、自进化的

Hermes 的 Skill 是「活的」——完成 5+ 次工具调用的复杂任务后，系统会自动提炼为 Skill（Markdown + YAML frontmatter），实现 Progressive Disclosure（只加载 frontmatter 判断是否触发，不污染上下文）。

关键区别：OpenClaw 的「智能」来自外部 LLM + 人工编写的 Skill；Hermes 的「智能」来自内生进化——使用越多，自动沉淀的技能越多，重复指令越少。

Profile 隔离与记忆管理

Hermes 支持多 Profile 彻底隔离，每个 Profile 拥有独立的记忆空间、技能库和模型配置。记忆持续演化，但过程是黑盒的——这是它与 OpenClaw 的共同局限，也是与 PilotDeck 等新兴白盒记忆框架的分水岭。

Claude Code
Claude Code

定位：规范可控的编程专家

Claude Code 是 Anthropic 官方推出的智能体，以 Plan Mode 为标志性 UX——先输出完整计划，经用户批准后才动手执行。它强调用户控制、代码安全与权限管理，是 IDE 集成场景下的首选。

核心架构：QueryEngine + 三层权限检查

Claude Code 的核心是 QueryEngine（2026 年 3 月泄露的技术细节），启动时先加载 CLAUDE.md + Auto Memory，然后执行三层权限检查，可选子 Agent 委托：

  • Explore Agent：使用 Haiku 模型，只读搜索代码，成本低
  • Plan Agent：只读生成方案，不修改文件
  • General-purpose Agent：全能型，可读写执行

Plan Mode：用户控制的终极体现

Claude Code 的 Plan Mode 是其区别于其他智能体的核心 UX。用户输入需求后，系统首先生成详细的执行计划（包括修改哪些文件、执行哪些命令），用户确认无误后才进入执行阶段。这种设计避免了「AI 乱改代码」的风险，特别适合生产环境。

适用场景：对代码安全要求高的团队、需要审计追踪的企业环境、初学者学习编程时的安全网。

Skill 系统：规范 + 动态注入

Claude Code 的 Skill 系统最「规范」——采用开放标准，支持 Live Change Detection（修改 Skill 文件立即生效）和 Dynamic Context Injection（Skill 中写 !git diff HEAD 即可自动注入当前 diff，不是静态文本而是运行时动态内容）。

Codex CLI
Codex CLI

定位：工程化批处理利器

Codex CLI 是 OpenAI 推出的开源智能体，采用 Rust 编写，核心是 Agent Loop Harness。它显式区分「turn」（单次用户输入到响应的循环）与「thread」（完整对话线程），追求批处理效率与零数据留存（ZDR）合规。

核心架构：Agent Loop Harness + Rust 性能

Codex CLI 的架构围绕「批处理高效」设计，Rust 带来的内存安全与高性能使其在大规模代码迁移、批量测试生成等场景下表现卓越。

按需启动 vs 长驻守护

与 OpenClaw / Hermes 的 Gateway 长驻守护进程不同，Codex CLI 采用「按需启动」短进程模型——每次交互独立启动 CLI，适合 IDE 集成但每次需要冷启动开销。这种设计牺牲了一点延迟，换来了更好的资源隔离与安全性。

Skill 系统：最工程化的实现

Codex 的 Skill 系统是最「工程化」的——独立的 Rust crate（codex-core-skills），包含 8+ 模块文件（loader / manager / injection / render / model / config_rules / system / remote）。loader.rs 解析 YAML frontmatter，manager.rs 管理发现注册，injection.rs 负责 prompt 注入，模块化程度最高。

Rust Crate 结构
codex-core-skills/
├── loader.rs      # 解析 YAML frontmatter
├── manager.rs     # 发现、注册、生命周期
├── injection.rs   # Prompt 注入逻辑
├── render.rs      # 模板渲染
├── model.rs       # 数据模型定义
├── config_rules.rs # 配置规则
├── system.rs      # 系统级 Skill
└── remote.rs      # 远程 Skill 同步

快速选型指南

  • 要接入多个通讯平台 → OpenClaw（50+ 平台原生支持，插件最多）
  • 要多个独立 Agent 人格长期并存 → Hermes（Profile 隔离最彻底，记忆持续演化）
  • 主要在 IDE/终端写代码 → Claude Code / Codex CLI（前者 Claude 模型 + 多 IDE 扩展；后者开源 + OpenAI 模型）
  • 对 Token 成本敏感 → Hermes 手动 Profile 配模型；PilotDeck 智能路由自动省约 70%
  • 不想折腾，开箱即用 → Claude Code / Codex CLI（订阅/API Key 即用，无部署成本）

2026 下半年趋势

  • 记忆管理成为核心竞争维度：无记忆 → 黑盒记忆 → 项目级手动记忆 → 白盒记忆（可追溯、可编辑、可回滚）
  • 开源 vs 商业的分界线在记忆透明度：商业框架模型能力更强，开源框架记忆隔离和可审计性更灵活
  • 成本控制从手动走向自动：智能路由根据任务复杂度自动选择模型，是未来方向

写在最后

四大智能体没有绝对的优劣，只有场景的匹配。OpenClaw 适合打造「全能数字管家」，Hermes 适合培养「专属数字员工」，Claude Code 适合作为「安全编程伙伴」，Codex CLI 适合充当「批量代码工人」。理解它们的架构差异与设计哲学，才能在实际项目中做出最合适的技术选型。

延伸思考：智能体的终极形态可能是「Gateway 统一调度 + 白盒记忆 + 自动成本控制」的融合体。2026 年下半年，关注 PilotDeck 等新兴框架在白盒记忆与智能路由上的突破。
