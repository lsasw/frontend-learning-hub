0RAG 全景总览

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将外部知识检索与大语言模型生成相结合的技术架构。它通过在推理时从外部知识库中检索相关信息，将检索到的上下文注入到 Prompt 中，从而增强 LLM 的回答质量，减少幻觉，实现知识的动态更新。

为什么需要 RAG？

🧠LLM 的固有局限

知识截止日期后的事件一无所知；训练数据中未覆盖的私有领域知识无法回答；参数化知识无法实时更新。

✨RAG 的核心价值

外部知识库可实时更新无需重新训练；回答有据可查可溯源引用；私有数据不出本地保证安全。

📊vs Fine-tuning

RAG 适合事实性知识更新，成本低见效快；Fine-tuning 适合风格/格式调整。二者可互补而非互斥。

💡
一句话理解 RAG：给 LLM 配一个「开卷考试」的资料库。模型不再仅凭记忆答题，而是先查阅资料再组织答案 —— 这就是 RAG 的本质。

1RAG 基本概念与工作原理

1.1 核心定义

RAG = Retrieval（检索）+ Augmented（增强）+ Generation（生成）。它不修改 LLM 的参数，而是在推理阶段动态注入外部知识，使模型基于检索到的上下文生成回答。

📦Naive RAG

标准三步走：Index → Retrieve → Generate。结构简单，开发快，但检索质量依赖原始 Chunk 和 Embedding 质量，容易召回噪声。

入门级

🔧Advanced RAG

三阶段优化：检索前（Query 改写）、检索中（混合检索）、检索后（Rerank + 压缩）。系统性地提升每个环节质量。

生产级

🧩Modular RAG

模块化编排：检索可跳过、可迭代（Self-RAG）、可路由（Router RAG）。支持多轮检索、反思纠错等复杂模式。

研究级
🔬
RAG 发展趋势：从 Naive → Advanced → Modular，核心趋势是从「一次性检索」走向「自适应检索」。Self-RAG 让模型自行判断是否需要检索，CRAG 在检索质量差时触发网络搜索兜底，这些都在模糊 RAG 与 Agent 的边界。

3嵌入模型改进

Embedding 模型是 RAG 的「理解引擎」—— 它将文本映射到向量空间，直接决定了检索的质量上限。模型不行，后面的检索、重排再怎么优化都是亡羊补牢。

🔢
选择更强的 Embedding 模型
基础 · 影响大 ▼

核心思路：在 MTEB 榜单上选择适合任务的模型，平衡效果、维度、速度和成本。

  • 闭源 API：OpenAI text-embedding-3-large（3072d）、Voyage AI voyage-3（1024d）、Cohere embed-v3（1024d）
  • 开源中文：BGE-M3（多语言/多功能）、bge-large-zh-v1.5、GTE-large-zh、E5-mistral-7b
  • 选择维度：语言（中/英/多语言）、维度（768/1024/1536/3072）、场景（通用/代码/金融）、是否支持稀疏向量
  • 验证方法：构建领域 QA 测试集，计算 Recall@K（K=1,3,5,10）对比模型表现
💡
实践建议：中文场景首推 BGE-M3（同时输出 dense + sparse + multi-vector），它支持最长 8192 tokens 输入，多语言效果接近商业模型。英文场景推荐 voyage-3 或 text-embedding-3-large。
🎯
领域微调（Domain Fine-tuning）
进阶 · 影响大 ▼

核心思路：通用 Embedding 模型在垂直领域（医疗、法律、金融）可能表现不佳，使用领域数据微调可以让向量空间更贴合业务语义。

  • 方法一：对比学习微调 — 构建 (Query, 正例, 负例) 三元组，拉近正例推远负例
  • 方法二：LLM2Vec — 用 LLM 作为 Encoder，通过适配层转化为句向量模型
  • 数据构建：从业务 FAQ / 人工标注 / LLM 合成 获取训练对
  • 框架推荐：FlagEmbedding（BGE 官方微调框架）、Sentence-Transformers

⚡
Matryoshka Representation（嵌套维度）
高级 · 省存储 ▼

核心思路：Matryoshka Embedding 在训练时让向量的前 N 维就能保持较好区分度。这样可以在存储时截断到更低维度，牺牲少量精度换取大幅存储和检索加速。

  • 原理：类似俄罗斯套娃，外层维度编码粗粒度语义，内层逐步细化
  • 效果：3072d → 768d 截断后，检索精度仅下降 2-3%，但存储减少 75%
  • 适用场景：大规模向量库（亿级以上）、边缘设备部署
  • 支持模型：OpenAI text-embedding-3 系列、Nomic-Embed-Text
🔀
指令化 Embedding（Instruction-Tuned）
高级 · 提升精度 ▼

核心思路：在 Embedding 输入前添加任务指令前缀，让同一个模型适应不同检索场景（如问答、相似度匹配、聚类）。

python 
# E5 / BGE 指令前缀
query = "Represent this sentence for retrieving relevant documents: " + query
passage = "Represent this sentence for searching relevant passages: " + passage

# BGE-M3 不需要指令前缀，但 BGE-large 需要
# E5 模型也需要 "query: " 和 "passage: " 前缀

embedding = model.encode([query, passage])

效果：正确使用指令前缀可以提升 5-15% 的检索准确率，但用错前缀（如对称任务用了非对称前缀）反而会降低效果。

✅
选型建议：不要上来就微调。先用 MTEB 榜单上的强基座模型跑通 Pipeline，构建领域评测集量化基线效果。只有在通用模型确实不达标时，才值得投入微调成本。模型选择 > 数据质量 > 微调方法。

4Chunk 分割策略

Chunk 分割是 Indexing 阶段最关键的决策之一。分块太大 → 噪声多、语义稀释；分块太小 → 上下文断裂、信息碎片化。好的 Chunk 策略要在「语义完整性」和「检索精度」之间找平衡。

4.2 六大 Chunk 策略详解

📏
固定长度分割（Fixed-Size Chunking）
最简单 ▼

原理：按固定字符数（如 500/1000 chars）或 Token 数（如 256/512 tokens）切分。

  • 优点：实现极简，速度快，不需要理解文档结构
  • 缺点：可能在句子/段落中间截断，破坏语义完整性
  • 适用：快速原型验证、格式统一的短文本
  • 典型参数：chunk_size=500, chunk_overlap=50
🔄
递归字符分割（Recursive Character Splitting）
LangChain 默认 ▼

原理：按分隔符优先级递归切分：段落（\n\n）→ 句子（\n）→ 空格 → 字符。优先在自然边界处切割。

  • 优点：兼顾语义边界和长度限制，是 LangChain 的默认策略
  • 缺点：对非结构化文本（表格、代码）效果一般
  • 适用：大多数文本文档（文章、报告、文档）
  • 典型参数：chunk_size=1000, chunk_overlap=200, separators=["\n\n","\n"," ",""]
🧠
语义分割（Semantic Chunking）
效果最好 ▼

原理：先按句子切分，计算相邻句子的 Embedding 相似度。当相似度骤降（语义转折点）时切分，保证每个 Chunk 内语义连贯。

  • 优点：Chunk 边界对齐语义变化，召回质量显著提升
  • 缺点：需要额外 Embedding 计算，处理速度慢，Chunk 大小不可控
  • 适用：对召回质量要求高的场景（问答系统、知识检索）
  • 工具：LangChain SemanticChunker、LlamaIndex SemanticSplitter
💡
加速技巧：语义分割的瓶颈在 Embedding。可以用轻量模型（如 MiniLM）做分割判断，重模型做最终索引，实现两阶段加速。
📊
结构感知分割（Structure-Aware）
文档专用 ▼

原理：利用文档结构（Markdown 标题、HTML 标签、PDF 段落）进行层级化切分。先按 H1/H2/H3 分节，再在节内按段落切分。

  • 优点：保留文档层级信息，可生成「父子 Chunk」做摘要+细节两级检索
  • 缺点：需要文档有清晰结构，纯文本无效
  • 适用：技术文档、API 文档、法律合同、学术论文
  • 典型工具：MarkdownHeaderTextSplitter、HTMLSectionSplitter
🔗
父子分块（Parent-Child / Small-to-Big）
进阶推荐 ▼

原理：检索时用小 Chunk（精确匹配），生成时用大 Chunk（完整上下文）。小 Chunk 是大 Chunk 的子集，通过 ID 关联。

  • 优点：兼顾检索精度和生成上下文完整性，解决「小 Chunk 上下文不足」问题
  • 缺点：需要额外存储父 Chunk，索引结构更复杂
  • 适用：长文档问答、需要完整段落的场景
  • 实现：LlamaIndex AutoMergingRetriever、LangChain ParentDocumentRetriever
文档段落（Parent）
→
小 Chunk（Child）→ 索引
→
检索 Child
→
返回 Parent 给 LLM
🤖
LLM 辅助分割（Agentic Chunking）
前沿 ▼

原理：让 LLM 阅读文档，自主判断在哪里切分、如何命题每个 Chunk。甚至可以生成 Chunk 摘要和提问预设。

  • 优点：语义理解最深，Chunk 质量最高
  • 缺点：成本极高（大量 LLM 调用）、速度慢、不可复现
  • 适用：高价值文档（核心知识库）、小规模精标数据
  • 趋势：Proposition-based Chunking — 让 LLM 将每段文档拆解为独立命题（proposition），每个命题自包含完整语义
⚠️
参数调优经验：chunk_size 没有万能值。经验起点是 256-512 tokens（约 400-800 中文字），配合 10-20% 的 overlap。关键是在你的领域评测集上做消融实验：固定其他变量，只改 chunk_size，看 Recall@5 曲线的拐点。

5检索优化策略

检索是 RAG 的「瓶颈环节」—— 检索不到的，LLM 再强也生成不出来。优化检索的目标是提高 Recall（召回率），让相关 Chunk 尽可能进入候选集。

5.1 检索方式对比

检索方式原理优势劣势适用场景
稠密向量检索

5.2 Query 侧优化（检索前）

🔄Query 改写（Rewriting）

用户提问可能模糊、口语化或含指代。用 LLM 改写为更利于检索的形式。

  • 指代消解：「它的价格」→「iPhone 15 Pro 的价格」
  • 口语化转正式：「咋算 WACC」→「WACC 加权平均资本成本计算方法」
  • 子问题拆解：复杂问题拆为多个可检索子问题

🎯HyDE（假设性文档嵌入）

先让 LLM 生成一个「假设答案」，用这个答案的 Embedding 去检索，而不是用原始 Query。

  • Query 和 Document 在向量空间分布不同（不对称检索）
  • HyDE 让检索从 Query→Doc 变为 Doc→Doc（对称检索）
  • 适合短 Query 长答案的场景

🔀多路召回（Multi-Query）

让 LLM 从不同角度生成多个 Query 变体，分别检索后合并去重。

  • 「WACC 怎么算」→「WACC 公式」「加权平均资本成本计算」「股权成本和债务成本加权」
  • 用 RRF（Reciprocal Rank Fusion）合并排名
  • 显著提升 Recall，代价是多倍检索调用

🗺️Step-back Prompting

让 LLM 先把具体问题抽象为更高层的问题，检索背景知识后再回答原问题。

  • 原问题：「2024 年 Apple 的 WACC 是多少」
  • 退后问题：「如何计算科技公司的 WACC」
  • 两者都检索，背景知识 + 具体数据组合

5.3 混合检索详解

python 
# RRF (Reciprocal Rank Fusion) 融合算法
def rrf_fuse(ranked_lists, k=60):
    """k 是平滑常数，通常 60 是经验最优值"""
    scores = {}
    for ranked_list in ranked_lists:
        for rank, doc in enumerate(ranked_list, 1):
            scores[doc] = scores.get(doc, 0) + 1 / (k + rank)
    # 按融合分数降序排列
    return sorted(scores, key=scores.get, reverse=True)

# 使用示例：dense_results 和 sparse_results 各自 Top-10
fused = rrf_fuse([dense_results, sparse_results], k=60)
final_candidates = fused[:5]  # 取融合后 Top-5
💡
为什么 RRF 有效：RRF 不依赖原始分数（Dense 和 Sparse 的分数量纲不同），只看排名。一个文档在两个列表中都靠前，它的融合分数就高。k=60 的平滑常数避免 Top-1 过度主导。

6重排序机制（Reranking）

检索阶段追求 Recall（不漏），召回 Top-K（如 K=20-50）会包含噪声。Rerank 阶段追求 Precision（不错），用更强的模型从候选集中精选 Top-N（如 N=3-5）给 LLM。

6.1 为什么需要 Rerank？

6.2 Bi-Encoder vs Cross-Encoder

维度Bi-Encoder（双塔）Cross-Encoder（交叉编码）
架构Query 和 Doc 各自编码，最后计算相似度Query 和 Doc 拼接后联合编码
交互层无（只在最后做点积/Cosine）有（Transformer 各层全交互）
速度极快 Doc 可预编码慢 每对都要实时计算
精度中等高 深层语义匹配
用途第一阶段检索（召回 Top-K）第二阶段精排（精选 Top-N）
典型模型BGE / E5 / OpenAI EmbeddingBGE-Reranker / Cohere Rerank / ms-marco-MiniLM
✅
两阶段检索范式：Bi-Encoder 做初筛（快但粗），Cross-Encoder 做精排（慢但准）。这是当前 RAG 系统的标准配置，效果比单独使用任一方案都好。先召回 50 条，再 Rerank 到 5 条，是常见的参数配置。

6.3 Rerank 方法演进

Cross-Encoder Rerank
用 Cross-Encoder 逐对打分重排。经典方案，效果好但速度慢，适合候选集 ≤100 的场景。BGE-Reranker-Large 是中文场景首选。
ColBERT（延迟交互）
Token 级别的 Bi-Encoder。Doc 可预编码（保留所有 token 向量），Query 来了做 token-to-token 的 MaxSim 聚合。速度接近 Bi-Encoder，效果接近 Cross-Encoder。
LLM-as-Reranker
直接用 LLM 做 Rerank：给 LLM 看候选列表，让它输出排名。如 RankGPT（滑动窗口重排）、Cohere Rerank-3。效果好但成本高，适合候选集 ≤20 的小规模精排。
Listwise Rerank
不再逐对打分，而是让模型同时看所有候选，输出一个整体排序。比 Pointwise 方式更准，因为能感知候选间的相对优劣。
python 
# BGE-Reranker 使用示例
from FlagEmbedding import FlagReranker

reranker = FlagReranker('BAAI/bge-reranker-large',
                        use_fp16=True)

query = "如何计算 WACC？"
candidates = [
    "WACC 计算公式的完整推导过程",
    "资本成本的构成要素介绍",
    "DCF 估值模型的基本原理",
]

pairs = [[query, c] for c in candidates]
scores = reranker.compute_score(pairs)

# 按分数降序排列，取 Top-3
ranked = sorted(zip(candidates, scores),
               key=lambda x: x[1], reverse=True)[:3]

8最佳实践与评估

8.1 RAG 评估框架

📊检索质量评估

  • Recall@K：Top-K 中包含正确答案的比例（最重要）
  • Precision@K：Top-K 中相关文档的占比
  • MRR：正确答案的倒数排名平均
  • NDCG：考虑位置加权的排名质量

📝生成质量评估

  • Faithfulness：回答是否忠于检索内容（防幻觉）
  • Answer Relevance：回答与问题的相关度
  • Context Relevance：检索上下文与问题的相关度
  • 工具：RAGAS / TruLens / DeepEval
python 
# RAGAS 评估示例
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_precision,
    context_recall,
)

result = evaluate(
    dataset=test_dataset,
    metrics=[faithfulness, answer_relevancy,
             context_precision, context_recall],
)
print(result)
# {'faithfulness': 0.92, 'answer_relevancy': 0.88,
#  'context_precision': 0.85, 'context_recall': 0.79}

8.2 工程化最佳实践清单

🏗️架构层面

  • 建立评测集再开发 —— 没有度量就没有优化
  • 离线索引与在线检索解耦
  • 向量库支持增量更新而非全量重建
  • 检索结果缓存（相同 Query 命中缓存）
  • 多级索引：摘要索引 + 细节索引

🔧调优层面

  • Chunk 大小在评测集上调参，不要拍脑袋
  • Top-K（召回）和 Top-N（Rerank）分开调
  • 混合检索权重按领域动态调整
  • Prompt 中明确要求引用来源
  • 对低置信度回答触发兜底策略

8.3 常见反模式

反模式问题正确做法
盲目增大 Top-K更多噪声进入 Prompt，LLM 被干扰先扩大召回，再用 Rerank 收窄
Chunk 过大语义稀释，向量区分度下降256-512 tokens 起步，评测集调参
只看生成不看检索生成质量差但不知是检索还是 LLM 的问题分层评估：先测 Recall@K 再测 Faithfulness
过早微调投入大但效果可能不如换个强模型先穷尽模型选型 + Prompt 优化
忽视元数据无法按时间/来源/类型过滤为 Chunk 打上 metadata 标签，支持过滤检索
无引用溯源用户无法验证回答可靠性Prompt 要求标注来源 Chunk ID

RAG 的未来方向：1）Agentic RAG — 将 RAG 嵌入 Agent 循环，模型自主决定何时检索、检索什么；2）Self-RAG — 模型自评估检索质量，低质量时触发重新检索或网络兜底；3）Long-context RAG — 利用百万 token 上下文窗口，减少分块需求。RAG 正从「固定 Pipeline」走向「自适应智能体」。
