1 微调技术全景概览

大语言模型（LLM）的微调（Fine-tuning）是指在预训练基座模型的基础上，使用特定领域或任务的数据进行二次训练，使模型更好地适应下游任务。随着模型参数规模从数十亿扩展到数千亿，微调策略也在不断演进。

2 全量微调（Full Fine-tuning）

原理简述

全量微调是指在预训练模型的所有参数上继续训练。每一轮反向传播都会更新模型中的每一个权重，使模型完全适配目标领域的数据分布。

⚠️ 灾难性遗忘（Catastrophic Forgetting）：全量微调可能导致模型遗忘预训练阶段学到的通用知识。学习率设置和训练步数需要谨慎控制。

典型配置示例

Python - Transformers 
# 全量微调 Qwen2-7B 示例
from transformers import AutoModelForCausalLM, TrainingArguments, Trainer
from peft import get_peft_model # 这里演示全量微调，不使用 PEFT

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2-7B",
    torch_dtype="auto",
    device_map="auto"
)

training_args = TrainingArguments(
    output_dir="./qwen2-ft",
    per_device_train_batch_size=1,       # 全量微调 batch 很小
    gradient_accumulation_steps=16,        # 梯度累积模拟大 batch
    learning_rate=2e-5,                       # 1e-5 ~ 5e-5 常用
    warmup_ratio=0.03,
    num_train_epochs=3,
    logging_steps=10,
    save_strategy="epoch",
    bf16=True,                            # BF16 混合精度节省显存
    gradient_checkpointing=True,          # 梯度检查点，以时间换空间
    deepspeed="ds_config_zero3.json"      # DeepSpeed ZeRO-3
)

trainer = Trainer(model=model, args=training_args, train_dataset=dataset)
trainer.train()

✅ 优点

  • 效果最好：所有参数参与调整，理论上能达到该任务的最优性能
  • 灵活度高：不受额外结构限制，模型可以完全自由地学习
  • 简单直接：标准训练流程，无需特殊技术改造
  • 生态成熟：DeepSpeed ZeRO、Megatron-LM 等工具链完善

❌ 缺点

  • 资源开销巨大：需要保存完整模型副本、优化器状态、梯度
  • 灾难性遗忘：可能丢失预训练学到的通用能力
  • 存储成本高：每个下游任务都需要一份完整模型
  • 部署不灵活：多任务场景下需要维护多个全量模型
✅ 适用场景：拥有充足 GPU 资源（A100 80G × 4+），追求极致性能的垂直领域特化任务，如医疗/法律/金融专有模型。训练数据量在 100K+ 条时效果最稳定。

3 参数高效微调（PEFT）

PEFT（Parameter-Efficient Fine-Tuning）的核心理念是：冻结原始模型参数，仅训练少量新增参数，从而大幅降低计算和存储开销。主流方法包括 LoRA、QLoRA、Adapter、Prefix-tuning 等。

3.1 LoRA（Low-Rank Adaptation）

核心原理

LoRA 的核心思想基于一个关键洞察：模型适配过程中的权重更新矩阵 ΔW 具有低"内在秩"（Intrinsic Rank）。因此可以用两个低秩矩阵 A 和 B 的乘积来近似 ΔW，而不需要更新原始权重 W。

h = W₀x + ΔWx = W₀x + BAx

其中 W₀ ∈ ℝᵈˣᵏ（冻结），B ∈ ℝᵈˣʳ，A ∈ ℝʳˣᵏ，秩 r ≪ min(d,k)

🔑 关键参数 rank (r)：常用的 r 取值 8、16、32、64。r 越大表达能力越强但参数也越多。r=16 在大多数场景下是性价比最优的选择。

代码配置示例

Python - PEFT 
from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2-7B",
    torch_dtype="auto",
    device_map="auto"
)

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,         # 因果语言模型
    r=16,                                # LoRA 秩（核心超参）
    lora_alpha=32,                         # 缩放因子，一般取 2×r
    lora_dropout=0.05,                       # Dropout 防止过拟合
    target_modules=[                      # 应用 LoRA 的模块
        "q_proj", "k_proj", "v_proj",  # Attention QKV
        "o_proj",                         # Attention 输出投影
        "gate_proj", "up_proj",       # FFN 层
        "down_proj"                       # FFN 输出
    ],
    bias="none",                        # 不训练 bias
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# 输出: trainable params: 33,554,432 || all params: 7,077,441,536 || trainable%: 0.4741

关键超参数调优指南

参数推荐值作用调优建议
r8~16低秩矩阵的秩简单任务 r=8 足够，复杂任务可升至 32~64
lora_alpha16~32LoRA 输出的缩放系数通常设为 r 的 2 倍，调大 = 放大 LoRA 贡献
lora_dropout0.05~0.1防止过拟合数据少时适当调大；数据多可设为 0
target_modulesQKV+FFN应用 LoRA 的模块覆盖越多效果越好但参数也越多
lr1e-4~5e-4学习率通常比全量微调高一个数量级

✅ LoRA 优点

  • 可训练参数减少 99%+，显存大幅下降
  • 训练速度提升 2~3 倍
  • 推理时可以合并到原始权重，无推理延迟
  • 支持多 LoRA 热切换（切换 adapter 即可）
  • 存储友好：每个 LoRA adapter 仅几 MB

❌ LoRA 缺点

  • 表达能力受限于 rank，复杂任务可能欠拟合
  • 目标模块选择需要领域经验
  • 与全量微调相比有轻微性能差距（通常 < 2%）
  • 不适合需要大规模知识注入的场景
🎯 最佳实践：先使用 r=16, alpha=32 进行实验；若效果不足再增加 r 到 32~64。始终对 QKV 和 FFN 层应用 LoRA。在 LoRA 训练完成后，可通过 model.merge_and_unload() 合并权重以消除推理延迟。

3.2 QLoRA（Quantized LoRA）

QLoRA 是 LoRA 的量化版本，由华盛顿大学在 2023 年提出。它通过 4-bit NormalFloat (NF4) 量化 + 双重量化 + 分页优化器 三大技术，使得在单张消费级 GPU 上微调 65B 模型成为可能。

核心技术组件

原始模型 FP16 → NF4 量化 → 双重量化 → LoRA 训练 → 分页优化器 → 合并/部署

① NF4 量化 (4-bit NormalFloat)

一种针对正态分布权重优化的 4-bit 数据类型。假设预训练权重服从 N(0,σ²)，将值域按分位数分成 16 个区间，每个区间映射到一个 4-bit 值。

② 双重量化 (Double Quantization)

不仅量化模型权重，还对量化过程中产生的缩放因子（scaling factors）再次量化，进一步节省约 0.5 bit/参数的存储。

Python - QLoRA 
from transformers import BitsAndBytesConfig
from peft import LoraConfig, get_peft_model

# 4-bit 量化配置
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                          # 启用 4-bit 量化
    bnb_4bit_compute_dtype=torch.bfloat16,    # 计算时使用的精度
    bnb_4bit_quant_type="nf4",                # NormalFloat4
    bnb_4bit_use_double_quant=True,            # 双重量化
)

model = AutoModelForCausalLM.from_pretrained(
    "mistralai/Mixtral-8x7B-v0.1",
    quantization_config=bnb_config,            # 应用量化
    device_map="auto"
)

# LoRA 配置 (与标准 LoRA 相同)
lora_config = LoraConfig(
    r=16, lora_alpha=32,
    target_modules=["q_proj","k_proj","v_proj","o_proj"],
    lora_dropout=0.05, bias="none",
    task_type="CAUSAL_LM"
)

model = get_peft_model(model, lora_config)
# Mixtral 8×7B 在单张 48GB GPU 上即可微调！

3.3 Adapter 方法

Adapter 方法在 Transformer 的每一层中插入小型瓶颈网络（Bottleneck），训练时只更新这些 Adapter 的参数。这是最早被广泛研究的 PEFT 方法之一（Houlsby et al., 2019）。

Adapter 类型对比

类型插入位置结构特点代表工作
Serial Adapter每层 Attention + FFN 之后Down-Proj → GeLU → Up-Proj经典方案，结构简单Houlsby 2019
Parallel Adapter与 Attention/FFN 并行同上，输出与主路径相加减少推理延迟He et al. 2021
Compacter同上Kronecker 积分解参数量进一步减少参数Mahabadi 2021
LoRA替换 Attention QKV 投影低秩矩阵 A·B可合并无延迟Hu et al. 2021
📌 现状：Adapter 曾是 PEFT 的主流方案，但由于 LoRA 可以合并权重消除推理延迟，目前 LoRA 已成为事实标准。Adapter 在需要严格模块化多任务切换的场景中仍有价值。

4 指令微调（Instruction Tuning）

核心思想

指令微调的核心在于训练数据格式的改变：将自然语言任务描述（指令）+ 输入映射到期望输出，使模型学会"理解并执行指令"，而非简单的补全文本。这是让基座模型变为"助手"的关键一步。

数据格式对比

❌ 传统预训练格式

北京是中国的首都。上海是中国的经济中心。深圳是中国的...

模型学到的：预测下一个 token

✅ 指令微调格式

### Instruction:
请解释"光合作用"的过程，并用一个比喻帮助理解。

### Response:
光合作用是植物利用光能将二氧化碳和水转化为...

模型学到的：理解指令 → 生成有帮助的回答

常见数据格式

JSON - Alpaca 格式 
{
  "instruction": "将以下中文翻译为英文",
  "input": "人工智能正在改变世界",
  "output": "Artificial intelligence is changing the world"
}
JSON - ShareGPT / ChatML 格式 
{
  "conversations": [
    {"role": "user", "content": "什么是量子计算？"},
    {"role": "assistant", "content": "量子计算是利用量子力学原理..."}
  ]
}

def create_instruction_prompt(example):
    # 构建 ChatML 格式的 prompt
    messages = [
        {"role": "user", "content": example["instruction"]},
        {"role": "assistant", "content": example["output"]}
    ]
    text = tokenizer.apply_chat_template(messages, tokenize=False)
    return {"text": text}

# DataCollator: 仅对 assistant 部分计算 loss
class DataCollatorForCompletionOnlyLM:
    def __call__(self, examples):
        # 将 user 部分的 labels 设为 -100（忽略）
        batch = tokenizer(examples, padding=True, ...)
        labels = batch["input_ids"].clone()
        for i, assistant_start in enumerate(assistant_positions):
            labels[i, :assistant_start] = -100  # 忽略 user 部分
        batch["labels"] = labels
        return batch
📊 数据质量优于数量：研究表明，精选的 1K 高质量指令数据的微调效果往往优于 50K+ 低质量数据。数据多样性（覆盖不同任务类型）远比单纯的数量重要。

5 RLHF（基于人类反馈的强化学习）

RLHF（Reinforcement Learning from Human Feedback）是让大模型对齐人类偏好的关键技术。ChatGPT 的成功很大程度上归功于 RLHF。其核心流程分为三个阶段：

三阶段流程

DPO：RLHF 的简化替代方案

Direct Preference Optimization (DPO) 省去了训练独立奖励模型的步骤，直接从人类偏好数据中优化策略。数学等价于 RLHF 但实现更简单、训练更稳定。

Python - DPO Trainer 
from trl import DPOTrainer, DPOConfig

# DPO 数据格式: 每条包含 prompt, chosen (更好回答), rejected (更差回答)
train_dataset = [
    {
        "prompt": "解释什么是机器学习",
        "chosen": "机器学习是人工智能的一个分支...",    # 人类更喜欢的回答
        "rejected": "机器学习就是让电脑学习东西。"      # 人类不喜欢的回答
    },
    # ...更多偏好对
]

dpo_config = DPOConfig(
    output_dir="./dpo-model",
    per_device_train_batch_size=1,
    gradient_accumulation_steps=8,
    learning_rate=5e-6,                    # DPO 学习率通常较低
    beta=0.1,                            # KL 散度约束系数
    num_train_epochs=1,
    max_prompt_length=512,
    max_length=1024,
    logging_steps=10,
)

trainer = DPOTrainer(
    model=model,                                       # SFT 后的模型
    ref_model=ref_model,                               # 参考模型 (通常是 SFT 模型本身)
    args=dpo_config,
    train_dataset=train_dataset,
    tokenizer=tokenizer,
)

trainer.train()
# DPO 训练不需要独立的奖励模型，直接优化偏好

8 实践建议与常见问题

微调实战 Checklist

🔍 数据准备

  • 数据清洗：去除重复、低质量、格式错误样本
  • 多样性检查：确保覆盖预期场景的全部类型
  • 长度分布分析：避免过长样本占用过多显存
  • 数据量建议：指令微调 1K~10K，领域微调 10K~100K

⚙️ 训练配置

  • 先小规模实验（100 条数据，1 epoch）验证流程
  • 学习率从 2e-5（全量）或 2e-4（LoRA）起步
  • 使用 warmup + cosine decay 学习率调度
  • BF16 混合精度 + 梯度检查点节省显存

📊 评估验证

  • 保留 10%~20% 数据作为验证集
  • 除 loss 外，添加任务特定指标（BLEU/ROUGE/准确率）
  • 人工抽检：尤其关注安全性和幻觉问题
  • 对比基座模型 baseline 确认微调收益

常见问题 FAQ

Q: LoRA 的 rank 设多大合适？

A: 一般来说，简单任务（如文本分类、情感分析）r=8 即可；中等复杂度任务（如指令微调）r=16 是性价比较高的选择；复杂任务（如代码生成）可以尝试 r=32~64。建议先用 r=8 跑通流程，再根据验证集效果适当调大。

Q: 什么时候用全量微调，什么时候用 LoRA？

A: 经验法则：如果 GPU 显存 ≥ 80GB 且数据量 ≥ 50K 条，可以尝试全量微调；否则首选 LoRA。全量微调主要优势在于极限性能，但在大多数场景下，LoRA 的效果差距不超过 2%，而成本只有 1/10 甚至更低。

Q: RAG 和微调应该如何选择？

A: RAG 适合需要实时知识更新、引用溯源、知识库频繁变更的场景；微调适合改变模型行为风格、学习特定领域推理模式、减少 prompt 长度。二者不互斥 — 最佳实践往往是 RAG + 微调 组合：微调让模型"学会怎么用检索结果"，RAG 提供实时准确的知识。

Q: 如何避免微调后的"灾难性遗忘"？

A: (1) 使用较低学习率（尤其是全量微调）；(2) 在训练数据中混入少量通用数据（5%~10%）；(3) 使用 LoRA 等 PEFT 方法天然减少遗忘风险；(4) 监控模型在通用 benchmark 上的表现，及时早停；(5) 考虑使用 Elastic Weight Consolidation (EWC) 等正则化技术。

Q: DPO 能完全替代 RLHF 吗？

A: 在大多数应用场景中，DPO 已经足够好 — 训练更稳定、实现更简单、成本更低。但对于追求极致对齐效果（如 ChatGPT/Gemini 级别），传统 RLHF 的在线探索能力使其理论上限更高。当前趋势是 DPO 在快速普及，但 RLHF 在顶级产品中仍有不可替代的价值。

Q: 微调数据需要多少条才够？

A: 质量远比数量重要。指令微调：500~5K 高质量样本即可看到明显效果；领域知识注入：10K~100K 样本；行为对齐 (DPO)：1K~10K 偏好对。建议从最小可用数据集开始（~500条），验证效果后再决定是否需要扩充数据。
