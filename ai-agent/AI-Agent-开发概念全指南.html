1Agent 核心架构

理解单个 Agent 的内部运转机制 -- 这是一切复杂体系的基础

🧠 推理引擎 (Reasoning)

Agent 的"大脑"，基于 LLM 进行 Chain-of-Thought (CoT) 推理。核心范式是 ReAct：思考(Reason) - 行动(Act) - 观察(Observe) 循环，直到任务完成。

ReAct 是当前 Agent 最主流的推理范式，将"推理"和"行动"交织进行

👁️ 感知层 (Perception)

接收用户输入，解析意图。包括自然语言理解、多模态输入处理（图像/语音/视频）、意图分类、任务拆解。

好的感知层 = 准确的意图识别 = 更少无效工具调用

🎯 输出层 (Action)

将推理结果转化为可执行操作：返回文本、调用工具、触发工作流、与其他 Agent 通信。

输出不只是"说"，更是"做" -- 这正是 Agent 区别于 Chatbot 的关键

ReAct 循环详解

# ReAct: Reasoning + Acting 交替循环

Thought: 用户想查询上海明天的天气，我需要调用天气 API
Action: weather_api
Action Input: {"city": "上海", "date": "明天"}

Observation: {"temp": 28, "weather": "多云转晴", "humidity": 65}

Thought: 获取到天气数据，用户还问要不要带伞，湿度 65% 且多云转晴，不需要带伞
Action: finish
Action Input: "上海明天 28度，多云转晴，湿度 65%，不需要带伞。"

# 循环结束，返回最终结果
2多 Agent 协作体系

从单打独斗到团队协作 -- 多 Agent 的编排模式、通信机制与角色分工

📋 中心化编排 (Orchestrator)

一个"主 Agent"负责任务分配和结果汇总，其他 Agent 作为执行者。类似项目经理 + 团队成员的模式。

✓ 控制清晰✓ 易调试✗ 单点瓶颈

🔄 去中心化 (Peer-to-Peer)

Agent 之间直接通信，无中心节点。每个 Agent 自主决策何时与谁协作。更灵活但更难控制。

✓ 高可用✓ 可扩展✗ 难调试

🏭 流水线 (Pipeline)

Agent 按顺序依次处理，上一个的输出是下一个的输入。适合有明确阶段的任务。

✓ 流程清晰✗ 串行延迟

⚖️ 辩论式 (Debate)

多个 Agent 对同一问题给出不同观点，互相批判，最终由裁判 Agent 做决策。适合需要多角度分析的场景。

✓ 多角度✗ Token 消耗大
角色职责典型能力类比
Orchestrator任务分解、分配、结果汇总规划、路由、决策项目经理
Researcher信息检索、数据收集搜索 API、网页抓取、DB 查询调研员
Executor执行具体操作代码执行、API 调用、文件操作工程师
Reviewer审核结果、质量把控对比、校验、评分QA / 审核员
Critic提出反对意见、发现风险逆向推理、边界分析魔鬼代言人
Summarizer整合多 Agent 输出摘要、格式化、翻译编辑 / 文案
设计原则：角色要有明确边界 -- 一个 Agent 只做一件事，做透。避免"全能 Agent"导致 prompt 膨胀和决策混乱。
# 基于 LangGraph 的多 Agent 中心化编排示例
from langgraph.graph import StateGraph, END
from typing import TypedDict

class AgentState(TypedDict):
    task: str
    research_results: str
    analysis: str
    final_report: str

# 1. 定义各角色 Agent
def researcher_agent(state: AgentState) -> dict:
    """调研 Agent：负责信息检索"""
    task = state["task"]
    results = llm.invoke(f"请调研以下主题并收集关键信息：{task}")
    return {"research_results": results}

def analyst_agent(state: AgentState) -> dict:
    """分析 Agent：负责数据深度分析"""
    data = state["research_results"]
    analysis = llm.invoke(f"基于以下数据做深度分析：\n{data}")
    return {"analysis": analysis}

def writer_agent(state: AgentState) -> dict:
    """写作 Agent：负责最终报告"""
    report = llm.invoke(f"基于分析结果写报告：\n{state['analysis']}")
    return {"final_report": report}

# 2. 构建编排图（流水线模式）
workflow = StateGraph(AgentState)
workflow.add_node("researcher", researcher_agent)
workflow.add_node("analyst", analyst_agent)
workflow.add_node("writer", writer_agent)

# 3. 定义执行顺序
workflow.set_entry_point("researcher")
workflow.add_edge("researcher", "analyst")
workflow.add_edge("analyst", "writer")
workflow.add_edge("writer", END)

# 4. 编译并执行
app = workflow.compile()
result = app.invoke({"task": "2025 年 AI Agent 发展趋势"})
3兜底机制 (Fallback)

当 Agent 执行失败时，如何优雅降级而非直接崩溃

🔄 模型降级

主模型不可用时切换到备用模型。如 GPT-4o - GPT-4o-mini - 本地小模型。

备用模型能力可能不足，需要调整 prompt 或简化任务

📋 规则兜底

所有 AI 链路都失败时，回退到硬编码规则。如 FAQ 匹配、关键词路由、默认回复。

规则兜底是最后一道防线，保证用户体验不中断

💾 缓存兜底

对相似/历史查询返回缓存结果。即使实时计算不可用，也能返回"足够好"的答案。

缓存要有 TTL 和失效策略，避免返回过时信息

兜底策略代码示例

class FallbackChain:
    """多级兜底链：主模型 > 降级模型 > 规则引擎"""

    def __init__(self):
        self.chain = [
            {"name": "gpt-4o", "timeout": 15, "handler": self.gpt4o_call},
            {"name": "gpt-4o-mini", "timeout": 10, "handler": self.gpt4o_mini_call},
            {"name": "rule-based", "timeout": 2, "handler": self.rule_based},
        ]

    async def execute(self, query: str) -> dict:
        errors = []
        for layer in self.chain:
            try:
                result = await asyncio.wait_for(
                    layer["handler"](query),
                    timeout=layer["timeout"]
                )
                return {"answer": result, "served_by": layer["name"],
                        "degraded": layer != self.chain[0]}
            except Exception as e:
                errors.append({"layer": layer["name"], "error": str(e)})
                logger.warning(f"Layer {layer['name']} failed: {e}, trying next...")

        return {"answer": "抱歉，服务暂时不可用，请稍后重试。",
                "served_by": "default", "degraded": True, "errors": errors}
4重试策略 (Retry)

网络抖动、API 限流、临时不可用 -- 重试是分布式系统的基础生存技能

⏰ 指数退避 (Exponential Backoff)

最常用的重试策略。每次重试间隔按指数增长：1s - 2s - 4s - 8s，避免重试风暴。

import asyncio
import random

async def retry_with_backoff(
    func, max_retries=5,
    base_delay=1.0, max_delay=60.0
):
    for attempt in range(max_retries):
        try:
            return await func()
        except Exception as e:
            if attempt == max_retries - 1:
                raise
            # 指数退避 + 随机抖动
            delay = min(
                base_delay * (2 ** attempt),
                max_delay
            )
            jitter = random.uniform(0, delay * 0.1)
            await asyncio.sleep(delay + jitter)

🎯 重试条件判断

不是所有错误都该重试。需要区分可重试错误和不可重试错误。

错误类型是否重试
429 Too Many Requests✓ 重试
500/502/503✓ 重试
超时 / 连接重置✓ 重试
400 Bad Request✗ 不重试
401 Unauthorized✗ 不重试
内容安全拦截✗ 不重试
陷阱：对 401 错误重试只会浪费 Token，还会触发安全告警。务必先判断错误码！

Agent 工具调用的重试流程

❌ 常见重试反模式
  • 固定间隔重试（无退避）- 雪崩效应
  • 对所有错误都重试 - 浪费资源
  • 无限重试 - 用户超时等待
  • 重试时不加抖动 - 同步重试风暴
  • 重试时不记录日志 - 问题无法追踪
✅ 正确的重试姿势
  • 指数退避 + 随机抖动
  • 只对可重试错误重试（5xx/超时/429）
  • 设置最大重试次数（通常 3-5 次）
  • 配合熔断器模式使用
  • 每次重试都记录上下文日志
5A2A 协议 (Agent-to-Agent)

Google 提出的开放协议 -- 让不同框架、不同厂商的 Agent 能够互相发现、通信、协作

📡 A2A 是什么？

A2A (Agent-to-Agent) 是 Google 在 2025 年提出的开放协议，定义了 Agent 之间的标准通信方式。类似于 HTTP 之于 Web，A2A 之于 Agent 网络。

核心理念：不同框架（LangChain / CrewAI / AutoGen / Dify）的 Agent，只要实现 A2A 协议，就能互相调用。

🔧 核心概念

Agent Card - Agent 的"名片"，描述能力和接口
Task - Agent 之间的工作单元
Message - 通信消息，可含多模态内容
Artifact - 任务产出物（文件/图片/数据）
Streaming - SSE 流式响应支持

Agent Card 示例

// /.well-known/agent.json - Agent 的标准名片
{
  "name": "data-analyst-agent",
  "description": "专业数据分析 Agent，支持 CSV/Excel 处理与可视化",

    "streaming": true,

6Skills 技能体系

将 Agent 能力模块化、可复用、可组合 -- Skills 是 Agent 的"技能树"

🎯 Skill 是什么？

Skill 是对 Agent 某项具体能力的标准化封装。包含：名称、描述、输入输出 schema、执行逻辑。Agent 通过 Skill 描述自动判断何时调用哪个 Skill。

Skill vs Tool 的区别：Tool 是原子操作（如"调用天气 API"），Skill 是更高层的复合能力（如"规划旅行路线"可能调用多个 Tool）。

🏗️ Skill 结构

# skill.yaml - 技能声明文件
name: web-research
description: """搜索网络、提取信息、
  生成结构化摘要"""
version: 1.0.0

inputs:
  - name: query
    type: string
    required: true
    desc: "搜索关键词"
  - name: depth
    type: enum
    options: [quick, deep]
    default: quick

outputs:
  - name: summary
    type: string
  - name: sources
    type: array

tools_used:
  - web_search
  - url_fetcher
  - text_summarizer

Skill 生命周期

1
📝 声明 (Declare)

编写 skill.yaml，定义名称、描述、输入输出 schema、依赖工具

2
🔌 注册 (Register)

将 Skill 注册到 Agent 的技能库中，Agent 自动将 Skill 描述注入 system prompt

3
🧭 匹配 (Match)

LLM 根据用户意图，从技能库中自动选择最匹配的 Skill（语义路由）

4
⚡ 执行 (Execute)

按 Skill 定义的流程执行，可能调用多个 Tool，支持中间结果观察和重试

5
📤 输出 (Output)

按 schema 返回结构化结果，供 Agent 进行下一步推理或直接返回用户

6
🔄 迭代 (Iterate)

根据使用反馈持续优化 Skill 的 prompt、流程和参数

🧩 复合 Skill

Skill 可以嵌套调用其他 Skill，形成能力树。如"市场分析"Skill 内部调用"数据采集"和"图表生成"两个子 Skill。

🏷️ Skill 路由

当技能库中有几十个 Skill 时，需要语义路由：先用 embedding 检索 top-K 相关 Skill，再让 LLM 精确选择。

📊 Skill 版本管理

每个 Skill 有版本号，支持灰度发布、A/B 测试。旧版本不删，标记 deprecated，平滑迁移。

7上下文管理 (Context Management)

LLM 的上下文窗口是有限的 -- 如何在有限窗口内塞入最有效的信息

🗜️ 压缩策略

当对话超出窗口时，将早期对话摘要化。保留关键事实和决策，丢弃寒暄和冗余。

async def compress_context(messages):
    # 按轮次分组
    chunks = chunk_messages(messages, size=10)

    # 对早期对话做摘要
    if total_tokens(messages) > MAX_TOKENS:
        old = chunks[:-3]  # 保留最近3轮
        summary = await llm.summarize(old)
        return [{"role": "system",
                 "content": f"历史摘要: {summary}"}]
               + chunks[-3:]
    return messages

🎯 Token 预算分配

将上下文窗口当作"预算"来管理，给每个部分分配额度：

区域预算占比说明
System Prompt~10%角色、规则、约束
Skills/Tools~15%技能描述和 schema
RAG 检索~25%知识库检索结果
记忆~15%历史摘要和偏好
近期对话~25%最近几轮交互
输出预留~10%给 LLM 生成空间
实际比例需根据场景调整。RAG 密集型任务可给 RAG 更多预算
8记忆系统 (Memory)

让 Agent 拥有跨会话的"长期记忆" -- 短期、长期、向量三层架构

📝 记忆写入流程

1. 交互发生

用户与 Agent 对话，产生新信息

2. 信息提取

LLM 从对话中提取关键事实和偏好

3. 分类存储

结构化 -> DB；非结构化 -> 向量库

4. 索引更新

更新 embedding 索引，供后续检索

🔍 记忆读取流程

1. 意图分析

分析当前查询需要什么记忆

2. 多路召回

同时查 DB（精确）和向量库（语义）

3. 相关性排序

对召回结果按相关性 + 时效性排序

4. 注入上下文

将 Top-K 记忆注入 prompt

🧹 记忆维护

去重

相同信息合并，避免重复存储

衰减

长期未使用的记忆降低权重

遗忘

过时/低价值记忆定期清理

整合

多条碎片记忆合并为完整知识

💻 记忆系统代码示例

class AgentMemory:
    """三层记忆架构：短期 > 长期 > 向量"""

    def __init__(self):
        self.short_term = []          # 当前会话消息列表
        self.long_term = RedisDB()    # 结构化记忆
        self.vector = VectorStore()   # 语义记忆

    async def write(self, message: dict):
        # 1. 写入短期记忆
        self.short_term.append(message)

        # 2. 提取关键信息写入长期记忆
        if is_important(message):
            facts = await llm.extract_facts(message)
            for fact in facts:
                await self.long_term.set(
                    fact["key"], fact["value"],
                    ttl=fact.get("ttl", 86400 * 30)
                )

        # 3. 向量化写入语义记忆
        embedding = await embed_model.encode(message["content"])
        await self.vector.upsert(
            id=message["id"],
            embedding=embedding,
            metadata={"timestamp": now(), "type": message.get("type")}
        )

    async def recall(self, query: str, top_k: int = 5) -> list:
        # 多路召回：精确 + 语义
        exact = await self.long_term.search(query)
        semantic = await self.vector.search(
            await embed_model.encode(query), top_k=top_k
        )
        # 合并 + 去重 + 排序
        return merge_and_rank(exact, semantic)
9工具调用 (Tool Calling)

Function Calling -- 让 Agent 能够"动手"而不仅是"动嘴"

✅ 工具设计最佳实践

1
描述清晰

name 和 description 要让 LLM 一眼看懂何时用

2
参数最小化

只暴露必要参数，有默认值的不要让 LLM 填

3
错误可读

工具报错信息要让 LLM 能理解并自我修正

4
幂等设计

同一参数重复调用应返回相同结果

5
超时控制

每个工具设置独立超时，避免拖垮整个 Agent

⚠️ 常见工具调用问题

LLM 幻觉调用不存在的工具
LLM 编造了一个 schema 中不存在的工具名
解决：在 system prompt 中明确列出可用工具；对未知工具返回友好错误而非崩溃
参数格式错误
LLM 传入了字符串而非数字，或缺少必填参数
解决：工具入口做参数校验 (Pydantic)；错误信息回传 LLM 让它修正后重试
无限循环调用
LLM 反复调用同一工具，不收敛
解决：设置最大工具调用次数（如 10 次）；检测重复调用模式并强制终止
工具选择错误
LLM 选了不太合适的工具
解决：优化工具 description；减少相似工具数量；增加 few-shot 示例
10插件体系 (Plugin System)

插件是 Agent 能力扩展的标准化机制 -- 安装即用、隔离安全、版本管理

🔌 插件 vs 工具 vs Skill

概念粒度包含内容类比
Tool原子一个函数 + schema螺丝刀
Skill复合多个 Tool + 流程 + prompt维修手册
Plugin系统Skill + Tool + 配置 + UI + 依赖工具箱
Plugin 是最大的能力单元，一个插件可能包含多个 Skill 和 Tool，甚至自带 UI 组件

11常见问题与排查

Agent 开发中高频出现的问题及其排查思路和解决方案

🔥Agent 陷入无限循环，不断调用工具但不收敛▶

现象：Agent 反复调用同一工具或在不同工具间跳转，不产出最终答案。

根因分析：

  • 工具返回结果不明确，LLM 无法判断任务是否完成
  • system prompt 缺少"何时停止"的明确指令
  • 工具间有功能重叠，LLM 在多个工具间犹豫
  • 上下文过长导致 LLM "忘记"原始目标

解决方案：

  • 设置最大工具调用次数硬限制（如 max_iterations=10）
  • 在 system prompt 中明确"任务完成的标志"
  • 检测连续重复调用模式，超过 3 次相同调用强制终止
  • 每次工具调用后让 LLM 做一次"进度自评"
  • 使用 LangGraph 等框架的条件边精确控制流转
💸Token 消耗暴涨，成本不可控▶

现象：单次对话消耗数万甚至数十万 Token，费用远超预期。

根因分析：

  • 上下文不压缩，每轮对话都带完整历史
  • RAG 检索结果过多，注入大量无关文本
  • 工具 schema 描述冗长，占用大量 system prompt
  • 多 Agent 协作时重复传递相同信息
  • 未使用 prompt caching，每次都全量发送

解决方案：

  • 实现上下文压缩策略（摘要化旧对话）
  • RAG 结果限制 top-K（通常 3-5 条）
  • 精简工具描述，合并相似工具
  • 启用 prompt caching / KV cache 复用
  • 设置每会话 Token 上限，超限自动降级模型
  • 多 Agent 间传递摘要而非完整上下文
🧠Agent "失忆"，跨会话不记得用户信息▶

现象：新会话中 Agent 不记得之前聊过的内容、用户偏好、历史决策。

根因分析：

  • 只有短期记忆（会话内），没有长期记忆层
  • 记忆写入逻辑缺失或条件太严格
  • 记忆检索时相关性排序不佳，召回了无关记忆
  • 用户 ID 关联缺失，记忆无法归属到正确用户

解决方案：

  • 建立三层记忆架构：短期（内存）+ 长期（DB）+ 向量（Embedding）
  • 每次对话结束后自动提取关键事实写入长期记忆
  • 使用 user_id 作为记忆关联键
  • 记忆检索时同时使用精确匹配 + 语义搜索
  • 定期整理记忆：去重、整合、衰减
⚡响应延迟过高，用户体验差▶

现象：Agent 从收到消息到返回结果超过 30 秒，用户等待时间过长。

根因分析：

  • 多步 ReAct 循环，每步都等 LLM 完整响应
  • 工具调用同步阻塞，未做并行化
  • 上下文过长导致 LLM 推理慢
  • 多 Agent 串行编排而非并行
  • 没有使用流式输出（SSE），用户等全部完成才看到

解决方案：

  • 使用 SSE 流式输出，让用户"边生成边看"
  • 无依赖的工具调用并行化（asyncio.gather）
  • 多 Agent 并行编排：可同时执行的节点并行跑
  • 压缩上下文减少推理时间
  • 简单查询走快速路径，不进 Agent 循环
  • 使用更快的模型处理简单步骤（如 GPT-4o-mini 做路由）
🔒安全问题：Agent 执行了危险操作▶

现象：Agent 调用工具执行了不该执行的操作，如删除文件、发送邮件、修改数据库。

根因分析：

  • 工具权限过宽，没有操作级别限制
  • 缺少 Human-in-the-Loop 审批机制
  • Agent prompt 被注入恶意指令（Prompt Injection）
  • 工具输入参数缺少校验

解决方案：

  • 对危险操作强制 Human-in-the-Loop 审批
  • 工具权限分级：只读 / 写入 / 危险操作
  • 对用户输入做 Prompt Injection 检测
  • 工具参数用 Pydantic 严格校验
  • 操作日志全记录，支持审计和回滚
  • 沙箱环境执行代码类工具
🎭多 Agent 通信混乱，信息丢失或重复▶

现象：多 Agent 协作时，信息传递不准确，有的 Agent 收到重复消息，有的漏收关键信息。

根因分析：

  • 没有统一的消息格式和协议
  • Agent 间状态共享方式不一致（有的用共享内存，有的用消息传递）
  • 消息队列没有去重机制
  • Orchestrator 没有做结果合并和冲突检测

解决方案：

  • 使用统一的状态管理（如 LangGraph 的 StateGraph）
  • 定义标准消息格式：type / source / content / timestamp
  • 消息 ID 去重，防止重复处理
  • Orchestrator 负责结果合并和冲突检测
  • 采用 A2A 协议标准化 Agent 间通信
  • 每个 Agent 有明确的输入输出 schema
🎯Agent 输出质量不稳定，时好时差▶

现象：同一个问题不同时间问，得到的答案质量差异大。

根因分析：

  • LLM 本身的随机性（temperature 过高）
  • 上下文组装顺序不一致，影响 LLM 注意力
  • RAG 检索结果不稳定，不同时刻检索到不同内容
  • 缺少 few-shot 示例，LLM 理解偏差

解决方案：

  • 适当降低 temperature（0.3-0.7），减少随机性
  • 固定上下文组装顺序：System > Memory > RAG > Dialog
  • RAG 检索结果做 re-ranking，保证 top 结果稳定
  • 在 system prompt 中加入 few-shot 示例
  • 增加 Reviewer Agent 做质量审核，不合格则重试
  • 关键场景使用结构化输出（JSON mode）
