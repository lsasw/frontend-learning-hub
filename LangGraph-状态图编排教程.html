LangGraph 是什么 —— 为什么需要它

LangChain 链式调用的天花板

LangChain 的 线性 Chain 擅长 A → B → C 的流水线。但真实应用往往是：

  • 根据中间结果 条件分支（if answer is code → execute; else → explain）
  • 循环 执行直到满足条件（Agent 的 Think-Act-Observe 循环）
  • 需要 人工审批 后才能继续（发送邮件前确认）
  • 多个 Agent 并行/串行 协作

这些场景下，线性 Chain 力不从心。LangGraph 用 有向图（DAG / 循环图） 来建模工作流。

🔗 LangChain（链）

  • 线性 Pipeline：A → B → C
  • 无状态（每次调用独立）
  • 适合：翻译、摘要、简单 RAG

🕸️ LangGraph（图）

  • 任意有向图：A → B → C → (循环回 A)
  • 有状态（State 贯穿图执行）
  • 适合：Agent 循环、多步推理、人工审批

StateGraph —— 核心概念

三大核心元素

State（状态）

一个 TypedDict，贯穿整个图的执行。每个节点都可以读写它。类似 Redux 的 store。

Nodes（节点）

执行具体逻辑的函数：调用 LLM、调用工具、判断条件... 接收 State，返回 State 的更新部分。

Edges（边）

节点之间的转移规则。分普通边（无条件）和条件边（根据 State 内容路由）。

PYTHON
from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages

# 1. 定义 State（状态结构）
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]   # 消息自动累加
    next_step: str                            # 控制路由
    approved: bool                            # 审批状态

# 2. 定义 Node（节点函数）
def call_model(state: AgentState) -> AgentState:
    """调用 LLM，返回新消息"""
    llm = ChatOpenAI(model="gpt-4o")
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

# 3. 构建图
builder = StateGraph(AgentState)

builder.add_node("agent", call_model)
builder.set_entry_point("agent")         # 入口节点
builder.add_edge("agent", END)            # agent → 结束

graph = builder.compile()
result = graph.invoke({"messages": [HumanMessage(content="你好!")]})
🔑
Annotated[list, add_messages] 是关键魔法add_messages 是一个 reducer 函数——当多个节点都返回 messages 时，新消息会追加到已有列表中，而不是覆盖。没有这个 reducer，后执行的节点返回的 messages 会覆盖前一个的。

坑 #1：忘记用 Annotated 导致 State 被覆盖

在 TypedDict 中，如果字段没有 Annotated + reducer，新的返回值会 完全覆盖 旧值。很多人把 messages: list 当成会自动追加的列表，结果只剩下最后一条消息。

任何需要累积的字段（如消息列表、中间结果列表），必须使用 Annotated[list, operator.add] 或 Annotated[list, add_messages]。不需要累积的字段（如 next_step, approved）可以不用 Annotated，直接覆盖即可。

图编译与运行

PYTHON
# 编译图
graph = builder.compile(
    checkpointer=MemorySaver(),   # 启用持久化（支持暂停/恢复）
    interrupt_before=["approval"]  # 在执行 approval 节点前暂停
)

# 同步调用
result = graph.invoke(
    {"messages": [HumanMessage(content="查一下今天天气")]},
    config={"configurable": {"thread_id": "conv-001"}}
)

# 流式输出
for event in graph.stream(input_data, stream_mode="values"):
    print(event["messages"][-1])

# 异步调用
result = await graph.ainvoke(input_data)

坑 #2：checkpointer 未设置导致无法暂停/恢复

如果编译图时没有传 checkpointer，graph 是完全无状态的。这意味着 interrupt_before / interrupt_after 不生效，每次 invoke 都是独立的。很多人在做 Human-in-the-loop 时忘记这个。

做中断/暂停/恢复功能时，编译图时一定要传入 checkpointer（如 MemorySaver() 或 SqliteSaver），并在 invoke 时指定唯一的 thread_id。

条件分支与循环 —— 图的核心能力

条件边 (Conditional Edges)

根据 State 的内容，动态决定下一个要执行的节点。这是 Agent 循环的基础。

PYTHON
from langgraph.graph import StateGraph, END

# 构建 ReAct Agent 图
builder = StateGraph(AgentState)

# 添加节点
builder.add_node("agent", call_model)        # LLM 推理
builder.add_node("tools", ToolNode(tools))    # 工具调用

builder.set_entry_point("agent")

# 条件边：根据 agent 的输出来路由
def should_continue(state: AgentState) -> Literal["tools", END]:
    last_message = state["messages"][-1]
    # 如果 LLM 要求调用工具 → 跳转到 tools 节点
    if last_message.tool_calls:
        return "tools"
    # 否则结束
    return END

builder.add_conditional_edges(
    "agent",                # 从 agent 节点出发
    should_continue,          # 条件判断函数
    {"tools": "tools", END: END}  # 路由映射
)

builder.add_edge("tools", "agent")  # tools 执行完回到 agent

graph = builder.compile()

坑 #3：条件函数返回的字符串不在路由映射中

条件函数返回的 Literal 值必须精确匹配路由映射的 key。如果条件函数返回 "tool" 但路由映射里写的是 "tools"，图会直接报错。

使用 Python Literal 类型注解约束返回值（会自动检查），并确保路由映射的 key 和返回值严格一致。

坑 #4：Agent 循环没有出口条件 → 无限循环

ReAct Agent 的实现中，如果 LLM 每次都返回 tool_calls（即使工具返回了错误），图就会在 agent ↔ tools 之间无限循环。

在图编译时设置 recursion_limit（默认 25）：graph.compile(recursion_limit=10)。同时在条件函数中检查 total_steps，超过阈值直接 return END。

并行执行与子图

PYTHON
# 并行节点：Send API
from langgraph.types import Send

class State(TypedDict):
    tasks: list[str]
    results: Annotated[list, operator.add]  # 并行结果自动汇总

def continue_to_jokes(state: State):
    """为每个任务生成一个并行分支"""
    return [Send("process_task", {"task": t}) for t in state["tasks"]]

def process_task(state: dict):
    """并行处理单个任务"""
    result = llm.invoke(f"处理任务: {state['task']}")
    return {"results": [result.content]}

# 图的构建
builder.add_node("process_task", process_task)
builder.add_conditional_edges("planner", continue_to_jokes, ["process_task"])
builder.add_edge("process_task", "aggregator")

坑 #5：Send 并行中忘记用 Annotated + reducer → 结果互相覆盖

N 个并行分支都写入同一个 State 字段时，如果没设 reducer，最后一个完成的会覆盖所有。

并行写入的字段必须用 Annotated[list, operator.add]（或自定义 reducer）来实现聚合，而非简单覆盖。

Human-in-the-Loop —— 人机协作

中断 → 审批 → 恢复

这是 LangGraph 最强大的特性之一。让图在执行关键操作前暂停，等待人工确认。

PYTHON
from langgraph.checkpoint.memory import MemorySaver

builder = StateGraph(AgentState)

builder.add_node("draft", draft_email)      # 草拟邮件
builder.add_node("approval", human_approval)  # 人工审批节点
builder.add_node("send", send_email)         # 发送邮件

builder.set_entry_point("draft")
builder.add_edge("draft", "approval")
builder.add_edge("approval", "send")
builder.add_edge("send", END)

# 关键：在 approval 节点前暂停
graph = builder.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["approval"]  # 👈 此处中断
)

# 首次运行 → 在 approval 前自动暂停
config = {"configurable": {"thread_id": "email-001"}}
result = graph.invoke(
    {"messages": [HumanMessage(content="起草一封发给全员的周报邮件")]},
    config
)
print(result["messages"][-1].content)  # 查看草稿

# 人工审批通过 → 恢复执行
graph.update_state(config, {"approved": True})
result = graph.invoke(None, config)  # 传入 None 继续执行
🔑
interrupt_before vs interrupt_after
interrupt_before=["send"]：执行 send 节点之前暂停（还没发邮件）
interrupt_after=["draft"]：draft 节点执行完之后暂停（已经生成了草稿，等待确认）
还可以用 NodeInterrupt 异常在节点内部主动中断。

坑 #6：用 invoke(None, config) 恢复时传入错误数据

恢复执行时 invoke(None, config) 的 None 表示"不新增输入"。但如果在 update_state 时 state 结构写错了（比如把 approved 放在了错误的层级），恢复后的图会读到错误的状态。

update_state 接受的字典格式和 TypedDict 必须一致。可以先 graph.get_state(config) 查看当前 state 结构，再精确更新。

坑 #7：thread_id 用错导致读取到别人的会话状态

thread_id 是隔离不同会话的关键。如果多个用户共享同一个 thread_id，它们的对话历史和状态会混在一起。

每个用户/会话使用唯一的 thread_id（如 f"user-{user_id}-{session_uuid}"）。使用 SqliteSaver 而非 MemorySaver 做生产持久化。

多 Agent 协作模式

四种协作模式

模式结构适用场景复杂度
Supervisor一个主管 Agent 调度多个 Worker任务分发、客服路由⭐ 中
Hierarchical多层嵌套的 Agent 团队复杂项目管理、多层审批⭐⭐ 高
SequentialA → B → C → D 串联流水线处理、文档审核⭐ 低
Debate多个 Agent 互相校验代码审查、安全审计⭐⭐ 高
PYTHON
# Supervisor 模式 —— 最实用的多 Agent 模式
from langgraph_supervisor import create_supervisor
from langgraph.prebuilt import create_react_agent

# 创建两个独立 Agent
research_agent = create_react_agent(
    llm, [search_tool, web_scrape_tool],
    state_modifier="你是研究员，擅长搜集资料。"
)
code_agent = create_react_agent(
    llm, [python_repl, bash_tool],
    state_modifier="你是程序员，擅长写代码。"
)

# 创建主管 Agent，把两个子 Agent 当工具
supervisor = create_supervisor(
    agents=[research_agent, code_agent],
    model=llm,
    prompt="你是项目主管。接到任务后分派给合适的人。"
)

# 使用
result = supervisor.invoke({
    "messages": [HumanMessage(content="调研 LangGraph 最新版本变化，并写一个对比代码示例")]
})

坑 #8：Supervisor 模式下子 Agent 的上下文不互通

每个子 Agent 都有独立的 State。如果 Research Agent 查到了关键信息，Code Agent 默认看不到。Supervisor 必须显式地把信息从输出中提取并传递给下一个 Agent。

在 Supervisor prompt 中明确要求它"提取前一个 Agent 的关键结果并传递给下一个 Agent"。或者用 SharedState 模式：定义一个共享 State，所有子 Agent 读写同一个 state。

LangGraph vs LangChain 选型指南

✅ 直接用 LangChain（不用 LangGraph）

  • 翻译、摘要、简单 RAG 问答
  • 单次调用、无状态、无分支
  • 代码量少、可读性优先

✅ 必须用 LangGraph

  • Agent 需要 Think-Act-Observe 循环
  • 需要条件分支 / 循环 / 并行
  • 需要 Human-in-the-loop（审批/确认）
  • 多 Agent 协作
✅
最佳实践 1. 从简单开始：先用 LCEL 实现基础流程，复杂了再迁移到 LangGraph
2. State 设计是核心：花时间设计好 TypedDict 结构，后续改动成本低
3. 善用 .get_graph().draw_mermaid_png() 可视化图结构
4. 生产环境用 SqliteSaver 或 PostgresSaver 做 checkpointer
5. 始终设置 recursion_limit 防止死循环
