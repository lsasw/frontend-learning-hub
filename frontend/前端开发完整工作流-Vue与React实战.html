1需求分析阶段
拿到需求不要立刻写代码，先花 20% 时间做需求拆解，能省后续 80% 的返工

📋 需求拆解

  • 通读 PRD，列出所有功能点
  • 区分 MVP 必做 与 二期优化
  • 梳理用户路径：登录→首页→详情→下单...
  • 识别模糊点，列 TODO 找 PM 确认

🎨 UI 评审

  • 交互细节：动画时长、loading 状态、空状态
  • 响应式：移动端 / 平板 / 桌面三端适配
  • 组件复用：哪些可抽公共组件？
  • 设计交付：Figma 标注 / Zeplin 切图

🔌 接口对齐

  • 列出所有需要的接口（RESTful / GraphQL）
  • 字段含义、数据类型、是否必填
  • 分页格式、错误码规范
  • 联调环境 / Mock 数据策略

⚠️ 常见坑：需求不清晰就开始写

写完发现某个字段是后端动态生成 / 某个交互要支持手势回退 / 异常态没考虑... 返工成本是初始的 3-5 倍。建议：在需求评审会上把所有模糊点都暴露出来，哪怕多花 1 小时，也比写一半推倒重来划算。

🧩 组件设计三大原则

1️⃣ 单一职责

一个组件只做一件事。
✅ UserAvatar
❌ UserCardWithAvatarAndBadgeAndFollow

2️⃣ 受控 / 非受控

明确组件状态由谁管理：
· 受控：value 由父组件传入
· 非受控：内部 state + ref 暴露

3️⃣ 容器 / 展示分离

· 展示组件：纯 props 进、event 出
· 容器组件：处理数据获取、状态
关注点分离，便于复用和测试

4状态管理阶段
什么时候用全局状态？什么时候用组件内 state？分清楚这 5 层就够了

🪜 状态分层的 5 个层级

层级方案典型场景
L1 · 组件内useState / ref表单输入、UI 开关、临时标记
L2 · 父子共享props + emit / 回调列表项 → 父组件
L3 · 跨组件共享provide/inject / Context主题、国际化、当前用户基础信息
L4 · 全局共享Pinia / Redux / Zustand购物车、登录态、权限、全局通知
L5 · 服务端缓存React Query / SWR / Vue Query列表数据、详情数据、自动重试
⚠️
不要把所有状态都塞进全局 Store 很多新手把所有 useState 都提到 Pinia / Redux，结果每次 setState 都触发整个 store 订阅者重渲染，性能反而变差。原则：能用组件内 state 解决的，绝不上升到全局。

1Vue 核心知识点
Vue 3 + Composition API + TypeScript 已成为主流标配
💚 响应式原理：ref / reactive / computed / watch
🟢 Vue 3 响应式（基于 Proxy）
  • ref(x) → 包装基本类型，.value 访问
  • reactive(obj) → 深度代理整个对象
  • computed(fn) → 自动追踪依赖的缓存值
  • watch(source, cb) → 副作用监听
  • watchEffect(fn) → 自动收集依赖
  • 底层用 Proxy 拦截 get/set
🟡 Vue 2 响应式（Object.defineProperty）
  • 用 Object.defineProperty 递归劫持
  • 无法检测对象属性的添加/删除
  • 数组需要重写 7 个方法（push/pop/...）
  • Vue 3 改进后这些限制全部消除
🔧 Composition API 三大核心
// 1. ref / reactive —— 响应式数据
import { ref, reactive, computed } from 'vue'

const count = ref(0)            // 基本类型用 ref
const state = reactive({           // 对象用 reactive
  name: 'Vue',
  list: []
})
const double = computed(() => count.value * 2)

// 2. watch / watchEffect —— 副作用
import { watch, watchEffect } from 'vue'

watch(count, (newVal, oldVal) => {
  console.log(`count: ${oldVal} → ${newVal}`)
})

watchEffect(() => {
  console.log('count =', count.value)  // 自动追踪 count
})

// 3. 生命周期 onMounted / onUnmounted ...
import { onMounted, onUnmounted } from 'vue'
onMounted(() => { console.log('组件挂载') })
onUnmounted(() => { console.log('组件卸载') })

function useMouse() {
  const x = ref(0), y = ref(0)
  onMounted(() => {
    const handler = (e) => { x.value = e.clientX; y.value = e.clientY }
    window.addEventListener('mousemove', handler)
    onUnmounted(() => window.removeEventListener('mousemove', handler))
  })
  return { x, y }
}

// stores/user.ts —— setup 风格（推荐）
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useUserStore = defineStore('user', () => {
  // state
  const token = ref('')
  const profile = ref(null)

  // getters
  const isLogin = computed(() => !!token.value)
  const userName = computed(() => profile.value?.name ?? '游客')

  // actions
  async function login(form) {
    const { data } = await apiLogin(form)
    token.value = data.token
    profile.value = data.user
  }

  function logout() {
    token.value = ''
    profile.value = null
  }

  return { token, profile, isLogin, userName, login, logout }
})

// 组件中使用
import { useUserStore } from '@/stores/user'
const userStore = useUserStore()
console.log(userStore.isLogin, userStore.userName)
await userStore.login(form)
3Vue 如何调用后端接口
Axios 二次封装 + 拦截器 + 错误处理是工程化标配
🛠 1. Axios 封装（企业级标准方案）
// utils/request.ts
import axios from 'axios'
import { ElMessage } from 'element-plus'
import { useUserStore } from '@/stores/user'
import router from '@/router'

const service = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 15000,
  headers: { 'Content-Type': 'application/json' }
})

// ── 请求拦截器 ──
service.interceptors.request.use(
  (config) => {
    const userStore = useUserStore()
    if (userStore.token) {
      config.headers.Authorization = `Bearer ${userStore.token}`
    }
    return config
  },
  (error) => Promise.reject(error)
)

// ── 响应拦截器 ──
service.interceptors.response.use(
  (response) => {
    const res = response.data
    // 假设后端约定：{ code, message, data }
    if (res.code !== 0) {
      ElMessage.error(res.message || '请求失败')
      // 401 token 失效
      if (res.code === 401) {
        useUserStore().logout()
        router.push('/login')
      }
      return Promise.reject(new Error(res.message))
    }
    return res.data  // 直接返回 data，组件不用再解一层
  },
  (error) => {
    // HTTP 状态码错误
    if (error.response?.status === 404) ElMessage.error('资源不存在')
    else if (error.response?.status === 500) ElMessage.error('服务器错误')
    else if (error.code === 'ECONNABORTED') ElMessage.error('请求超时')
    else ElMessage.error(error.message)
    return Promise.reject(error)
  }
)

export default service

1React 核心知识点
函数组件 + Hooks + 不可变数据是 React 18 的核心范式
⚛️ 核心理念：UI = f(state)

React 的本质

React 不是框架，而是声明式 UI 库。每次 state 变化，React 重新执行 render()，Diffing 算法决定真实 DOM 怎么变。开发者只需关心"状态是什么"，不必关心"DOM 怎么改"。

🎣 5 个最常用 Hooks
import { useState, useEffect, useMemo, useCallback, useRef } from 'react'

// 1. useState —— 状态
const [count, setCount] = useState(0)
const [user, setUser] = useState<User | null>(null)

// 2. useEffect —— 副作用（数据请求/订阅/DOM 操作）
useEffect(() => {
  const sub = apiGetUser().then(setUser)
  return () => sub.unsubscribe()  // 清理函数
}, [userId])  // 依赖数组

// 3. useMemo —— 缓存计算结果（依赖变化才重算）
const filtered = useMemo(
  () => list.filter(x => x.status === filter),
  [list, filter]
)

// 4. useCallback —— 缓存函数引用（避免子组件重渲染）
const handleClick = useCallback((id: string) => {
  doSomething(id)
}, [])

// 5. useRef —— 拿到 DOM 引用 / 跨渲染保存值
const inputRef = useRef<HTMLInputElement>(null)
useEffect(() => { inputRef.current?.focus() }, [])

// 额外：useReducer 适合复杂 state 逻辑
const [state, dispatch] = useReducer(reducer, initialState)

🧠 渲染机制核心

📊 虚拟 DOM & Diffing

  • JS 对象描述 UI（虚拟 DOM）
  • state 变 → 重 render → 新虚拟 DOM
  • Diff 算法对比新旧树（O(n) 启发式）
  • 找到差异后只更新真实 DOM 必要部分

🚀 React 18 并发特性

  • createRoot：启用并发模式
  • useTransition：标记非紧急更新
  • useDeferredValue：延迟更新
  • <Suspense>：异步加载占位
  • 自动批处理：多次 setState 合并一次渲染

// stores/useUserStore.ts —— 极简 API，比 Redux 轻 10 倍
import { create } from 'zustand'
import { apiLogin, apiGetUserInfo } from '@/api/user'

interface UserState {
  token: string
  profile: User | null
  isLogin: boolean
  login: (form: LoginForm) => Promise<void>
  logout: () => void
  fetchProfile: () => Promise<void>
}

export const useUserStore = create<UserState>(set => ({
  token: localStorage.getItem('token') || '',
  profile: null,
  isLogin: false,

  async login(form) {
    const { token, user } = await apiLogin(form)
    localStorage.setItem('token', token)
    set({ token, profile: user, isLogin: true })
  },

  logout() {
    localStorage.removeItem('token')
    set({ token: '', profile: null, isLogin: false })
  },

  async fetchProfile() {
    const user = await apiGetUserInfo()
    set({ profile: user, isLogin: true })
  }
}))

// 组件中按需订阅（避免不相关 state 变化触发重渲染）
const token = useUserStore(s => s.token)
const login = useUserStore(s => s.login)

🎣 2. 自定义 Hook 封装数据请求
// hooks/useRequest.ts —— 极简版
import { useState, useEffect, useCallback } from 'react'

export function useRequest<T, P extends any[]>(
  service: (...args: P) => Promise<T>,
  deps: P = [] as any
) {
  const [data, setData] = useState<T | null>(null)
  const [loading, setLoading] = useState(false)
  const [error, setError] = useState<Error | null>(null)

  const run = useCallback(async (...args: P) => {
    setLoading(true)
    setError(null)
    try {
      const res = await service(...args)
      setData(res)
      return res
    } catch (e) {
      setError(e as Error)
      throw e
    } finally {
      setLoading(false)
    }
  }, deps)

  useEffect(() => { run() }, [run])

  return { data, loading, error, run }
}

// 使用
function OrderList() {
  const { data, loading, error, run } = useRequest(
    () => apiGetOrderList({ page: 1, size: 20 }),
    []
  )
  if (loading) return <Skeleton />
  if (error) return <ErrorPage />
  return <Table dataSource={data?.list} />
}
🚀 3. 推荐：React Query（生产级方案）
// 解决：缓存、重复请求、loading、错误、自动 refetch、乐观更新...
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query'

// 查询订单列表
function useOrders(params: { page: number; size: number }) {
  return useQuery({
    queryKey: ['orders', params],   // 缓存 key
    queryFn: () => apiGetOrderList(params),
    staleTime: 5 * 60 * 1000,     // 5 分钟内不重新请求
    refetchOnWindowFocus: true  // 切回窗口时刷新
  })
}

// 删除订单（变更）
function useDeleteOrder() {
  const qc = useQueryClient()
  return useMutation({
    mutationFn: apiDeleteOrder,
    onSuccess: () => {
      qc.invalidateQueries({ queryKey: ['orders'] })  // 让列表自动重查
    }
  })
}

// 组件中使用 —— 简洁得让人感动
function OrderList() {
  const { data, isLoading, error, refetch } = useOrders({ page: 1, size: 20 })
  const del = useDeleteOrder()

  if (isLoading) return <Skeleton />
  if (error) return <ErrorPage />
  return (
    <>
      <Button onClick={refetch}>刷新</Button>
      <Table dataSource={data.list} />
    </>
  )
}

· staleTime 控制缓存新鲜度

⚡Vue 容易被忽略的重要知识点
这些点会让你的 Vue 代码从"能跑"变成"优雅"

1️⃣ nextTick —— 异步更新队列

Vue 在同一事件循环内多次修改数据，DOM 不会立即更新，而是批量处理。要拿到更新后的 DOM，必须用 nextTick。

import { nextTick } from 'vue'

count.value = 100
console.log(refDOM.value.innerText)  // 还是旧值
await nextTick()
console.log(refDOM.value.innerText)  // 新值

2️⃣ shallowRef / shallowReactive

大数据列表用深响应式性能差，用 shallowRef 只代理 .value 引用，内部交给 triggerRef 手动通知更新。

const bigList = shallowRef([])
// bigList.value.push(x) ❌ 不会触发更新
bigList.value = [...bigList.value, x]  // ✅ 整体替换
// 或
bigList.value.push(x); triggerRef(bigList)  // ✅ 手动通知

3️⃣ markRaw —— 标记永不响应式

第三方类实例（Echarts、地图、Monaco）被 reactive 代理后会失效。用 markRaw 跳过代理。

import { markRaw } from 'vue'
import * as echarts from 'echarts'

const chart = markRaw(echarts.init(dom))
chart.setOption({...})  // 正常调用

4️⃣ defineAsyncComponent 异步组件

大型组件（如富文本编辑器、图表库）按需加载，减小首屏体积。

import { defineAsyncComponent } from 'vue'
const RichEditor = defineAsyncComponent(() =>
  import('@/components/RichEditor.vue')
)
// 配 loadingComponent / errorComponent

5️⃣ keep-alive 缓存组件

列表 → 详情 → 返回时，组件默认重建，状态丢失。<keep-alive> 缓存已加载过的组件。

<keep-alive :include="['Home','List']" :max="5">
  <component :is="currentView" />
</keep-alive>
// 配合 onActivated / onDeactivated 监听激活/休眠

6️⃣ v-memo 性能优化（Vue 3.2+）

大列表中，只对"依赖变化"的节点重新渲染。

<div v-for="item in list" :key="item.id"
     v-memo="[item.id, item.selected]">
  <Heavy :data="item" />
</div>
// 当 id 和 selected 都没变时，跳过该子树的 diff

7️⃣ 自定义指令封装 DOM 操作

当某些 DOM 行为在多个组件都需要（如 v-permission、v-debounce、v-copy），抽成自定义指令。

app.directive('permission', {
  mounted(el, binding) {
    if (!hasPerm(binding.value)) {
      el.parentNode?.removeChild(el)
    }
  }
})
<button v-permission="'order:delete'">删除</button>

8️⃣ 插槽透传（inheritAttrs）

默认情况下，父作用域的非 props 属性会"穿透"到根元素。多根组件需要 defineOptions({ inheritAttrs: false })，再用 $attrs 手动控制。

<script setup>
defineOptions({ inheritAttrs: false })
</script>
<template>
  <label>姓名</label>
  <input v-bind="$attrs" />  // 显式绑定
</template>

9️⃣ defineModel（Vue 3.4+）

新版 v-model 写法，不需要再手写 props + emit。

<script setup>
const model = defineModel<string>()
</script>
<template>
  <input v-model="model" />
</template>
// 父组件：<MyInput v-model="name" /> 默认 prop='modelValue' event='update:modelValue'

🔟 watch 的 4 个细节

  • 默认懒执行，要立刻执行传 { immediate: true }
  • 监听 reactive 对象无法拿到 oldValue
  • 监听多个源：watch([a, b], ([na, nb], [oa, ob]) => ...)
  • 深度监听大对象耗性能，用 { deep: 2 } 限制层级
⚡React 容易被忽略的重要知识点
这些坑 80% 的 React 项目都犯过，知道了少走 2 年弯路

1️⃣ useState 的"看似不变"陷阱

React 用 Object.is 比较新旧值。直接修改对象/数组 不会触发更新！

// ❌ 错误
const [list, setList] = useState([])
list.push(x)        // 不更新
user.name = 'new'  // 不更新

// ✅ 正确 —— 不可变数据
setList([...list, x])
setUser({ ...user, name: 'new' })
⚠️
进阶方案：用 Immer（produce()）可写"可变"语法但生成不可变结果。

2️⃣ useEffect 依赖数组的"老值"陷阱

如果不把用到的所有 state/props 加进依赖，会拿到陈旧闭包。

function Timer() {
  const [count, setCount] = useState(0)
  useEffect(() => {
    const t = setInterval(() => {
      console.log(count)  // 永远是 0！
      setCount(count + 1)  // 永远 +1 然后停
    }, 1000)
    return () => clearInterval(t)
  }, [])  // ❌ 缺了 count

  // ✅ 改用函数式更新
  setCount(c => c + 1)  // 不依赖外部 count
}

3️⃣ useMemo / useCallback 不是"越用越快"

它们本身有开销（依赖比较 + 缓存）。简单计算用它们反而更慢！

// ❌ 过度优化
const sum = useMemo(() => a + b, [a, b])  // 加法比 useMemo 便宜

// ✅ 真正值得用的场景
// 1. 重计算（千条数据 filter / sort）
// 2. 引用稳定性（传给 memo 包裹的子组件）
// 3. 引用类型作为 useEffect 依赖

4️⃣ key 属性的"硬核作用"

很多人以为 key 只是为了消除 warning。错！key 决定了 Diffing 时如何识别节点身份。

// ❌ 用 index 作为 key
{list.map((item, i) => <Item key={i} />)}
// 问题：删除中间项后，后续组件的 state 会"串号"

// ✅ 用稳定 ID 作为 key
{list.map(item => <Item key={item.id} />)}

5️⃣ React.memo 的失效场景

父组件每次都传新的函数/对象，memo 直接破功。

// ❌ 父组件
<Child onClick={() => doX()} />  // 每次新函数
<Child config={{ a: 1 }} />  // 每次新对象

// ✅ 父组件
<Child onClick={handleClick} />        // useCallback 缓存
<Child config={memoConfig} />          // useMemo 缓存

// 或者：把 Child 的 props 改成 children 渲染 / render props

6️⃣ StrictMode 双调用的陷阱

开发模式下 React 故意双调用函数组件和 useEffect，帮你发现副作用问题。

// 如果 useEffect 里有"非幂等"操作，会被双执行：
// - 写 localStorage 用 setItem 而非 getItem 后改回
// - 订阅没清理：双重订阅 → 内存泄漏
// - 请求没防抖：发两次请求

// 解决：保证 effect 是幂等的 + 一定有 cleanup
useEffect(() => {
  const sub = subscribe(cb)
  return () => sub.unsubscribe()  // ⭐ 必须
}, [])

7️⃣ 受控 vs 非受控组件

表单输入，状态归谁管是个哲学问题。

// 受控 —— React 全权管理（推荐）
<input value={name} onChange={e => setName(e.target.value)} />

// 非受控 —— DOM 自行管理
<input ref={inputRef} />  // inputRef.current.value 取值

// 推荐：受控 + 第三方库（react-hook-form / formik）

8️⃣ 错误边界（ErrorBoundary）

没有错误边界，组件报错会导致整棵树卸载。v18 必须用 class 组件实现。

class ErrorBoundary extends React.Component {
  state = { hasError: false }
  static getDerivedStateFromError() {
    return { hasError: true }
  }
  componentDidCatch(error, info) {
    logToSentry(error, info)
  }
  render() {
    if (this.state.hasError) return <ErrorPage />
    return this.props.children
  }
}
<ErrorBoundary><MyWidget /></ErrorBoundary>

9️⃣ Suspense + 异步渲染

v18 的 Suspense 配合 React.lazy / use() 实现声明式异步 UI。

// 路由级代码分割
<Suspense fallback={<Skeleton />}>
  <Routes>
    <Route path="/orders" element={<OrderList />} />
  </Routes>
</Suspense>

// use() + Promise（v18.3+ 实验性）
function Page({ promise }) {
  const data = use(promise)  // 挂起直到 Promise resolve
  return <div>{data.name}</div>
}

🔟 useTransition 区分优先级

输入框打字要"立刻响应"，过滤列表可以"慢一点"。

function Search() {
  const [input, setInput] = useState('')
  const [filter, setFilter] = useState('')
  const [isPending, startTransition] = useTransition()

  const onChange = (e) => {
    setInput(e.target.value)  // 紧急：立即更新
    startTransition(() => {
      setFilter(e.target.value)  // 非紧急：可中断
    })
  }
  return <input value={input} onChange={onChange} />
}

🎯 选型建议

🟢 选 Vue 的场景

  • 团队前端水平参差，需要快速上手
  • 项目以中后台 / 表单密集为主
  • 需要平滑从 Vue 2 迁移
  • 喜欢"约定优于配置"的开发体验

🔵 选 React 的场景

  • 项目规模大、状态复杂、需要严格类型
  • 需要 React Native 做跨端 App
  • 对接海外生态 / 大量第三方库
  • 团队熟悉函数式编程理念
✨通用最佳实践（Vue / React 都适用）
不分框架，前端工程师通用的工程素养

🚫 性能反模式

  • 把所有 state 都放全局
  • 列表渲染不写 key 或用 index
  • 在 render 函数里发请求 / 写大计算
  • 频繁操作 DOM（应批量 / 虚拟化）
  • 大文件直接 import（应动态 import）

✅ 性能正模式

  • 路由级 + 组件级双重懒加载
  • 图片懒加载 + WebP + 响应式 srcset
  • 列表用虚拟滚动（> 100 条）
  • 输入防抖、滚动节流
  • SSR / 预渲染提升首屏

🔐 安全基线

  • 永远不要 v-html / dangerouslySetInnerHTML 渲染未净化内容
  • 用户输入做前端 + 后端双校验
  • Token 存 httpOnly Cookie 而非 localStorage
  • 接口加 CSRF 防护，CORS 严格配置
  • 依赖定期 npm audit 扫描

🧪 测试金字塔

  • 单元测试（最多）：工具函数 / composable / Hook
  • 组件测试：核心交互组件
  • 集成测试：关键业务流程
  • E2E 测试（最少）：登录、支付等核心路径
🎯
最后一条建议 框架只是工具，工程能力（需求拆解、组件设计、状态分层、性能优化、监控部署）才是前端的真正竞争力。两套框架的核心思想是相通的：声明式 UI + 状态驱动 + 组件化 + 不可变数据。掌握一个，另一个上手只需 1-2 周。
