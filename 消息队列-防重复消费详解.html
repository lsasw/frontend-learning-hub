1 "不重复消费"的两种含义

这个问题要分两层看："不重复消费"本身有两种含义——一种是"多个实例别抢同一条消息"（负载均衡层面），另一种是"消息被重复投递后怎么保证业务不出错"（幂等层面）。两者都得处理，缺一不可。

🔗 第一层：MQ 层面

防止多个实例抢同一条消息

通过分区/队列分配机制，确保同一条消息只投递给一个消费者实例

Kafka → Consumer Group + Partition
RocketMQ → CLUSTERING 模式
RabbitMQ → 竞争消费

🛡️ 第二层：业务层面

消费了也保证业务不出错

MQ 的"至少一次投递"语义下，消息可能被重复投递，必须做幂等

唯一键约束 / Redis 标记 / 业务状态机

⚠️
关键认知：只做第一层是不够的——线上大部分"重复消费导致数据错乱"的事故，根源都是漏了第二层的幂等保护。
2 Kafka：Consumer Group + 分区独占

核心机制：同一消费者组内，一个分区只能被一个实例消费

3 个实例用 相同的 group.id，Topic 设 3 个或更多分区，Kafka 自动做分区分配（Rebalance），天然保证不会有两个实例同时消费同一条消息。

# Topic 创建时指定分区数 ≥ 实例数
# bin/kafka-topics.sh --create --topic orders --partitions 3 --replication-factor 2
3 RocketMQ：集群模式 + 队列负载均衡

核心机制：集群模式下 Broker 按 Queue 分配，一条消息只给一个实例

RocketMQ 默认就是 CLUSTERING（集群）模式。同一 Consumer Group 的多个实例，Broker 按队列做负载均衡，效果等价于 Kafka 的分区分配。

java 
// ✅ 集群模式（默认），同组 3 个实例不会抢同一条消息
consumer.setMessageModel(MessageModel.CLUSTERING);

// ❌ 广播模式——故意让每个实例都消费一遍，别配错！
// consumer.setMessageModel(MessageModel.BROADCASTING);
⚠️
如果配成 BROADCASTING 模式，每个实例都会收到全量消息——这跟你的"避免重复消费"目标正好相反，确认不要用广播模式。

4 RabbitMQ：竞争消费 + Prefetch

核心机制：一条消息被推给某个实例后，其它实例不会再收到同一条消息

RabbitMQ 没有"分区"概念，靠 Broker 端的竞争消费天然保证一条消息只推给一个消费者。但要注意 prefetch count 设置。

💡
prefetch count 别设太大——某个实例一次拉走一大批消息、自己处理不过来，会造成消费不均衡（不是重复消费问题，是负载不均问题）。推荐设 1~10。
yaml 
# RabbitMQ 关键配置
spring.rabbitmq.listener.simple.prefetch: 5            # ✅ 每次只拉 5 条，防止不均衡
spring.rabbitmq.listener.simple.acknowledge-mode: manual # ✅ 手动 ACK
spring.rabbitmq.listener.simple.concurrency: 3          # 并发消费者数
⚠️
如果消费失败或超时未 ACK，消息会被 重新入队再分配，这可能导致同一条消息在不同时间点被不同实例消费——这属于"重复投递"，需要第二层幂等兜底。
5 即便 MQ 做了保证，为什么还会重复？

At-Least-Once 语义下，重复投递是正常现象，不是 Bug

MQ 保证的是"至少一次投递"（at-least-once），不是"恰好一次"（exactly-once）。以下场景都会触发重复：

场景 1：消费成功但 ACK 发送前，实例挂了 / 网络抖动

业务逻辑执行完了，但 ACK 没发出去。MQ 判定消费失败，把消息重新投给另一个实例 → 同一条消息被消费了两次。

场景 2：消费者组发生 Rebalance

新增/减少实例触发 Rebalance。过程中 offset 提交不及时，部分消息被重复分配。

场景 3：生产者重试导致重复发送

网络超时，生产者不确定消息是否已送达，重试后又发了一条 → MQ 队列里同一条消息出现了两份。

🔥
结论：这些场景光靠 MQ 的分区/负载均衡机制是解决不了的，必须在业务层做幂等处理。
6 业务层保证幂等的三种做法

方式一：唯一键 + 数据库唯一约束（最简单可靠）

每条消息带上唯一的 messageId，数据库建唯一索引。重复插入直接抛异常，天然拦截。

java 
// 每条消息带上全局唯一的 messageId 或业务唯一键
@Transactional
public void consume(String messageId, OrderData data) {
    try {
        // 利用数据库唯一索引，重复插入会抛异常，天然拦截重复消费
        consumeLogMapper.insert(messageId);  // message_id 字段加唯一索引
        // 执行真正的业务逻辑
        orderService.process(data);
    } catch (DuplicateKeyException e) {
        log.info("消息{}已消费过，跳过", messageId);
        return;
    }
}
✅
优点：实现简单，不依赖外部组件，数据库天然保证一致性。
注意：确保 insert 和业务逻辑在同一个事务中（加 @Transactional），否则可能 insert 成功但业务逻辑失败。

方式二：Redis 做幂等标记（适合高并发，比数据库快）

利用 Redis 的 SETNX 原子操作做消费标记，只有第一次设置成功才执行业务。

java 
public void consume(String messageId, OrderData data) {
    // SETNX：只有 key 不存在时才设置成功，天然做到"只处理一次"的判断
    Boolean isFirstTime = redisTemplate.opsForValue()
        .setIfAbsent("mq:consumed:" + messageId, "1", Duration.ofHours(24));

    if (Boolean.FALSE.equals(isFirstTime)) {
        log.info("消息{}重复投递，跳过", messageId);
        return;
    }
    orderService.process(data);
}
💡
关键参数：过期时间要设得比消息的最大重试周期长（比如设 24h）。太短的话 Redis key 过期后，后续重试消息查不到标记会被重复处理。
缺点：SETNX 成功但业务逻辑失败时，需要回滚 Redis 标记（或者用 Lua 脚本保证原子性）。

方式三：业务本身天然幂等（最优雅，零额外开销）

把业务操作设计为状态机式，重复执行的结果完全一样，不需要任何额外记录。

❌ 反例：不幂等
account.setBalance(
    account.getBalance() - amount
);
// 重复消费 → 余额被扣两次！
✅ 正例：状态机式判断，天然幂等
UPDATE orders
SET status = 'PAID'
WHERE order_id = ?
  AND status = 'UNPAID';
// 第二次执行时 status
// 已经不是 UNPAID 了
// SQL 影响行数 = 0
// 不会重复扣款
🏆
推荐优先考虑这个方式——不用额外建表、不用 Redis、不用 catch DuplicateKeyException。只要你的业务模型支持状态流转，这在设计上是最干净的解法。
7 3 实例场景下的完整配置方案

生产环境两层的完整对比

MQ 类型 第一层：MQ 配置 关键注意点
Kafka 3 个实例用同一个 group.id，Topic 分区数 ≥ 3 分区数不够时多余实例空转；Rebalance 期间可能有短暂重复
RocketMQ 3 个实例用同一个 Consumer Group，模式设为 CLUSTERING 别配成 BROADCASTING（广播模式会全量投递）
RabbitMQ 3 个实例监听同一个 Queue，天然竞争消费 prefetch count 设 1~10，防止某个实例一次性拉太多
第二层：幂等方式 实现 适用场景
唯一键约束 DB 建消费记录表 + message_id 唯一索引 通用，最可靠，适合中等并发
Redis 标记 SETNX + 过期时间 > 最大重试周期 高并发，比数据库快，需处理 SETNX 成功但业务失败的场景
业务状态机 UPDATE ... WHERE status = 'OLD' 式判断 最优雅，有明确状态流转的业务优先考虑

🎯 一句话总结

  • MQ 层面的机制解决的是 "正常情况下别抢重"
  • 业务层幂等解决的是 "异常情况下重复来了也不出错"
  • 生产环境这两层都得有，只做第一层是不够的
  • 线上大部分"重复消费导致数据错乱"的事故，根源都是漏了第二层的幂等保护
