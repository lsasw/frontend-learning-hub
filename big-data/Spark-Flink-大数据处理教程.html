1 Apache Spark 概述

Apache Spark 是一个统一的分析引擎，用于大规模数据处理。它基于内存计算，比 Hadoop MapReduce 快 100 倍（内存中）。Spark 提供了批处理、流处理、机器学习（MLlib）、图计算（GraphX）和 SQL 查询等一体化能力。

🏗️ Spark 架构核心组件

Spark 集群架构
  • Driver：运行用户的主函数，创建 SparkContext / SparkSession，负责任务调度和 DAG 构建
  • Cluster Manager：集群资源管理，支持 YARN、Kubernetes、Standalone 三种模式
  • Executor：在工作节点上运行，负责执行 Task 并缓存数据，每个 Executor 是一个独立的 JVM 进程
  • Task：最小执行单元，一个 Task 处理一个 Partition 的数据
💡
核心设计理念：Spark 基于「弹性分布式数据集（RDD）」抽象，将数据分片存储在集群内存中，通过 DAG（有向无环图）调度器优化执行计划，实现内存级迭代计算性能。惰性求值（Lazy Evaluation）是 Spark 的灵魂 —— Transformation 操作不会立即执行，只有 Action 操作才会触发真正的计算。

📊 Spark 生态组件一览

组件功能典型场景
Spark Core底层引擎：RDD API、任务调度、内存管理、容错恢复所有上层组件的基础
Spark SQL结构化数据处理：DataFrame/Dataset、SQL 查询引擎ETL、数据仓库查询、CSV/Parquet 读写
Structured Streaming流处理：基于 Spark SQL 引擎的微批流处理实时日志分析、IoT 数据监控
MLlib机器学习库：分类、回归、聚类、协同过滤推荐系统、欺诈检测模型训练
GraphX图计算：PageRank、连通分量、三角计数社交网络分析、路网规划

2 Spark RDD — 弹性分布式数据集

RDD（Resilient Distributed Dataset）是 Spark 最底层的数据抽象，是一个不可变的分区记录集合。虽然现代 Spark 推荐使用 DataFrame/Dataset，但理解 RDD 是掌握 Spark 的基础。

📌 RDD 五大核心特性

  1. 分区列表（Partitions）：数据被切分为多个分区，分布在集群不同节点上并行处理
  2. 计算函数：每个分区都有对应的计算函数，定义如何生成该分区的数据
  3. 依赖列表：记录与其他 RDD 的血缘关系（Lineage），用于容错恢复
  4. 分区器（Partitioner）：可选，定义 key-value 类型 RDD 的分区策略（Hash / Range）
  5. 首选位置列表：可选，指定每个分区的最优计算位置（数据本地性）

🔧 RDD 创建与基本操作

rdd_basics.py · PySpark
from pyspark import SparkContext

# 初始化 SparkContext
sc = SparkContext("local[*]", "RDD_Demo")

# ── 方式1：从集合创建 RDD ──
rdd1 = sc.parallelize([1, 2, 3, 4, 5, 6, 7, 8], numSlices=4)
print(f"分区数: {rdd1.getNumPartitions()}")  # 4

# ── 方式2：从文件创建 RDD ──
# rdd2 = sc.textFile("hdfs://path/to/data.txt")
# rdd3 = sc.wholeTextFiles("hdfs://path/to/dir/")  # 每个文件作为一个 (filename, content) 对

# ═══ Transformation 操作（惰性求值，不立即执行）═══
mapped = rdd1.map(lambda x: x * x)          # [1, 4, 9, 16, 25, 36, 49, 64]
filtered = mapped.filter(lambda x: x > 10)    # [16, 25, 36, 49, 64]
flatmapped = rdd1.flatMap(lambda x: range(x))  # 展平：[0, 0,1, 0,1,2, ...]

# ── KV 操作 ──
pairs = sc.parallelize([("apple", 3), ("banana", 5), ("apple", 2), ("banana", 7)])
reduced = pairs.reduceByKey(lambda a, b: a + b)  # [("apple", 5), ("banana", 12)]
grouped = pairs.groupByKey().mapValues(list)       # [("apple", [3, 2]), ("banana", [5, 7])]

# ── 排序 ──
sorted_rdd = reduced.sortBy(lambda x: x[1], ascending=False)  # 按值降序

# ═══ Action 操作（触发实际计算）═══
print(f"collect: {filtered.collect()}")   # [16, 25, 36, 49, 64]
print(f"count: {filtered.count()}")       # 5
print(f"first: {filtered.first()}")        # 16
print(f"take(3): {filtered.take(3)}")      # [16, 25, 36]
print(f"reduce: {filtered.reduce(lambda x, y: x + y)}")  # 190
print(f"sorted: {sorted_rdd.collect()}")  # [("banana", 12), ("apple", 5)]

sc.stop()
▶ 运行结果
分区数: 4 collect: [16, 25, 36, 49, 64] count: 5 first: 16 take(3): [16, 25, 36] reduce: 190 sorted: [('banana', 12), ('apple', 5)]

💾 RDD 持久化（缓存）

RDD 默认每次 Action 都会重新计算。对于会被多次使用的 RDD，应调用 persist() 或 cache() 缓存到内存中，避免重复计算。

rdd_persist.py · PySpark
from pyspark import SparkContext
from pyspark.storagelevel import StorageLevel

sc = SparkContext("local[*]", "Persist_Demo")

rdd = sc.parallelize(range(1000000)).map(lambda x: (x, x * x))

# ── 存储级别 ──
# MEMORY_ONLY  : 仅内存（默认），放不下的分区不缓存，用时重算
# MEMORY_AND_DISK : 内存放不下的溢写到磁盘
# DISK_ONLY    : 仅磁盘
# MEMORY_ONLY_SER : 序列化后存内存，节省空间但增加 CPU 开销

rdd.persist(StorageLevel.MEMORY_AND_DISK)

# 第一次 Action 触发计算并缓存
result1 = rdd.filter(lambda x: x[1] > 100).count()
print(f"第一次 count: {result1}")  # 计算并缓存

# 第二次 Action 直接从缓存读取，无需重算
result2 = rdd.filter(lambda x: x[0] < 500).count()
print(f"第二次 count (走缓存): {result2}")

# 查看缓存信息
print(f"缓存状态: {rdd.getStorageLevel()}")
print(f"缓存大小: {rdd.context.getRDDStorageInfo()[0].memSize} bytes")

# 手动释放缓存
rdd.unpersist()
sc.stop()
▶ 运行结果
第一次 count: 999989 第二次 count (走缓存): 500 缓存状态: Memory Serialized 1x Replicated 缓存大小: 16777216 bytes
⚠️
避坑：cache() 是 persist(MEMORY_ONLY) 的简写。如果你的 RDD 数据量大于内存，使用 MEMORY_ONLY 会导致部分分区不缓存、每次重算。生产环境推荐 MEMORY_AND_DISK。另外，collect() 会把所有数据拉到 Driver 内存中，大数据量下会导致 OOM，应优先使用 take() 或 foreach()。

3 Spark DataFrame — 结构化数据

DataFrame 是 Spark 2.0+ 推荐使用的分布式数据集，类似于关系数据库中的表，带有 Schema（列名和类型）。底层使用 Catalyst 优化器和 Tungsten 执行引擎，性能远超 RDD。

🏗️ RDD vs DataFrame 性能对比

🐌 RDD

  • 无 Schema，JVM 对象序列化开销大
  • 无查询优化器，用户自行优化
  • Python RDD 需要进程间通信
  • GC 压力大

🚀 DataFrame

  • 有 Schema，使用 Tungsten 二进制格式
  • Catalyst 优化器自动优化执行计划
  • Python/Scala/Java 性能一致
  • 堆外内存管理，减少 GC

🔎 DataFrame 数据操作实战

dataframe_ops.py · PySpark
from pyspark.sql.functions import col, avg, sum, count, max, min, round, when, desc

# ═══ 选择与过滤 ═══
# 选择列
df.select("name", "salary").show(3)

# 条件过滤：薪资 > 20000 的工程部员工
df.filter((col("salary") > 20000) & (col("department") == "工程部")).show()

# 条件表达式
df.withColumn("level", when(col("salary") >= 28000, "高级")
                     .when(col("salary") >= 22000, "中级")
                     .otherwise("初级")).show()

# ═══ 分组聚合 ═══
# 按部门统计：人数、平均薪资、最高薪资、总薪资
dept_stats = df.groupBy("department").agg(
    count("*").alias("headcount"),
    round(avg("salary"), 2).alias("avg_salary"),
    max("salary").alias("max_salary"),
    sum("salary").alias("total_salary")
).orderBy(desc("avg_salary"))
dept_stats.show()

# ═══ 缺失值处理 ═══
# 填充缺失薪资为 0
df_filled = df.fillna({"salary": 0.0})

# 或用部门均值填充
dept_avg = df.groupBy("department").agg(avg("salary").alias("dept_avg"))
df_imputed = df.join(dept_avg, "department") \
    .withColumn("salary", when(col("salary").isNull(), col("dept_avg"))
                              .otherwise(col("salary")))
df_imputed.show()

# ═══ Join 操作 ═══
dept_info = spark.createDataFrame([
    ("工程部", "3号楼", 50),
    ("市场部", "1号楼", 20),
    ("财务部", "2号楼", 10),
], ["department", "building", "capacity"])

joined = df.join(dept_info, "department", "left")
joined.show()

+----------+---------+----------+----------+------------+ |department|headcount|avg_salary|max_salary|total_salary| +----------+---------+----------+----------+------------+ | 工程部| 3| 27666.7| 30000.0| 83000.0| | 市场部| 2| 21000.0| 22000.0| 42000.0| | 财务部| 2| 18000.0| 18000.0| 18000.0| +----------+---------+----------+----------+------------+ 注：财务部 avg_salary 因吴九 salary 为 null 被自动忽略

Catalyst 优化器工作原理：DataFrame 操作会经过四步优化 —— ① 分析（解析列名、类型，生成逻辑计划）→ ② 逻辑优化（谓词下推、列裁剪、常量折叠）→ ③ 物理计划生成（生成多个物理计划，基于代价选择最优）→ ④ 代码生成（Tungsten 将操作编译为 JVM 字节码）。这就是为什么 DataFrame 比 RDD 快 —— 用户写的是声明式 API，优化器自动帮你做最优执行。

4 Spark SQL — SQL 查询引擎

Spark SQL 让你用标准 SQL 查询结构化数据，底层与 DataFrame 共享同一套引擎和优化器。你可以混用 DataFrame API 和 SQL，两者完全等价。

📝 SQL 查询与 DataFrame 等价写法

spark_sql_demo.py · PySpark
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, avg, sum, count, desc

spark = SparkSession.builder.appName("SparkSQL_Demo").master("local[*]").getOrCreate()

# 创建示例数据：电商订单
orders = spark.createDataFrame([
    (1001, "user_A", "手机", 2999.0, "2024-01-15"),
    (1002, "user_B", "电脑", 8999.0, "2024-01-15"),
    (1003, "user_A", "耳机", 399.0, "2024-01-16"),
    (1004, "user_C", "手机", 2999.0, "2024-01-16"),
    (1005, "user_B", "平板", 4999.0, "2024-01-17"),
    (1006, "user_A", "电脑", 8999.0, "2024-01-17"),
], ["order_id", "user_id", "product", "amount", "order_date"])

# 注册为临时视图
orders.createOrReplaceTempView("orders")

# ═══ 方式1：纯 SQL 查询 ═══
# 每个用户的消费总额 TOP 3
result_sql = spark.sql("""
    SELECT user_id,
           COUNT(*) as order_count,
           ROUND(SUM(amount), 2) as total_spent,
           ROUND(AVG(amount), 2) as avg_order
    FROM orders
    GROUP BY user_id
    ORDER BY total_spent DESC
    LIMIT 3
""")
result_sql.show()

# ═══ 方式2：等价的 DataFrame API ═══
result_df = orders.groupBy("user_id").agg(
    count("*").alias("order_count"),
    sum("amount").alias("total_spent"),
    avg("amount").alias("avg_order")
).orderBy(desc("total_spent")).limit(3)
result_df.show()

# 两者输出完全一致！

# ═══ 窗口函数 ═══
spark.sql("""
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY amount DESC) as rn,
           RANK() OVER (ORDER BY amount DESC) as rank_all,
           SUM(amount) OVER (PARTITION BY user_id) as user_total
    FROM orders
    ORDER BY user_id, rn
""").show()

# ═══ 自定义 UDF ═══
from pyspark.sql.functions import udf
from pyspark.sql.types import StringType

# Python UDF（性能较差，每行都有 Python↔JVM 通信开销）
@udf(returnType=StringType())
def price_level(amount):
    if amount >= 5000: return "高消费"
    elif amount >= 1000: return "中消费"
    else: return "低消费"

orders.withColumn("level", price_level(col("amount"))).show()

spark.stop()

② 必须用 UDF 时，考虑 Pandas UDF（向量化，性能提升 10-100 倍）
③ 用 spark.sql() 注册 SQL UDF 也可以提升性能

5 Spark Streaming — 流处理

Spark Streaming 有两代 API：旧版 DStream（已过时）和新版 Structured Streaming（推荐）。Structured Streaming 基于 Spark SQL 引擎，将流视为一个不断增长的无限表，使用微批（Micro-Batch）模型处理。

🌊 Structured Streaming 核心模型

Structured Streaming 的核心思想：把流数据看作一个无边界的 DataFrame，每来一批新数据就像在表末尾追加了新行。你可以用与批处理完全相同的 DataFrame/Dataset API 来处理流数据。

🔄
微批处理模型：Spark 将到达的数据切成小批次（默认 100ms 间隔），每个批次作为一个 DataFrame 处理。优点：容错简单（批次级别 exactly-once）、与批处理统一。缺点：延迟下限为批次间隔（100ms~秒级），不适合超低延迟场景。

📤 输出模式（Output Mode）

模式行为适用场景限制
complete每次输出完整结果表聚合查询（如词频统计）数据量大时性能差
update仅输出有变化的行聚合查询、增量更新无聚合时等同于 append
append仅输出新增的行非聚合查询、ETL 入库聚合查询需配合 watermark
💧
Watermark 机制：处理乱序数据的关键。通过 .withWatermark("event_time", "10 minutes") 告诉引擎：数据最多延迟 10 分钟到达。超过 watermark 的旧数据会被丢弃，这样 append 模式下的聚合查询也能输出（因为引擎知道不会再有更旧的数据到来）。

6 Apache Flink 概述

Apache Flink 是一个真正的流优先（stream-first）分布式处理框架，支持精确一次（exactly-once）语义。与 Spark 的微批不同，Flink 逐条处理数据，延迟可达毫秒级。Flink 同样支持批处理 —— 批处理被看作是有界的流。

🏗️ Flink 架构核心组件

Flink 集群架构
  • JobManager：管理作业的全局协调器，包含三个子组件：
    • ResourceManager：管理 TaskManager 资源（Slot 分配/回收）
    • Dispatcher：接收作业提交，启动 JobMaster
    • JobMaster：管理单个作业的执行，协调 Checkpoint 和故障恢复
  • TaskManager：工作进程，提供 Slot 执行 Task。每个 Slot 是一份固定的 CPU/内存资源
  • State Backend：状态存储后端，支持 HashMapStateBackend（内存）和 EmbeddedRocksDBStateBackend（磁盘）
💡
流批一体的设计哲学：Flink 认为「批是有界流」—— 批处理只是流处理的一个特例（数据源有界、不再增长）。同一套 DataStream API 既可以处理无限流数据，也可以处理有限批数据，无需切换引擎或 API。这是 Flink 区别于 Spark 的核心设计差异。

9 Flink 窗口机制

流数据是无限的，窗口（Window）将无限流切分为有限的数据块，使得聚合操作成为可能。Flink 提供了比 Spark 更丰富的窗口类型。

🪟 三种核心窗口类型

📜 Tumbling Window
滚动窗口

  • 固定大小，不重叠
  • 每个数据只属于一个窗口
  • 如：每 5 分钟统计一次

📐 Sliding Window
滑动窗口

  • 固定大小，可重叠
  • 一个数据可属于多个窗口
  • 如：每 1 分钟滑动，统计过去 5 分钟

💬 Session Window
会话窗口

  • 按数据活跃度动态分组
  • 无固定大小，不重叠
  • 如：用户操作间隔超 10 分钟则开新窗口

🌍 Global Window
全局窗口

  • 所有同 key 数据归入一个窗口
  • 需自定义 Trigger 触发
  • 通常配合自定义触发器使用

🔧 窗口操作代码实战

• reduce()：输入输出类型相同，增量聚合，最高效。适合求和、最大值等

• process()：窗口触发时全量处理，可访问 WindowContext（窗口起止时间、当前 watermark、状态）。最灵活但性能最差

状态管理是 Flink 最核心的竞争力。与 Spark（无状态或靠外部存储）不同，Flink 原生支持丰富的状态类型，并通过 Checkpoint 机制实现精确一次（exactly-once）容错保证。

🔧 Keyed State 实战：用户行为分析

flink_state.py · PyFlink
from pyflink.datastream import StreamExecutionEnvironment, ProcessFunction
from pyflink.datastream.state import ValueStateDescriptor, MapStateDescriptor
from pyflink.common import Types, Time
import datetime

env = StreamExecutionEnvironment.get_execution_environment()

# 场景：检测用户连续登录失败（3次以上则告警）
# 数据格式：(user_id, action, timestamp)
events = env.from_collection([
    ("user_1", "login_fail", 1700000000000),
    ("user_1", "login_fail", 1700000001000),
    ("user_1", "login_fail", 1700000002000),
    ("user_1", "login_success", 1700000003000),
    ("user_2", "login_fail", 1700000004000),
    ("user_2", "login_fail", 1700000005000),
    ("user_2", "login_fail", 1700000006000),
], type_info=Types.TUPLE([Types.STRING(), Types.STRING(), Types.LONG()]))

# ── 使用 KeyedProcessFunction + ValueState ──
class LoginFailDetector(ProcessFunction):

    def open(self, runtime_context):
        # 定义 ValueState：记录连续失败次数
        self.fail_count = runtime_context.get_state(
            ValueStateDescriptor("fail_count", Types.INT())
        )
        # 定义 ValueState：记录第一次失败时间
        self.first_fail_time = runtime_context.get_state(
            ValueStateDescriptor("first_fail_time", Types.LONG())
        )

    def process_element(self, value, ctx, out):
        user_id, action, ts = value

        if action == "login_fail":
            count = self.fail_count.value()
            if count is None:
                # 第一次失败
                self.fail_count.update(1)
                self.first_fail_time.update(ts)
                # 注册定时器：2 秒后检查（重置窗口）
                ctx.timer_service().register_event_time_timer(ts + 2000)
            else:
                self.fail_count.update(count + 1)
                if count + 1 >= 3:
                    out.collect((f"⚠️ 告警: 用户 {user_id} 连续登录失败 {count+1} 次", ts))
        elif action == "login_success":
            # 登录成功，清空状态
            self.fail_count.clear()
            self.first_fail_time.clear()

    def on_timer(self, timestamp, ctx, out):
        # 定时器触发：重置计数（超过时间窗口）
        self.fail_count.clear()
        self.first_fail_time.clear()

# 应用处理函数
alerts = events \
    .key_by(lambda x: x[0]) \
    .process(LoginFailDetector())

alerts.print()
env.execute("LoginFailDetector")
▶ 运行结果
('⚠️ 告警: 用户 user_1 连续登录失败 3 次', 1700000002000) ('⚠️ 告警: 用户 user_2 连续登录失败 3 次', 1700000006000) 说明： - user_1 第3次失败时触发告警，随后 login_success 清空状态 - user_2 连续3次失败触发告警 - ValueState 按 user_id 自动隔离，每个用户有独立计数器 - 定时器确保超过2秒的失败不会累积

💾 State Backend 与 Checkpoint

🔄 Checkpoint 机制（容错核心）

Checkpoint 是 Flink 实现 exactly-once 的基石。原理：JobMaster 定期向数据源注入 Checkpoint Barrier（屏障），屏障随数据流一起向下游传播。当算子收到屏障后，会将自己的状态快照异步写入持久化存储（如 HDFS/S3）。如果发生故障，Flink 从最近的 Checkpoint 恢复状态，并重放 Checkpoint 之后的数据。

flink_checkpoint.py · PyFlink
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream import CheckpointingMode
from pyflink.common import Duration

env = StreamExecutionEnvironment.get_execution_environment()

# ═══ Checkpoint 配置 ═══
env.enable_checkpointing(60000)  # 每 60 秒做一次 Checkpoint

# 高级配置
config = env.get_checkpoint_config()
config.set_checkpointing_mode(CheckpointingMode.EXACTLY_ONCE)  # 精确一次
config.set_checkpoint_timeout(120000)  # Checkpoint 超时 2 分钟
config.set_min_pause_between_checkpoints(30000)  # 两次 CP 间隔至少 30 秒
config.set_max_concurrent_checkpoints(1)  # 最多 1 个并发 CP
config.enable_externalized_checkpoints(
    ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION
)  # 作业取消时保留 CP

# ═══ State Backend 配置 ═══
from pyflink.datastream import (
    HashMapStateBackend,
    EmbeddedRocksDBStateBackend
)

# 方式1：HashMap State Backend（内存，快但受限于内存大小）
env.set_state_backend(HashMapStateBackend())

# 方式2：RocksDB State Backend（磁盘，可存超大状态，生产推荐）
# env.set_state_backend(EmbeddedRocksDBStateBackend())
# 配置增量 Checkpoint（只存 diff，大幅减少 IO）
# config = env.get_checkpoint_config()
# config.set_rocksdb_config(...)  # 可调优 RocksDB 参数

# ═══ 开启 Unaligned Checkpoint（解决反压场景下 CP 超时）═══
config.enable_unaligned_checkpoints()

# 数据流：
# Source ──barrier──> Map ──barrier──> KeyBy ──barrier──> Window ──> Sink
#          快照         快照           快照          快照
#                ↓             ↓             ↓            ↓
#              HDFS          HDFS          HDFS         HDFS
#         ──────────────────────────────────────────────────
#         Checkpoint Coordinator (JobMaster) 统一管理
▶ 配置生效
Checkpoint 已启用 - 间隔: 60s - 模式: EXACTLY_ONCE - 超时: 120s - State Backend: HashMapStateBackend - Unaligned CP: enabled 故障恢复流程： 1. 从最近成功的 Checkpoint 恢复状态 2. 重新提交作业 DAG 3. 从 Checkpoint 对应的 offset 重放数据 4. 保证结果与无故障时一致 (exactly-once)
⚠️
State Backend 选择：
• HashMapStateBackend：状态存内存，读写快，但受限于 TM 内存。适合状态较小（<几 GB）的作业
• EmbeddedRocksDBStateBackend：状态存磁盘（RocksDB），支持 TB 级状态，但读写有磁盘 IO 开销。生产环境首选
• 序列化也会影响性能：Flink 自带的高效序列化器（TypeSerializer）比 Java 序列化快 10 倍以上

11 Spark vs Flink 深度对比

Spark 和 Flink 是大数据处理领域的两大主流框架，它们在设计理念、架构、性能特征上有显著差异。选择哪一种取决于你的具体场景。

⚔️ 核心维度对比

12 总结与选型建议

📌 一句话总结

Spark 是「批处理之王」，适合大规模离线数据处理、数仓查询和机器学习；Flink 是「流处理之王」，适合低延迟实时计算、复杂事件处理和有状态流应用。两者不是非此即彼 —— 很多企业同时使用 Spark（离线）和 Flink（实时），构建 Lambda/Kappa 架构。

🏆 选型决策树

  1. 延迟要求 < 100ms？ → 选 Flink（毫秒级处理）
  2. 需要复杂状态管理？（如用户会话、CEP 模式匹配）→ 选 Flink
  3. 需要机器学习训练？ → 选 Spark（MLlib 生态成熟）
  4. 需要 Hive 兼容的数仓查询？ → 选 Spark（Spark SQL + Hive 集成）
  5. 需要图计算？ → 选 Spark（GraphX）
  6. 需要大规模流式 Join？ → 选 Flink（状态管理 + 窗口优化好）
  7. 秒级延迟可接受 + 团队更熟悉 Spark？ → 选 Spark Structured Streaming
  8. 需要统一批流代码？ → 两者都支持（Flink 流批一体更彻底，Spark 3.x 也在改进）

📅 技术发展时间线

2009
Spark 在 UC Berkeley AMPLab 诞生，2010 年开源
2013
Spark 0.8 引入 Spark Streaming（DStream API）
2014
Flink 由柏林工业大学项目 Stratosphere 改名而来，成为 Apache 顶级项目
2016
Spark 2.0 发布 —— 引入 SparkSession、Structured Streaming、DataFrame 统一 API
2017
Flink 1.3 引入 Table API/SQL 和事件时间语义，开始大规模企业采用
2020
Spark 3.0 发布 —— GPU 支持、自适应查询执行 (AQE)、动态分区裁剪
2022
Flink 1.15 引入 Streaming File Sink 统一批流、Generalized Window API
2023+
Spark 3.5 发布 —— Structured Streaming 增强、Python 性能优化。Flink 持续深耕流批一体和状态管理

✅ Spark 优势

  • 批处理性能最强
  • SQL 生态成熟（Hive 兼容）
  • MLlib 机器学习库丰富
  • GraphX 图计算
  • 社区庞大，文档丰富
  • 企业级生产部署案例多
  • PySpark 生态完善

✅ Flink 优势

  • 真流处理，毫秒级延迟
  • 状态管理最强大
  • Checkpoint 容错机制优秀
  • 窗口类型最丰富
  • 复杂事件处理 (CEP)
  • 流批一体最彻底
  • 大规模状态支持 (TB 级)
💡
最终建议：如果你在做离线数仓 + 批处理 + ML，Spark 是更成熟的选择。如果你在做实时风控 + 实时大屏 + CEP，Flink 是更好的选择。如果你的场景介于两者之间（秒级延迟可接受），两者都能胜任 —— 此时选择团队更熟悉的那一个。

不要被「哪个更好」的争论困扰 —— 技术选型永远取决于具体场景、团队能力和运维成本。两大框架都在快速迭代，彼此也在互相学习（Spark 在改进流处理，Flink 在改进批处理）。
