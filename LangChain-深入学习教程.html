LangChain 是什么 —— 框架定位与核心设计哲学

一句话定义

LangChain 是一个用于构建 LLM 驱动应用 的框架。它不是模型本身，而是一套"胶水代码"——帮你把 LLM 和数据源、工具、记忆、向量数据库等等串联起来，形成可运行的 Pipeline。

🎯 核心设计理念

  • Composability（可组合性）：一切皆可链式组合（Chain）
  • Model-agnostic（模型无关）：同一套代码切换 OpenAI / Claude / 本地模型
  • Runnable 接口：统一抽象，所有组件都是 "Runnable"

❌ LangChain 不是什么

  • 不是模型服务：不做推理托管，只做编排
  • 不是向量数据库：它是向量库的"客户端"，不是引擎
  • 不是一站式平台：你仍需要选择模型 / 数据库 / 部署方案

六大核心组件详解

2.1 Model I/O —— 模型输入输出

LangChain 中最基础、最常用的组件。统一了 LLM（文本补全）和 Chat Model（对话接口）的调用方式。

PYTHON
# 方式一：Chat Model（推荐）
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, SystemMessage

llm = ChatOpenAI(model="gpt-4o", temperature=0.3)
messages = [
    SystemMessage(content="你是一个 Python 专家。"),
    HumanMessage(content="什么是 GIL？")
]
response = llm.invoke(messages)
print(response.content)

# 方式二：Prompt Template + 链式调用
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是 {role} 专家。"),
    ("user", "{question}")
])
chain = prompt | llm
result = chain.invoke({"role": "Python", "question": "什么是 GIL？"})

坑 #1：用错 model 类型导致输出格式错误

很多初学者分不清 OpenAI()（LLM, 返回纯文本）和 ChatOpenAI()（Chat Model, 返回 Message 对象）。Chat Model 返回的不是字符串，是 AIMessage 对象，需要 .content 才能拿到文本。

始终用 ChatOpenAI（而不是 OpenAI），LangChain 0.3+ 已逐步弃用旧 LLM 接口。

2.2 Prompt Templates —— 提示词模板

PYTHON
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

# 基础模板
prompt = ChatPromptTemplate.from_messages([
    ("system", "你是 {expertise} 专家，回答不超过 {max_words} 字。"),
    MessagesPlaceholder(variable_name="history"),   # 对话历史占位
    ("user", "{input}")
])

# Few-shot 模板（带示例）
from langchain_core.prompts import FewShotChatMessagePromptTemplate

examples = [
    {"input": "苹果的英文", "output": "Apple"},
    {"input": "香蕉的英文", "output": "Banana"},
]

2.3 Chains —— 链式编排

Chains 是 LangChain 的灵魂。把多个组件以管道方式串联，上一个的输出自动成为下一个的输入。

PYTHON
# LCEL 链式语法（推荐）
chain = (
    prompt
    | llm
    | StrOutputParser()
)

# 多条链组合
from langchain_core.runnables import RunnableParallel

# 两个分支并行执行
planner = ChatPromptTemplate.from_template("为这个问题制定计划：{question}") | llm
executor = ChatPromptTemplate.from_template("根据计划回答问题：{question}") | llm

parallel_chain = RunnableParallel(
    plan=planner,
    answer=lambda x: executor.invoke(x)
)

坑 #2：Chains 嵌套过深导致调试困难

旧版 LangChain（0.1.x）使用 SequentialChain / LLMChain 等类，嵌套 3 层以上就难以追踪数据流。LangChain 0.2+ 已用 LCEL 替代，但大量旧教程仍在使用旧 API。

始终使用 | 管道运算符（LCEL），而非旧的 Chain 子类。LCEL 的 .with_config() 或 LangSmith Tracing 可以方便地可视化数据流。

2.4 Memory —— 对话记忆

让 LLM 在多轮对话中记住上下文。有短期 (ConversationBufferMemory) 和长期 (VectorStore-backed) 两种策略。

PYTHON
from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

store = {}  # 生产环境用 Redis/DB

def get_session_history(session_id: str):
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

chain_with_history = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",
    history_messages_key="history",
)
# 多轮对话自动管理
response = chain_with_history.invoke(
    {"input": "我叫七智"},
    config={"configurable": {"session_id": "user-001"}}
)

坑 #3：Memory 无限膨胀吃掉 Token

ConversationBufferMemory 会把所有历史都塞进 Prompt，对话超过 20 轮就可能超 Token 上限（或费用暴涨）。

使用 ConversationSummaryMemory（摘要压缩）或 ConversationBufferWindowMemory（只保留最近 K 轮）。RAG 场景可以考虑用向量存储做长期记忆。

2.5 Retriever —— 检索器

把文档从向量数据库或其他存储中检索出来，是 RAG 的核心。LangChain 封装了 50+ 种 Retriever。

PYTHON
from langchain_community.document_loaders import TextLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_chroma import Chroma
from langchain_openai import OpenAIEmbeddings

# 1. 加载文档
loader = TextLoader("docs/readme.txt")
docs = loader.load()

# 2. 分割
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=100
)
chunks = splitter.split_documents(docs)

# 3. 向量化 + 存储
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=OpenAIEmbeddings(model="text-embedding-3-small")
)

# 4. 创建检索器
retriever = vectorstore.as_retriever(
    search_type="mmr",        # MMR 避免相似冗余
    search_kwargs={"k": 5}
)

坑 #4：chunk_size 和 chunk_overlap 选错导致检索质量差

chunk_size=1000 过大 → 检索精确度低、召回噪声大；chunk_size=100 过小 → 丢失上下文、无法回答完整问题。很多人默认用 1000，但不同文档类型需要不同策略。

代码文档推荐 200-500，长文推荐 500-1000。始终保留 10%-20% 的 overlap。可以和 metadata filtering 配合，先用关键词粗筛再用向量检索。

2.6 Tools & Agents —— 工具与智能体

PYTHON
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent

# 自定义工具
@tool
def get_weather(city: str) -> str:
    """获取指定城市的天气。"""
    return f"{city} 今天晴，25°C"

@tool
def calculator(expression: str) -> str:
    """计算数学表达式。"""
    return str(eval(expression))

tools = [get_weather, calculator]
agent = create_react_agent(llm, tools)

坑 #5：Agent 选了不合适的工具导致无限循环

Agent 可能在调用工具时陷入循环（调用 → 结果不对 → 再调用 → 仍然不对...），特别是工具描述不清晰时。默认的 ReAct Agent 最多尝试 15 次。

工具描述（docstring）必须清晰说明功能、输入格式、返回值。设置 max_iterations 限制步数。对复杂场景考虑用 LangGraph 做精确的状态机控制。

LCEL —— LangChain 表达式语言（现代写法）

什么是 LCEL？

LangChain Expression Language 是 LangChain 0.2+ 的核心。用 | 管道符把组件串联起来，自动处理 streaming、async、并行等。

PYTHON
# LCEL 基础语法
chain = prompt | llm | output_parser

# 更复杂的例子：RAG with source
from langchain_core.runnables import RunnablePassthrough, RunnableLambda

def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# Streaming
async for chunk in rag_chain.astream("什么是 GIL？"):
    print(chunk, end="", flush=True)
💡
重点理解：LCEL 中的 RunnablePassthroughRunnablePassthrough() 是 LCEL 中的"透传"操作。上面的例子中，用户问题既传给 retriever 去检索，又作为 question 传给 prompt。这种"一对多"分流是 LCEL 的核心模式。

坑 #6：RunnablePassthrough.assign() 误用导致属性覆盖

RunnablePassthrough.assign(foo=bar) 会向 dict 中注入新字段，但如果 key 已存在会被覆盖。很多人在复杂链中不小心覆盖了上游传递的关键数据。

在 assign 前先打印上游 dict 确认字段名不冲突。对复杂场景，优先使用 RunnableParallel 显式声明数据流。

RAG 全流程实战

标准 RAG Pipeline

PYTHON
# 完整 RAG 实战
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import WebBaseLoader
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough

# 1. 加载文档
loader = WebBaseLoader("https://python.langchain.com/docs/introduction/")
docs = loader.load()

# 2. 分块
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=100,
    separators=["\n\n", "\n", " ", ""]
)
splits = text_splitter.split_documents(docs)

# 3. 向量存储
vectorstore = Chroma.from_documents(
    splits, OpenAIEmbeddings(model="text-embedding-3-small")
)
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# 4. Prompt 模板
template = """基于以下上下文回答问题。如果上下文中没有答案，就说不知道。

上下文：
{context}

问题：{question}

回答："""
prompt = ChatPromptTemplate.from_template(template)
llm = ChatOpenAI(model="gpt-4o-mini")

# 5. 构建 RAG Chain
rag_chain = (
    {"context": retriever | (lambda docs: "\n\n".join(d.page_content for d in docs)),
     "question": RunnablePassthrough()}
    | prompt | llm
)

print(rag_chain.invoke("LangChain 的 core 模块包含哪些内容？").content)

坑 #7：直接对 HTML 做 embedding 导致大量噪声

从网页爬取的 HTML 里含大量 CSS/JS/导航栏等噪声文本，直接做 chunk 和 embedding 会让检索结果满是垃圾信息。

使用 BSHTMLLoader 或自行用 beautifulsoup4 提取主体内容后再 embedding。可以考虑先清洗（strip tags, remove scripts）再分块。

坑 #8：Retriever 返回的文档和问题语义不匹配

这是 RAG 最常见的失败模式。用户问"怎么部署？"但文档写的是"部署指南"，语义向量距离不够近，导致检索结果不相关。

使用 HyDE（假设性文档嵌入）、Multi-Query（生成多个查询变体）、或 重排序（reranker 二次排序）。对于专业领域，微调 embedding 模型效果提升显著。

Agent 智能体开发

从 Tool 到 Agent 到多 Agent 协作

PYTHON
# 使用 LangGraph 创建 ReAct Agent（推荐，LangChain 0.3+）
from langgraph.prebuilt import create_react_agent
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o")

@tool
def search_knowledge_base(query: str) -> str:
    """搜索内部知识库，返回相关文档内容。"""
    docs = retriever.invoke(query)
    return "\n".join(d.page_content[:200] for d in docs[:3])

@tool
def run_sql(sql: str) -> str:
    """执行只读 SQL 查询。仅支持 SELECT 语句。"""
    if not sql.strip().upper().startswith("SELECT"):
        return "错误：仅支持 SELECT 查询"
    return "查询结果: [{'id':1,'name':'test'}]"  # 示例

agent = create_react_agent(
    llm, [search_knowledge_base, run_sql],
    state_modifier="你是数据分析助手，优先用 SQL 查数据。"
)

# 运行
result = agent.invoke({
    "messages": [HumanMessage(content="帮我查一下最近一周的销售总额")]
})

坑 #9：@tool 装饰的函数没有类型注解

LangChain 依赖函数的类型注解和 docstring 来告诉 LLM 工具的输入输出格式。如果没写类型注解，LLM 可能传错参数格式导致调用失败。

始终给 tool 函数加完整的 Python 类型注解（str, int, float）和清晰的 docstring。docstring 第一行是工具描述，会被 LLM 看到。

坑 #10：在 Agent 工具中做有副作用的操作

Agent 可能在一次推理中多次调用同一个工具。如果你的 send_email() 或 create_order() 工具没有幂等保护，后果不堪设想。

所有有副作用的工具必须设计成幂等的：用 idempotency_key、检查是否已执行、或用 human_in_the_loop 中断确认（见 LangGraph 教程）。

十大踩坑速查表

#坑现象解法
1用错模型类型AIMessage 对象当成字符串输出统一用 ChatOpenAI + .content
2使用旧版 Chain API代码冗长、调试困难改用 LCEL 管道语法 |
3Memory 无限膨胀超 Token 上限 / 费用暴涨用 SummaryMemory 或 WindowMemory
4chunk 大小不当检索噪音大或丢失上下文按文档类型调参：200-500（代码）500-1000（长文）
5Agent 无限循环工具反复调、不收敛清晰的 tool docstring + max_iterations
6RunnablePassthrough 覆盖上游字段被意外覆盖检查字段冲突，优先用 RunnableParallel
7HTML 噪声检索结果全是网页导航栏文本用 BSHTMLLoader 或自行清洗
8语义匹配失败用户问题检索不到相关文档HyDE / Multi-Query / Reranker / 微调 Embedding
9Tool 缺类型注解参数传递错误、Agent 调用失败完整的类型注解 + 清晰 docstring
10副作用工具无幂等重复发邮件/重复下单幂等键 / 执行前检查 / human_in_the_loop
