🎯 常见面试问题深度分析

Java 核心开发 · 高频面试题
JVM 内存模型是怎样的？如何排查和解决 OOM 问题？ JVM ▼

核心考点：

  • JVM 内存分区：堆(新生代Eden+S0+S1/老年代)、方法区(元空间)、虚拟机栈、本地方法栈、程序计数器
  • 堆外内存：DirectByteBuffer、Netty的堆外内存泄漏排查
  • OOM 类型：java.lang.OutOfMemoryError: Java heap space / GC overhead limit exceeded / Direct buffer memory
💡 调优实战：
① jps -l 定位进程 → ② jmap -heap PID 查看堆配置 → ③ jmap -histo:live PID 查看对象分布 → ④ jstack PID 分析线程 → ⑤ 调整 -Xmx -Xms -Xmn参数 → ⑥ GC日志分析（-XX:+PrintGCDetails）
🔧 调优参数速查：
-Xms2g -Xmx2g 初始/最大堆 · -Xmn1g 新生代 · -XX:+UseG1GC G1收集器 · -XX:MaxGCPauseMillis=200 目标暂停时间
如何设计一个支持千万级并发的秒杀系统？ 高并发 ▼

核心考点：系统分层架构与流量管控

🏗️ 分层架构方案：
① 前端层：CDN静态化 + 按钮防重复(置灰+Token) + 验证码削峰
② 网关层：Nginx限流(limit_req) + 黑名单/IP频控
③ 服务层：Sentinel熔断降级 + 本地缓存(Caffeine) + Redis预减库存
④ 数据层：Redis Lua原子扣减 + 异步MQ下单 + 数据库最终一致性
🔑 关键技术点：
热点数据分离 · 读写分离 · 库存预热到Redis · 订单异步化 · 兜底方案(熔断后返回友好提示) · 全链路压测验证
Spring Bean 的生命周期是怎样的？循环依赖如何解决？ Spring ▼

核心考点：Spring IoC 容器原理

  • Bean 生命周期：实例化 → 属性填充 → Aware回调 → BeanPostProcessor前置 → InitializingBean → 自定义init → BeanPostProcessor后置 → 就绪 → DisposableBean → 自定义destroy
  • 循环依赖解决：三级缓存机制
🔄 三级缓存原理：
singletonObjects(一级·成品) → earlySingletonObjects(二级·半成品) → singletonFactories(三级·工厂)
A ← → B 循环依赖时：A实例化→放入三级缓存→填充属性时发现依赖B→B实例化→B填充属性时从三级缓存获取A的早期引用→B完成→A完成
synchronized 和 ReentrantLock 的区别？AQS 原理是什么？ 并发 ▼

核心考点：锁机制与AQS框架理解

  • synchronized：JVM内置，自动加锁/释放，非公平，1.6后引入偏向锁/轻量级锁优化
  • ReentrantLock：API层面，支持公平锁、可中断、tryLock超时、多条件变量Condition
  • AQS原理：volatile int state + CLH队列(FIFO) + CAS修改状态。acquire()尝试获取锁→失败进入等待队列→前驱节点唤醒
分布式架构 · 高频面试题
分布式事务如何保证数据一致性？Seata AT/TCC/Saga 模式对比 分布式事务 ▼

核心考点：分布式事务方案选型与对比

📊 三种模式对比：
AT 模式(自动)：基于undo_log回滚，业务无侵入，但依赖数据库ACID，适合单服务多DB
TCC 模式(Try-Confirm-Cancel)：业务需实现三阶段接口，性能好，适合高并发扣减类
Saga 模式(长事务)：正向补偿+逆向补偿编排，适合长流程，最终一致性
🏆 选型建议：
资金类操作 → TCC · 普通业务 → AT · 长流程调用链 → Saga · 简单场景 → 本地消息表 + 最大努力通知
Redis 缓存穿透/击穿/雪崩的区别与解决方案？ Redis ▼

核心考点：缓存三大经典问题

问题场景解决方案
缓存穿透查询不存在的数据布隆过滤器 + 空值缓存(短TTL)
缓存击穿热点key过期瞬间互斥锁(mutex) + 逻辑过期 + 永不过期
缓存雪崩大量key同时过期/Redis宕机过期时间随机化 + 集群高可用 + 限流降级

核心考点：SQL优化与执行计划分析

  • 慢查询定位：slow_query_log 开启慢日志 → long_query_time 设置阈值 → mysqldumpslow/pt-query-digest分析
  • Explain 关键字段：type(ALL→index→range→ref→eq_ref→const) · key(实际使用的索引) · rows(扫描行数) · Extra(Using filesort/Using temporary 表示需优化)
消息队列如何保证消息不丢失、不重复消费？ 消息队列 ▼

核心考点：MQ可靠性保障

📨 消息不丢失（全链路）：
① 生产者：确认机制(publisher-confirm) + 失败重试
② Broker：持久化 + 镜像集群 + 同步刷盘
③ 消费者：手动ACK + 消费完成后再确认
🔄 防重复消费（幂等性）：
① 数据库唯一约束(insert ignore) · ② Redis setnx · ③ 业务唯一ID + 状态机 · ④ 乐观锁(version)

向量数据库的选型考量？Milvus vs Pinecone vs Qdrant 对比 向量数据库 ▼

核心考点：向量数据库技术选型

  • Milvus：开源、高性能、分布式架构、支持多种索引(IVF/HNSW/DiskANN)、适合大规模生产环境
  • Pinecone：全托管SaaS、零运维、自带Embedding、适合快速起步和中小规模
  • Qdrant：Rust编写、高性能、支持过滤+向量混合查询、量化压缩、适合中等规模
📋 选型决策树：

🛡️ 技术风险 · 线上故障排查与方案设计

以下梳理了后端系统中最常见的线上故障场景，每个场景包含故障现象、排查思路、应急方案、根因修复四个维度，覆盖从"救火"到"防火"的完整链路。

🔥 CPU 突然飙升 100%
故障现象

服务响应变慢→超时→上游熔断→雪崩。CPU被打满，所有请求排队。

排查思路
  • top -Hp PID 定位高CPU线程
  • printf '%x\n' TID 转16进制 → jstack PID | grep -A20 0xTID
  • 检查GC日志：是否频繁Full GC
  • Arthas thread -n 3 快速定位
应急方案

限流降级→重启应急→回滚可疑发布→扩容缓冲

💧 内存泄漏导致 OOM
故障现象

服务运行一段时间后OOM Kill，堆内存使用率持续上升不下降，Full GC后仍接近上限。

排查思路
  • jmap -dump:live,file=heap.hprof PID 导出堆转储
  • MAT/JProfiler 分析：按Retained Size找大对象
  • ThreadLocal未清理、静态集合持续增长、连接池泄漏
  • Arthas vmtool 动态检查对象引用链
应急方案

重启服务→调整-Xmx临时缓解→修复后灰度发布

🐢 数据库慢查询拖垮服务
故障现象

接口超时→数据库连接池耗尽→服务不可用。常由新上线的未优化SQL或数据量增长触发。

排查思路
  • 慢查询日志定位具体SQL
  • EXPLAIN分析执行计划 → 关注type/rows/Extra
  • 检查是否缺索引或索引失效(隐式类型转换/函数)
  • 检查连接数：SHOW PROCESSLIST / SHOW STATUS LIKE 'Threads_connected'
应急方案

kill慢查询→紧急加索引→限流→读写分离→分库分表（长期）

🔴 Redis 集群故障
故障现象

缓存命中率骤降→请求直击DB→数据库压力暴增→全链路超时。

排查思路
  • 检查Redis连通性：redis-cli PING
  • 检查内存使用：INFO memory → maxmemory是否触发淘汰
  • 检查主从同步：INFO replication → offset差距
  • 慢日志：slowlog get 10 → keys*/hgetall大key
  • 大Key扫描：redis-cli --bigkeys
应急方案

故障转移(Sentinel自动)→DB限流保护→缓存预热→大Key拆分

📨 消息队列积压
故障现象

消费者处理速度跟不上生产速度→消息大量积压→下游业务延迟甚至数据丢失。

排查思路
  • 检查消费者是否存活(节点宕机/消费线程死锁)
  • 检查消费逻辑是否有慢操作(DB慢查询/外部API超时)
  • 检查prefetch count是否合理
  • 检查是否有消息被Nack后无限重试
应急方案

临时扩容消费者→批量消费优化→降级跳过非关键消息→死信队列兜底

🤖 LLM 调用超时/幻觉
故障现象

AI接口响应超时(30s+)、返回内容不可靠(幻觉)、Token消耗暴增、线上回答质量波动。

排查思路
  • 监控LLM API延迟分布(P50/P95/P99)
  • 检查Prompt是否因上下文过长导致Token激增
  • RAG场景检索结果是否相关(评估召回率/准确率)
  • 温度参数是否过高导致输出不稳定
应急方案

超时降级返回缓存/默认回答→切备用模型→缩短上下文→强制引用溯源→人工兜底

🌀 Agent 多步推理失控
故障现象

Agent陷入无限循环、调用链过长导致Token枯竭、多个Agent产生冲突决策、用户等待时间过长。

排查思路
  • 设置max_iterations硬限制防止死循环
  • 记录每步thought+action+observation用于回溯
  • 监控Token消耗速率 → 检测异常agent
  • 检查Agent间通信是否有丢失/重复
应急方案

强制中断超时Agent→降级为单Agent或直接LLM→缩短规划窗口→人工介入复杂决策

📉 向量检索精度下降
故障现象

RAG系统回答质量下降、检索结果与Query不匹配、用户反馈"答非所问"增多。

排查思路
  • 检查Embedding模型是否变更(版本/维度)
  • 检查Chunk切分策略是否适合当前文档类型
  • 对比RAGAS评估指标(忠实度/相关性)
  • 检查向量索引是否需要重建
应急方案

A/B测试切换检索策略→调整Chunk大小→引入Reranker→回滚Embedding模型版本

🚨 故障排查通用方法论

① 止损优先：先恢复服务，再排查根因（限流→降级→回滚→重启→扩容）
② 三板斧：监控看板(整体) → 日志查询(异常) → 链路追踪(定位) — Grafana + ELK + SkyWalking
③ 黄金信号：关注 RED 指标（Rate请求速率、Errors错误率、Duration延迟）
④ 事后复盘：5Why分析法 + 改进措施落地 + 监控告警补齐
