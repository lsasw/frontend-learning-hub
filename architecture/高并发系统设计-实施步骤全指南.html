1 需求分析 — QPS 预估与数据量估算

🎯 核心原则：先量化，再设计

高并发系统设计的第一步不是选技术栈，而是量化业务指标。没有精确的数字估算，所有的架构决策都是空中楼阁。需要回答三个核心问题：系统要承受多少并发？数据量增长多快？延迟要求是什么？

1.1 QPS 预估方法论

1 日均请求量推算

公式：日总请求 = DAU × 人均请求次数

示例：DAU 1000万，人均 50 次 → 日总请求 5亿次

2 峰值 QPS 计算

公式：峰值 QPS = 日总请求 ÷ 有效秒数 × 峰谷比

流量通常集中在上架时间的 3-4 小时内

3 读写分离估算

公式：读 QPS : 写 QPS ≈ 10:1 ~ 20:1

电商场景读多写少，IM 场景可能读写均衡

4 容量冗余规划

公式：设计容量 = 峰值 QPS × 1.5~2 倍冗余

为突发流量（秒杀、推广）留出 Buffer

💡
实战经验：电商大促场景的流量分布通常遵循「28 原则」——80% 的流量集中在 20% 的时间段。以双 11 为例，零点前 10 分钟的 QPS 可能是全天平均 QPS 的 50-100 倍。峰值估算公式：峰值 QPS = 日均 QPS × 峰谷比（通常 5~10 倍，大促场景 50~100 倍）。

📊 数据量估算的关键维度

结构化MySQL/PG：用户、订单、支付 — 按 行数 × 平均行大小 估算
半结构化MongoDB/ES：日志、行为 — 按 文档数 × 平均文档大小 × 压缩比(0.3~0.5)
非结构化OSS/S3：图片、视频 — 按 文件数 × 平均文件大小
缓存Redis：热点数据 — 按 key 数 × 平均 value 大小 × 1.2(元数据开销)
1.3 延迟要求
接口类型P99 延迟P999 延迟典型场景技术手段
超低延迟< 10ms< 50ms行情推送、风控决策本地缓存 + 内存计算
低延迟< 100ms< 200ms商品详情、搜索Redis 缓存 + ES
普通延迟< 500ms< 1s订单创建、支付DB 读写 + MQ 异步
可容忍延迟< 3s< 5s报表导出、批量操作异步任务 + 轮询
⚠️ 常见误区
  • 只看平均值不看分位数：平均延迟 100ms 不代表用户体验好，P99 可能已经 2s。必须关注 P99/P999。
  • 忽略长尾效应：GC 停顿、网络抖动、慢查询会导致尾部延迟飙升。1% 的慢请求可能影响 20%+ 的用户感知。
  • 低估数据增长速度：按当前数据量设计，半年后单表就超警戒线。必须按 1 年后的预期数据量 设计分片规则。
  • 混用读延迟和写延迟要求：读写延迟要求不同，架构设计应分别处理。不要用同一套方案应对所有场景。
  • 忽略网络开销：跨可用区延迟 1-2ms，跨地域 30-100ms。微服务拆分过细会导致网络延迟累积。
✅ 实施要点
  • 建立三档容量规划：日常容量 / 大促容量 / 极限容量，分别对应不同的扩缩容策略
  • 用历史数据 + 业务预测双维度估算，不要只看历史趋势，要考虑业务增长曲线
  • 把峰值 QPS × 2作为设计目标，预留突发流量 Buffer
  • 制定数据归档策略：热数据（3个月）→ 温数据（1年）→ 冷数据（归档到 OSS/HDFS）
  • 定期做全链路压测验证容量估算，至少每季度一次

2 架构选型 — 微服务/单体、同步/异步

🏗️ 核心决策：没有银弹，只有权衡

架构选型的本质是在复杂度和扩展性之间找平衡点。过早微服务化会增加运维成本，过晚拆分会阻碍团队协作。关键判断标准：团队规模、业务复杂度、并发量级。

2.1 单体 vs 微服务
🏢 单体架构

适用阶段：0 → 1 创业期，DAU < 50万

优势：

  • 开发简单，部署一个包搞定
  • 模块间调用是本地方法，零网络开销
  • 调试方便，一个 IDE 看全貌
  • 运维成本低，一套监控即可

劣势：

  • 代码膨胀后难以维护
  • 技术栈受限，无法混用语言
  • 扩容只能整体扩，浪费资源
  • 局部故障可能拖垮全局
🔧 微服务架构

适用阶段：业务成熟期，DAU > 100万，团队 > 20人

优势：

  • 独立部署、独立扩缩容
  • 技术栈自由，按场景选型
  • 故障隔离，局部不拖全局
  • 团队边界清晰，并行开发

劣势：

  • 分布式事务复杂度高
  • 网络调用有延迟和不可靠性
  • 运维成本骤增（监控/日志/链路）
  • 服务间依赖管理困难
📐
拆分时机判断标准（满足 2 条以上再拆）：
① 代码量超过 10 万行，编译时间 > 5 分钟
② 团队超过 15 人，出现跨团队协作瓶颈
③ 不同模块的并发量差异超过 10 倍
④ 不同模块的技术栈需求不同（如 AI 模块需要 Python，交易模块需要 Java）
⑤ 局部故障频发，影响全局可用性
2.2 同步 vs 异步
维度同步调用异步消息选型建议
响应延迟 所有下游 RT 之和 仅本地处理时间 用户感知路径用同步，后台处理用异步
可用性 任一下游宕机即失败 MQ 宕机才失败（MQ 可集群） 核心链路降级为异步可提升可用性
数据一致性 强一致（同事务） 最终一致（需补偿） 资金交易用同步，通知/日志用异步
吞吐量 受最慢下游限制 MQ 削峰后可线性扩展 秒杀/大促场景必须异步削峰
复杂度 低，调用链清晰 高，需处理幂等/乱序/积压 异步引入的复杂度不容低估
用户请求 → 同步：创建订单 → 返回成功 ⇢ 异步：扣库存 ⇢ 异步：发优惠券 ⇢ 异步：发通知 ⇢ 异步：数据统计

🔑 架构选型黄金法则

能单体就别微服务，能同步就别异步。 每引入一个分布式组件，就多一个故障点。但一旦并发量或团队规模到了临界点，就必须果断拆分——拖延只会让技术债越滚越大。正确的演进路径是：单体 → 模块化单体 → 服务化 → 微服务，而不是一步到位。

⚠️ 常见误区
  • 过早微服务化：5 个人的团队搞 20 个微服务，80% 时间在处理服务间通信问题。
  • 拆分粒度过细：一个 CRUD 操作拆成一个服务，网络开销远大于业务逻辑耗时。
  • 盲目追新：K8s + Service Mesh + 事件驱动，技术栈看起来很酷，但团队 hold 不住。
  • 异步滥用：把本应同步返回结果的操作异步化，导致用户体验变差，还要额外做轮询。
  • 忽略数据一致性：异步化后忘记做幂等和补偿，导致数据不一致。

3.1 接入层：负载均衡

DNS 轮询

最外层分流，一个域名映射多个 IP。简单但无法感知服务器健康状态。

CDN 边缘缓存

静态资源（图片/JS/CSS）缓存到边缘节点，减少源站压力。命中率 > 95% 是目标。

四层负载 (LVS/F5)

基于 IP+端口转发，吞吐量极高（百万级 PPS），不感知协议内容。

七层负载 (Nginx)

基于 HTTP 协议转发，支持路径路由、灰度发布、SSL 卸载、限流。

API 网关

统一鉴权、限流、熔断、日志。Spring Cloud Gateway / Kong / APISIX。

健康检查

主动探测 + 被动检测，自动剔除不健康节点，恢复后自动加回。

nginx.conf 
# Nginx 七层负载均衡 + 健康检查 + 限流
upstream api_backend {
    least_conn;                    # 最少连接数策略
    
    server 10.0.1.1:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 10.0.1.2:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 10.0.1.3:8080 weight=2 max_fails=3 fail_timeout=30s backup;  # 备用节点
    
    keepalive 32;                    # 保持长连接，减少握手开销
}

# 限流：每个 IP 每秒最多 10 个请求
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

server {
    listen 443 ssl http2;
    
    location /api/ {
        limit_req zone=api_limit burst=20 nodelay;  # 突发流量允许 20 个排队
        proxy_pass http://api_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_connect_timeout 2s;     # 连接超时
        proxy_read_timeout 5s;        # 读取超时
    }
}
3.2 应用层：无状态化与缓存策略

🔓 无状态化设计

核心思想：应用服务器不存储任何会话状态，所有状态外部化到 Redis/DB。这样任何一台应用服务器宕机，请求可以被无缝转发到其他节点。

Session存入 Redis（Spring Session / JWT Token），不要存本地内存
缓存本地缓存仅做只读缓存（启动时加载），可变数据走 Redis
文件上传到 OSS/S3，不要存本地磁盘
配置走配置中心（Nacos/Apollo），不要写死在本地
多级缓存架构
请求进入 → L1: 本地缓存 Caffeine (1ms) → L2: Redis 集群 (2-5ms) → L3: DB 查询 + 回写缓存
缓存层技术选型容量延迟适用场景失效策略
L1 本地缓存Caffeine / Guava百MB级< 1ms配置字典、热点数据TTL + 主动失效
L2 分布式缓存Redis ClusterGB~TB级2-5ms用户会话、商品信息TTL + LRU 淘汰
L3 CDN 缓存CloudFlare / 阿里CDNTB级< 5ms静态资源、HTML 页面边缘 TTL
Java 
// 多级缓存实现：Caffeine + Redis 协同
@Service
public class MultiLevelCache {
    
    // L1: 本地缓存 — Caffeine
    private final Cache<String, Object> localCache = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(60, TimeUnit.SECONDS)  // 本地缓存 60s
        .recordStats()
        .build();
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    @Autowired
    private ProductMapper productMapper;
    
    public Product getProduct(Long id) {
        String key = "product:" + id;
        
        // L1: 先查本地缓存
        Product product = (Product) localCache.getIfPresent(key);
        if (product != null) return product;
        
        // L2: 再查 Redis
        product = (Product) redisTemplate.opsForValue().get(key);
        if (product != null) {
            localCache.put(key, product);  // 回填 L1
            return product;
        }
        
        // L3: 查 DB 并回写缓存（加锁防击穿）
        synchronized (this) {
            product = productMapper.selectById(id);
            if (product != null) {
                redisTemplate.opsForValue().set(key, product, 30, TimeUnit.MINUTES);
                localCache.put(key, product);
            }
        }
        return product;
    }
}
🔥 缓存穿透
  • 查询不存在的 key，每次打到 DB
  • 解法：空值缓存 + 布隆过滤器
💥 缓存击穿
  • 热点 key 过期瞬间，大量请求打到 DB
  • 解法：互斥锁 + 永不过期 + 后台刷新
❄️ 缓存雪崩
  • 大量 key 同时过期，DB 瞬间过载
  • 解法：TTL 加随机偏移 + 多级缓存
3.3 数据层：分库分表与读写分离

A 读写分离

主库写，从库读，通过 binlog 同步。读 QPS 可线性扩展。

注意：主从延迟 0.1-1s，写后读需走主库

B 垂直分库

按业务拆分：用户库、商品库、订单库。降低单库压力和耦合。

注意：跨库 JOIN 失效，需应用层组装

C 水平分表

单表数据量超 2000万行时按 sharding key 拆分。order_id % 16 = 16 张表。

注意：跨页查询、聚合统计变复杂

D 分片键选择

选择查询最频繁的字段作为分片键（如 user_id）。

注意：避免热点分片，均匀分布数据

SQL / ShardingSphere

⚠️
分库分表后的难题与对策：
① 跨库 JOIN：禁止跨库 JOIN，改用应用层组装或数据冗余
② 分布式事务：用 Seata AT/TCC 或事务消息保证最终一致
③ 全局 ID：用雪花算法（Snowflake）或 Leaf 生成唯一 ID
④ 跨分片查询：分页查询需先查各分片再合并，深度翻页用游标方案
⑤ 统计报表：分片数据同步到 ES/ClickHouse 做离线分析
⚠️ 常见误区
  • 过早分库分表：单表 500万行就拆，实际上 MySQL 优化后千万级也能扛。拆了就回不去了。
  • 分片键选错：按 order_id 分片，但 90% 的查询按 user_id，导致全分片扫描。
  • 忽略主从延迟：写入后立即从从库读取，读到旧数据。必须写后读走主库。
  • 缓存和 DB 不一致：先更新 DB 再删缓存 vs 先删缓存再更新 DB，需根据场景选择并加延迟双删。
  • 忽视连接池配置：微服务拆分后 DB 连接数爆炸。10 个服务 × 20 实例 × 20 连接 = 4000 连接，MySQL 默认最大 151。
✅ 实施要点
  • 接入层：四层 + 七层负载均衡组合使用，LVS 处理高吞吐，Nginx 处理智能路由
  • 应用层：必须无状态化，Session 外部化到 Redis，支持随时水平扩缩容
  • 缓存层：L1 本地缓存 + L2 Redis，热点数据本地化，减少网络开销
  • 数据层：读写分离 → 垂直分库 → 水平分表，按瓶颈逐步推进
  • 连接管理：统一连接池配置，按服务实例数计算总连接数，确保 < MySQL max_connections × 0.6

4 关键技术 I — 限流、降级、熔断

🛡️ 系统稳定性的三道防线

高并发系统的核心挑战不是「跑得快」，而是「在异常情况下还能跑」。限流控制入口流量、降级牺牲非核心功能、熔断切断故障链路。三者协同构成系统的自我保护机制。

🚦 限流 (Rate Limiting)

目标：控制请求速率，防止系统过载

层次：网关限流 → 应用限流 → DB 限流

策略：QPS 限流 / 并发数限流 / 资源限流

📉 降级 (Degradation)

目标：主动牺牲非核心功能保核心链路

方式：自动降级（超时/异常率）+ 手动降级（开关）

原则：先降级写操作，再降级读操作

⚡ 熔断 (Circuit Breaker)

目标：切断对异常服务的调用，快速失败

状态：Closed → Open → Half-Open

恢复：半开状态试探性放行，成功则恢复

4.2 降级策略设计

📉 分级降级策略

级别触发条件降级动作影响范围
L1 轻度CPU > 70%关闭推荐/广告等非核心功能用户体验略降
L2 中度CPU > 80% 或 P99 > 1s关闭评论/评价/搜索排序部分功能不可用
L3 重度CPU > 90% 或 DB 连接耗尽拒绝写请求，只保留核心读只能浏览不能操作
L4 极端系统濒临崩溃返回静态页面/兜底数据基本不可用但存活
⚠️ 常见误区
  • 限流只做入口：只在网关限流，内部服务无保护，一个慢查询拖垮整个调用链。应该每层都有限流。
  • 熔断不恢复：熔断后一直 Open，忘记 Half-Open 探测恢复。下游早已恢复但流量还是被拒绝。
  • 降级粒度过粗：一键降级所有非核心功能，但没区分哪些可降、哪些不可降。应按功能维度细分开关。
  • 降级未演练：线上从未触发过降级，真到用时发现降级逻辑有 bug，反而加剧故障。
  • 忽略超时设置：不设超时或超时设置过长（如 30s），导致线程池耗尽。HTTP 调用超时建议 < 3s。
✅ 实施要点
  • 建立多级限流：网关限流（QPS）→ 服务限流（并发数）→ 资源限流（DB 连接/线程池）
  • 熔断器配置三个核心参数：错误率阈值（如 50%）、最小请求数（如 20）、恢复超时（如 10s）
  • 降级开关配置化：通过配置中心动态开关，不需要发版即可降级
  • 定期做混沌工程演练：主动注入故障，验证限流降级熔断是否生效
  • 所有外部调用都必须有超时 + 重试 + 熔断三件套，缺一不可

秒杀请求 10万 QPS → MQ 削峰（缓冲队列） → 消费端 1万 QPS → DB 写入 ⇢ 异步通知用户

5.2 秒杀场景 MQ 削峰实现
Java / RocketMQ 
// 秒杀场景：MQ 削峰 + 异步处理

// ===== 生产者：接收秒杀请求，写入 MQ =====
@RestController
public class SeckillController {
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    @Autowired
    private RedisTemplate redisTemplate;
    
    @PostMapping("/seckill")
    @SentinelResource("seckill", blockHandler = "seckillBlock")
    public Result seckill(SeckillRequest req) {
        // 1. 原子扣减库存（Redis Lua 脚本）
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent("seckill:user:" + req.getUserId(), "1", 24, TimeUnit.HOURS);
        if (!success) return Result.fail("您已抢购过");
        
        Long stock = redisTemplate.opsForValue().decrement("seckill:stock:" + req.getGoodsId());
        if (stock == null || stock < 0) return Result.fail("已售罄");
        
        // 2. 发送 MQ 消息（异步创建订单）
        Message<SeckillOrder> msg = new Message<>(
            "seckill_topic",
            "create_order",
            new SeckillOrder(req.getGoodsId(), req.getUserId())
        );
        mqTemplate.asyncSend(msg, new SendCallback() {
            @Override public void onSuccess(SendResult r) { log.info("消息发送成功"); }
            @Override public void onFailure(Throwable e) { log.error("消息发送失败", e); }
        });
        
        // 3. 立即返回，告知用户"排队中"
        return Result.ok("正在排队处理，请稍候");
    }
}

// ===== 消费者：按 DB 承受能力消费消息 =====
@Component
@RocketMQMessageListener(
    topic = "seckill_topic",
    consumerGroup = "seckill_consumer_group",
    consumeMode = ConsumeMode.CONCURRENTLY,
    consumeThreadMax = 50          // 控制消费速度 = 削峰
)
public class SeckillConsumer implements RocketMQListener<SeckillOrder> {
    
    @Override
    public void onMessage(SeckillOrder order) {
        try {
            // 幂等检查（防止重复消费）
            if (orderService.existOrder(order.getUserId(), order.getGoodsId())) {
                return;
            }
            // 创建订单（DB 写入）
            orderService.createOrder(order);
        } catch (Exception e) {
            log.error("消费失败", e);
            throw e;  // 抛出后 RocketMQ 自动重试
        }
    }
}
5.3 消费者核心配置

消费速度控制

consumeThreadMax 控制最大消费线程数

pullBatchSize 控制每次拉取消息数

根据 DB 承受能力动态调整

幂等消费

MQ 保证 at-least-once，必须幂等

方案：唯一键去重 / 状态机检查 / Redis 标记

消费前检查，消费后标记

重试与死信

消费失败自动重试（默认 16 次）

超过重试次数进入死信队列

死信需人工介入或定时补偿

顺序消费

同一订单的创建/支付/发货需顺序

用 orderId 作为 sharding key

同一 key 路由到同一队列

⚠️ 常见误区
  • 不处理消息积压：生产速度远大于消费速度，积压越来越多。需监控积压量，达到阈值自动扩容消费者。
  • 忘记幂等：MQ 至少投递一次，不做幂等会导致重复下单/重复扣款。
  • 消费失败不重试：异常消息直接 ack 掉，数据永久丢失。应设重试 + 死信队列。
  • MQ 做强一致：把 MQ 当数据库用，期望消息不丢不重。MQ 的定位是「最终一致」，不是「强一致」。
  • topic 过多：每个业务场景建一个 topic，topic 数上千后 Kafka 性能骤降。应该用 tag 区分而非新建 topic。
✅ 实施要点
  • 削峰关键：生产者无阻塞写入 MQ，消费者按 DB 能力限速消费。队列长度 = 削峰缓冲区
  • 幂等三重保障：业务唯一键 + Redis 去重标记 + DB 唯一索引
  • 积压监控：积压量 > 1万条自动告警，> 10万条自动扩容消费者实例
  • 消息可靠性：生产端用同步发送 + 重试，消费端手动 ack，MQ 开启持久化
  • 灰度消费：新功能上线时，按比例消费消息，灰度验证后再全量

6 关键技术 III — 分布式锁

🔒 为什么需要分布式锁

单机时代用 synchronized 或 ReentrantLock 即可保证线程安全。但在分布式系统中，多个服务实例运行在不同机器上，本地锁失效——需要分布式锁来保证跨进程的互斥访问。

典型场景：库存扣减、防重复提交、定时任务防重、资源配额控制。

💡
选型建议：99% 的场景用 Redis + Redisson 即可（高性能 + 可重入 + WatchDog 续期）。对锁可靠性要求极高（如金融级）用 ZooKeeper。简单低频场景用 MySQL 行锁。
6.2 Redisson 分布式锁实现
Java / Redisson 
// Redisson 分布式锁：可重入 + 自动续期 + 公平锁

@Service
public class StockService {
    
    @Autowired
    private RedissonClient redisson;
    
    @Autowired
    private StockMapper stockMapper;
    
    public boolean deductStock(Long goodsId, Integer quantity) {
        String lockKey = "lock:stock:" + goodsId;
        RLock lock = redisson.getLock(lockKey);
        
        try {
            // 尝试加锁：最多等待 3 秒，锁自动过期 10 秒
            // 注意：如果不传 leaseTime，会启用 WatchDog 自动续期（默认 30s 续一次）
            boolean locked = lock.tryLock(3, 10, TimeUnit.SECONDS);
            if (!locked) {
                log.warn("获取锁失败，goodsId={}", goodsId);
                return false;
            }
            
            // === 临界区：扣减库存 ===
            Stock stock = stockMapper.selectByGoodsId(goodsId);
            if (stock.getQuantity() < quantity) {
                return false;  // 库存不足
            }
            stock.setQuantity(stock.getQuantity() - quantity);
            stockMapper.updateById(stock);
            return true;
            
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        } finally {
            // 确保释放锁（且锁是当前线程持有）
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}

// WatchDog 续期原理（不传 leaseTime 时自动启用）：
// 1. 默认锁过期时间 30 秒
// 2. 每 10 秒检查一次，如果锁还被当前线程持有，续期到 30 秒
// 3. 如果线程宕机，WatchDog 停止续期，锁 30 秒后自动释放
6.3 Redis 分布式锁的安全隐患
⚠️ 主从切换丢锁
  • 主节点加锁成功，同步到从节点前宕机
  • 从节点晋升为主，锁数据丢失
  • 其他线程可以再次获取同一把锁

解法：RedLock 算法（多节点过半数加锁）或用 ZK

⚠️ 锁过期 + 业务未完成
  • 业务执行时间超过锁过期时间
  • 锁自动释放，其他线程获取到锁
  • 两个线程同时执行临界区

解法：WatchDog 自动续期 / 合理设置过期时间

⚡
RedLock 争议：Redis 作者提出 RedLock 算法解决主从切换丢锁问题，但被 Martin Kleppmann 批评存在时钟漂移问题。实际工程中，大多数场景用 Redisson 单节点锁 + WatchDog 足够。如果业务对锁绝对敏感（如金融），用 ZooKeeper 或 etcd 更可靠。
⚠️ 常见误区
  • 用 SETNX 手写分布式锁：不处理锁续期、不可重入、无公平性。直接用 Redisson，不要自己造轮子。
  • 锁粒度过粗：锁整个方法而不是锁具体资源。如扣库存应锁 goodsId 而不是锁整个方法。
  • 忘记 finally 释放锁：异常导致锁未释放，其他线程永远获取不到锁。必须 try-finally。
  • 锁释放错对象：A 线程的锁过期后 B 获取到锁，A 执行完把 B 的锁释放了。必须检查 isHeldByCurrentThread()。
  • 用分布式锁替代数据库事务：锁只保证互斥，不保证原子性。锁内操作仍需事务。
✅ 实施要点
  • 锁粒度最小化：按具体资源加锁（如 lock:stock:{goodsId}），不要锁整个方法或类
  • 必须设超时：tryLock(waitTime, leaseTime, unit)，避免无限等待死锁
  • WatchDog 续期：不传 leaseTime 自动启用，确保业务未完成时锁不过期
  • 释放前检查持有者：lock.isHeldByCurrentThread() 防止释放他人的锁
  • 锁内逻辑尽量短：锁内不要做 RPC 调用、IO 操作，只做核心互斥逻辑

7 关键技术 IV — 最终一致性方案

🔄 为什么不用强一致

分布式系统中，强一致（2PC/XA）代价极高：性能差、可用性低、实现复杂。CAP 定理告诉我们，网络分区不可避免时，只能在 C 和 A 之间取舍。高并发系统几乎都选择 AP + 最终一致性，用异步补偿保证数据最终正确。

7.1 最终一致性五大方案

① 本地消息表

业务 + 消息写入同一事务，定时扫描发送

可靠 简单

适合简单异步场景

② 事务消息

RocketMQ 半消息 + 本地事务回查

可靠 高性能

适合电商交易

③ TCC

Try-Confirm-Cancel 三阶段补偿

强一致 复杂

适合资金交易

④ Saga

长事务拆分为多个子事务 + 补偿

灵活 补偿复杂

适合跨服务长流程

⑤ 最大努力通知

消息投递 + 定时重试 + 对账兜底

简单 不保证

适合通知类场景

⑥ 对账补偿

定时跑批比对数据差异并修复

兜底 延迟高

所有方案的最终防线

7.2 RocketMQ 事务消息实现
1.发送半消息 → 2.执行本地事务 → 3.提交/回滚 ⇢ 4.事务回查 → 5.消费者消费
Java / RocketMQ Transaction 
// RocketMQ 事务消息：保证本地事务与消息发送的最终一致

// ===== 事务生产者 =====
@Service
public class OrderTransactionService {
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    @Autowired
    private OrderService orderService;
    
    public void createOrder(OrderRequest req) {
        Message msg = MessageBuilder.withPayload(new OrderEvent(req))
            .setHeader("bizId", req.getBizId())
            .build();
        
        // 发送事务消息
        mqTemplate.sendMessageInTransaction("order_topic", msg, req);
    }
}

// ===== 事务监听器：执行本地事务 + 事务回查 =====
@RocketMQTransactionListener
public class OrderTransactionListener implements RocketMQLocalTransactionListener {
    
    @Autowired
    private OrderService orderService;
    
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        OrderRequest req = (OrderRequest) arg;
        try {
            // 执行本地事务（创建订单 + 扣库存）
            orderService.createOrderAndDeductStock(req);
            return RocketMQLocalTransactionState.COMMIT;    // 提交：消息可被消费
        } catch (Exception e) {
            log.error("本地事务执行失败", e);
            return RocketMQLocalTransactionState.ROLLBACK;   // 回滚：消息丢弃
        }
    }
    
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 事务回查：检查本地事务是否执行成功
        String bizId = msg.getProperty("bizId");
        Order order = orderService.getByBizId(bizId);
        
        if (order != null && order.getStatus() == OrderStatus.CREATED) {
            return RocketMQLocalTransactionState.COMMIT;    // 事务已执行，提交消息
        }
        if (order != null && order.getStatus() == OrderStatus.FAILED) {
            return RocketMQLocalTransactionState.ROLLBACK;   // 事务失败，回滚消息
        }
        return RocketMQLocalTransactionState.UNKNOWN;      // 未知，等待下次回查
    }
}
7.3 TCC 补偿模式
阶段动作示例（转账）失败处理
Try预留资源（冻结）冻结转出方 100 元释放冻结
Confirm确认执行（扣款）扣除冻结的 100 元重试直到成功
Cancel取消执行（解冻）解冻 100 元重试直到成功
💡
TCC 三大约束：
① 幂等：Confirm 和 Cancel 可能被重试，必须幂等
② 空回滚：Try 未执行但 Cancel 被调用，需检测并跳过
③ 悬挂：Cancel 先于 Try 到达，Try 后不再执行（需记录 Cancel 状态）
⚠️ 常见误区
  • 过度追求强一致：用 2PC/XA 做分布式事务，性能下降 10 倍以上。99% 场景最终一致就够了。
  • 不做对账：只依赖 MQ 保证一致性，没有对账兜底。MQ 也有极低概率丢消息。
  • 补偿逻辑不可靠：补偿操作本身失败后没有重试机制。补偿必须可重试 + 幂等。
  • 忽略幂等：消费者收到重复消息后重复处理数据。所有消费逻辑都必须幂等。
  • 事务范围过大：把多个无关操作放在一个事务里，任一失败全部回滚。应按业务边界拆分。
✅ 实施要点
  • 选型原则：能用事务消息就不用 TCC，能用 TCC 就不用 Saga，复杂度递增
  • 所有异步操作必须幂等：唯一键 + 状态机 + Redis 去重三重保障
  • 建立对账系统：T+1 跑批比对，发现差异自动补偿 + 人工告警
  • 记录操作日志：每一步操作记录流水，方便回溯和补偿
  • 设计补偿顺序：正向操作和补偿操作配对，按逆序执行补偿

⚠️ 常见误区
  • 监控指标太多：Dashboard 上几百个指标，看不出重点。应该用 USE/RED 方法论筛选核心指标。
  • 告警风暴：一个 DB 慢查询触发 50 条告警。应做告警收敛和聚合，同一故障只通知一次。
  • 日志无结构：日志全用字符串拼接，无法检索。必须结构化日志（JSON）+ TraceId。
  • 日志级别滥用：正常流程打 ERROR，异常不打日志。日志级别应严格遵循：ERROR=需人工介入、WARN=自动恢复、INFO=关键节点、DEBUG=调试用。
  • 没有业务监控：只有技术指标（CPU/QPS），没有业务指标（订单量/支付成功率）。业务指标往往比技术指标更早发现问题。
✅ 实施要点
  • RED 方法论：每个服务监控 Rate（请求率）、Errors（错误率）、Duration（延迟）三个核心指标
  • USE 方法论：每个资源监控 Utilization（利用率）、Saturation（饱和度）、Errors（错误）
  • 黄金信号：延迟、流量、错误、饱和度——Google SRE 四大黄金信号
  • 告警分级：P0（电话+短信）、P1（短信+IM）、P2（IM）、P3（邮件），不同级别不同响应
  • TraceId 全链路贯穿：从网关到 DB，所有日志都带 TraceId，一键串联全链路
  • 大盘 + 告警 + 日志联动：Grafana 大盘看到异常 → 告警通知 → 点击跳转 Kibana 查日志 → SkyWalking 看链路

9 演进路线 — 从单机到十万级并发

🗺️ 高并发不是一蹴而就，是逐步演进

所有高并发系统都是从单机开始的。正确的演进节奏比一步到位更重要——过早引入复杂度会拖死团队，过晚优化会错失业务机会。以下是一条经过实战验证的演进路线图。

阶段演进路线图
阶段QPS 量级架构形态关键动作团队规模
阶段 1 0 ~ 1,000 单体应用 + 单库 关注业务功能实现，单体足矣 3-5 人
阶段 2 1,000 ~ 10,000 单体 + 读写分离 + Redis 缓存 引入缓存层、MySQL 主从、CDN 静态加速 5-10 人
阶段 3 10,000 ~ 50,000 垂直拆分 + 负载均衡 + MQ 按业务垂直拆分、Nginx 负载均衡、MQ 异步化 10-20 人
阶段 4 50,000 ~ 100,000 微服务 + 分库分表 + 限流降级 服务化拆分、ShardingSphere 分片、Sentinel 限流 20-50 人
阶段 5 100,000+ 云原生 + 多活 + 单元化 K8s 容器化、异地多活、单元化部署、Service Mesh 50+ 人
单体 → +缓存+读写分离 → +垂直拆分+MQ → +微服务+分片+限流 → +多活+单元化
每个阶段的核心原则

📌 阶段 1-2：快速验证

核心原则：用最简单的技术栈快速验证业务模式

技术栈：Spring Boot + MySQL + Redis

禁忌：不要过早微服务化、不要引入 MQ

优化重点：SQL 优化 + 索引 + 合理缓存

📌 阶段 3：系统化优化

核心原则：开始面向高并发做架构调整

技术栈：+ Nginx + RocketMQ + ELK

禁忌：不要过度拆分微服务

优化重点：异步化 + 缓存策略 + 读写分离

📌 阶段 4：分布式架构

核心原则：全面分布式化，建立稳定性保障

技术栈：+ ShardingSphere + Sentinel + SkyWalking

禁忌：不要忽视监控和运维体系建设

优化重点：分库分表 + 限流降级 + 全链路监控

📌 阶段 5：极致弹性

核心原则：面向极端流量做容灾和弹性

技术栈：+ K8s + 异地多活 + Service Mesh

禁忌：不要盲目追求技术先进性

优化重点：多活容灾 + 单元化 + 自动化运维

🎯 总结：高并发系统设计的核心心法

第一步量化：用数字说话，QPS/数据量/延迟，不拍脑袋
第二步分层：接入层 → 应用层 → 数据层，逐层消化并发
第三步缓存：多级缓存是高并发第一利器，命中率 > 95% 是目标
第四步异步：MQ 削峰填谷，把同步链路缩短到极致
第五步保护：限流 + 降级 + 熔断，系统自我保护三件套
第六步监控：Metrics + Logging + Tracing，可观测性是底线
第七步演进：按需迭代，不过度设计，也不欠技术债
🎯
最后一句话：高并发系统设计的本质不是堆砌技术，而是在正确的阶段做正确的事。10万 QPS 不是目标，而是在业务驱动下自然演进的结果。保持架构的简洁性和可演进性，比追求某个技术指标更重要。
