Docker · 01

核心概念：镜像、容器、仓库

Docker 的世界由三个核心概念构建 —— 理解它们之间的关系，就理解了 Docker 的全部基础。

📦 镜像 (Image)

镜像是容器的模板 —— 一个只读的、分层的文件系统，包含了运行应用所需的一切：代码、运行时、库、环境变量、配置文件。

  • 分层存储 (Layered FS)：每条指令产生一层，层与层之间共享
  • 不可变 (Immutable)：构建后不可修改，修改 = 新建层
  • 通过 Dockerfile 定义构建步骤
  • 通过命名标签 (Tag) 管理版本，如 nginx:1.25-alpine

🏃 容器 (Container)

容器是镜像的运行实例 —— 在镜像之上添加一个可写层，分配独立的进程空间、网络和文件系统，形成一个轻量级的隔离环境。

  • 每个容器有独立 PID、网络栈、挂载点
  • 启动速度秒级（vs 虚拟机分钟级）
  • 一个镜像可以同时运行多个容器实例
  • 容器停止后可写层数据丢失 —— 需用 Volume 持久化

🏛️ 仓库 (Registry)

仓库是集中存储和分发镜像的服务。开发者将构建好的镜像 push 到仓库，部署时从仓库 pull 镜像到目标机器运行。

  • Docker Hub：官方公共仓库，类似 GitHub for images
  • Harbor：企业级私有仓库，支持权限管理与镜像扫描
  • 云厂商：AWS ECR / 阿里云 ACR / 腾讯云 TCR
  • 一个仓库 (Repository) 下通过 Tag 管理多个版本

Docker · 02

容器 vs 虚拟机：为什么选择容器化？

容器不是虚拟机的替代品，而是不同层次的隔离方案。理解差异才能正确选型。

架构对比：虚拟机 vs Docker 容器

🚀 启动速度

容器启动只需秒级（直接复用宿主机内核），虚拟机需要启动完整 OS，分钟级。100 个容器 vs 100 个 VM 的启动差距是质变的。

💾 资源效率

容器共享宿主机内核，无需 Guest OS 开销。单机可运行数百容器，而 VM 通常只能跑十几个。资源利用率提升 5-10 倍。

📦 一致性

「在我机器上能跑」的问题被彻底消灭。开发、测试、生产环境完全一致 —— 因为运行的是同一个镜像。

🔄 版本管理

镜像是不可变的、分层的、可标签的。回滚 = 切换 Tag。版本管理像 Git 一样自然。

🏗️ 微服务友好

每个微服务一个容器，独立构建、部署、扩缩容。容器天然适配微服务架构的拆分粒度。

⚡ CI/CD 加速

构建即镜像，部署即拉取。Docker 让持续交付流水线更简洁、更可靠、更快速。

Docker · 04

Dockerfile 编写指南

Dockerfile 是镜像的源代码 —— 一组按顺序执行的指令，每条指令构建一层。

Dockerfile 
# ===== 多阶段构建：Node.js 应用 =====

# --- 阶段1: 构建阶段 ---
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --production
COPY . .
RUN npm run build

# --- 阶段2: 运行阶段 (最终镜像) ---
FROM node:20-alpine AS runtime
WORKDIR /app

# 安装 dumb-init 实现信号转发 (PID 1 问题)
RUN apk add --no-cache dumb-init

# 创建非 root 用户运行应用
RUN addgroup -S app && adduser -S app -G app

# 只拷贝构建产物 (不带 node_modules 源码)
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json .

USER app
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

ENTRYPOINT ["dumb-init", "--"]
CMD ["node", "dist/index.js"]

FROM

指定基础镜像。优先选 alpine 变体（~5MB），其次 slim。多阶段构建时用 AS 命名阶段。

COPY vs ADD

优先用 COPY（语义明确）。ADD 仅在需要自动解压 tar 或从 URL 拉取时使用。

RUN 优化

多条命令用 && 连接为一层。apt/apk 后加 --no-cache 避免缓存残留。

.dockerignore

类似 .gitignore，排除 node_modules、.git、测试文件等，避免构建上下文过大和缓存失效。

CMD vs ENTRYPOINT

ENTRYPOINT 固定入口程序，CMD 提供默认参数。组合使用实现「可覆盖参数」的灵活设计。

多阶段构建

构建工具不进入最终镜像。一个 Node 应用从 900MB → 80MB 是常见优化效果。

⚓ Kubernetes · 容器编排平台

K8s · 01

K8s 集群架构总览

K8s 集群由 控制平面 (Control Plane) 和 工作节点 (Worker Node) 组成 —— 前者是「大脑」负责决策，后者是「手脚」负责执行。

Kubernetes 集群架构全景图

🧠 控制平面组件

kube-apiserver

集群唯一入口，所有组件都通过它通信。提供 RESTful API，负责认证、授权、准入控制。kubectl 命令本质就是调用 API Server。

etcd

高可用分布式键值存储，保存集群所有状态数据（Pod/Service/ConfigMap 等）。Raft 协议保证一致性。是 K8s 唯一的有状态组件。

kube-scheduler

负责决定 Pod 运行在哪个 Node 上。评估资源请求、亲和性规则、污点容忍、数据卷拓扑等约束，做出最优调度决策。

controller-manager

运行各种控制器的进程。核心思想：不断观察实际状态，驱动它向期望状态收敛 (Reconcile Loop)。

💪 工作节点组件

kubelet

每个 Node 上的代理。接收 API Server 下发的 Pod 规格，调用容器运行时创建/销毁容器，并定期汇报节点状态和资源使用。

kube-proxy

维护节点上的网络规则。实现 Service 的负载均衡和流量转发，底层使用 iptables 或 IPVS。

Container Runtime

负责拉取镜像、创建/运行容器的引擎。K8s 通过 CRI 接口与之交互。主流：containerd、CRI-O（已弃用 Docker Shim）。

K8s · 02

核心对象模型

K8s 通过「声明式 API」管理对象 —— 你告诉系统「期望的状态」，K8s 负责驱动实际状态向期望状态收敛。

🫛 Pod 最小调度单元

Pod 是 K8s 中最小可部署单元，不是容器。一个 Pod 可以包含一个或多个紧密协作的容器，它们共享网络和存储命名空间。

  • Pod 内容器共享 IP 地址和端口空间
  • Pod 内容器通过 localhost 互相通信
  • Pod 是临时的 —— 重建后 IP 会变
  • Sidecar 模式：主容器 + 辅助容器（日志/代理/监控）
  • 每个 Pod 含一个 pause 容器维持命名空间

🔁 Deployment 最常用

Deployment 管理 ReplicaSet，ReplicaSet 管理 Pod。它提供了声明式的 Pod 副本管理、滚动更新和回滚能力。

  • 声明期望副本数，Controller 自动维持
  • 滚动更新 (RollingUpdate)：逐个替换 Pod 实现零停机
  • 版本回滚：kubectl rollout undo
  • 支持暂停/恢复 rollout
  • 金丝雀发布：通过多 Deployment + 标签实现

🌐 Service 网络抽象

Service 为一组功能相同的 Pod 提供固定的访问入口和负载均衡。Pod 的 IP 是易变的，Service 的 IP 是稳定的。

  • ClusterIP：集群内部访问（默认）
  • NodePort：在节点上开放端口供外部访问
  • LoadBalancer：云厂商提供外部负载均衡器
  • 通过 Label Selector 关联后端 Pod
  • Ingress：七层路由，基于域名/路径分发流量

📄 ConfigMap

非敏感配置数据存储，将配置与镜像解耦。

  • ::before 存储 key-value 或文件内容
  • ::before 通过环境变量或 Volume 注入
  • ::before 修改不需重新构建镜像
  • ::before 明文存储 —— 不放密码！
yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "debug"
  MAX_CONN: "1000"
  config.yaml: |
    server:
      port: 8080

🔒 Secret

敏感数据存储（密码、Token、证书），Base64 编码。

  • ::before Base64 编码（非加密！）
  • ::before 可配合 RBAC 限制访问
  • ::before 类型：Opaque / tls / docker-registry
  • ::before 生产环境需配合 KMS/Sealed Secrets
yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  password: cGFzczEyMw==
  # echo -n "pass123" | base64

其他重要对象

📊

Namespace

逻辑隔离单位。一个集群划分为多个命名空间，用于环境隔离（dev/staging/prod）或团队隔离。资源配额可按 NS 设置。

💾

PV / PVC

PersistentVolume (存储资源) + PersistentVolumeClaim (存储申请)。将存储与 Pod 生命周期解耦，Pod 重建后数据仍在。

⚡

HPA

HorizontalPodAutoscaler：根据 CPU/内存/自定义指标自动扩缩 Pod 副本数。流量大时自动扩容，空闲时自动缩容。

📦

StatefulSet

管理有状态应用。每个 Pod 有稳定身份（名称/序号）和独立存储。适用于数据库、消息队列等需要持久化标识的场景。

🕐

CronJob

定时任务，类似 Linux crontab。按 Cron 表达式周期性创建 Pod 执行任务，如数据备份、报表生成、清理脚本。

🚪

Ingress

七层路由入口。基于域名/路径将外部流量路由到集群内 Service。需要 Ingress Controller（如 nginx-ingress）。

K8s · 03

声明式部署：YAML 实战

K8s 的核心理念是「声明式」—— 你用 YAML 描述期望状态，K8s 负责实现。以下是完整的部署示例。

deployment.yaml 
# ===== Deployment: 管理 Pod 副本 =====
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web
spec:
  replicas: 3                    # 期望 3 个 Pod 副本
  selector:
    matchLabels:
      app: web
  strategy:                       # 滚动更新策略
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1                # 最多多创建 1 个
      maxUnavailable: 0          # 不允许减少 (零停机)
  template:                       # Pod 模板
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: registry.cn/myapp:2.1.0
        ports:
        - containerPort: 8080
        resources:                    # 资源限制
          requests:                   # 最低保障
            cpu: "100m"            # 0.1 核
            memory: "128Mi"
          limits:                     # 上限
            cpu: "500m"
            memory: "512Mi"
        env:                          # 环境变量
        - name: REDIS_HOST
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: redis_host
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password
        readinessProbe:               # 就绪探针
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:                # 存活探针
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 15
          periodSeconds: 20
---
# ===== Service: 提供稳定访问入口 =====
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  type: ClusterIP
  selector:
    app: web                      # 关联上面的 Pod
  ports:
  - port: 80                      # Service 端口
    targetPort: 8080             # Pod 端口
---
# ===== HPA: 自动扩缩容 =====
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70   # CPU 超 70% 自动扩容
💡 声明式 vs 命令式

命令式：「帮我启动 3 个 nginx 容器」—— 你关心怎么做。
声明式：「我需要 3 个 nginx Pod 在运行」—— 你只关心结果，K8s 负责实现和维持。当某个 Pod 挂了，K8s 会自动创建新的来补足 3 个 —— 这就是 自愈。

综合

Docker 与 Kubernetes 的关系

Docker 解决「如何打包和运行一个应用」，K8s 解决「如何管理成百上千个应用」—— 它们是互补关系，不是替代关系。

🐳 Docker

容器化工具 —— 打包、运行单个容器

  • 定位：容器运行时 + 镜像构建工具
  • 能力：构建镜像、运行容器、端口映射、数据卷
  • 规模：单机或少量机器 (docker-compose)
  • 编排：docker-compose (单机多容器)
  • 网络：bridge/host/overlay (基本)
  • 适用：开发、测试、CI、小型部署

⚓ Kubernetes

容器编排平台 —— 管理大规模容器集群

  • 定位：分布式系统编排平台
  • 能力：调度、自愈、扩缩容、滚动发布、服务发现
  • 规模：数百节点、数万容器
  • 编排：原生集群级编排
  • 网络：CNI 插件 (Calico/Cilium)、Service Mesh
  • 适用：生产环境、大规模微服务

从开发到生产：Docker + K8s 协作流程

✓ 一句话总结

Docker 是集装箱 —— 把应用打包成标准化、可运输的单元。K8s 是港口调度系统 —— 管理成千上万个集装箱的装卸、堆放、运输、故障处理。没有集装箱就没有现代航运，没有调度系统就无法管理大规模航运。

K8s · 04

Pod 生命周期与自愈机制

理解 Pod 从创建到销毁的完整流程，是掌握 K8s 编排能力的关键。

1

用户提交 YAML

kubectl apply -f deployment.yaml → 请求发送到 API Server，经过认证/鉴权/准入后写入 etcd。

2

Controller 创建 Pod

Deployment Controller 监听到新 Deployment，创建 ReplicaSet → ReplicaSet 创建 Pod 对象（此时 Pod 状态为 Pending）。

3

Scheduler 调度

Scheduler 评估所有节点的资源、亲和性、污点等约束，选择最合适的 Node，将 Pod 绑定到该节点（Pod 仍为 Pending）。

4

kubelet 拉起容器

目标节点的 kubelet 监听到 Pod 绑定，调用容器运行时拉取镜像、创建容器、挂载卷。Pod 状态变为 Running。

5

探针检查与就绪

readinessProbe 通过后，Pod 变为 Ready，Service Endpoints 加入该 Pod，开始接收流量。

6

自愈与恢复

如果 Pod 崩溃、节点宕机或健康检查失败，Controller 自动在健康节点上重新创建 Pod 替换故障实例 —— 这就是 K8s 的自愈能力。
