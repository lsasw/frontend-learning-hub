Module 01

架构设计原则

设计原则是架构决策的底层价值观。它们不是教条，而是在约束条件下做权衡时的指南针 —— 当两条路都看起来可行时，原则告诉你应该偏向哪一边。

🎯

KISS — Keep It Simple, Stupid

简单是架构的第一美德。每增加一层抽象都在增加认知成本和故障面。在满足需求的前提下，选择最简单的方案。

简化认知负载
♻️

DRY — Don't Repeat Yourself

每一个知识点在系统中应当有唯一、明确、权威的表示。重复不仅浪费工时，更会导致修改时的不一致风险。

去重一致性
✂️

YAGNI — You Aren't Gonna Need It

不要为「将来可能需要」的功能提前设计。过度设计比设计不足更危险 —— 它冻结了当前灵活性，增加了维护成本。

反过度设计精益
📐

SoC — Separation of Concerns

关注点分离：每个模块只做一件事，且做得好。这是分层架构、微服务拆分的理论根基，也是单一职责的宏观版本。

模块化解耦
🧩

DDD — Domain-Driven Design

以业务领域为核心驱动设计。通过统一语言、限界上下文、聚合根等概念，让架构与业务模型高度对齐。

领域建模统一语言
🔄

Convention over Configuration

约定优于配置。通过合理的默认约定减少决策疲劳，只在偏离约定时才需要显式配置。Rails/Spring Boot 的哲学。

约定效率

SOLID 原则详解

S

Single Responsibility

单一职责原则

一个类/模块应该只有一个被修改的原因。如果一个类承担了多个职责，任何一个职责的变化都会影响其他职责的稳定性。

O

Open-Closed

开闭原则

对扩展开放，对修改关闭。通过抽象和多态实现新功能的扩展，而非修改已有代码。这是稳定 API 设计的基石。

L

Liskov Substitution

里氏替换原则

子类型必须能够替换其基类型而不破坏程序正确性。继承不是代码复用的工具，而是「is-a」契约的表达。

I

Interface Segregation

接口隔离原则

不应强迫客户端依赖它不使用的方法。多个细粒度接口优于一个胖接口 —— 类似微服务的「小而专」理念。

D

Dependency Inversion

依赖倒置原则

高层模块不应依赖低层模块，二者都应依赖抽象。依赖注入、六边形架构、Clean Architecture 的理论根基。

CAP 定理与 PACELC 扩展

在分布式系统中，一致性 (Consistency)、可用性 (Availability)、分区容错性 (Partition tolerance) 三者不可同时满足。网络分区是客观存在的，因此实际选择是 CP 还是 AP。

CP — 强一致性

分区时拒绝不一致的写入，牺牲可用性。适用于金融交易、库存扣减等场景。代表：ZooKeeper, etcd, HBase。

AP — 高可用性

分区时允许各节点继续服务，接受最终一致性。适用于社交 Feed、推荐等场景。代表：Cassandra, DynamoDB, Eureka。

⚡ PACELC 定理 — CAP 的进阶

如果发生分区 (P)，则在 A 和 C 之间选择；否则 (E)，在 L (延迟) 和 C 之间选择。即：PA/EL 系统（如 Cassandra）优先延迟和可用性，PC/EC 系统（如 Spanner）优先一致性。

Module 02

常见架构模式

架构模式是针对特定上下文中反复出现的问题的标准化解决方案。每种模式都有其适用边界和代价 —— 没有「最好的」模式，只有「最合适的」。

01

分层架构 (Layered / N-Tier) 经典

最通用的架构模式，将系统按职责水平切分为若干层

核心特征

  • 每层只与相邻层通信（封闭层）或跨层通信（开放层）
  • 上层依赖下层，下层不感知上层
  • 关注点分离的直观体现
✓ 优势
  • 简单易懂，上手快
  • 职责清晰，易于测试
  • 标准化程度高
✗ 劣势
  • 单体膨胀风险
  • 层间性能开销
  • 修改穿透多层
02

微服务架构 (Microservices) 热门

将系统拆分为一组小型、自治的服务，各自独立部署和演进

核心特征

  • 每个服务拥有独立数据库 (Database per Service)
  • 通过 API 或消息队列通信，不共享内存
  • 独立部署、独立扩缩容、独立技术栈
  • 围绕业务能力组织 (Conway's Law)
✓ 优势
  • 独立部署与扩展
  • 技术栈自由
  • 故障隔离
✗ 劣势
  • 分布式事务复杂
  • 运维成本高
  • 服务间调用延迟
03

事件驱动架构 (Event-Driven) 热门

系统组件通过生产/消费事件进行异步通信，实现松耦合

核心特征

  • 生产者不关心谁消费事件（-fire-and-forget）
  • 消费者订阅感兴趣的事件类型
  • 支持事件溯源 (Event Sourcing)
  • 天然解耦，易于增加新消费者
✓ 优势
  • 极高解耦度
  • 异步高吞吐
  • 易于水平扩展
✗ 劣势
  • 调试链路复杂
  • 事件顺序保证难
  • 最终一致性延迟
04

CQRS — 命令查询职责分离 进阶

将写入操作 (Command) 与读取操作 (Query) 分离到不同的模型和路径

核心特征

  • 写模型关注业务规则与一致性，读模型关注查询效率
  • 常与事件溯源 (Event Sourcing) 结合使用
  • 读写可以使用不同的数据存储引擎
  • 写侧通过事件同步到读侧
✓ 优势
  • 读写各自优化
  • 独立扩展能力
  • 审计日志天然支持
✗ 劣势
  • 架构复杂度倍增
  • 读写最终一致延迟
  • 学习曲线陡峭
05

六边形架构 (Hexagonal / Ports & Adapters) 进阶

将业务逻辑置于中心，通过端口与外部适配器交互，实现技术无关性

核心特征

  • 领域逻辑位于六边形中心，完全隔离技术细节
  • 端口 (Port) = 接口定义；适配器 (Adapter) = 接口实现
  • 驱动端口 (Driving)：外部调用领域（如 REST Controller）
  • 被驱动端口 (Driven)：领域调用外部（如 Repository）
✓ 优势
  • 领域逻辑高度可测
  • 技术替换成本低
  • DDD 实践的理想载体
✗ 劣势
  • 接口设计要求高
  • 简单系统过度设计
  • 间接调用有开销
06

管道-过滤器架构 (Pipe & Filter)

数据流经一系列处理过滤器，每个过滤器执行特定变换

核心特征

  • 每个过滤器独立工作，不共享状态
  • 管道负责数据传递，可以是同步或异步
  • 过滤器可自由组合、重用、重排
  • Unix 管道、ETL 管线、编译器流水线的理论模型
✓ 优势
  • 高度可组合
  • 过滤器可并行执行
  • 易于增删处理步骤
✗ 劣势
  • 错误处理传播难
  • 数据转换开销
  • 不适合交互式场景
07

MVC 架构模式 经典

Model-View-Controller，将用户界面逻辑与业务数据分离

核心特征

  • Controller 负责接收用户输入并调度
  • Model 封装业务状态与数据
  • View 负责将 Model 渲染为用户可感知的界面
  • MVP、MVVM 是 MVC 的演进变体
✓ 优势
  • UI与逻辑分离
  • 可独立测试 Model
  • 多视图共享模型
✗ 劣势
  • Controller 易膨胀
  • 紧密耦合风险
  • 大规模时维护困难
08

空间架构 (Space-Based) 高并发

通过内存网格消除数据库瓶颈，实现极高并发与低延迟

核心特征

  • 数据驻留在内存网格中，非数据库
  • 每个处理单元 (PU) 同时包含计算和数据副本
  • 通过数据复制实现高可用，而非持久化
  • 异步写入数据库做持久化备份
✓ 优势
  • 极高吞吐量
  • 超低延迟
  • 线性扩展
✗ 劣势
  • 内存成本高
  • 数据一致性挑战
  • 调试复杂

Module 03

系统设计关键质量属性

质量属性是架构师与利益相关者之间的契约语言。它将模糊的「好系统」转化为可度量、可测试、可权衡的具体指标。

⚡

可用性 Availability

系统在需要时可正常服务的概率

通过冗余部署、健康检查、故障转移、灰度发布等手段保障。核心指标是「几个9」。

99.9% → 8.76h/年宕机 99.99% → 52.6min/年 99.999% → 5.26min/年
📈

可扩展性 Scalability

系统在负载增长时保持性能的能力

水平扩展（加机器）vs 垂直扩展（加配置）。无状态服务易于水平扩展，有状态需分片。

水平扩展 垂直扩展 分片/Sharding 读写分离
🚀

性能 Performance

响应时间、吞吐量、资源利用率

延迟 (Latency) 与吞吐 (Throughput) 是核心指标。缓存、异步化、批量处理是常见优化手段。

P99 延迟 QPS/TPS 缓存策略 连接池
🔒

安全性 Security

机密性、完整性、可用性 (CIA)

从认证授权到数据加密，从网络隔离到审计追踪。安全是纵深防御体系，不是单一措施。

OAuth 2.0 TLS/mTLS 零信任 最小权限
🔧

可维护性 Maintainability

修复缺陷、增加功能、适应变化的成本

模块化、低耦合、代码可读性、文档完备性。技术债管理是长期可维护性的关键。

模块化 自动化测试 CI/CD 技术债管理
👁️

可观测性 Observability

从外部行为推断内部状态的能力

三大支柱：Metrics（指标）、Logging（日志）、Tracing（链路追踪）。可观测性是运维效率的基石。

Metrics Logging Tracing 告警体系
⚖️ 质量属性之间的权衡

质量属性之间往往存在张力：强一致性 vs 高可用性（CAP）、性能 vs 安全性（加密开销）、可扩展性 vs 简单性（分布式复杂度）。架构师的核心能力就是在特定业务上下文中找到最优的平衡点，而非追求所有属性最大化。

垂直扩展 vs 水平扩展

Module 04

技术选型考量维度

技术选型不是选「最好的」技术，而是选「最合适的」技术。每个选型决策都是在团队能力、业务需求、技术生态、成本约束之间的多维博弈。

① 业务匹配度

技术是否真正解决当前业务问题？是否过度或不足？业务场景的读写比、一致性要求、延迟容忍度是首要考量。

② 成熟度与生态

生产验证程度、社区活跃度、文档质量、周边工具链。技术栈的生态决定了你能「站在多高的肩膀上」。

③ 团队熟悉度

团队的学习曲线和已有经验。引入全新技术栈的隐性成本远高于表面许可费用 —— 培训、试错、招聘都需算入。

④ 性能与资源

在目标负载下的吞吐、延迟表现。资源占用（CPU/内存/磁盘/网络）直接影响基础设施成本。

⑤ 运维成本

部署复杂度、监控支持、故障恢复、升级路径。一个难以运维的「先进」技术是负债而非资产。

⑥ 许可与成本

开源协议 (MIT/Apache/GPL)、商业许可费用、云服务定价模型。合规风险不可忽视。

Module 05

架构决策流程

好的决策过程比好的决策结果更重要 —— 因为上下文会变，但可重复的决策方法能持续产出合理结果。ADR (Architecture Decision Record) 是沉淀决策上下文的最佳实践。

1

识别问题与上下文

明确要解决的架构问题是什么，它的业务背景、技术约束、利益相关者是谁。这一步决定了后续所有分析的边界。

业务背景 约束识别 利益相关者
2

分析约束与质量属性

列出硬约束（预算、合规、已有系统）和软约束（团队偏好、时间窗口）。确定哪些质量属性是必须满足的，哪些是可以妥协的。

硬约束 软约束 质量优先级
3

探索候选方案

至少生成 2-3 个候选方案（包含「不做」选项）。避免只考虑一个方案就做决定 —— 没有比较就没有权衡。用 Spike/PoC 验证高风险假设。

多方案生成 Spike 验证 PoC 原型
4

评估与权衡分析

用评估矩阵量化对比各方案在不同维度上的表现。明确每个方案的代价和风险。关注「不可逆决策」(Type 1) vs 「可逆决策」(Type 2)，前者需要更谨慎。

加权矩阵 风险评估 可逆性分析
5

决策与记录 (ADR)

做出最终决定并编写 ADR 文档。ADR 不是「证明我选对了」，而是「记录我当时为什么这么选」。包含上下文、决策、理由、后果。

ADR 文档 决策会议 利益相关者确认
6

实施与持续演进

架构决策不是一次性事件。随着上下文变化（业务增长、团队扩大、技术演进），需要定期回顾已有决策，必要时通过新的 ADR 进行修订或废弃。

定期回顾 演进式架构 废弃机制

ADR (架构决策记录) 模板

ADR 是轻量级的 Markdown 文档，每个文件记录一个架构决策。建议存放在代码仓库的 /docs/adr 目录下。

# ADR-001: 使用 Kafka 作为核心消息中间件 ## 状态 (Status) Accepted # Proposed | Accepted | Deprecated | Superseded by ADR-002 ## 上下文 (Context) # 描述问题背景、约束条件、业务场景 当前系统日均处理 5 亿条事件消息，峰值 QPS 达 10 万。 现有 RabbitMQ 在峰值期间出现消息积压，P99 延迟超过 2s。 业务场景包括：实时风控、订单事件流、日志收集。 团队有 2 名工程师具备 Kafka 运维经验。 ## 决策 (Decision) # 清晰陈述做了什么决定 选择 Apache Kafka 作为核心消息中间件， 替代现有 RabbitMQ 用于高吞吐事件流场景。 RabbitMQ 保留用于低延迟命令式消息（如通知推送）。 ## 理由 (Rationale) # 为什么做这个选择？权衡了什么？ - 吞吐量：Kafka 在 10 万 QPS 下延迟稳定在 10ms 以内 ✓ - 持久化：Kafka 的 append-only log 模型天然支持事件回溯 ✓ - 生态：Kafka Connect / Streams 生态成熟，降低开发成本 ✓ - 代价：运维复杂度增加，需要专门的 Kafka 集群运维 ✗ ## 后果 (Consequences) # 这个决策带来的正面和负面影响 正面: - 解决消息积压问题，P99 延迟降至 < 50ms - 支持事件溯源，为后续 CQRS 改造奠定基础 负面: - 需要新增 Kafka 运维能力，预估 2 周学习曲线 - 消息顺序性保证仅在 partition 内，需调整消费者逻辑 - 基础设施成本增加约 30%（3 broker 集群） ## 合规性 (Compliance) # 如何验证决策是否被遵循 - 所有新事件流必须使用 Kafka，需在架构评审中确认 - RabbitMQ 仅保留通知场景，禁止新增事件流消费 - 每季度回顾一次本 ADR 的有效性
📖 Bezos 的两类决策

Type 1 决策（不可逆）：一进门就关上了门 —— 如核心数据库选型、主通信协议。需要充分分析、谨慎决策。Type 2 决策（可逆）：进出自由的门 —— 如缓存方案、前端框架。可以快速决策、快速试错、快速调整。架构师的关键能力之一是正确分类决策类型，避免对 Type 2 决策过度分析，也避免对 Type 1 决策草率行事。
