1 什么是消息队列？

消息队列（Message Queue）是一种异步通信中间件。生产者把消息放进队列，消费者从队列取出处理——两者不需要同时在线，也不需要知道对方的存在。

🔗

解耦

⚡

异步

耗时操作异步化，用户体验不阻塞：下订单后发短信、发邮件全扔给 MQ

🏔️

削峰

秒杀流量先入队列排队，后端按自己的节奏慢慢消费，数据库不会被冲垮

📖 一个经典的场景

用户下单后，系统需要：扣库存 → 创建订单 → 发短信通知 → 发优惠券 → 写操作日志。

如果没有 MQ：每一步都串行等待，用户等 3 秒才看到"下单成功"。

有了 MQ：主流程只做「扣库存 + 创建订单」→ 把"发短信、发券、写日志"三条消息丢进队列 → 立即返回"下单成功"。后台慢慢消费即可。

3 消息模型

📥 点对点（P2P / Queue）

每条消息只能被一个消费者处理。适合任务分发场景。

代表：RabbitMQ Queue、RocketMQ 普通消息

📢 发布/订阅（Pub/Sub / Topic）

每条消息可以被多个订阅者独立消费。适合广播场景。

代表：Kafka Topic、RocketMQ 广播模式

🧠 关键区别：P2P 是「一条消息一次消费」，Pub/Sub 是「一条消息多次消费」。Kafka 用 Consumer Group 把两者统一了——同组内 P2P，跨组 Pub/Sub。
4 消息可靠性：不丢、不重

消息可能在哪丢失？

→
Broker
宕机 / 磁盘满
→
消费端
自动 ACK / 异常吞没
📤

生产端可靠

同步发送 + Broker 落盘确认（如 RocketMQ 的 SYNC_FLUSH），不用异步发送

💾

Broker 持久化

多副本（Replica）+ 同步刷盘 + ISR 机制，确保消息写入多数节点

✅

消费端确认

处理完成后手动 ACK，不自动提交 offset，避免"消费了但没处理完"

三种投递语义

语义含义代价
At-most-once 最多一次，可能丢消息 最低，适合日志/metrics 等可丢失数据
At-least-once 至少一次，可能重复 中等，消费端需幂等处理，最常用
Exactly-once 精确一次，不丢不重 最高，需事务支持（如 Kafka 事务 / RocketMQ 事务消息）
⚠️ 实战经验：绝大多数场景做到 At-least-once + 消费端幂等 就足够了。Exactly-once 代价极高，只在金额交易等严格场景使用。
5 消费模式

Push 模式（推）

Broker 主动把消息推给 Consumer。

✅ 实时性好，延迟低
❌ 消费慢时会积压在 Consumer 内存，OOM 风险

代表：RabbitMQ、RocketMQ 集群模式

Pull 模式（拉）

Consumer 主动从 Broker 拉取消息。

✅ 消费端自主控速，不会被打垮
❌ 轮询间隔造成延迟，短间隔浪费 CPU

代表：Kafka、RocketMQ 广播模式

Consumer Group（消费者组）

同组内：各 Consumer 分摊不同分区，一条消息只被组内一个 Consumer 消费。

跨组间：不同 Consumer Group 之间互不影响，类似 Pub/Sub。

消费乱序 ✗

多分区并行消费 + 重试机制 = 全局消息天然乱序。

✅ 全局有序

只用一个分区，只有一个消费者。简单但吞吐量低。

⚠️ 一般只在严格顺序场景使用（如 MySQL binlog 同步）

✅ 分区有序

相同 Key 的消息路由到同一分区，保证局部有序。

💡 推荐做法：用 orderId 做 Key，同一订单的消息在同一个分区内有序
Java 
// RocketMQ：发送顺序消息
Message msg = new Message("OrderTopic", "order:1001", body);
// 相同 orderId → 同一 MessageQueue → 局部有序
SendResult result = producer.send(msg, new MessageQueueSelector() {
    public MessageQueue select(...) {
        return mqs[orderId.hashCode() % mqs.length];
    }
}, "order:1001");
7 高级特性
⏰

延迟消息

消息发出去后不立即投递，而是等指定时间后再推送给消费者。
场景：订单 30 分钟未支付自动取消

🔄

事务消息

先发半消息（half message），本地事务执行完后决定提交或回滚。
场景：下单 + 扣库存分布式事务

💀

死信队列（DLQ）

消息重试 N 次仍然失败后，进入死信队列等待人工处理。
场景：消息格式错误、下游服务长期不可用

8 常见问题 & 解决方案
▶ ❌ 消息丢失
丢失环节原因解法
生产端异步发送失败无感知同步发送 + 回调 / 事务消息
Broker异步刷盘，宕机丢数据同步刷盘 + 多副本
消费端自动 ACK，处理前就确认了手动 ACK，处理完再确认
▶ 🔄 消息重复消费

原因：At-least-once 语义下，消费成功但 ACK 超时，Broker 重复投递。

解法：消费端幂等

// 用唯一 ID 做幂等：数据库唯一键 / Redis SETNX
if (redis.setnx("msg:" + msgId, "1", 3600)) {
    // 首次消费，执行业务
    processOrder(msg);
} else {
    // 重复消息，直接跳过
    return ACK;
}
▶ 📊 消息堆积

原因：生产速度 > 消费速度，队列越积越多。

解法：

  • 临时增加 Consumer 实例数（水平扩容）
  • 消费者内部批量处理，减少 IO 开销
  • 将不重要消息降级（直接写日志跳过）
  • 长期方案：评估 Topic 分区数是否够用
▶ ⚡ 消息积压导致系统雪崩

消息堆积过多 → 消费者内存打满 → 消费者挂 → 堆积更多 → 死循环。

解法：流控 & 限流

  • 消费端限流（如 RocketMQ pullBatchSize）
  • 设置消息 TTL，过期自动丢弃
  • 监控报警：堆积量超过阈值立刻通知
