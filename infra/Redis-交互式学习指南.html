0Redis 核心架构总览

1数据类型 & 使用场景

String — 最基础的类型，二进制安全

最大 512MB。可以存字符串、整数、浮点数、二进制数据（图片/序列化对象）。底层用 SDS（Simple Dynamic String），预分配空间 + 惰性释放。

🔥 典型场景

  • 缓存对象：JSON 序列化后存入，读时反序列化
  • 计数器：INCR 原子自增，统计 PV/点赞数
  • 分布式锁：SET key value NX EX 30
  • Session 共享：分布式下统一存储 Session
  • 限流：INCR + EXPIRE 实现滑动窗口
Redis 
# 基本操作
SET user:1 '{"name":"张三","age":28}'
GET user:1                        # 获取
SET user:1 '{"name":"张三","age":29}' XX  # 存在才更新
SET lock:order:1001 1 NX EX 30    # 分布式锁

# 计数器
INCR article:view:10086            # 原子 +1
INCRBY user:score:1 50           # +50
DECR stock:sku12345                # 库存扣减

List — 有序可重复的字符串链表

底层用 QuickList（LinkedList + ZipList 混合），两端操作 O(1)。最多 2³²-1 个元素。

🔥 典型场景

  • 消息队列：LPUSH + RPOP / BRPOP（阻塞消费）
  • 最新列表：LPUSH + LTRIM 保持固定长度（如最新 100 条消息）
  • 时间线：按时间排序的动态流
  • 栈：LPUSH + LPOP
Redis 
LPUSH timeline "msg1" "msg2"      # 左侧插入
RPUSH queue "task1" "task2"       # 右侧插入
LPOP queue                        # 左侧弹出
BRPOP queue 5                    # 阻塞弹出，超时5秒
LRANGE timeline 0 9            # 取最新 10 条
LTRIM timeline 0 99           # 只保留前 100 条

Hash — 字段-值映射表

适合存储对象。小 Hash 用 ZipList 压缩存储（节省内存），大 Hash 用 HashTable。

🔥 典型场景

  • 对象存储：用户信息、商品详情（比 String JSON 更灵活 — 可部分修改）
  • 购物车：HSET cart:user1 sku123 3（用户ID → 商品ID → 数量）
  • 计数器组：文章的各维度统计（阅读、点赞、收藏）
Redis 
HSET user:1 name "张三" age 28  # 设置多字段
HGET user:1 name                 # 获取单字段
HMGET user:1 name age city     # 批量获取
HINCRBY user:1 age 1          # 原子自增
HGETALL user:1                 # 获取所有字段
HDEL user:1 city               # 删除字段

Set — 无序不重复集合

底层用 HashTable（值全为 NULL）或 IntSet。支持交/并/差集运算。

🔥 典型场景

  • 标签系统：文章标签去重
  • 共同好友：SINTER user:1:friends user:2:friends
  • 抽奖/随机：SRANDMEMBER / SPOP 随机抽取
  • 去重统计：独立访客 UV（比 HyperLogLog 精确但耗内存）
Redis 
SADD tags:article:1 "Redis" "Go" SADD tags:article:2 "Redis" "Java"
SINTER tags:article:1 tags:article:2  # 交集 → {"Redis"}
SUNION tags:article:1 tags:article:2  # 并集
SDIFF tags:article:1 tags:article:2   # 差集 → {"Go"}
SISMEMBER tags:article:1 "Redis"   # 是否存在
SCARD tags:article:1                  # 集合大小

ZSet (Sorted Set) — 有序不重复集合

每个成员关联一个 score（double 类型），按 score 排序。底层用 Skiplist + HashTable。

🔥 典型场景

  • 排行榜：游戏积分榜、文章热度榜（ZREVRANGE 取 Top N）
  • 延迟队列：score = 当前时间 + 延迟秒数，定时 ZRANGEBYSCORE
  • 带权重的去重集合：如搜索热词 + 搜索次数
  • 滑动窗口限流：score = 时间戳，ZREMRANGEBYSCORE 清理过期
Redis 
ZADD leaderboard 1000 "player1" 850 "player2"
ZINCRBY leaderboard 50 "player1"     # 加分
ZREVRANGE leaderboard 0 9 WITHSCORES # Top10
ZRANK leaderboard "player1"            # 排名
ZSCORE leaderboard "player1"           # 分数
ZREMRANGEBYSCORE delay 0 1620000000 # 移除过期

4缓存三大问题 & 解决方案

🕳️ 缓存穿透

现象：查询不存在的数据，缓存和 DB 都查不到，大量请求直接打到 DB。

✅ 解决方案：

  • ① 布隆过滤器：将所有存在的 key 预先存入 Bloom Filter，请求先过 BF
  • ② 缓存空值：查不到也缓存（如 null），设短过期时间（如 60s）
  • ③ 参数校验：非法 ID 直接拦截
伪代码 
if bloomFilter.mightContain(key) {
  val = cache.get(key)
  if (val == null) { val = db.query(); cache.set(key, val) }
} else {
  return "not found"   // 布隆过滤
}

🌪️ 缓存雪崩

现象：大量缓存 key 同时过期 / Redis 宕机，所有请求瞬间打到 DB。

✅ 解决方案：

  • ① 过期时间加随机值：TTL = 3600 + random(0,300)
  • ② Redis 高可用：主从 + 哨兵 / Cluster
  • ③ 多级缓存：本地缓存（Caffeine）+ 远程 Redis
  • ④ 限流 + 降级：缓存不可用时走限流

💥 缓存击穿

现象：热点 key 过期瞬间，大量并发请求同时查 DB。

✅ 解决方案：

  • ① 互斥锁：查 DB 前先 SETNX 锁，只有一个线程查 DB，其余等待
  • ② 逻辑过期：缓存永不过期，另起线程异步更新
  • ③ 热点不过期：对热点 key 不设 TTL，手动更新
Java 
// 缓存击穿 · 互斥锁方案
public String queryWithMutex(String key) {
    String val = redis.get(key);
    if (val != null) return val;

    String lockKey = "lock:" + key;
    try {
        // 抢锁：set lock:key 1 NX EX 10
        if (redis.setnx(lockKey, "1", 10)) {
            // Double Check：加锁后再查一次缓存
            val = redis.get(key);
            if (val != null) return val;
            // 查 DB 并回写缓存
            val = db.query(key);
            redis.set(key, val, 3600 + new Random().nextInt(300));
            return val;
        } else {
            Thread.sleep(50);       // 没抢到锁，短暂等待
            return queryWithMutex(key); // 递归重试
        }
    } finally { redis.del(lockKey); }
}
💡 一句话区分：穿透 = 查不存在的数据 · 雪崩 = 大量 key 同时过期 · 击穿 = 单个热点 key 过期。三者的共性：缓存失效 → 流量直冲 DB。

8数据分片：哈希槽




  
🎯 16384 个哈希槽

  
Redis Cluster 将整个键空间划分为 16384 个哈希槽（slot），每个 Master 节点负责一部分槽。键通过 CRC16(key) % 16384 映射到具体槽位。


  

    
  


  

    📌
    为什么是 16384？槽位信息通过 Gossip 协议的心跳包传播。16384 个槽的位图占用 2KB（16384/8=2048 bytes），刚好适配心跳包大小（约 8KB）限制。如果像 Couchbase 一样用 1024 个 vBucket，粒度太粗；如果用 65536 个槽，心跳包放不下全部槽信息。
  






  
🏷️ 哈希标签 (Hash Tag)

  
多键操作（MGET、事务、Lua 脚本）要求所有键在同一槽位。通过 {} 哈希标签 可以强制相关键落到同一个槽。

  
  

    

      
原理

      

        Hash Tag 只对大括号 {...} 内的内容计算 CRC16。

        如果 key 中包含 {...} 模式，只对花括号内的字符串计算哈希。

        花括号必须匹配，且内部不能为空：{user:1001}:name 和 {user:1001}:age 会落入同一槽。
      

    

    

      
Redis Cluster

# 无 Hash Tag — 不同槽位，MGET 报错
MGET user:1001:name user:1001:age
# → (error) CROSSSLOT Keys don't hash to the same slot

# 使用 Hash Tag — 强制同一槽位 ✅
MGET {user:1001}:name {user:1001}:age
# → 1) "张三"  2) "28"

# 事务中必须用 Hash Tag
MULTI
HSET {order:2001} status "paid"
INCRBY {order:2001} total 99
EXEC

    

  


  

    ⚠️
    注意：Hash Tag 会导致数据倾斜——如果某个 tag 值对应的 key 过多（如热门用户），该槽所在节点会成为热点。需要结合业务合理设计 tag 粒度。
  










9Gossip 节点通信




  
🌐 Gossip 协议原理

  
Redis Cluster 采用去中心化的 Gossip 协议 进行节点间元数据交换。每个节点通过 Cluster Bus（端口 = 普通端口 + 10000）与其他节点通信，无需中心协调器。


  

    
  


  

    

      
消息发送时机携带信息作用

      

        MEET
        新节点加入
        发送者基本信息（ID、IP、Port、epoch）
        请求加入集群
      

      

        PING
        每 100ms 随机选节点
        自身状态 + 已知的其他节点信息（Gossip 段）
        心跳检测 + 传播元数据
      

      

        PONG
        收到 PING/MEET 后
        响应节点状态 + Gossip 段
        确认存活 + 回传元数据
      

      

        FAIL
        PFAIL → FAIL
        故障节点 ID
        广播节点下线
      

    

  


  

    🔬
    Gossip 段：每个 PING/PONG 消息中携带了发送者已知的随机几个其他节点的状态（而非全部），通过"一传十、十传百"的方式，最终整个集群都能达成一致。这类似于流行病传播模型，收敛速度与节点数对数成正比。
  










10故障转移




  
🚨 PFAIL → FAIL → 自动切换

  
Redis Cluster 的故障检测采用 两阶段判定：主观下线（PFAIL）和客观下线（FAIL），防止误判。


  

    

      ① 心跳超时→
      ② PFAIL
本节点判定→
      ③ Gossip 广播 PFAIL→
      ④ 超半确认→
      ⑤ FAIL
客观下线
    

    

      ⑥ Slave 发起选举→
      ⑦ 请求投票
超半赞成→
      ⑧ Slave→Master
接管槽位→
      ⑨ 广播新配置
    

  


  
关键参数

# PFAIL 超时时间（毫秒）— 默认 5 秒无响应判定 PFAIL
cluster-node-timeout 5000

# 故障转移触发条件 — Master 被多数主节点标记为 FAIL
# 公式：N/2 + 1 个主节点一致同意（含自身）

# 选举延迟 — Slave 的延迟排名（越小越先发起）
# delay = 500ms + random(0, 500ms) + SLAVE_RANK * 1000ms
# Slave Rank = 复制偏移量越大排名越小，数据越新的优先级越高

  


  

    ✅
    选举优先级：数据最新的 Slave 优先当选（复制偏移量最大 → 延迟最小 → 最先发起选举）。这确保了切换后数据丢失最少。
  






  
⚠️ 一致性保证

  
Redis 集群的复制是异步的，存在 数据丢失窗口——Master 写入后、同步到 Slave 前宕机，该写操作将丢失。


  

    

      
🟡 异步复制风险

      
Master 确认写入后异步复制给 Slave。
若 Master 在复制前宕机，已确认的写入会丢失。

    

    

      
🟠 脑裂丢失

      
网络分区时旧 Master 收不到票数仍可接受写入，新 Master 当选后，旧 Master 上的写入丢失。

    

    

      
🟢 min-replicas-to-write

      
设置 min-replicas-to-write 1 和 min-replicas-max-lag 10：若至少 1 个 Slave 延迟＞10s，Master 拒绝写入，防止脑裂丢失。

    

  


  
redis.conf

# 至少 1 个 Slave，复制延迟不超过 10 秒，Master 才接受写入
min-replicas-to-write 1
min-replicas-max-lag 10

# 脑裂恢复后，旧 Master 自动同步新 Master 数据
# cluster-replica-validity-factor 10

🔑 大 Key 与热 Key

  

    

      
🐘 大 Key 问题

      
单个 key 的 value 过大（String > 10KB / 集合 > 10000 元素）

      
    
        
  • 危害：网络带宽打满、主从同步延迟、DEL 阻塞（用 UNLINK 异步删除）
    • 
        
  • 拆分：拆成多个小 key，如 user:1:info:0、user:1:info:1
    • 
        
  • 检测：redis-cli --bigkeys 扫描大 key
    • 
      

    

    

      
🔥 热 Key 问题

      
某个 key 被极高频率访问（如秒杀商品、热点新闻）

      
    
        
  • 危害：单节点 CPU 打满、该槽位所在节点过载
    • 
        
  • 多副本：在多个节点创建热点 key 副本，使用随机 key 读取
    • 
        
  • 本地缓存：应用层加 Caffeine/Guava Cache 短TTL 本地缓存
    • 
        
  • 检测：redis-cli --hotkeys 扫描热 key
    • 
      

    

  






  
🔀 扩缩容：槽迁移

  
集群扩缩容的本质是 槽在节点间迁移，Redis 提供 redis-cli --cluster reshard 命令自动化这个过程。

  
  

    
  


  
扩缩容命令

# 添加新 Master 节点
redis-cli --cluster add-node new-node:6379 existing-node:6379

# 重新分配槽位（交互式）
redis-cli --cluster reshard existing-node:6379
# 输入：要迁移多少槽 → 目标节点 ID → 源节点 ID → done

# 添加 Slave 节点
redis-cli --cluster add-node new-slave:6379 existing-node:6379 --cluster-slave

# 删除节点（需先迁移其槽位）
redis-cli --cluster del-node existing-node:6379 <node-id>

  






  
🔄 MOVED / ASK 重定向

  
当客户端请求的 key 不在当前节点时，Redis Cluster 返回重定向信息。客户端需要 更新路由表 并重试请求。


  

    

      
类型触发场景响应格式客户端行为

      

        MOVED
        槽已永久迁移到其他节点
        -MOVED 3999 127.0.0.1:6381
        永久更新本地槽映射表，后续请求直连新节点
      

      

        ASK
        槽正在迁移中（部分 key 已迁移）
        -ASK 3999 127.0.0.1:6381
        临时重定向：先发送 ASKING 命令到目标节点，再重发原命令。不更新槽映射表
      

    

  


  
客户端处理流程

try {
  result = client.execute(key);
} catch (MOVED_REDIRECT e) {
  // MOVED: 永久更新槽→节点映射
  slotMap.put(e.slot, e.targetNode);
  return retry(e.targetNode, key);
} catch (ASK_REDIRECT e) {
  // ASK: 临时重定向（迁移中的槽）
  conn = getConnection(e.targetNode);
  conn.send("ASKING");        // 必须先发 ASKING
  return conn.execute(key);   // 再发原命令
}

  


  

    💡
    Smart Client 推荐：生产环境务必使用支持 Cluster 模式的客户端（如 JedisCluster、Lettuce、redis-py-cluster、go-redis），它们内置了槽映射表缓存、MOVED/ASK 自动重定向、连接池管理等能力，大幅减少重定向开销。避免使用普通单机客户端连接集群。
